# spec/ — 設計層

> 構築の唯一の入力。このディレクトリの仕様に基づいて仮PJ・提供PJ・模範解答・全ドキュメントが生成される。
> プロジェクト全体の哲学は `../CLAUDE.md` を参照。

## 位置づけ

```
CLAUDE.md（哲学: WHO/WHY/WHAT/HOW/MAP）
  └── spec/（設計: プロダクト定義・機能仕様・データ設計・チケット設計）
        └── 実装（provided/ + answer/ + docs/）
```

---

## ファイル構成と責務

| ファイル | 責務 | 単一の問い |
|---------|------|----------|
| `overview.md` | プロダクト定義 | このLMSは何か？ |
| `features/{name}/` | 機能単位の詳細仕様 | この機能は何を満たし、どう実装するか？ |
| `database.md` | データ層の設計 | データはどう構造化されるか？ |
| `routes.md` | HTTP層の設計 | URLとコントローラの対応は何か？ |
| `tickets.md` | チケット設計 | 受講生は何をするか？何を壊すか？ |

### 各ファイルのセクション構成

**overview.md**
- ロール定義
- コンテンツ階層
- Feature一覧（features/ へのインデックス）
- コーディング規約（README転記用）

**database.md**
- エンティティ概要 + リレーション図
- テーブル定義（カラム・型・制約・インデックス）
- シーダー仕様（データ件数・内容・特殊データ条件）

**routes.md**
- ルート定義テーブル（method / URI / controller@action / middleware / feature）
- Bladeマッピング（ルート → Bladeファイルパス）

**tickets.md**
- チケット全体像（カテゴリ → 種類 → 配分）
- お手本機能の選定
- AI対策ガイドライン
- 深さの設計指針
- 個別チケット一覧（受講生向け記述 + 劣化アクション）

## features/{name}/ テンプレート

各featureは以下の3ファイルで構成。ifield-lmsのSDD構造をLaravel向けに軽量化したもの。

### requirements.md — 受け入れ基準

```markdown
# 要件定義: {Feature名}

## はじめに

本仕様は〜を定義する。

**スコープ外**: 〜は別仕様で扱う。

## 要件

### 1. {機能名}

**目的:** {ロール}として、{やりたいこと}。それにより、{得られる価値}。

#### 受け入れ基準

1. The Module shall 〜する
2. When 〜の場合, the Module shall 〜する
3. If 〜の場合, then the Module shall 〜する
4. While 〜の間, the Module shall 〜する

## スコープ外

- 〜は本仕様の対象外
- 〜は {別feature名} で扱う
```

**EARS形式（受け入れ基準の構文）:**

| パターン | 構文 | 用途 |
|---------|------|------|
| 常時 | `The Module shall [action]` | 常に満たす要件 |
| イベント駆動 | `When [event], the Module shall [action]` | 特定イベント時の振る舞い |
| 条件付き | `If [condition], then the Module shall [action]` | 条件分岐 |
| 状態依存 | `While [state], the Module shall [action]` | 特定状態の間の振る舞い |

**ルール:**
- 要件見出しには数字IDを付与する（例: `### 1. 認証`, `### 2. ロール制御`）
- 受け入れ基準は検証可能な文で書く
- 各基準は独立してテスト可能であること

### design.md — 技術設計

```markdown
# 技術設計: {Feature名}

## 概要

{何を実現するか。2-3行}

### ゴール
- {目的1}
- {目的2}

### スコープ外
- {やらないこと}

## コントローラ設計

| メソッド | 責務 | 入力 | 出力 |
|---------|------|------|------|

## モデル設計

| モデル | リレーション | スコープ | アクセサ/ミューテタ |
|--------|-----------|--------|-------------------|

## ビジネスルール

- {ルール1: 条件 → 振る舞い}
- {ルール2: 条件 → 振る舞い}

## エラーハンドリング

| エラー種別 | 条件 | レスポンス |
|-----------|------|----------|
| バリデーション | {条件} | 422 + エラーメッセージ |
| 認可 | {条件} | 403 |
| Not Found | {条件} | 404 |

## Bladeが期待するもの

| Blade | 渡す変数 | フォームフィールド |
|-------|---------|----------------|
```

**ルール:**
- requirements.md の受け入れ基準を実現する設計であること
- コーディング規約（overview.md）との整合性を明記
- エッジケースを明示する
- 複雑な機能ではMermaid図を任意で追加可能

### tasks.md — 実装タスク

```markdown
# 実装タスク: {Feature名}

- [ ] 1. {タスク名}
  - {詳細}
  - _Requirements: 1.1, 1.3_

- [ ] 2. {タスク名} (P)
  - {詳細}
  - _Requirements: 2.1, 2.2_

- [ ] 2.1 {サブタスク名} (P)
  - {詳細}
  - _Requirements: 2.1_

- [ ]* 3. {テストタスク名}
  - {詳細}
  - _Requirements: 1.1, 2.1_
```

**マーカー:**

| マーカー | 意味 |
|---------|------|
| `(P)` | 他タスクと並列実行可能 |
| `*`（`- [ ]*`） | テスト等の延期可能タスク |

**ルール:**
- 各タスクは requirements.md の要件番号に紐づける（`_Requirements: 1.1, 2.3_`）
- 依存関係のあるタスクは順序を明示
- メジャータスク + サブタスクの階層構造を使い分ける

## 作成ルール

- **各ファイルは独立して読める**ようにする（過度な相互参照を避ける）
- **features/ が全成果物の派生元**（SDD的: requirements → design → tasks → 実装）
- **作成順**: overview.md → features/ → database.md → routes.md → tickets.md

## feature の粒度基準

- **ドメイン（関心事）で分割**する。ロールで分割しない
- 1 featureは**1つの明確な関心事**に対応する
- requirements.md が長くなりすぎる場合（目安: 10要件超）は分割を検討
- 逆に受け入れ基準が2-3個しかない場合は他featureとの統合を検討
