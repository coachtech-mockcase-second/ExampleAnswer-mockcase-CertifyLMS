# チケット設計

> 個別チケットの定義は全featureの仕様確定後に行う。本ファイルではチケット枠（種類・配分）を先に定義する。

## チケット種類（3カテゴリ × 8種類）

| カテゴリ | 種類 | 定義 |
|---------|------|------|
| バグ修正 | データの不正 | 出力結果が期待と異なる（集計誤り、データ起因の0除算等） |
| | アクセス制御の不備 | 見えてはいけないものが見える・操作できる（Policy未適用、認証漏れ等） |
| | 機能の不全 | 動かない・エラーになる（バリデーション不備、状態遷移条件の不備等） |
| 機能開発 | 既存機能の修正 | 既存の振る舞いを変更する |
| | 既存機能の拡張 | 既存の機能に新しい要素を追加する（新規モデルなし） |
| | 新規機能の構築 | 新規モデル・コントローラ・ルートをゼロから作成（テスト必須） |
| リファクタリング | コード構造 | 可読性・保守性の改善（Fat Controller分割、Scope抽出、定数化等） |
| | パフォーマンス | 実行効率の改善（N+1解消はBasic、インデックス/キャッシュはAdvance） |

| カテゴリ | 種類 | Basic | Advance | 計 |
|---------|------|-------|---------|---|
| バグ修正 | データの不正 | 2 | - | 2 |
| | アクセス制御の不備 | 1 | - | 1 |
| | 機能の不全 | 2 | - | 2 |
| 機能開発 | 既存機能の修正 | 1 | - | 1 |
| | 既存機能の拡張 | 2 | - | 2 |

| | パフォーマンス | 1 | 1 | 2 |

## ボリューム目標

| 区分 | チケット数 | 推定時間 | 評価比率 |
|------|----------|---------|---------|
| Basic | 15 | ~43h | ~80% |
| Advance | 3 | ~18h | ~20% |
| 合計 | 18 | ~61h | 100% |

基準: BookShelf Basic（~43h）と同等。Basic全問正解で全体の75-80%。

## 難易度分布（Basic）

| 難易度 | 件数 | 時間目安 | 対応する種類 |
|--------|------|---------|------------|
| 易（1-2h） | 5 | 明確な1箇所の修正 | バグ修正、N+1解消 |
| 中（3h） | 7 | 複数ファイル跨り | 機能修正・拡張、リファクタ構造 |
| 難（5h） | 3 | 設計+実装+テスト | 新規機能の構築 |

## 提供テストの範囲

提供プロジェクトの一部の機能にはテストが含まれる。受講生に明示はせず、コードリーディングの中で自分で発見し、パターンを読み取らせる。

| 候補 | テストを含める理由 |
|------|----------------|
| 受講登録/解除 | シンプルなCRUDでテストの基本パターンを示しやすい |
| カテゴリ管理 | 管理者のCRUD操作 + 認可テストのパターンを含む |

確定は個別チケット定義時。

## AI対策ガイドライン

### チケット設計での対策

| 手法 | 具体例 |
|------|--------|
| 50%要件 | 要件・チケット記述を意図的に曖昧にする（「コース一覧が重い」等）。コーチへのヒアリングで明確化させる |
| 規約準拠 | プロジェクト固有の規約・パターンに合わせる |
| 既存コード参照 | 「既存テストを参考にテストを書け」（どのテストかは自分で探す） |

### PR記述の必須化

各チケットのPRに以下を必須記載:

```
## 調査内容
（何を読んだか、どこを確認したか）

## 原因分析 / 設計判断
（バグ: 何が原因だったか / 新機能: なぜこの設計にしたか）

## 実装内容
（何を変えたか、簡潔に）

## テスト・動作確認
（何をどう確認したか）
```

## 深さの設計指針

チケット対象の機能には以下のような「深さ」を意図的に組み込む。

| 深さの種類 | 具体例 | チケットとの関係 |
|-----------|--------|--------------|
| 業務ルール | 「公開Courseのみ受講登録可」「受講登録済みでないとSection閲覧不可」 | バグ（ルール実装漏れ） |
| 状態遷移 | Course: draft → published → archived。Chapter/Sectionが0なら公開不可 | バグ（遷移条件不備） |
| 集計ロジック | ダッシュボードの統計（平均進捗率、人気Top5） | リファクタ（foreachからクエリへ） |
| エッジケース | Chapter 0件のCourse、受講者0での平均計算（0除算） | バグ（データ起因） |
| 権限の複雑さ | コーチは自分のCourseのみ編集可。管理者は全Course操作可 | バグ（認可漏れ） |
| 検索の複雑さ | キーワード + カテゴリ + ソート（新着/人気）の組み合わせ | リファクタ（Scope化）、パフォーマンス |
