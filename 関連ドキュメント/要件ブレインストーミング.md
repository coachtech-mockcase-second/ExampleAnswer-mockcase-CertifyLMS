### 模範解答 PJ の前提

`模範解答プロジェクト/` 配下には **既に完成形（受講生がすべて実装した状態相当）の Laravel 実装** がある（19 Feature の大半が実装済、一部 ai-chat / dashboard 仕上げ中）。チケット設計では:
- 模範解答 PJ に **既に実装されているもの** → 受講生は提供 PJ で「壊れた状態」「未実装 Blade のみ」を受け取る前提で、要件シートに従って実装・修正

## 1. 前提コンセプト（CLAUDE.md からの抜粋）

### 1.1 Certify LMS の位置づけ

- COACHTECH 受講生向け模擬案件②、**既存 PJ 参画 + 曖昧要件 + AI 耐性** の 3 課題を解く
- 題材: マルチ資格対応のプラン受講型 LMS
- 受講生は **提供 PJ をクローン → 抽象的チケットを受領 → コーチ（PM 役）へのヒアリングで詳細化 → 実装** のフロー（実務 PJ 参画体験）
- BookShelf 等の新規構築型 PJ との **根本的な違い**: 受講生はゼロから設計せず、既存コード + ONBOARDING + ヒアリングで詳細化

### 1.2 要件定義の修了条件（3 条件）

| # | 条件 | 値 |
|---|---|---|
| 1 | **量** | 想定 **250 時間**（BookShelf 150h 比 約 1.67 倍）。要件シート定義の最後で各チケットに工数仮置き → 合計が **250h ± 10%** に収まるかを点検 |
| 2 | **難易度** | **Basic** = 教材範囲 + ContactForm / BookShelf 既習範囲の **内側** に収まる / **Advance** = 指定なし、教材外 OK |
| 3 | **評価設計** | **60% = A / 80% = S** / **Basic 配点 = ジャスト 60% 固定** / **Advance 配点 = 40%** / Basic 満点で A ジャスト、S には Advance から 20% 以上必須 |

### 1.3 評価シート構造（3 大項目、振る舞いベース）

**評価項目の粒度** = 基本的に振る舞いベース。② 横断コード品質だけは「明らかに不適切でないことを担保するチェックリスト」として例外配置。

## 3. 論点・未確定事項

### 論点 L: Sanctum + 通知 API 案 ⚠️💬 **最優先**

**背景**: ユーザー案「Sanctum を用いた公開 API + 通知 API」は **C-1 制約**（2026-05-16 確定で Sanctum SPA / 公開 JSON API 不採用）と矛盾。一方、模範解答 PJ への新規実装の可能性も検討対象。

**選択肢**:
- **L-1**: C-1 制約維持。`S-B-06` を **API キー方式の素データ API**（`X-API-KEY` ヘッダ）に置換。Advance 側のフロント JS 連携（`S-A-06`）も API キー API を JS で叩く方向
- **L-2**: **C-1 制約を変更**し、Sanctum 採用を再解禁。`S-B-06` を「Sanctum 認証 + 通知 API」とし、模範解答 PJ に新規実装する。Advance 側で SPA 風フロント JS でこの API を叩く（BookShelf の Sanctum 経験を踏襲できる）
- **L-3**: 両方併存。Basic に API キー方式の素データ API、Advance に Sanctum + フロント JS

**影響範囲**:
- L-2 を選ぶ場合: `tech.md` / `CLAUDE.md` / 既存 spec（analytics-export 等）の Sanctum 関連記述を全部見直し
- L-1 を選ぶ場合: BookShelf で扱った Sanctum 経験との接続が薄れる

**判断材料**:
- Certify LMS の AI 耐性核として Sanctum を使う価値があるか
- 受講生が BookShelf からの連続性を感じるか（Sanctum 既習を再登場させる意義）
- 時間配分（Sanctum 実装 = +N 時間、250h ± 10% 内に収まるか）

### 論点 M: 受講生（student）向け機能は Bug 主体にする方針 💬

**ユーザーコメント**: 「ユーザー側は基本バグのみとかでもいいかも」

**論点**: 受講生（student ロール）向けの機能は **新規 Story を作らず、Bug 修正 + Advance 拡張のみ** にする方針で合意するか。これだと:
- Basic Story = admin / coach 向けの管理 UI 中心（プランマスタ管理 / 面談プランマスタ管理 等）
- 受講生向け（教材閲覧 / 問題演習 / 模試受験 等）は既存実装 + 各所のバグ修正
- 通知も受講生向けだが、Laravel Notification の実装自体は admin / coach 視点（通知発火）からも触れる

→ 受講生体験は「壊れた状態の機能を直す」「コーチ視点での機能追加」が主軸になる。

### 論点 N: 追加 Bug 候補（`B-B-02`, `B-B-07`）と Task 候補（`T-B-02`）の決定 💬

具体的に何を仕込むか:
- `B-B-02` 認可系（封鎖パターン）: 候補 — 他コーチの予約をキャンセル可 / 他者の qa 投稿を編集可 / 他者の chat メッセージを編集可 / 他者の個人目標を編集可 等
- `B-B-07` 計算/表示系: 候補 — ダッシュボード集計（合格可能性スコア / 弱点ヒートマップ）/ プラン残日数表示 / 残面談回数表示 等
- `T-B-02` N+1 解消対象: 候補 — 受講生一覧 / dashboard / 教材一覧 / 模試結果一覧 等

### 論点 O: 採点における懸念（振る舞いベース原則の例外） 💬 **重要**

**ユーザー懸念**:
- **認可系**: Blade 側（提供 PJ）で既に表示制御されている → 表示が「ない」ことだけ確認しても、サーバ側で実装されているとは限らない → コードベース確認が必要
- **N+1 や `chunk()` 等のパフォーマンスチューニング系**: 振る舞いだけでは「最適化されているか」が分からない → コードベース確認が必要

**論点**: 振る舞いベース原則をどう運用するか。3 案:

| 案 | 説明 | メリット | デメリット |
|---|---|---|---|
| **O-1** 振る舞いベース厳守、半振る舞い化 | 認可は「直接 URL を叩いて 403 か」で証明、パフォーマンスは「`sail artisan db:monitor` でクエリ数を計測」で証明 | 振る舞いベース原則を維持 | 計測手順を要件・採点で明示する必要 |
| **O-2** 該当チケットのみコードベース採点を許容 | 認可系・パフォーマンス系チケットは評価項目に「コード確認」を含める | 採点が確実 | 振る舞いベース原則の例外が増える |
| **O-3** ② 横断コード品質に集約 | 認可系・パフォーマンス系の「コード確認部分」は ② 横断コード品質に寄せ、① チケット完了は振る舞いベース純粋に維持 | 大項目構造が綺麗 | 「このチケットの認可は OK か」と「全体の認可実装品質」が分離してしまう |

**推奨**: **O-1**。具体的に:
- 認可: 採点者が `curl` / Postman でリクエストを直接送り、サーバ側レスポンスを確認
- パフォーマンス: 受講生に `sail artisan db:monitor` の実行結果スクショ or 動画を PR 動作確認に含めることを要件化
- これらは **実装方法は問わず（Policy か Middleware か等）、結果（HTTP ステータス / SQL クエリ数）を評価**

## 5. 進め方の提案

### 5.1 議論フェーズ（現在地）

```
[現在地] § 2 チケット一覧（粗粒度）+ § 3 論点 を提示
   │
   ├─→ ① § 3 論点を順次合意（最優先: L → O → M → 残り）
   │     L (Sanctum) は他チケット定義に影響大、最初に決める
   │     O (採点方法) は評価シート設計の根幹、L の次に決める
   │     M (受講生機能 Bug 主体) は Basic Bug 件数に影響
   │     残りは平行で合意可能
   │
   ├─→ ② § 2 チケット一覧を 全部 ✅ に
   │     未確定箇所（B-B-02 / B-B-07 / T-B-02 の対象、S-A-06 内容、S-A-05 配置、Q の S 必須 3 件目）を具体的に決める
   │
   └─→ ③ 詳細化フェーズへ
```

### 5.2 詳細化フェーズ（合意後）

```
① § 2 確定 → ② 各チケットの抽象度設計（100% 版）
                ・受け入れ条件 / 完了条件
                ・実装方針
                ・やらないこと
                ・動作確認手順
                ・想定ヒアリング Q&A
                ・工数仮置き
              ↓
              ③ 250h ± 10% 点検 → 件数調整
              ↓
              ④ 評価シート骨格作成（中項目 = 各チケット）
              ↓
              ⑤ 30% 版変換ルール確定 + 30% 版作成
              ↓
              ⑥ 正式な要件シート 4 ファイル + 評価シートを `関連ドキュメント/要件シート_詳細度100%/` 等に配置
                 本ファイル（要件ブレインストーミング.md）は議論ログとして保持
```

### 5.3 並行作業候補

詳細化フェーズで並行可能な作業:
- 提供 PJ の **README.md / ONBOARDING.md** の章立て設計（要件シート詳細化と並行）
- **02_開発プロセス.md** の DoD 部分（テスト要件 / Pint / PR 7 セクション等）の整理（BookShelf シート 2 踏襲）

### 5.4 まずは論点 L から議論しませんか

L が決まらないと:
- `S-B-06` の内容が確定しない
- `S-A-06` の内容も連動して未確定
- 模範解答 PJ への新規実装スコープが見えない
- C-1 制約変更の波及（spec / steering の見直し）が判断できない

L 合意 → O 合意 → M 合意 → 残り（N / P / Q / S）の順で進めるのが効率的と提案します。
