## 1. 前提コンセプト（CLAUDE.md からの抜粋）

### 1.1 Certify LMS の位置づけ

- COACHTECH 受講生向け模擬案件②、**既存 PJ 参画 + 曖昧要件 + AI 耐性** の 3 課題を解く
- 題材: マルチ資格対応のプラン受講型 LMS
- 受講生は **提供 PJ をクローン → 抽象的チケットを受領 → コーチ（PM 役）へのヒアリングで詳細化 → 実装** のフロー（実務 PJ 参画体験）
- BookShelf 等の新規構築型 PJ との **根本的な違い**: 受講生はゼロから設計せず、既存コード + ONBOARDING + ヒアリングで詳細化

### 1.2 要件定義の修了条件（3 条件）

| # | 条件 | 値 |
|---|---|---|
| 1 | **量** | 想定 **250 時間**（BookShelf 150h 比 約 1.67 倍）。要件シート定義の最後で各チケットに工数仮置き → 合計が **250h ± 10%** に収まるかを点検 |
| 2 | **難易度** | **Basic** = 教材範囲 + ContactForm / BookShelf 既習範囲の **内側** に収まる（既習範囲を全部使う意味ではなく、上限制約）/ **Advance** = 指定なし、教材外 OK |
| 3 | **評価設計** | **60% = A / 80% = S** / **Basic 配点 = ジャスト 60% 固定** / **Advance 配点 = 40%** / Basic 満点で A ジャスト、S には Advance から 20% 以上必須 |

### 1.3 評価シート構造（3 大項目、振る舞いベース）

| 大項目 | 中項目 | 配点比率目安 |
|---|---|---|
| **① チケット完了** | 各チケット（`{Feature略称}-{番号}: 概要` 形式、同 Feature 隣接配置）| 65-70% |
| **② 横断コード品質** | Pint / 命名 / Eloquent / N+1 / Clean Architecture 準拠 / 型宣言（応用）等 | 15-20% |
| **③ 横断ドキュメント** | README 改修 / ONBOARDING 改修 / 全 PR の 7 セクション記述率 / カバレッジ目標 等 | 10-15% |

**評価項目の粒度** = **基本的に チケット完了条件 / 受け入れ条件レベルの振る舞いベース**（実装方法は問わず「機能が要件通り動くか」を評価）。

- 理由: ① 実務フロー再現 / ② AI 耐性 / ③ 既存パターン踏襲との二重化回避 / ④ 実装方法の判断力養成
- 例外: 大項目 ② 横断コード品質 だけは「**明らかに不適切でないことを担保するチェックリスト**」レベルの形式的品質指標を配置（配点抑制）
- 「テスト」「PR 7 セクション」「動作確認動画」は **各チケットの完了条件として ① に内包**（独立大項目化しない）

### 1.4 要件シート構造（4 ファイル構成）

```
関連ドキュメント/要件シート_詳細度100%/
├── 01_概要.md       (ターム内容 + 開発プロセス + 環境構築手順)
├── 10_Story.md      (機能開発チケット集合、1 チケット = 1 Markdown セクション)
├── 20_Bug.md        (バグ修正チケット集合)
└── 30_Task.md       (リファクタリング・既存改修チケット集合)

関連ドキュメント/要件シート_詳細度30%/   # 上記 4 ファイル × 抽象化変換、受講生用
```

**詳細仕様の所在分離**: テーブル仕様 / API 仕様 / バリデーション / 画面遷移 等の **詳細設計書は要件シートに含めない**。提供 PJ のコード + `ONBOARDING.md` + コーチヒアリングで詳細化する。

### 1.5 PR 品質（AI 耐性の中核）

- **PR 7 セクション必須**（関連チケット / 調査内容 / 原因分析・設計判断 / 実装内容 / 自動テスト / 動作確認 / レビュー観点・自己評価）
- **動作確認**: 動的機能（タイマー / 状態遷移 / リアルタイム / モーダル / 非同期更新）は **動画必須**、静的 UI はスクショ、バグ修正は修正前後比較

### 1.6 既存方針との制約（要件シート作成時に守るべき確定事項）

| # | 制約 | 出所 | 影響 |
|---|---|---|---|
| C-1 | **Sanctum SPA / 公開 JSON API / `HasApiTokens` は LMS 全体で不採用**（2026-05-16 確定）| `docs/steering/tech.md` | 旧「Sanctum 公開 API」チケット案は採用不可 → analytics-export の API キー方式に置換 |
| C-2 | **Alpine.js / Livewire 不採用** | tech.md | FE Advance は素の JS + fetch のみ |
| C-3 | **修了は受講生「修了証を受け取る」ボタンで自己完結** | product.md v3 | 修了承認フローは不存在 |
| C-4 | **担当コーチは資格に紐づく N:N**（`certification_coach_assignments`） | product.md v3 | 1on1 担当変更動線は持たない |
| C-5 | **面談予約は時刻スロット選択 → コーチ自動割当** | product.md v3 | 申請承認フローは不存在 |

## 3. 論点・未確定事項

### 論点 A: Sanctum 公開 API 案の置換 💬

- ユーザー Basic 案「Sanctum を用いた公開 API 作成」は **C-1 制約**（2026-05-16 確定で Sanctum SPA / 公開 JSON API 不採用）と矛盾
- 修正案: **analytics-export の API キー方式** (`X-API-KEY` ヘッダ + `.env` 共通キー) に置換
  - `S-B-06`: 素データ API 提供（Basic）
  - `S-A-02`: GAS スプシ連携（Advance）の 2 段構え

**論点**: この置換で合意してよいか。Basic に「API キー認証 Middleware の実装」を含めるか、提供 PJ で完成済み扱いにするか。

### 論点 B: mock-exam の Basic 化 💬

- ユーザー案には漏れていた
- `product.md` で「資格 LMS 中核 / 未実装（Blade のみ）」と定義
- 受講生が新規実装する Basic Story の主力候補

**論点**: `S-B-05` として Basic に追加で合意してよいか。

### 論点 C: plan-management の扱い 💬

- ユーザー案には漏れていた
- `product.md` で「未実装、プラン受講モデルの中核」と定義
- Basic Story（`S-B-07`）として追加するか、提供 PJ 完成済み扱いにするか

**論点**: Basic 必須機能（プラン受講モデルの根幹）なので受講生が実装すべきか、それともコース受講モデルは既に整備済みとして提供すべきか。

### 論点 D: default-enrollment の扱い 🔒

- ユーザー案には漏れていた
- cross-cutting infrastructure（サイドバー Switcher + `ResolveDefaultEnrollment` Middleware）
- Basic / Advance どちらにも収まりにくい

**論点**: チケット化するか、提供 PJ 実装済み扱いにするか。保留して後で判断する案を提示。

### 論点 E: S 必須 Advance チケット 3 件の確定 💬

候補:
- ⭐ `S-A-01` リアルタイムチャット（Pusher Broadcasting）
- ⭐ `S-A-02` GAS スプシ連携
- ⭐ `S-A-03` Google Calendar 連携（面談予約）

**論点**: この 3 件を「S 取得必須 Advance」として確定してよいか。それとも `S-A-04` Gemini や `S-A-05` Stripe を入れ替えるか。

### 論点 F: 「既存パターン準拠」の採点方法 💬

ユーザー懸念「採点方法が想像つかない」への提案:

評価シート ② 横断コード品質 の中項目「Clean Architecture 準拠」は「**明らかな逸脱の有無**」二値評価:
- Controller に 50 行超のビジネスロジックが書かれていないか
- 1 業務操作 = 1 Action 原則からの明らかな逸脱がないか
- FormRequest / Policy / Action / Service / Repository のディレクトリ配置が既存パターンと一致するか

「Action と Service の分け方として正しい設計か」のような深い設計議論は **採点対象外**（実装方法は問わない方針との整合）。

**論点**: この採点方法で合意してよいか。

### 論点 G: Bug カテゴリの最終件数 💬

- ユーザー案: Bug = 0 件
- Claude 補強案: 4-5 件
- 「3 カテゴリ × 8 種類」のメモリ記述からは Story : Bug : Task = 4-5 : 2-3 : 2 が目安

**論点**: § 2 の Bug 候補（`B-B-01` 〜 `B-B-04` + `B-A-01`）で過不足ないか。種別バランスは適切か。

### 論点 H: チケット名規則 💬

評価シート中項目で「`{Feature略称}-{番号}: 概要`」形式を想定（CLAUDE.md 記載）。

**論点**: § 2 で使っている `S-B-01` / `B-B-01` / `T-B-01` 系の ID 規則と、Feature 略称を含む表示名（例: `qa-board-01: 質問掲示板の認可実装`）の両方を持つか、どちらか一方に統一するか。

### 論点 I: 想定 250h ± 10% への収め方 💬

- 現状候補 22 チケット × 平均 11h ≈ 242h（範囲内）
- 個別チケットの工数仮置きは詳細化時（受け入れ条件設計後）に行う

**論点**: 件数感は妥当か。Story 14 件は多すぎないか。

### 論点 J: 修了証 PDF の配置 Feature 💬

- `S-A-06`「修了証 PDF 出力」は certification-management / enrollment どちらの Feature に配置するか
- `product.md` では certification-management 内に「修了証発行 + PDF 出力」と定義
- ただし発火元は enrollment の `ReceiveCertificateAction`

**論点**: チケットの主担当 Feature をどちらにするか。
