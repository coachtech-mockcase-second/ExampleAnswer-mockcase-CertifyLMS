### 模範解答 PJ の前提

`模範解答プロジェクト/` 配下には既に完成形相当の Laravel 実装があるが、**新規実装の可能性も含めて検討する**（例: Sanctum + 通知 API は模範解答 PJ に未実装、新規でチケット化されたので模範解答 PJ にも実装が必要）。

### `default-enrollment` Feature の扱い

`product.md` では「未実装」と記載されているが、cross-cutting infrastructure（サイドバー Switcher + `ResolveDefaultEnrollment` Middleware）でチケット粒度に収まりにくいため、**提供 PJ で実装済として渡し、受講生は使う側として実装に触れない** 方針で確定（論点 R 決着）。

### 5.3 詳細化フェーズの進め方提案

詳細化フェーズの進め方として、ユーザーの好みに合わせて 3 つの選択肢:

| 案 | 説明 | メリット | デメリット |
|---|---|---|---|
| **A. ジャンル別に進める** | Story 全件 → Bug 全件 → Task 全件 の順で詳細化 | 種別固有の記述スタイル（Bug は完了条件のみ等）を一括で習得 | Feature 横断で見たときに整合性チェックがしにくい |
| **B. Feature 別に進める** | qa-board の全チケット → chat の全チケット ... の順 | Feature 内整合性が取れる | Feature 横断バランスが取りにくい |
| **C. 優先度順に進める** | 重要・複雑なチケットから先（S-B-06 Sanctum / S-A-02 OAuth / B-B-03 mock-exam 等）| 工数感を早期に掴める | 進めながらの一覧把握がしにくい |

おすすめは **A（ジャンル別）**。Story / Bug / Task の記述スタイルが大きく異なるため、種別単位で記述ルールを固めながら進める方が効率的。

加えて、まず **1 チケットだけ詳細化** してフォーマットの叩き台を作る → ユーザー合意 → 残りを一気に詳細化、という流れが安全です。叩き台候補:
- `S-B-01 / qa-board-01` 質問掲示板の実装（典型的な新規 Story、認可ヒアリング要素あり）
- `B-B-02 / enrollment-01` 他者の個人目標編集 → 認可追加（典型的な認可系 Bug）
- `T-B-01 / user-management-01` Admin ユーザー管理画面 N+1（典型的なパフォーマンス Task）

3 種別の代表 1 件ずつを先に詳細化 → フォーマット合意 → 残り 24 件を一気に詳細化、が効率的かと思います。
