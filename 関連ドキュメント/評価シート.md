# 評価シート Certify LMS（マルチ資格対応 資格取得 LMS／模擬案件②）

## 評価方式

- **判定**: 全体配点に対する **取得率** で評価する。**60% 以上 = A 評価 / 80% 以上 = S 評価**。
- **採点方法（振る舞い優先）**: 採点は実機動作・動作確認（振る舞い）ベースを原則とする。PR の diff / コードを読むのは「手段的にそれしかない項目」（N+1 のクエリ数、リファクタの構造確認、冪等性など内部処理の検証）に限る。Basic / Advance で技術範囲が重なる箇所は「採点で気をつける」運用ではなく、チケット構造の側で解消する。
- **確認方法タグ**: 自動テストで判定する一部の採点行のみ、行内に「確認方法（テスト）」を併記している（同梱テストの pass / 受講生が実装するテストの該当ケースの pass）。
- **拒否系の判定**: 認可は「権限のないロール・状態で実際にアクセス・操作できないこと」で判定する（他人のリソースの URL や管理画面を開いて入れない / ボタン・入力欄が出ない）。状態ガードによる操作拒否は元画面に戻ってのフラッシュエラー表示で判定する。エラー画面の見た目（403 / 404 / 案内画面）は環境設定で変わるため判定基準にしない。HTTP ステータスコードを直接の判定基準とするのは、ブラウザではなく API 直叩き（curl / DevTools）で確認する行（S-A-05 の通知 JSON API / S-A-03 の Webhook）のみ。
- **2 大項目構造**:
  - **チケット要件** — 各チケットの受け入れ条件を「1 項目 = 1 採点行」に展開。採点者は対象 PR を開き、画面動作 / file Changes / 同梱テストで可否を判定する。
  - **横断品質** — 全 PR を横断する品質・プロセスの安全網。**受講生が追加・変更した差分**（各 PR の file Changes）と機械的チェック（Pint / 全テスト pass）・PR 記述で判定し、**既存コードの品質は採点対象としない**（提供プロジェクトに最初から含まれるコードは受講生の成果ではないため）。
- **配点（評価点）**: チケット要件は 1 採点行 = 1〜3 点（標準の振る舞い = 1 点、重要・複雑・難所 = 2 点、並行性 / 決済冪等性など最重要の correctness = 3 点）。横断品質は全 PR を横断する判定のため行ごとに重みを置く（段階評価を含む）。
- **Basic / Advance**: チケット ID で識別（`S-B-` / `B-B-` / `T-B-` = Basic、`S-A-` / `B-A-` / `T-A-` = Advance）。横断品質は行ごとに区分し、応用行に「（応用）」を付記。Basic 配点だけでは S（80%）に届かない設計（AI 丸投げ耐性）。
- **テストの採点（種別 2 バケット）**:
  - **Bug / パフォーマンス Task** = 模範解答の同梱テストが提供 PJ に残っており（修正前は赤）、その pass を採点行とする。
  - **Story / リファクタ Task** = 受講生がテストを記述（同梱不可）。「本チケットの機能に対するテストが実装されている」を採点行とする。
  - テストの **全 pass / 中身の質 / カバレッジ** は横断品質で別途評価する。
- **可否 / 点数**: 採点時に「可否」を `TRUE`（達成）に切り替えると「点数」に評価点が入る（未達成は `FALSE` = 0 点）。横断品質の段階評価行は「大半の PR で達成 = TRUE / 軽微でない欠落 = FALSE」を目安に判定する。

> 採点の入力源は `関連ドキュメント/要件シート_詳細度100%/` 配下の各チケット「受け入れ条件」セクション（本シートと 1:1 対応）。チケットの背景・実装方針・想定 Q&A はコーチ（PM 役）がヒアリング応答に用いる別レイヤーで、採点では通常参照しない。

> **やむを得ない依存の着手順・採点前提**: 一部の Advance チケットは他チケットの実装を前提とする。`T-A-04`（外部 API 連携のモックテスト追加）は `S-A-01` / `S-A-02` / `S-A-03` の実装後に着手する（モック対象の 3 連携が存在している前提）。`T-A-05`（通知・メール配信の非同期化）は `S-B-04` / `S-B-09` / `S-B-10` の通知・配信処理が存在する前提で着手する。採点でも、前提チケットが未達のまま依存先 Task だけが提出された場合は、依存先 Task の受け入れ条件を満たせないものとして扱う。
