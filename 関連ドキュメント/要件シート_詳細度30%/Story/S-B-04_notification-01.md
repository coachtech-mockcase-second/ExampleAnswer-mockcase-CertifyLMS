## 背景・目的

チャットや質問掲示板、面談予約といった既存のやり取り機能は動いているが、相手方や担当コーチへの通知が無く、画面を開かないと新着や状態変更に気付けない。受講生は学習中の取りこぼし、コーチは複数受講生のフォロー漏れのリスクを抱える。

本チケットは、業務イベントが起きた時にアプリ内通知とメールで当事者へ即時に知らせる基盤を新規構築する。通知一覧画面と既読化の動線を備える。これが無いと受講生・コーチが受動的に画面更新を待つ運用になり、LMS としての体裁が整わない。

## ユーザーストーリー

- **受講生(student)として**、自分宛の通知を一覧で確認したい。なぜなら、複数の機能を横断して未対応の事項を取りこぼしたくないから。
- **コーチ(coach)として**、自分宛の通知を一覧で確認したい。なぜなら、複数の受講生を効率的にフォローしたいから。
- **受講生 / コーチとして**、通知をクリックしたら既読化と該当画面への遷移が同時に走ってほしい。なぜなら、既読化忘れで未読件数が減らない事態を避けたいから。
- **受講生 / コーチとして**、まとめて既読にする操作が欲しい。なぜなら、未読が大量に残っているとき個別クリックでは追いつかないから。

## 要件

### 通知の確認・既読化（受講生・コーチ）

- サイドバーの「通知」から、自分宛の通知一覧を開いて確認できる
- 通知をクリックすると既読になり、関連する業務画面へ移動する
- まとめて既読にする操作ができる
- 他人宛の通知は閲覧・既読化できない

### 通知の発火

- 業務イベントが起きた時に、その当事者へアプリ内通知とメールで知らせる

### 配信対象の制御

- 利用状態やロールによって、通知の受信対象にならない場合がある

## スコープ外

- JSON API（認証なし）
- Cookie 認証 + JS フロントの通知ポップオーバー / リアルタイム push
- 管理者によるお知らせ一斉配信
- 面談リマインダー通知
- 管理者向けの通知（本 MVP では扱わない）
- 通知種別ごとの ON / OFF 設定 UI
- 通知の削除動線（既読化のみ提供）
- モバイルプッシュ通知
- メール配信のキュー非同期化

## インターフェース

| HTTP | パス | 認可 |
|---|---|---|
| GET | `/notifications` | 認証ユーザー |
| POST | `/notifications/{notification}/read` | 認証ユーザー（本人の通知のみ） |
| POST | `/notifications/read-all` | 認証ユーザー |

入口はサイドバーの「通知」項目。

## 初期データ

- 固定の受講生・コーチに、既読・未読を混在させた通知を投入する（一覧とページネーションの動作を確認できる状態にする）
