## 背景・目的

受講生の残面談回数は受講プラン契約時に初期付与される分のみで、使い切った後は次の契約更新まで増やせない。残数 0 で予約不可になった受講生は学習リズムを崩しコーチ面談を諦めるしかなく、運営も追加収益機会を取り逃している。

本チケットでは、受講生が残数 0 になっても自力でその場で追加面談パックを購入でき、決済完了後すぐに予約画面で残数加算が反映される状態を達成する。運営は管理者で面談パックを管理するだけで、価格 / 回数の運用を柔軟にコントロールできる。

学習継続率の維持と追加収益動線の構築に加え、外部の決済サービス連携(決済画面の委譲・決済結果の通知受信とその正当性の検証・同じ通知が重複しても二重計上しない仕組み)という実務で頻出する題材を扱う。

## ユーザーストーリー

- **受講生(student)として**、残面談回数が 0 になっても、その場で追加面談を購入してすぐ予約できる状態を期待する。なぜなら、学習リズムを崩したくないから。
- **受講生として**、購入の流れが直感的で、決済画面は信頼できる決済プラットフォームに任されている安心感を期待する。なぜなら、自分のカード情報を学習サービス側に直接渡したくないから。
- **受講生として**、決済完了後すぐに残面談回数が反映され、続けて面談予約に進める導線を期待する。なぜなら、購入してからまた残数が反映されるのを待つ体験はストレスだから。
- **受講生として**、決済が失敗 / 中断した場合は残数が変わらないことを期待する。なぜなら、課金されていないのに残数が増えるのは不安、逆に課金されたのに残数が増えないのも困るから。
- **管理者(admin)として**、決済サービスからの決済完了の知らせが二重で届いても残数が二重加算されないことを期待する。なぜなら、決済プロバイダ側の再送で残数会計が崩れたら監査が成立しないから。
- **管理者として**、決済の改ざんを構造的に防げる仕組みを期待する。なぜなら、決済サービスからの知らせを偽装した不正な残数加算を防ぎたいから。
- **コーチ(coach) / 管理者として**、自分の画面に「追加面談を購入」動線が表示されない。なぜなら、購入動線は受講生専用機能で、自分のロールに関係ないから。

## 要件

### 追加面談購入動線(受講生・学習中のみ)

- 公開中の面談パック一覧を閲覧できる(名前 / 面談回数 / 価格)
- 面談パックを選んで Stripe の外部決済画面へ進み、決済する
- 決済完了後は完了画面が表示され、ダッシュボードへ続けて進める導線がある
- 決済をキャンセル / 中断した場合はダッシュボードに戻り、残数は変わらない

### 購入対象・購入者のガード

- 公開中でない面談パックは、購入動線に並ばないだけでなく、URL を直接指定しても購入できない
- 学習中でない受講生は購入できない

### 決済イベントの受信(決済サービスからの通知）

> ここでの「イベント」は外部決済サービス(Stripe)がサーバ間で送ってくる通知を指す(LMS のアプリ内通知 / メールとは別物)。

- 決済サービスから送られてくる決済結果の通知を受け取る公開窓口を用意し、通知が改ざんされていないか検証する
- 決済完了の通知を受けて、購入回数分の残面談回数を加算する
- 決済失敗 / 期限切れの通知を受けた場合は残数を加算しない
- 同じ決済完了の通知が複数回届いても、残数加算は 1 回だけに保つ
- 対応する購入記録がない通知や、扱う必要のない種別の通知は、受信成功として無視する

### 共通の振る舞い

- 残数が加算された瞬間から、受講生の残数(ダッシュボード / 予約画面の数値)に反映される
- 決済額 / 購入回数は購入時点の値を控えとして保存し、後からマスタを変更しても過去の購入を監査できる
- 同じ決済に対して購入記録が重複して作られない

### 非機能要件

- 決済通知の改ざん防止(署名検証を必須化)
- 通貨は円のみ、都度購入(自動再課金・サブスクリプションなし)

## スコープ外

- 面談回数のサブスクリプション(自動再課金)— 都度購入のみ
- 面談回数の有効期限 — 無期限(受講プラン期間内)
- 面談回数の他者への譲渡
- 円以外の通貨対応 — 円のみ
- Stripe 以外の決済プロバイダ(PayPal / Square / Pay.jp 等)
- 返金フロー(受講生主導)— 管理者が Stripe ダッシュボードから手動操作
- 面談パックの在庫管理(デジタル商品で無限在庫)
- 管理者による面談回数の手動付与 UI — 本チケットでは購入動線のみを扱う
- 残数ゼロ通知 / 購入リマインダーのプッシュ通知
- 面談パックのまとめ買い割引 / クーポンコード — 単品購入のみ
- 領収書 / 請求書 PDF 自動発行 — Stripe ダッシュボードからの手動操作で代替
- 多段階の承認フロー(法人契約等)

## インターフェース

| HTTP | パス | 認可 |
|---|---|---|
| GET | `/meeting-quota/checkout` | 受講生(学習中)のみ |
| POST | `/meeting-quota/checkout` | 受講生(学習中)のみ |
| GET | `/meeting-quota/success` | 受講生(学習中)のみ |
| POST | `/webhooks/stripe` | 認証なし(署名検証のみ) |

購入の入口は追加面談の購入画面。決済完了 / キャンセルの戻り先はそれぞれ完了画面 / ダッシュボード。決済サービスからの通知窓口は認証なしの公開エンドポイントで、署名検証だけが正当性の担保となる。
