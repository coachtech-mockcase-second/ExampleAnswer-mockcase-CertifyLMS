## シート1: ターム内容

| 項目 | 内容 |
|---|---|
| タイトル | 模擬案件_資格学習プラットフォーム Certify LMS 開発参画 |
| 目的 | 模擬案件を通して、**稼働中の既存プロジェクトに途中参画して開発する実務プロセス**（既存コードリーディング、チケットベース開発、PM へのヒアリング、Pull Request による成果提出）を経験すること |
| 期間 | コーチと相談の上決めてください。<br>※ 目安総工数は約 220 時間です（各チケットの目安工数はシート 4 参照） |
| やること | チケットベースのバックエンド開発（既存コードリーディング、新機能実装、バグ修正、リファクタリング・パフォーマンス改善、テスト作成）<br>※ 画面（Blade テンプレート・CSS・JavaScript）は完成品として提供されるため、フロントエンドの実装は原則不要です（一部チケットで JavaScript からの API 利用を扱います）<br><br>本書では「何を作るか（WHAT）」は示しますが、「どう作るか（HOW）」の詳細はあえて記載していません。データ構造、バリデーションルール、画面遷移、認可の細部などは、**既存コードのパターンを読み解き、自分で設計し、不明点はコーチ（PM）にヒアリング**して詰めてください。詳しくはシート 2 を確認してください。<br><br>【2 フェーズ構成】<br>本模擬案件は以下の 2 フェーズに分かれています。<br><br>**フェーズ 1: Basic チケット（27 件）**<br>`basic` ブランチを起点に開発し、`basic` に向けて PR を提出します。<br><br>**フェーズ 2: Advance チケット（14 件）**<br>Basic 完了後に着手します。フェーズ 2 開始時に `basic` での成果を `advance` ブランチへ取り込み、以降は `advance` に向けて PR を提出します。応用機能向けの画面もあらかじめプロジェクトに組み込まれており、対応する機能を実装すると画面に現れます。外部 API 連携（Google Calendar / Gemini / Stripe）、Sanctum 認証、キュー、キャッシュなどの応用技術に取り組みます。 |
| 作成物 | 各チケットの実装を含む **Pull Request（1 チケット = 1 PR、全 41 件）**<br><br>【システム概要】<br>開発対象の Certify LMS は、マルチ資格対応の資格学習プラットフォームです。受講生は資格ごとの教材で学習し、演習問題・模擬試験で理解度を確かめながら、コーチの面談サポートを受けて資格取得を目指します。受講生・コーチ・管理者の 3 ロールが登場し、招待制でユーザーが管理されます。 |
| ルール | 原則質問チャットサポートの利用は禁止です。<br><br>留意点：<br>1. 教材やブラウザで検索した記事を参考にすることは可能です。<br>2. GitHub でのエラーが解決できず、模擬案件の提出が困難な場合はコーチに相談してください。<br>3. 質問の利用数は COACHTECH Pro の合格基準とさせていただきますので、できるだけ質問対応は利用しないようにしましょう！<br><br>※ 仕様・要件の確認をコーチ（PM）へヒアリングすることは上記の「質問」とは別であり、本課題の中核プロセスとして推奨されます（シート 2 の Step 3 参照）。 |

## シート2: 教材の設定と進め方

| 項目 | 内容 |
|---|---|
| 教材の設定 | あなたは開発会社のエンジニアです。クライアント企業が運営する、稼働中の資格学習プラットフォーム「Certify LMS」の開発チームに途中参画しました。サービスには多数の機能が既に実装されています。あなたには新機能開発（Story）・バグ修正（Bug）・改善（Task）の **41 件のチケット** がアサインされています。チケットの要件は詳細まで固められていません。**コーチ = クライアント PM** として、あなたは PM にヒアリングを行い、仕様を詰めながら開発を進めていきます。 |
| Step 0: 参画準備 | 自リポジトリを生成して環境構築します（シート 3）。続けてリポジトリ直下の `ONBOARDING.md`（コードリーディングガイド）を読み、プロジェクト構造・ドメインモデル・既存パターンの全体像を掴みます。 |
| Step 1: チケットの把握 | 着手するチケットの要件を読み、関連する画面を実際に操作し、関連する既存コードを確認します。 |
| Step 2: 思考と設計 | 「この機能を実現するには、どんなデータが必要か？」「どんなバリデーションが必要か？」「成功時 / 失敗時の画面遷移はどうすべきか？」といった**詳細な仕様を自分自身で考え、言語化**してください。**既存コードの類似機能が最大のヒント**です。提供された Blade テンプレートのフォーム項目（入力タイプ、属性、プレースホルダー等）も重要な手がかりになります。 |
| Step 3: PM へのヒアリング | 自分で考えた設計案を PM（コーチ）に提示し、「この仕様で問題ないか？」「他に考慮すべき点はあるか？」といった確認を行います。不明点があれば積極的に質問してください。<br><br>このとき、「〜はどうしますか？」と受け身で聞くのではなく、**「〜という設計にしましたが、問題ないですか？」と自分の仮説を提案する形**を心がけると良いでしょう。PM も仮説ベースの方が回答しやすく、やり取りの効率が上がります。 |
| Step 4: 設計の確定 | PM からのフィードバックを元に設計をブラッシュアップし、最終的な仕様を確定させます。 |
| Step 5: 実装と提出 | 確定した設計を元に実装し、テスト・動作確認を行い、PR を提出します（シート 3 の PR 運用）。マージしたら次のチケットへ進みます。 |
| 既存プロジェクト参画の心得 | ゼロから作る課題と違い、**既存パターンの踏襲**が品質の基準になります。新しいコードを書く前に、必ず似た機能の既存実装を探して読み、命名・レイヤー構造・テストの書き方を倣ってください（`ONBOARDING.md` に「やりたいこと → 手本となる既存コード」の対応表があります）。 |

## シート3: 開発プロセス

| 項目 | 内容 |
|---|---|
| 自リポジトリの生成 | この課題のリポジトリは GitHub の **Template リポジトリ** として提供されます。リポジトリページの **「Use this template」→「Create a new repository」** から、自分のアカウントにリポジトリを生成してください。<br><br>**⚠️ 生成時に「Include all branches」へ必ずチェック**を入れてください（チェックしないと `advance` ブランチがコピーされず、フェーズ 2 に進めなくなります）。<br><br>※ `git clone` ではなく Template 生成を使う理由: 自分のリポジトリがないと PR の提出先が存在せず、成果物（PR）が成立しないためです。Template 生成なら fork ラベルも付かず、修了後もポートフォリオとしてそのまま残せます。 |
| 環境構築 | 生成した自リポジトリを `git clone` し、リポジトリ直下の **`README.md`「環境構築手順」** に従ってセットアップしてください。 |
| ブランチ運用 | デフォルトブランチは `basic` です。チケットごとに `basic` から `feature/{チケットID}`（例: `feature/S-B-01`）ブランチを切って作業し、`basic` に向けて PR を作成します（フェーズ 2 では `advance` に向けます）。**1 チケット = 1 PR** です。 |
| PR 記述（5 セクション） | PR 本文は次の 5 セクションで記述します（PR 作成時にテンプレートが自動で展開されます）。空欄やテンプレートの放置は減点対象です。<br><br>セクションの内訳は下の「PR 5 セクションの書き方」を参照してください。 |
| 動作確認の証跡 | PR の「動作確認」セクションには、確認手順とあわせて証跡を必ず添付します:<br>・静的な画面（一覧 / 詳細 / フォーム）= **スクリーンショット**<br>・動的な機能（状態遷移 / モーダル / 非同期更新など）= **動画**（Loom / 画面録画 / GIF）<br>・バグ修正・改修 = **修正前 / 修正後の比較**<br>・リファクタリング = テスト pass のスクリーンショットで代替可 |
| マージ運用 | PR のレビュー承認は不要です。提出前に自分で PR 内容を見直し、**セルフマージして次のチケットへ**進んでください（チケット間に依存があるため、マージせずに止まると後続チケットに着手できません）。 |
| コミット | コミットメッセージは「何をしたか」が明確にわかるように記述します（例: `qa-board: 質問投稿を実装`）。1 コミット = 1 論理変更を心がけましょう。 |
| コード品質 | Laravel Pint でコードを整形してからコミットします（`sail bin pint --dirty`）。採点では `sail bin pint --test` で「No fixable issues were found」となることが確認されます。命名・レイヤー構造・テストの書き方は**既存コードのパターンに合わせる**ことが品質基準です。 |
| テスト | 機能の実装・修正にはテストを伴わせ、`sail artisan test` で確認します。最終的に**全テストが pass** した状態が完成です（初期状態に失敗テストが含まれる点はシート 1「注意点」参照）。 |
| README への追記 | 外部サービス連携などで新しい環境変数やセットアップ手順が増えた場合は、**自リポジトリの README.md に追記**してください（チームの誰でも環境を再現できる状態を保つため。採点対象です）。 |
| 評価 | 評価は「各チケットの受け入れ状況」と「横断品質（コード品質 / テスト / README / PR 記述）」で行われます。全体配点に対する取得率（100 点満点換算）を、COACHTECH 標準の成績基準 **S: 95 点以上 / A: 85 点以上 / B: 70 点以上 / C: 50 点以上 / D: 50 点未満** に当てて評価されます。 |

### PR 5 セクションの書き方

| セクション | 書くこと |
|---|---|
| 概要 | チケット ID と、この PR で何ができるようになるか / 何が直るか（1〜2 行） |
| 調査・設計判断 | 何を読み・確認し、そこから何を判断したか。機能追加・改修なら「参考にした既存実装 → この設計にした理由・代替案を退けた理由」、バグ修正なら「調査の足取り → 特定した根本原因」 |
| 実装内容 | 主要な変更点を振る舞い単位で箇条書き（ファイル一覧ではなく「何ができるようになったか / 何が直ったか」） |
| 自動テスト | 追加・修正したテストと、それが保証する内容。テストを追加しない場合はその理由 |
| 動作確認 | 確認手順 + 証跡（スクリーンショット / 動画。上記「動作確認の証跡」参照） |

> 不安な箇所やレビューで重点的に見てほしい点があれば、PR 末尾に任意で追記して構いません。
