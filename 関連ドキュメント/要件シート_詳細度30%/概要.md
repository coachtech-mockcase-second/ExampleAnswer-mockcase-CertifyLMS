## シート1: 概要

### ターム内容

| 項目 | 内容 |
|---|---|
| タイトル | 模擬案件_資格学習プラットフォーム Certify LMS 開発参画 |
| 目的 | **稼働中の既存プロジェクトに途中参画して開発する実務プロセス**（既存コードリーディング、チケットベース開発、PM へのヒアリング、Pull Request による成果提出）を経験すること |
| シチュエーション設定 | あなたは開発会社のエンジニアです。クライアント企業が運営する、稼働中の資格学習プラットフォーム「Certify LMS」の開発チームに途中参画しました。サービスには多数の機能が既に実装されています。あなたには 41 件のチケットがアサインされています。**コーチ = クライアントの PM（プロジェクトマネージャー）** として、あなたは PM とやり取りしながら仕様を詰めて開発を進めます。 |
| 開発対象 | Certify LMS — マルチ資格対応の資格学習プラットフォーム。受講生・コーチ・管理者の 3 ロールが登場する招待制のサービスです。プロダクトの概要・機能・使用技術は、自リポジトリ直下の `README.md` を参照してください。 |

| 担当範囲 | チケットベースの**バックエンド開発**（既存コードリーディング、実装、バグ修正、リファクタリング・パフォーマンス改善、テスト作成）。<br>※ 画面（Blade テンプレート・CSS・JavaScript）は完成品として提供されるため、フロントエンドの実装は原則不要です（一部チケットで JavaScript からの API 利用を扱います）。 |
| 要件の詳細度 | 各チケットは**要件を高めの抽象度で示すにとどめ、詳細な仕様（データ構造・バリデーション・画面遷移・認可の細部など）はあえて固めていません**。詳細仕様は既存コードのリーディングと面談での PM ヒアリングで具体化し、それを踏まえた設計・実装はエンジニアとして自分で行います。要件の行間を読み、自ら思考し、コミュニケーションを取って仕様を詰めるプロセスそのものが重要な学習要素です。 |
| 想定期間 | コーチと相談の上で決めてください。 |

### 進め方

| ステップ | 内容 |
|---|---|
| Step 0: 参画準備 | 自リポジトリを生成して環境構築します（「開発ルール」参照）。続けてリポジトリ直下の **`README.md`**（プロダクト概要・環境構築）と **`ONBOARDING.md`**（コードリーディングガイド）を読み、プロジェクト構造・主要ドメインモデル・既存パターンの全体像を掴みます。 |
| Step 1: チケットの把握 | 着手するチケットの要件を読み、関連する画面を実際に操作し、関連する既存コードを確認します。 |
| Step 2: 思考と設計 | 「この機能を実現するにはどんなデータが必要か」「どんなバリデーションが必要か」「成功時 / 失敗時の画面遷移はどうすべきか」といった**詳細な仕様を自分自身で考え、言語化**します。**既存コードの類似機能が最大のヒント**です。提供された Blade テンプレートのフォーム項目（入力タイプ・属性・プレースホルダー等）も重要な手がかりになります。 |
| Step 3: 面談での PM ヒアリング | **コーチとの面談内**で、自分で考えた仕様案を PM（コーチ）に提示して確認します。「〜はどうしますか」と受け身で聞くのではなく、**「〜という仕様にしましたが問題ないですか」と仮説を提案する形**を心がけると、やり取りの効率が上がります。<br><br>**ヒアリングできるのは要件・仕様の確認に限ります**（機能の振る舞い、扱うデータ、バリデーション、画面遷移、認可の範囲など）。設計方針や実装方法（クラス構成・アルゴリズム・どのパターンを使うか等）は、エンジニアとして自分で判断・実装する領分であり、ヒアリングの対象ではありません。 |
| Step 4: 仕様の確定 | PM からのフィードバックを踏まえて仕様を確定させます。 |
| Step 5: 実装と提出 | 確定した仕様をもとに実装・テスト・動作確認を行い、PR を提出します（「開発ルール」の PR 運用参照）。マージしたら次のチケットへ進みます。 |
| 既存プロジェクト参画の心得 | ゼロから作る課題と違い、**既存パターンの踏襲**が基本です。新しいコードを書く前に、必ず似た機能の既存実装を探して読み、命名・レイヤー構造・テストの書き方を倣ってください（`ONBOARDING.md` に「やりたいこと → 手本となる既存コード」の対応表があります）。 |

### 開発ルール

| 項目 | 内容 |
|---|---|
| 自リポジトリの生成 | この課題は GitHub の **Template リポジトリ**として提供されます。<br>https://github.com/coachtech-prepared-file/AssignedProject-mockcase-CertifyLMS<br>ページの **「Use this template」→「Create a new repository」** から、自分のアカウントにリポジトリを生成してください。<br>※ `git clone` ではなく Template 生成を使うのは、PR の提出先となる自分のリポジトリが必要なためです（fork ラベルも付かず、修了後はポートフォリオとしてそのまま残せます）。 |
| 環境構築 | 生成した自リポジトリを `git clone` し、リポジトリ直下の **`README.md`「環境構築手順」** に従ってセットアップしてください。 |
| ブランチ・PR 運用 | デフォルトブランチは `main` です。チケットごとに `main` から `feature/{チケットID}`（例: `feature/S-B-01`）ブランチを切って作業し、`main` に向けて PR を作成します。**1 チケット = 1 PR** です。<br>フェーズはブランチで分けません。Basic チケットをひととおり終えてから Advance チケットへ進んでください。 |
| PR 記述（5 セクション） | PR 本文は 5 セクションで記述します（PR 作成時にテンプレートが自動展開されます）。内訳は下の「PR 5 セクションの書き方」を参照してください。 |
| 動作確認の証跡 | PR の「動作確認」セクションには、確認手順とあわせて証跡を添付します:<br>・静的な画面（一覧 / 詳細 / フォーム）= **スクリーンショット**<br>・動的な機能（状態遷移 / モーダル / 非同期更新など）= **動画**（Loom / 画面録画 / GIF）<br>・バグ修正・改修 = **修正前 / 修正後の比較** |
| マージ運用 | PR のレビュー承認は不要です。提出前に自分で内容を見直し、**セルフマージして次のチケットへ**進んでください（チケット間に依存があるため、マージせずに止まると後続に着手できません）。 |
| コード品質・テスト | 命名・レイヤー構造・テストの書き方は**既存コードのパターンに合わせる**ことが基本です。コードの整形やテスト実行の具体的な手順は自リポジトリの `README.md` に従ってください。 |
| README への追記 | 外部サービス連携などで新しい環境変数やセットアップ手順が増えた場合は、**自リポジトリの `README.md` に追記**してください（チームの誰でも環境を再現できる状態を保つため）。 |
| 提出方法 | LMS のテスト一覧画面から、生成した自リポジトリの URL を提出してください。 |

#### PR 5 セクションの書き方

| セクション | 書くこと |
|---|---|
| 概要 | チケット ID と、この PR で何ができるようになるか / 何が直るか（1〜2 行） |
| 調査・設計判断 | 何を読み・確認し、そこから何を判断したか。機能追加・改修なら「参考にした既存実装 → この設計にした理由・代替案を退けた理由」、バグ修正なら「調査の足取り → 特定した根本原因」 |
| 実装内容 | 主要な変更点を振る舞い単位で箇条書き（ファイル一覧ではなく「何ができるようになったか / 何が直ったか」） |
| 自動テスト | 追加・修正したテストと、それが保証する内容。テストを追加しない場合はその理由 |
| 動作確認 | 確認手順 + 証跡（スクリーンショット / 動画。上記「動作確認の証跡」参照） |

### 注意事項

| 項目 | 内容 |
|---|---|
| 質問チャットサポート | 原則、質問チャットサポートの利用は禁止です。<br>1. 教材やブラウザで検索した記事を参考にすることは可能です。<br>2. GitHub でのエラーが解決できず提出が困難な場合はコーチに相談してください。<br>3. 質問の利用数は COACHTECH Pro の合格基準となるため、できるだけ利用しないようにしましょう。<br>※ 仕様・要件をコーチ（PM）へ面談でヒアリングすることは上記の「質問」とは別で、本課題の中核プロセスとして推奨されます（「進め方」の Step 3 参照）。 |
| 外部 API の設定 | Advance では外部 API 連携（Google Calendar / Gemini / Stripe）を扱います。利用には各自でのキー取得・`.env` 設定が必要です。新しく必要になった設定は自リポジトリの `README.md` に追記してください。 |
| 初期状態のテスト | 環境構築直後に `sail artisan test` を実行すると**一部のテストが失敗します**が、環境構築の失敗ではありません（バグ修正チケットの対象箇所に対応する失敗で、該当チケットを完了すると pass するようになります）。最終的に全テスト pass を目指してください。 |
| 難易度 | 教材学習の集大成で、少し難易度が高いです。焦らず、一歩ずつ進めていきましょう。 |
