# T-A-04 外部 API 連携のモックを使ったテスト追加

## 概要

3 つの外部 API 連携機能(コーチの Google カレンダー連携による面談予約 / Gemini を用いた AI チャット / Stripe からの通知受信による追加面談購入)には正常系のテストはあるが、外部 API へ実際に接続せずに失敗系・境界系を検証するテストが薄い(通信エラー / 空応答 / 連携期限切れからのリフレッシュ・再試行 / 受信通知の署名検証や二重配信時の冪等性 等)。テスト実行のたびに本物の外部 API を叩くと、API キーの漏洩・利用枠の消費・通信状況による不安定な失敗を招く。

本チケットでは、3 連携それぞれに適切なモック手法を選択し、正常系に加えて異常系・境界系を網羅するテストを追加する。テスト実行時に外部 API への実通信が一切発生しない状態を担保し、外部依存のテストをまとめて分離実行できるようにする。**外部 API ごとに最適なモック手法が異なり、1 つのやり方では統一できない**点を実コードで扱うのが主眼。

## 要件

### Google カレンダー連携のテスト追加

- 認可フロー(連携の開始 / 連携情報の交換 / 連携の解除)と、カレンダー操作(空き時刻の取得 / 予定の作成 / 予定の削除)それぞれのモックテスト
- 連携の期限切れ → 自動リフレッシュ → 再試行、リフレッシュに失敗したときのフォールバック、削除済み予定の扱い、を検証
- 利用する側(空き枠の集計 / 予約 / キャンセル / 連携設定画面)は、外部ライブラリの低レベルな詳細ではなくカレンダー操作のまとまった単位でスタブ化してテスト

### Gemini AI チャットのテスト追加

- 正常系 / 通信エラー / 空応答 / 一時的なエラーからの再試行成功 / 送信内容(プロンプト構造)の検証

### Stripe からの通知受信のテスト追加

- 受信した通知が正規のものか検証するための署名生成ヘルパーを用意し、正規署名 / 不正署名 / 署名欠落、および同じ通知が二重に届いたときの冪等性を検証

### 外部 API への実通信の防止

- テスト実行時に、モックしていない外部通信が発生したらテストが失敗する仕組み
- 外部 API に依存するテストをまとめて分離実行(除外)できるグループ指定

## スコープ外

- 外部ライブラリ / API モデル / SDK のバージョンアップ・切替(テストの整備のみ)
- 本物の外部サービス(Google / Gemini / Stripe)を実際に叩く結合テストの追加(別経路で実施)
- カバレッジ計測ツールの導入(別タスク)
- リアルタイム配信のテスト(本チケットの対象 3 連携に含めない)
- 外部連携機能そのものの仕様変更(テスト追加のみ)
- 3 連携のモック手法を 1 つに統一する抽象化(それぞれに合った手法の使い分けを扱うことが目的)
