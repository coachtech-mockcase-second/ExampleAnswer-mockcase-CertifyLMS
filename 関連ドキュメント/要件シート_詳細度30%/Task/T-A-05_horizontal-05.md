# T-A-05 通知・メール配信の非同期化（キュー）

## 概要

通知(チャット / Q&A 返信 / 面談の予約・キャンセル・リマインダー / 管理者お知らせ)とメールの送信が、発火元の HTTP リクエスト内で同期実行されている。とくに管理者お知らせの一斉配信は対象受講生の人数だけ送信をループするため、人数に比例してリクエストがブロックされ、途中で 1 件でも送信が失敗するとリクエスト全体に伝播して、配信の信頼性と管理画面の応答性を損ねる。

本チケットでは、通知・メール送信をバックグラウンドのキューに逃がして非同期化する。発火元リクエストは送信をキューに積むだけで即応答し、実際の送信は worker が順次処理する。あわせて、一時的な送信失敗の自動リトライと、上限を超えた送信を失敗ジョブとして記録・再投入する運用を整える。

## 要件

### 通知・メール送信の非同期化

- 通知 / メール送信を発火元リクエストから切り離し、バックグラウンドのキューで処理する
- 一斉配信(管理者お知らせ)で対象受講生が多くても、発火元リクエストがブロックされない
- データ確定(トランザクション commit)後に送信をキューへ投入する(ロールバック時に送信が漏れない)

### キュー基盤の運用

- キューに積まれたジョブを処理する worker の起動手順を整備する
- 一時的な送信失敗時に段階的な待機を挟んで自動リトライする
- リトライ上限を超えた送信は失敗ジョブとして記録し、後から再投入できる

## スコープ外

- リアルタイム push 配信(Pusher / WebSocket)— 本チケットは「キューに積んで worker が送る」非同期化のみ
- Redis / SQS 等の外部ミドルウェアを使うキュードライバの採用 — DB ベースのキューで完結
- キュー監視ダッシュボード(Horizon 等)の導入
- 通知の送信内容 / 対象 / チャネルの変更 — 配信経路の非同期化のみが対象
- 定期実行のスケジューリング自体の変更 — 発火後の送信処理の非同期化が対象
