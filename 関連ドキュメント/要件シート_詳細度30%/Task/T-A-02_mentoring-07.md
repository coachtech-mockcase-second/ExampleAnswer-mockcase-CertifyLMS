## メタ情報

|---|---|

| 依存チケット | (なし) |

## 概要

面談機能の Controller の各メソッドが抱える業務ロジック・データ取得を、単一責務の Action クラスへ切り出すリファクタリング。状態変更系(予約 / キャンセル / メモ保存)も取得系(一覧 / コーチ向け一覧 / 詳細 / 空き枠取得)も Action に分離し、Controller をリクエスト受付 + 認可委譲 + レスポンス整形だけの薄いラッパーに統一する。本プロジェクトが全 Controller で踏襲している「1 Controller メソッド = 1 Action クラス」「Controller は薄く保つ」という軽量 Clean Architecture 規約に揃えるのが目的。

## 要件

- 状態変更系メソッド(受講生による面談予約 / 当事者によるキャンセル / コーチによる面談メモ記録)から業務ロジックを Action クラスへ抽出する
- 取得系メソッド(受講生向け一覧 / コーチ向け一覧 / 面談詳細 / 空き枠取得)のデータ取得処理も Action クラスへ抽出し、「1 メソッド = 1 Action」の既存パターンに揃える
- Controller メソッドは抽出した Action を呼ぶだけの薄いラッパーに書き直す(受付 / 認可委譲 / レスポンス整形のみ、業務ロジック・クエリ組み立ては 0 行)
- 状態変更を伴う Action はトランザクション境界で囲む(面談回数の増減・通知発火・外部カレンダー連動を同一境界に含める)
- 抽出した各 Action に単体テストを新規追加する

## スコープ外

- 予約フォーム表示メソッド — 認可 + 画面返却のみで業務ロジック・データ取得を持たないため、Action 化せず Controller に残す
- 定期実行(自動完了)から呼ばれる処理 — すでに分離済みで対象外
- Action が内部で呼ぶ協力 Service の責務見直し — Service 分離の別 Task と棲み分け
- 認可ロジックの Policy 抽出 — Policy は既存、変更しない
- 振る舞いを変える変更 — リクエスト・レスポンス・DB 副作用・通知文面はすべて同一
