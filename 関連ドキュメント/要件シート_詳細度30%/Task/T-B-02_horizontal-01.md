# T-B-02 大量レコードを扱う定期実行処理のメモリ最適化（chunk 処理）

## 概要

定期実行処理(プラン期間満了による受講生の自動修了 / 目標受験日を過ぎた受講登録の自動失敗 / 終了時刻を過ぎた面談の自動完了)は、対象を一定件数ずつに区切って処理してはいるものの、その分割方式が処理中の状態更新と両立しないため、対象が一定件数を超えると一部を取りこぼす。対象が将来的に万単位へ拡大すると、取りこぼしが顕在化して自動処理が漏れるおそれがある。

本チケットでは、この分割処理を状態更新と両立する方式に修正し、件数規模に依らず・メモリ使用量を一定に保ちつつ・全件を取りこぼしなく処理できる状態にする。対象 3 件はいずれも処理を進めながら「絞り込み条件となる状態」を更新するため、分割方式の選び方を誤ると処理対象を取りこぼす——という分割処理の典型的な落とし穴に対する判断軸(処理対象を更新するか / 件数規模)を扱う。

## 要件

### 分割方式の修正(対象 3 件)

- プラン期間が満了した受講生を修了状態に自動遷移する日次処理
- 目標受験日を過ぎた受講登録を不合格(学習中止)に自動遷移する日次処理
- 終了時刻を過ぎた面談を完了状態に自動遷移する分次処理

### 分割方式の選択

- 対象 3 件はいずれも処理を進めながら「絞り込み条件となる状態」を更新するため、処理を進めても取りこぼしが起きない分割方式を選択する
- 1 回に読み込む件数(チャンクサイズ)を固定する

### 動作確認

- 既存の定期実行処理の振る舞い(対象の絞り込み / 状態の遷移 / 出力)を変えず、件数を増やしても全件が処理されることを確認する

## スコープ外

- すでに分割処理で実装されている定期実行処理(面談リマインダー配信 / チャットメンバー同期 等)の改修
- 対象全件を 1 つのトランザクションで原子的に処理する業務要件を持つ処理(期限切れ招待の一括失効 + 連動退会 等)の分割処理化(原子性が崩れるため)
- 対象を一括でロックして処理する処理(滞留した学習セッションの強制クローズ 等)の分割処理化(ロックの範囲が崩れるため)
- 通知配信 / メール送信の非同期化(キュー化は別スコープ)
