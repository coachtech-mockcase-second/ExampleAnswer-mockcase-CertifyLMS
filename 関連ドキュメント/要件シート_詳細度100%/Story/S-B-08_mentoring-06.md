## 背景・目的

コーチは面談予約・面談に紐づく議事メモは記録できるが、面談以外の日常観察(「最近 chat の応答が遅れている」「Q&A で躓いた論点」「次回面談で確認したい事項」等)を構造化して残す場所がない。複数受講生を持つコーチは私的なメモ帳や別ツールに分散管理することになり、引き継ぎや管理者監督が困難になっている。

本チケットでは、コーチが担当資格に登録した受講生の受講登録単位でメモを CRUD でき、時系列に履歴を残せる状態にする。複数コーチ体制下では各コーチが自分のメモのみ編集 / 削除でき、管理者は全コーチのメモを越境管理できる。受講生本人にはメモが完全に見えない(コーチ → 管理者の業務記録の性質を保つ)。担当コーチが受講生の状況を継続観察するための業務記録基盤であり、本機能が整うとコーチ間の知見共有が促され、担当変更時の引き継ぎも円滑になる。新規エンティティの CRUD + ロール × 当事者の二重認可判定として Policy の典型実装が揃う構成でもある。

## ユーザーストーリー

- **コーチ(coach)として**、担当資格に登録した受講生の受講登録単位でメモを追加・編集・削除したい。なぜなら、面談以外の日常観察を構造化して残し、フォローの質を高めたいから。
- **コーチとして**、他コーチが書いたメモも閲覧したい。なぜなら、複数コーチ体制での申し送り事項を相互に把握したいから。介入(編集 / 削除)はせず、自分のメモのみ自分で管理する。
- **管理者(admin)として**、任意の受講生に対する全コーチのメモを閲覧・編集・削除したい。なぜなら、運用上の不適切記述の是正やコーチ離任時のメモ管理を一元的に扱いたいから。
- **管理者として**、自分自身もコーチと同じフォーマットでメモを残せることを期待する。なぜなら、運営観察も同じ場所に記録した方が一貫しているから。
- **受講生(student)として**、自分の受講登録配下にコーチ / 管理者が残したメモが自分には一切表示されないことを期待する。なぜなら、メモは業務記録であり、受講生に見えると素直な観察ができなくなるから。

## 要件

### メモの管理(コーチ + 管理者のみ操作可、受講生は閲覧含めすべて拒否)

- 追加: コーチは担当資格に登録した受講生の受講登録に本文を入力して追加でき、追加成功で元の画面に戻る。作成者として操作者が記録される
- 追加(管理者): 管理者は任意の受講登録にメモを追加でき、作成者には管理者が記録される
- 編集: コーチは自分が作成したメモのみ専用編集ページで本文を更新できる(管理者は任意のコーチ / 管理者が作成したメモを越境編集可)
- 削除: コーチは自分が作成したメモのみ削除できる(管理者は任意のメモを越境削除可)。削除は物理削除で履歴は残さない
- 作成者の不変性: 編集しても作成者は変更しない(誰が書いたかの履歴的整合性を保つ)

### メモの閲覧

- コーチ: 担当資格に登録した受講生の受講登録詳細画面でメモ一覧(時系列降順)を閲覧でき、自分のメモには編集 / 削除ボタンが表示される(他コーチのメモは閲覧のみ)
- 管理者: 任意の受講登録詳細画面でメモ一覧を閲覧でき、全メモに編集 / 削除ボタンが表示される
- 受講生本人: 自分の受講登録詳細画面にメモ一覧のセクション自体が表示されない

### 入力検証

- 本文: 必須 / 2000 文字以内

### アクセス制御(機能群共通)

- メモの閲覧 / 追加 / 編集 / 削除はコーチ(担当資格)と管理者のみ。受講生は閲覧含めすべて拒否
- 担当していない資格の受講登録に対するコーチのメモ操作は拒否
- 親の受講登録が論理削除された場合、その配下のメモは一覧から除外される

## スコープ外

- **受講生本人がメモを閲覧する動線** — 業務記録の性質を保つため、受講生は閲覧含め全拒否。設計上セクションが受講生画面で描画されない
- **メモの検索 UI**(本文部分一致検索等) — 時系列降順閲覧のみ
- **コーチ間のノート個別非公開フラグ** — 全コーチ + 管理者は他コーチのメモを常に閲覧可、個別非公開モードは持たない
- **メモへのコメント / リアクション機能** — 単純な本文のみ、対話機能なし
- **メモの達成マーク / ステータス管理** — メモはフラットな本文のみ(状態なし)
- **メモの論理削除 + 復元 UI** — 物理削除のみ(誤削除リスクは UI の確認ダイアログで対応)
- **添付ファイル(画像 / PDF)** — 本文テキストのみ
- **コーチ → 受講生宛のメッセージ** — chat Feature の責務、本機能は内部記録専用
- **コーチ離任時のメモ自動委譲 / 移管** — 自動的な作成者書き換えは行わない(履歴の連続性を重視、運用上必要なら管理者が新たにメモを追加して言及)

## 受け入れ条件

- [ ] 担当資格に登録した受講生の受講登録に対してコーチ(または管理者)がメモを追加すると保存され、元の受講登録詳細画面に戻ってフラッシュが表示される。作成者には操作したコーチ / 管理者が記録される
- [ ] 自分が作成したメモ(管理者は任意のメモ)を専用編集ページで本文更新でき、成功時に作成者は変わらず元の画面に戻ってフラッシュが表示される
- [ ] 自分が作成したメモ(管理者は任意のメモ)を物理削除でき、成功時に元の画面に戻ってフラッシュが表示される
- [ ] 担当資格に登録した受講生の受講登録詳細画面でメモが時系列降順で表示され、自分が作成したメモにのみ編集 / 削除ボタンが出る(他コーチのメモは閲覧のみ)。管理者は全メモに編集 / 削除ボタンが表示される
- [ ] 受講生が自分の受講登録詳細画面を開いてもメモのセクション自体が表示されず、メモ系ルートに直接アクセスしても 403 が返る
- [ ] コーチが担当していない資格の受講登録へメモ操作を試みた場合、および他コーチが作成したメモの編集 / 削除アクションにアクセスした場合はいずれも 403 が返る
- [ ] メモ追加 / 編集フォームにて、本文にバリデーションが行われ、ルール違反時に入力値の再表示とともに日本語のエラーメッセージが表示されるか:
  - 本文: 必須 / 2000 文字以内
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### インターフェース(必須)

**エンドポイント**:

| HTTP | パス | 認可 | 振る舞い |
|---|---|---|---|
| POST | `/enrollments/{enrollment}/notes` | 担当コーチ / 管理者(受講生・担当外コーチは 403) | メモ追加、成功時 `back()`(元の受講登録詳細等)へ redirect + フラッシュ「メモを追加しました。」 |
| GET | `/enrollment-notes/{note}/edit` | 作成者本人 / 管理者(他コーチ・受講生は 403) | 編集フォーム表示 |
| PATCH | `/enrollment-notes/{note}` | 作成者本人 / 管理者 | メモ更新(作成者は不変)、成功時 受講登録詳細へ redirect + フラッシュ「メモを更新しました。」 |
| DELETE | `/enrollment-notes/{note}` | 作成者本人 / 管理者 | 物理削除、成功時 `back()` へ redirect + フラッシュ「メモを削除しました。」 |

> route 名: `enrollments.notes.store`(親 Enrollment のネスト URL で POST)+ `enrollment-notes.{edit,update,destroy}`(個別メモは ID 解決可能なためフラット URL)。`auth` Middleware 配下の認証済ルート(ロール Middleware は不要 = Policy が当事者判定で完結)。GET 一覧 / 個別メモ詳細の単独画面は持たず、一覧は親 `enrollments.show` の partial として表示する。

### データモデル

**エンティティ**:

| エンティティ (Model) | 主要属性 | 関係性 | 制約 |
|---|---|---|---|
| 受講生メモ (`EnrollmentNote`) | 本文 / 作成者(業務語彙) | 受講登録(`Enrollment`)に N:1 所属(`Enrollment::notes()` HasMany)/ 作成者(`User`、`coach_user_id` 経由 `author()` BelongsTo) | ULID 主キー / `enrollment_id` FK `cascadeOnDelete`(親受講登録削除で連動物理削除)/ `coach_user_id` FK `restrictOnDelete`(作成者 User の物理削除を阻む = 退会は論理削除で行う)/ SoftDelete 不採用 = 物理削除のみ |

**Enum**: 本チケットで新規 Enum なし。

**インデックス用途**:

- `(enrollment_id, created_at)` 複合(受講登録配下の時系列降順一覧の高速化)
- `coach_user_id`(作成者別検索)

**初期データ Seeder(必須)**:

- `EnrollmentNoteSeeder`: 受講生 A × 資格 X 受講登録にコーチ X 作成メモ × 2(直近 / 数日前)+ 管理者作成の越境メモ × 1 / 受講生 B × 資格 X にコーチ X メモ × 1 / 受講生 C × 資格 Y にコーチ Y メモ × 1
- 動作確認用途: コーチ X 自身のメモの編集 / 削除ボタン活性 + 他コーチ・管理者メモの出し分け / 管理者越境 CRUD / 複数受講生間のメモ分離 / コーチ X からの担当外資格(資格 Y)メモ閲覧で 403
- DatabaseSeeder 順序: `UserSeeder` → `CertificationSeeder` → `EnrollmentSeeder`(`certification_coach_assignments` 含む)→ `EnrollmentNoteSeeder`

### コンポーネント

**Controller** (`app/Http/Controllers/`)
- `EnrollmentNoteController` — `store` / `edit` / `update` / `destroy`。`edit` / `destroy` は Controller 内で `$this->authorize(...)`、`store` / `update` は FormRequest::authorize() に集約

**FormRequest** (`app/Http/Requests/EnrollmentNote/`)
- `StoreRequest` / `UpdateRequest` — 本文の検証 + `authorize()` で Policy(`create` / `update`)を呼ぶ

**Action** (`app/UseCases/EnrollmentNote/`、※ Advance 範囲、Basic 受講生は Controller 内完結も可)
- `StoreAction`(作成者に操作者を記録して INSERT)/ `UpdateAction`(本文のみ更新、作成者不変)/ `DestroyAction`(物理削除)

**Policy** (`app/Policies/`)
- `EnrollmentNotePolicy` — `viewAny` / `view` / `create`(担当コーチ + 管理者越境)+ `update` / `delete`(作成者本人 + 管理者越境)。担当判定 / 越境 + 作成者判定を private ヘルパー(`canAccessEnrollmentForNotes` / `canModify`)に集約

**Model** (`app/Models/`)
- `EnrollmentNote`(新規)/ `Enrollment`(`notes()` HasMany を追加)

**View**(既存、ロック対象)
- `resources/views/enrollment-note/edit.blade.php`(編集専用ページ)/ `resources/views/enrollment-note/_list.blade.php`(親 `enrollments.show` に埋め込むメモ一覧 + 追加フォーム)

**Migration / Seeder**
- `database/migrations/*_create_enrollment_notes_table.php`
- `database/seeders/EnrollmentNoteSeeder.php`

**Routes** (`routes/web.php`)
- `enrollments.notes.store` + `enrollment-notes.{edit,update,destroy}`(`auth` 配下)

### 異常系

**入力検証**(FormRequest クラス名 + ルール記法):

- メモ追加 / 更新 (`EnrollmentNote\StoreRequest` / `EnrollmentNote\UpdateRequest`、同一ルール):
  - `body`: `required` / `string` / `max:2000`

**業務例外**:

- 状態ベースの業務例外はなし(認可拒否はすべて Policy 経由の 403)。受講登録の状態(`passed` / `failed` 等)はメモ CRUD を妨げない

### 設計判断

- **画面構成**: メモ一覧 + 追加フォームは親 `enrollments.show` 画面に partial で埋め込み、編集は専用ページ(`enrollment-notes.edit`)へ遷移する純 Laravel パターン(JS なし)。`EnrollmentGoal` と同じ構成
- **ルート命名の使い分け**: 追加は親 Enrollment のネスト URL(`enrollments.notes.store`)、編集 / 更新 / 削除は個別メモが ID で解決可能なためフラット URL(`enrollment-notes.*`)
- **作成者の保持**: 作成者は INSERT 時に操作者で設定し、UPDATE では変更しない(編集の `validated` に作成者を含めない)。「誰がこのメモを書いたか」の履歴的整合性を保つ
- **Policy の DRY**: 担当コーチ判定 + 管理者越境 + 作成者本人判定を private ヘルパー(`canAccessEnrollmentForNotes` / `canModify`)に集約。可読性と重複排除を両立
- **管理者の操作者記録**: 管理者がメモを追加すると作成者カラム(`coach_user_id`)に管理者が記録される(カラム名は `coach_user_id` だが実体は「作成者の User ID」)。リネームより、テーブル / Model / Blade / Action の一貫性を優先して現名のまま運用
- **受講生への完全非表示の二重防御**: Blade で `@can('viewAny', [EnrollmentNote::class, $enrollment])` によりセクション自体を出し分け + エンドポイント直叩きでも Policy が 403 を返す
- **`EnrollmentGoal` との役割対称**: 同じ「受講登録配下の子エンティティ」だが、受講生本人 CRUD(個人目標機能)/ コーチ + 管理者 CRUD(本チケット)で対称。Policy の責務委譲 + 二重認可判定パターンは両者で共通
- **テスト観点**: Policy の各メソッド × 各ロール × 当事者判定(担当コーチ × 自作成 true / 他コーチ作成 false(update/delete)/ 担当外コーチ false / 管理者 true 全パターン / 受講生 false 全パターン)の網羅 + 編集時の作成者不変 + 受講生画面でのセクション非表示

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| メモ本文の文字数上限は? | 2000 文字。必須 |
| 1 受講登録あたりのメモ数上限は? | 上限なし。コーチが自由に書き残す |
| 受講生本人にメモは見える? | 見えない。閲覧含めすべて 403。Blade 上でもセクション自体が出ない |
| 担当外コーチがメモを書ける / 閲覧できる? | できない。担当資格判定により担当外資格の受講登録に対しては 403(viewAny も同様) |
| 他コーチが作成したメモを編集 / 削除できる? | できない。コーチは自分が作成したメモのみ。他コーチのメモを編集 / 削除できるのは管理者のみ(越境) |
| 管理者は誰のメモでも編集 / 削除できる? | できる。全コーチ / 管理者が作成したメモを越境で編集 / 削除可 |
| 管理者がメモを追加すると作成者は誰になる? | 操作した管理者が作成者として記録される(作成者カラムの実体は「作成者の User ID」) |
| 編集すると作成者は変わる? | 変わらない。履歴的整合性のため編集時に作成者は不変 |
| メモは論理削除 / 物理削除? | 物理削除。誤削除リスクは UI の確認ダイアログで対応 |
| 親の受講登録が削除されたらメモは? | 連動して物理削除される(FK ON DELETE CASCADE)。論理削除された受講登録のメモは一覧から除外 |
| 作成者の User を退会させたら? | 作成者は物理削除できない(ON DELETE RESTRICT のため退会は論理削除で行う)。論理削除された作成者の名前は `withTrashed()` で解決できる |
| `passed` / `failed` の受講登録にもメモ追加できる? | できる。受講登録の状態判定は本機能の責務外(修了済 / 失敗済の受講生にも申し送りメモを残せる) |
| メモの並び順は? | 時系列降順(最新が上)|
| 検索 / 絞り込み UI はある? | ない。時系列降順の閲覧のみ |
| 編集はインライン / モーダル / 専用ページ? | 専用ページ遷移(Basic は JS なし、個人目標機能と同じ純 Laravel パターン)|
| 削除確認ダイアログは? | HTML 標準 `onsubmit="return confirm(...)"` で十分(JS ファイル不要)|
| フラッシュ文言の推奨は? | 追加「メモを追加しました。」/ 編集「メモを更新しました。」/ 削除「メモを削除しました。」(適切な日本語であれば細部は採点対象外)|
