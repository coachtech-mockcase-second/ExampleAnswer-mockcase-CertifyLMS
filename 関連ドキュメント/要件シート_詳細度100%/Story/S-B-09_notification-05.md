## 背景・目的

既存の通知基盤は業務イベント(chat 受信 / Q&A 回答 / 面談予約 / 面談キャンセル)に紐づく **受動的な通知発火** のみで、運営側が能動的に受講生集合へメッセージを届ける手段がない。メンテナンス予告 / 重要更新 / 学習キャンペーン告知などの運営連絡を chat 個別送信や外部メールツールで代替する状況。

本チケットは管理者(admin)が能動的に受講生集合へ一斉配信できるお知らせ機能を新規実装する。配信対象は「全受講生」「資格指定」「ユーザー指定」の 3 種類から選択でき、既存の通知基盤を利用して各受講生にデータベース通知 + メールを発火する。配信は不可逆(再配信 / 編集 / 取消なし)で、配信履歴は admin 専用画面で時系列降順に閲覧可能。本機能が揃うと外部メールツールへの依存が減り、運営からの能動連絡を LMS 内に集約する **運営コミュニケーション基盤** が成立する。

## ユーザーストーリー

- **管理者(admin)として**、全受講中の受講生に対して一斉お知らせを配信したい。なぜなら、メンテナンス予告 / 重要更新 / 学習キャンペーン告知を運営から直接届けたいから。
- **管理者として**、指定資格に登録した受講生だけに絞ってお知らせ配信したい。なぜなら、資格固有の試験変更 / 教材更新を該当受講生だけに届けたいから。
- **管理者として**、指定 1 ユーザーだけにお知らせ配信したい。なぜなら、個別のフォロー連絡を通知として残したいから(chat より明示的な「運営連絡」の格付けで残せる)。
- **管理者として**、配信履歴を時系列で閲覧し、各お知らせの配信件数 / 配信時刻を確認したい。なぜなら、配信実績を後から監査できる状態にしたいから。
- **管理者として**、誤配信防止のため再配信 / 編集 / 取消ができない設計を期待する。なぜなら、メール配信は取り消せない不可逆性を持つため、UI 上も同じ不可逆性で運用責任を明確化したいから。
- **受講生(student)として**、運営からのお知らせを通常の通知一覧で業務通知と並列に受信したい。なぜなら、運営連絡を見落としたくないから。

## 要件

### 配信作成・実行(管理者のみ操作可)

- 配信作成フォームの表示(タイトル(必須 / 200 文字以内)/ 本文(必須 / 5000 文字以内、プレーンテキスト)/ 配信対象タイプ(必須 / 全受講生・資格指定・ユーザー指定のいずれか)を入力、配信対象タイプに応じて配信先資格(資格指定時のみ必須 / 既存資格と整合)/ 配信先ユーザー(ユーザー指定時のみ必須 / 既存受講生と整合)フィールドを動的表示)
- 配信実行(配信対象集合を解決し各受講生に DB 通知 + メール送信、配信件数と配信時刻を記録)

### 配信対象の 3 種類

- **全受講生**: 受講中の全受講生に配信、資格 / ユーザー指定なし
- **資格指定**: 指定資格に学習中で登録している受講中の受講生に配信
- **ユーザー指定**: 指定 1 受講生のみに配信(指定ユーザーが受講生ロールでなければ拒否)

### 配信履歴閲覧(管理者のみ操作可)

- 配信履歴一覧(時系列降順、20 件 / ページ、配信件数 / 配信時刻 / 配信対象 表示)
- 配信詳細閲覧(タイトル / 本文 / 配信対象 / 配信件数 / 配信時刻 / 配信した管理者の名前)

### 配信の不可逆性

- 再配信不可(同じお知らせを再度配信する動線を持たない)
- 編集不可(配信後のお知らせのタイトル / 本文 / 対象 は変更不可)
- 取消不可(配信後のお知らせを削除する動線を持たない)

### 共通の振る舞い

- 配信対象集合は受講中ロールの受講生のみに限定(退会済 / 修了済 / 招待中 / コーチ / 管理者 は配信対象外)
- 配信されたお知らせは既存の通知基盤を経由して各受講生の通知一覧画面 / TopBar 未読バッジ / メール送信 すべてに乗る
- 通知行クリック時の遷移先は通知一覧画面(お知らせ単独画面ではない)
- メール件名は「【Certify LMS】」プレフィックス統一

## スコープ外

- **再配信 / 編集 / 取消** — 配信の不可逆性を仕様として確定
- **コーチ向けの配信** — 配信対象は受講生のみ(運営 → 学習者の連絡経路として位置付け)、コーチ向けは別 Feature の業務通知で間接的に発火
- **配信予約 / スケジュール送信** — MVP 外、配信は即時のみ
- **配信ターゲット内の追加絞り込み(複数資格 AND / 学習進捗フィルタ等)** — 単純な 3 種類(全 / 資格 / ユーザー)のみ
- **配信前の対象件数プレビュー** — MVP 外、配信実行時の配信件数で事後確認
- **お知らせの優先度フラグ / 既読率追跡 / 開封確認** — MVP 外
- **添付ファイル / リッチテキスト本文** — プレーンテキストのみ
- **お知らせ画面の受講生側専用詳細ページ** — 受講生は通常の通知一覧でデータを確認、お知らせ単独画面は持たない
- **配信失敗のリトライ** — 同期送信のためエラー時は管理者が再操作

## 受け入れ条件

- [ ] 管理者が `/admin/announcements` にアクセスすると配信履歴が時系列降順 + ページネーション(20 件 / ページ)で表示され、各行に配信件数 / 配信時刻 / 配信対象 が表示される
- [ ] 管理者が配信作成フォーム(`/admin/announcements/create`)を開くと、タイトル / 本文 / 配信対象タイプの入力欄が表示され、配信対象タイプの選択に応じて資格選択 / ユーザー選択フィールドが動的に出し分けされる
- [ ] 管理者が「全受講生」を指定して配信実行すると、受講中の全受講生に通知 + メールが発火し、配信件数と配信時刻が確定する
- [ ] 管理者が「資格指定」を指定して配信実行すると、指定資格に学習中で登録している受講中の受講生のみに通知 + メールが発火する
- [ ] 管理者が「ユーザー指定」を指定して配信実行すると、指定された 1 受講生のみに通知 + メールが発火する
- [ ] 受講中でない受講生(退会済 / 修了済 / 招待中)/ コーチ / 管理者 は配信対象集合から除外され、配信件数にも含まれない
- [ ] 管理者が配信詳細(`/admin/announcements/{announcement}`)を開くと、タイトル / 本文 / 配信対象 / 配信件数 / 配信時刻 / 配信した管理者の名前 が表示され、編集 / 削除 / 再配信ボタンは画面上に存在しない
- [ ] 配信されたお知らせは受講生の通知一覧画面に表示され、メール件名は「【Certify LMS】」プレフィックスで始まる
- [ ] 配信作成画面にて、各入力項目にバリデーションが行われ、ルール違反時に日本語のエラーメッセージが表示され配信は実行されないか:
  - タイトル: 必須 / 200 文字以内
  - 本文: 必須 / 5000 文字以内
  - 配信対象タイプ: 必須 / 全受講生・資格指定・ユーザー指定のいずれか
  - 配信先資格: 配信対象タイプが「資格指定」時のみ必須 / 既存資格と整合(その他のタイプでは指定不可)
  - 配信先ユーザー: 配信対象タイプが「ユーザー指定」時のみ必須 / 既存受講生と整合(その他のタイプでは指定不可)
- [ ] 受講生 / コーチが配信履歴 / 作成フォーム / 配信実行 / 配信詳細 のいずれかにアクセスすると 403 が返る
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### インターフェース(必須)

| HTTP | パス | 認可 | 振る舞い |
|---|---|---|---|
| GET | `/admin/announcements` | 管理者のみ(受講生 / コーチは 403) | 配信履歴一覧(時系列降順、20 件 / ページ、配信件数 / 配信時刻 / 配信対象 表示) |
| GET | `/admin/announcements/create` | 管理者のみ | 配信作成フォーム表示(タイトル / 本文 / 配信対象タイプ / 配信対象に応じた選択フィールド) |
| POST | `/admin/announcements` | 管理者のみ | 配信実行、成功時 `/admin/announcements/{announcement}` リダイレクト + フラッシュ「お知らせを配信しました (N 件)。」、対象受講生に通知 + メール発火 |
| GET | `/admin/announcements/{announcement}` | 管理者のみ | 配信詳細(タイトル / 本文 / 配信対象 / 配信件数 / 配信時刻 / 配信者) |

**ミドルウェア**: 全ルートに `auth` + `role:admin` を適用。`edit` / `update` / `destroy` ルートは登録しない(配信不可逆性を Route / Controller レベルで強制)。受講生 / コーチは通知一覧画面側で受信したお知らせを閲覧する(本 Feature の admin ルート群には到達不要)。

**ルート登録**: `routes/web.php` の admin プレフィックス内に `Route::resource(...)->only(['index','create','store','show'])` 相当の 4 ルートを登録。

### データモデル

| エンティティ (Model) | 主要属性 | 関係性 | 制約 |
|---|---|---|---|
| お知らせ (`Announcement`) | 配信した管理者 / タイトル(短文) / 本文(長文プレーンテキスト) / 配信対象タイプ / 配信先資格(資格指定時のみ) / 配信先ユーザー(ユーザー指定時のみ) / 配信件数 / 配信時刻 | User belongsTo(配信者、退会済管理者も解決可能) / Certification belongsTo(配信先資格) / User belongsTo(配信先ユーザー) | ULID 主キー / SoftDelete 不採用(配信履歴の不可逆性 + 削除動線なし) / FK は `restrictOnDelete`(配信者・配信先資格・配信先ユーザーの物理削除を拒否) / 配信時に確定後の状態遷移なし |

**Enum**:

- 配信対象タイプ (`AnnouncementTargetType`): 全受講生 (`All`) / 資格指定 (`Certification`) / ユーザー指定 (`User`)

**インデックス用途**:

- 配信対象タイプ × 配信時刻(対象別の配信履歴検索)
- 配信者(配信者別履歴)

**初期データ Seeder(必須)** (`AnnouncementSeeder`):

- 全受講生対象 × 1 件(「春の学習キャンペーン開始のお知らせ」/ 2 週間前配信)
- 資格指定対象 × 1 件(「[資格 X] 試験範囲変更のお知らせ」/ 1 週間前配信)
- ユーザー指定対象 × 1 件(「[受講生 A] 個別フォロー」/ 3 日前配信)
- 配信時の通知行: 各お知らせに紐づく受講生通知行を `NotificationSeeder` 経由で直接 INSERT(`type` = `Announcement\AnnouncementNotification` FQCN でリンク)
- 副作用抑制: Mail / Queue 副作用は発生させない(本 Seeder では通知行を直接 INSERT)
- DatabaseSeeder 順序: `UserSeeder` / `CertificationSeeder` / `EnrollmentSeeder` の後、`NotificationSeeder` の前

### 異常系

**入力検証**(FormRequest クラス名 + ルール記法):

- 配信作成 FormRequest (`Announcement\StoreRequest`):
  - `title`: `required` / `string` / `max:200`
  - `body`: `required` / `string` / `max:5000`
  - `target_type`: `required` / `string` / `in:all,certification,user`
  - `target_certification_id`: `required_if:target_type,certification` / `prohibited_unless:target_type,certification` / `nullable` / `ulid` / `exists:certifications,id`
  - `target_user_id`: `required_if:target_type,user` / `prohibited_unless:target_type,user` / `nullable` / `ulid` / `exists:users,id`

**業務例外**(状態ベースガード + HTTP ステータス):

- 受講生 / コーチが本 Feature の admin ルート群にアクセス → 403(`AnnouncementPolicy` 拒否)
- 配信対象不在(`target_type=certification` だが指定資格が存在しない 等)→ `Notification\AnnouncementTargetNotFoundException` を投げて 422
- 配信対象タイプと指定 FK 列の組み合わせ不整合 → `Notification\AnnouncementInvalidTargetException` を投げて 422
- 受講中でない受講生(退会済 / 修了済 / 招待中)/ コーチ / 管理者 → 配信対象集合から除外(配信件数にも含まれない)
- 受講生でないユーザー(コーチ / 管理者)を「ユーザー指定」で選択 → 配信集合解決時に受講生ロールチェックで除外(配信件数 0)
- 配信件数 0 件のお知らせ: 配信可能(誤操作の証跡として残す)、配信時刻も記録

### 設計判断

- **配信の不可逆性をコード 3 層で防御**: Controller を 4 method(`index` / `create` / `store` / `show`)に限定 + Route に `edit` / `update` / `destroy` を登録しない + UI 上にも編集 / 削除ボタンを配置しない 3 層で防御。メール配信は取り消せないため、UI 上も同じ不可逆性で運用責任を明確化
- **業務トランザクション + 通知発火の分離**: 業務トランザクション内では `announcements` INSERT + 配信件数 / 配信時刻 UPDATE までを原子化し、通知発火は `DB::afterCommit` のフックで実行(ROLLBACK 時のメール副作用残りを回避)
- **配信対象集合の解決を Action 内に集約**: 配信対象タイプに応じた受講生集合解決(全受講生 / 資格指定 / ユーザー指定 × 受講中 × 受講生ロール フィルタ)を `NotifyAnnouncementAction` 内で完結。Basic 受講生は `AnnouncementController::store` 内に直接書く実装でも振る舞いは満たせる
- **配信件数 0 件を許容する設計**: 「資格指定」「ユーザー指定」で対象受講生が 0 件の場合も配信履歴に件数 0 で残す(誤操作の証跡として残す意図)
- **テスト観点**: 配信対象タイプ 3 種(全受講生 / 資格指定 / ユーザー指定)× 受講中フィルタの組み合わせ網羅 + 配信対象タイプと指定 FK 列の整合性チェック(`required_if` / `prohibited_unless`)+ 退会済 / 修了済が配信対象集合から除外される Feature テストが本チケット固有の観点

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 配信対象の種類は? | 3 種類:「全受講生」「資格指定」「ユーザー指定」 |
| コーチ向け配信はできる? | できない。配信対象は受講生のみ |
| 配信対象は受講中のみ? | はい。退会済 / 修了済 / 招待中 のユーザーには配信されず、配信件数にも含まれない |
| タイトル / 本文の文字数上限は? | タイトルは 200 文字、本文は 5000 文字 |
| 本文に HTML / Markdown は使える? | 使えない。プレーンテキストのみ、リッチテキストは MVP 外 |
| 添付ファイル(画像 / PDF)は付けられる? | 付けられない。プレーンテキスト本文のみ |
| 配信したお知らせは再配信できる? | できない。配信は不可逆、再配信動線なし |
| 配信したお知らせは編集できる? | できない。タイトル / 本文 / 対象 すべて配信後は変更不可 |
| 配信したお知らせは取り消せる? | 取り消せない。メール配信は取り消せないため、UI 上も整合 |
| 配信件数はどう計上される? | 実際に通知行が INSERT された件数。受講中でないユーザーは集合から除外され、件数に含まれない |
| 配信時刻はどう記録される? | 配信実行時の現在時刻が記録される。INSERT 直後に UPDATE で確定する |
| 配信対象タイプと指定 FK 列の組み合わせ整合性は? | 「全受講生」の場合は資格 ID / ユーザー ID 両方 NULL、「資格指定」の場合は資格 ID 必須 / ユーザー ID NULL、「ユーザー指定」の場合はユーザー ID 必須 / 資格 ID NULL。不整合は 422 |
| 公開停止資格を「資格指定」で選んだら? | バリデーション層で資格存在チェックを通る場合は配信成功(資格自体は存在する)。受講生集合解決時に「学習中で登録」フィルタで絞り込むため、archived 資格に紐づく受講生も配信対象になり得る(資格の公開状態と受講登録状態は別概念)。詳細な絞り込み要件があれば別チケット |
| 受講生でないユーザー(コーチ / 管理者)を「ユーザー指定」で選んだら? | バリデーション + 配信集合解決時の受講生ロールチェックで除外される(422 または 配信件数 0) |
| 配信時のメール送信は同期 / 非同期? | 同期送信(Basic 段階は Queue 化なし、通知基盤と同じ設計)。配信実行画面が一瞬遅延するが、件数が極端に多い場合(全受講生数百名 等)はメール送信が直列に走る点を運用上認識する |
| 受講生はお知らせをどこで見る? | 通常の通知一覧画面(`/notifications`)。お知らせ単独画面は持たず、データベース通知 + メールで届く |
| 通知行クリック時の遷移先は? | 通知一覧画面。お知らせ詳細単独画面ではなく、通知一覧で確認する |
| 配信履歴の閲覧範囲は? | 管理者のみ。`/admin/announcements` で時系列降順 + ページネーションで一覧、各行クリックで詳細閲覧 |
| 配信件数 0 件のお知らせを配信できる? | できる(「資格指定」「ユーザー指定」で対象受講生が 0 件の場合)。配信件数 = 0 で配信履歴に残り、配信時刻も記録される(誤操作の証跡として残す) |
| 配信前の対象件数プレビューは出る? | 出ない。配信実行後の配信件数で確認するのみ。事前プレビューは MVP 外 |
| 配信者管理者を退会させたら? | 配信履歴の配信者リレーションは退会済管理者も解決可能。物理削除は FK 制約で阻まれる |
| メール件名の推奨は? | 「【Certify LMS】<お知らせタイトル>」プレフィックス統一(適切な日本語であれば文言の細部は採点対象外) |
| メール本文の構成は? | 挨拶「管理者からのお知らせ」+ お知らせ本文 + 「お知らせ一覧を開く」ボタン + 結びの簡潔な構成。専用 Mailable クラスは作らず Notification の Mail 組立 API のみで完結 |
| Pusher Broadcasting で配信時にリアルタイム反映できる? | Pusher 等のリアルタイム配信は本チケット範囲外。後続の拡張余地として broadcast チャネルを使う設計はあるが、Basic 段階では DB + メールのみ |
