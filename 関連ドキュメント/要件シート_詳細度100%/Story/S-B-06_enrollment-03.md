## 背景・目的

受講生は資格ごとに学習を進められるが、「いつまでに何を達成したいか」を LMS 内に記録する手段がない。モチベーション維持のために自分で目標を立てて進捗を可視化したくても動線がなく、目標は受講生個人の頭の中に閉じてしまう。担当コーチも、面談や Q&A 等の人手介入の場面以外では受講生の現在地を把握しづらい。

本チケットは受講登録(Enrollment)配下に個人目標を立て、達成状況をワンクリックでマークできる新規機能を構築する。受講生本人のみが目標を CRUD でき、達成 / 未達成の視覚的区別を伴う一覧を受講登録詳細画面に表示する。担当コーチ / 管理者は受講生の自主的な目標を閲覧でき(介入はせず閲覧のみ)、面談や chat での声かけの参考材料になる。新規エンティティの CRUD として Eloquent + Policy + FormRequest + Blade のフル組み立てが揃う構成でもある。

## ユーザーストーリー

- **受講生(student)として**、受講中の資格ごとに目標を CRUD したい。なぜなら、自分のペースで学習計画を立てて進捗を管理したいから。
- **受講生として**、達成した目標にワンクリックで達成マークを付けたい。なぜなら、達成感を可視化してモチベーションを維持したいから。
- **受講生として**、誤って達成マークしてしまった目標を取り消したい。なぜなら、間違って押した場合の補正動線が欲しいから。
- **コーチ(coach)として**、担当資格に登録した受講生の個人目標を閲覧したい。なぜなら、面談や chat で目標達成状況を踏まえて声をかけたいから(介入はしない)。
- **管理者(admin)として**、任意受講生の個人目標を閲覧したい。なぜなら、運用上の受講生フォロー状況を把握したいから(介入はしない)。

## 要件

### 個人目標の管理(受講生本人のみ)

- 目標の追加(タイトル + 詳細(任意) + 目標期日(任意)、受講登録詳細画面のフォームから)
- 目標の編集(専用ページでタイトル / 詳細 / 目標期日 を更新、達成状態はこのフォームでは変更しない)
- 目標の削除(物理削除、履歴は残さない)
- 達成マーク(未達成 → 達成済へ切替、達成日時を記録。既に達成済でもエラーにせず冪等)
- 達成解除(達成済 → 未達成へ戻す、達成日時をクリア。既に未達成でもエラーにせず冪等)

### 目標の閲覧

- 受講生本人: 受講登録詳細画面で自分の目標一覧を表示(未達成 / 達成済 の視覚的区別あり)
- コーチ: 担当資格(`certification_coach_assignments` 経由)に登録した受講生の受講登録詳細画面で目標一覧を閲覧(操作はできず閲覧のみ)
- 管理者: 任意受講生の受講登録詳細画面で目標一覧を閲覧(同上)
- 一覧の並び順: 未達成を優先(達成済は末尾)し、その中で目標期日が近い順、同期日は作成日の新しい順。目標期日が未設定の目標は期日ありの後ろに置く

### アクセス制御(機能群共通)

- 目標の追加 / 編集 / 削除 / 達成マーク / 達成解除 は受講生本人のみ実行可。コーチ / 管理者 / 他受講生は実行不可
- 他受講生は他人の受講登録詳細画面自体にアクセスできないため、配下の目標一覧にも到達できない
- 親の受講登録が削除された場合、配下の目標も連動して削除される
- 受講停止状態(修了済 / 退会済 / 招待中)の受講生は受講登録へのアクセス時点で制限され、目標操作にも到達しない

## スコープ外

- **目標テンプレートのマスタ管理** — 受講生個別の自由入力のみ
- **資格非紐づきの総合目標**(複数資格をまたぐ生活習慣等) — 個人目標は受講登録単位(資格紐づき)のみ
- **達成時の自動通知**(コーチ / 管理者宛 push)— 受講生の自己マークで完結
- **目標達成統計 / KPI ダッシュボード** — ダッシュボード側の責務、本チケットは個別 CRUD のみ
- **目標のリマインダー**(目標期日前のメール / プッシュ通知)— MVP 外
- **コーチ / 管理者による目標へのコメント / レビュー機能** — 閲覧専用、介入動線なし
- **目標の重要度 / 優先度フラグ** — フィールド最小化、視覚区別は達成 / 未達成のみ
- **目標の論理削除 + 復元 UI** — 物理削除のみ(誤削除リスクは画面の確認ダイアログで対応)
- **目標の状態遷移履歴**(達成 → 解除 → 達成のログ)— 達成日時 1 つでの状態管理のみ
- **チェックリスト形式のサブタスク** — 1 目標 = 1 行のフラット構造のみ

## 受け入れ条件

- [ ] 【受講生】目標の追加
  1. 自分の受講登録詳細画面で目標追加フォームを送信すると、その受講登録詳細画面へリダイレクトする
  2. 未達成状態(達成日時なし)で目標が作成され、フラッシュメッセージ「目標を追加しました。」が表示される
- [ ] 【受講生】目標の編集
  1. 本人が目標編集リンクから「目標を編集」ページに遷移して目標 / 目標期日 / 詳細 を更新できる
  2. 編集では達成 / 未達成の状態は変わらない
  3. 成功時に受講登録詳細画面へリダイレクトし、フラッシュメッセージ「目標を更新しました。」が表示される
- [ ] 【受講生】目標の削除
  1. 本人が目標を削除すると受講登録詳細画面へリダイレクトし、フラッシュメッセージ「目標を削除しました。」が表示される
- [ ] 【受講生】目標の達成マーク / 達成解除
  1. 本人が達成マーク / 達成解除で目標の達成状態をトグルでき、一覧の達成済 / 未達成の表示が切り替わる
  2. いずれも成功時に受講登録詳細画面へリダイレクトし、フラッシュ表示(「目標を達成済にしました。」「目標を未達成に戻しました。」)
- [ ] 【受講生・コーチ・管理者】目標一覧の並び順と表示制御
  1. 受講生本人の受講登録詳細画面で自分の目標が未達成優先(達成済は末尾) → 目標期日が近い順 で一覧表示される
  2. 未達成 / 達成済 の視覚的区別が付く
  3. コーチ / 管理者の閲覧時は一覧は見えるが CRUD / 達成マーク操作ボタンは表示されない
- [ ] 【受講生・コーチ・管理者】目標のアクセス制御
  1. 他受講生は他人の受講登録詳細画面にアクセスできず目標一覧にも到達できない
  2. コーチ / 管理者は目標の編集専用ページにアクセスできない(操作ボタンの非表示と合わせ、本人以外に目標を操作する経路がない)
- [ ] 【受講生】目標追加 / 編集フォームのバリデーション
  1. 違反時に日本語エラーが表示される(タイトル = 必須・100 文字以内 / 詳細 = 任意・1000 文字以内 / 目標期日 = 任意・日付形式)
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### インターフェース(必須)

| HTTP | パス | 認可 | 振る舞い |
|---|---|---|---|
| POST | `/enrollments/{enrollment}/goals` | 受講生本人のみ(他は 403) | 目標追加、成功時 `/enrollments/{enrollment}` リダイレクト + フラッシュ「目標を追加しました。」、達成日時 NULL で作成 |
| GET | `/enrollment-goals/{goal}/edit` | 受講生本人のみ(他は 403) | 編集フォーム表示 |
| PATCH | `/enrollment-goals/{goal}` | 受講生本人のみ(他は 403) | 編集更新、成功時 `/enrollments/{enrollment}` リダイレクト + フラッシュ「目標を更新しました。」、達成日時は変更しない |
| DELETE | `/enrollment-goals/{goal}` | 受講生本人のみ(他は 403) | 物理削除、成功時 `/enrollments/{enrollment}` リダイレクト + フラッシュ「目標を削除しました。」 |
| POST | `/enrollment-goals/{goal}/achieve` | 受講生本人のみ(他は 403) | 達成マーク、達成日時を現在時刻でセット、成功時 `/enrollments/{enrollment}` リダイレクト + フラッシュ「目標を達成済にしました。」、既達成でも冪等 |
| DELETE | `/enrollment-goals/{goal}/achieve` | 受講生本人のみ(他は 403) | 達成解除、達成日時を NULL に戻す、成功時 `/enrollments/{enrollment}` リダイレクト + フラッシュ「目標を未達成に戻しました。」、未達成でも冪等 |

> 認可拒否は 403、リソース不在は 404。認可は「受講生本人(`role === Student` かつ目標の親受講登録が本人所有)」で統一。閲覧(`viewAny` / `view`)のみ親受講登録にアクセスできるロール(本人 / 担当コーチ / 管理者)に開く。
>
> **route 名**: `enrollments.goals.store` / `enrollment-goals.{edit,update,destroy,markAchieved,unmarkAchieved}`。

**ミドルウェア**: 全ルートに `auth` 適用、ロール Middleware は不要(Policy が受講生本人判定で完結)。目標一覧 / 個別目標の単独画面は持たない(一覧は受講登録詳細画面 `enrollments.show` 内に partial で表示)。

### データモデル

| エンティティ (Model) | 主要属性 | 関係性 | 制約 |
|---|---|---|---|
| 個人目標 (`EnrollmentGoal`) | タイトル / 詳細(任意) / 目標期日(任意) / 達成日時(NULL = 未達成、datetime = 達成済) | 受講登録(`Enrollment`)に所属(`Enrollment::goals()` の HasMany) | ULID 主キー / FK `cascadeOnDelete`(親受講登録削除で連動物理削除) / SoftDelete 不採用 = 物理削除のみ |

**インデックス用途**:

- 受講登録 × 達成状態(`enrollment_id` × `achieved_at` の複合 INDEX、配下の目標一覧 + 達成 / 未達成フィルタ用)

**Cast**: `target_date` → `date` / `achieved_at` → `datetime`。Model に `isAchieved(): bool`(達成日時の有無で判定)ヘルパーを生やすと Blade で `@if ($goal->isAchieved())` と簡潔に書ける。

**初期データ**:

- 受講登録の初期データ生成と同時に個人目標を投入する
- 固定受講生の受講登録に達成済・未達成の目標を混在させる（視覚区別・達成マーク / 解除・編集 / 削除を確認）
- デモ受講生にも目標を散らす（コーチ / 管理者 / 他受講生からの閲覧の認可分岐を確認）

### コンポーネント

**Controller** (`app/Http/Controllers/`)
- `EnrollmentGoalController` — 受講生本人の目標操作(`store` / `edit` / `update` / `destroy` / `markAchieved` / `unmarkAchieved`)。コーチ / 管理者は閲覧専用のため CRUD エンドポイントを持たない

**FormRequest** (`app/Http/Requests/EnrollmentGoal/`)
- `StoreRequest` / `UpdateRequest` — 追加 / 編集の入力検証 + `authorize()` で `EnrollmentGoalPolicy` に委譲

**Action** (`app/UseCases/EnrollmentGoal/`、※ Advance 範囲、Basic 受講生は Controller 内完結も可)
- `StoreAction` / `UpdateAction` / `DestroyAction` / `MarkAchievedAction` / `UnmarkAchievedAction` — 各業務操作(達成マーク / 解除は達成日時を無条件 UPDATE する冪等処理)

**Policy** (`app/Policies/`)
- `EnrollmentGoalPolicy` — `viewAny` / `view` は親 `EnrollmentPolicy::view` に委譲、`create` / `update` / `delete` / `markAchieved` / `unmarkAchieved` は受講生本人判定

**Model** (`app/Models/`)
- `EnrollmentGoal`(`isAchieved()` + `Enrollment` への BelongsTo) / 親 `Enrollment` に `goals()` HasMany

**View**(既存、ロック対象)
- `resources/views/enrollment-goal/edit.blade.php` + 入力共通 partial `enrollment-goal/_form.blade.php`。目標一覧 + 追加フォームは受講登録詳細画面(`enrollments.show`)内に表示

**Migration / Seeder**
- `database/migrations/*_create_enrollment_goals_table.php`
- 初期データは `database/seeders/EnrollmentSeeder.php` 内で生成(独立 Seeder なし)

**Routes** (`routes/web.php`)
- 認証済グループ内に目標ルート 6 本(ネスト POST `enrollments/{enrollment}/goals` + フラット `enrollment-goals.*`)

### 異常系

**入力検証**(FormRequest クラス名 + ルール記法):

- **目標追加 / 編集** (`EnrollmentGoal\StoreRequest` / `UpdateRequest`、同一ルール):
  - `title`: `required` / `string` / `max:100`
  - `description`: `nullable` / `string` / `max:1000`
  - `target_date`: `nullable` / `date`

> 達成マーク / 達成解除は入力項目なし(パスパラメータ + 認証ユーザーのみで完結)のため FormRequest 不要。推奨エラーメッセージの日本語文言例は想定 Q&A に集約。

**業務例外**: なし。達成マーク / 解除は冪等な無条件 UPDATE のため状態ベースの例外を持たない。認可違反は `EnrollmentGoalPolicy` 拒否で 403。

### 設計判断

- **編集は専用ページ遷移(JS なし)**: 一覧 + 追加フォームは受講登録詳細画面に partial で埋め込み、編集だけ専用ページに遷移する純 Laravel パターン。インライン編集 / モーダル展開は採らない
- **達成マーク = POST `/achieve` / 達成解除 = DELETE `/achieve` の REST 風対称設計**: 状態切替を「達成扱いリソースを作る / 消す」とメタファー的に表現し、同一 URL を HTTP メソッドで切り替える
- **べき等性の無条件 UPDATE**: 達成マークは「既達成なら何もしない」分岐を持たず達成日時を無条件セット、達成解除も無条件 NULL クリア。重複呼び出しでも副作用が冪等
- **Policy の責務委譲**: `view` / `viewAny` は親 `EnrollmentPolicy::view` に委譲し目標固有の閲覧スコープを持たない(受講登録にアクセスできる権限なら目標も見える)。CRUD / 達成系は受講生本人判定を本 Policy 内で完結。Blade では `@can('create', [EnrollmentGoal::class, $enrollment])` / `@can('update', $goal)` 等で操作ボタンを出し分け
- **物理削除**: SoftDelete 不採用(進捗・履歴系テーブルは復旧概念がない)。誤削除は画面側の確認ダイアログ(HTML 標準 confirm)でカバー
- **受講登録の状態は目標 CRUD の可否に影響しない**: `passed` / `failed` 等でも目標 CRUD は許容(目標は進捗管理ツール、状態判定は本機能の責務外)
- **テスト観点**: Policy 各メソッド × 各ロール(受講生本人 / 他受講生 / 担当コーチ / 担当外コーチ / 管理者)の真偽網羅 + 達成マーク / 解除の冪等性 + 編集時の達成状態不変 が本チケット固有の観点

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 目標タイトル / 詳細の文字数上限は? | タイトルは 100 文字、詳細は 1000 文字 |
| 目標期日は必須? | 任意。期日を設定しない目標(継続的習慣等)も許容する |
| 過去日を目標期日に設定できる? | 可能(未来日制約は付けない)。受講生が過去の目標を後から記録したいケースを許容する |
| 1 受講登録あたりの目標数上限は? | 上限なし。受講生が自由に立てる |
| 受講生以外(コーチ / 管理者)が目標を追加できる? | できない。介入動線なし、コーチ / 管理者は閲覧専用 |
| 達成マークしたらコーチに通知が飛ぶ? | 飛ばない。MVP では受講生の自己マークで完結、通知連携なし |
| 達成済の目標を再編集できる? | 可能。タイトル / 詳細 / 目標期日 は達成後も更新可。達成日時は専用操作(達成マーク / 解除)でのみ変更 |
| 達成解除すると達成日時はどうなる? | NULL に戻る。「いつ達成したか」の履歴は保持しない |
| 目標を削除すると履歴は残る? | 残らない。物理削除のみ。誤削除リスクは画面の確認ダイアログで対応 |
| 親の受講登録が削除されたら目標はどうなる? | 連動して物理削除される(FK の連動削除) |
| 受講生が修了済 / 退会済の場合は目標 CRUD できる? | 受講登録へのアクセス時点で制限される。本機能側での追加判定は不要 |
| 達成マーク URL を直接叩いて他受講生の目標を達成にできる? | できない。受講生本人判定があり、他受講生 / 担当外コーチ / 管理者 は 403 |
| コーチ / 管理者の目標一覧表示でも操作ボタンが見える? | 見えない。認可がない場合はそもそもボタンを描画せず、URL を直叩きしても 403 |
| 編集フォームに「達成 / 未達成」チェックボックスを付けてよい? | 付けない。達成マーク / 解除は専用操作でのみ行う(編集 = 内容更新 / 達成マーク = ステータス操作 の意味的区別) |
| フラッシュ文言の推奨は? | 追加「目標を追加しました。」/ 編集「目標を更新しました。」/ 削除「目標を削除しました。」/ 達成「目標を達成済にしました。」/ 達成解除「目標を未達成に戻しました。」(適切な日本語であれば細部は採点対象外) |
| 一覧の並び順は? | 未達成を優先(達成済は末尾)し、その中で目標期日が近い順、同期日は作成日の新しい順に並ぶ。目標期日が未設定の目標は期日ありの後ろに置く |
