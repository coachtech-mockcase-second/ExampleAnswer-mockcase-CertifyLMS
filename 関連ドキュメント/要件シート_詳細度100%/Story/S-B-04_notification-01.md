## 背景・目的

提供 PJ には chat / Q&A / 面談予約 / 面談キャンセル の業務操作が完成しているが、当事者(やり取りの相手方 / 担当コーチ)への通知配信が無く、画面を開かないと新着 / 状態変更に気付けない。受講生は学習中の取りこぼし、コーチは複数受講生フォロー漏れのリスクを抱える。

本チケットは業務イベント発生時にデータベース通知 + メール送信で即時配信する基盤を新規構築する。通知一覧画面 + 既読化動線(行クリック既読化 + 全件既読化)を備え、chat メッセージ受信 / Q&A 回答受信 / 面談予約 / 面談キャンセル の 4 業務イベントを発火点として通知を配信する。本チケットが揃わないと、受講生 / コーチが受動的に画面更新を待つ運用になり LMS としての運用体裁が整わない。

## ユーザーストーリー

- **受講生(student)として**、自分宛の chat 新着メッセージ / Q&A 回答 / 面談キャンセル通知を一覧で確認したい。なぜなら、複数 Feature を横断して未対応事項を取りこぼしたくないから。
- **コーチ(coach)として**、自分宛の chat 新着メッセージ / 担当受講生の面談予約 / 面談キャンセル通知を一覧で確認したい。なぜなら、複数受講生を効率的にフォローしたいから。
- **受講生 / コーチとして**、通知行をクリックしたら既読化と該当画面遷移が同時に走ってほしい。なぜなら、既読化忘れで未読件数が永遠に減らない事態を避けたいから。
- **受講生 / コーチとして**、一括「全件既読にする」ボタンが欲しい。なぜなら、過去の未読が大量に残っているとき個別クリックでは追いつかないから。
- **管理者(admin)として**、自分宛の通知は受け取らない設計を期待する。なぜなら、運用情報は管理画面で集約確認するので通知ベルが鳴り続けると逆にノイズになるから。

## 要件

### 通知一覧 / 既読化(受講生 / コーチ共通、サイドバー「通知」項目から遷移)

- サイドバー「通知」項目クリックで通知一覧画面(`/notifications`)に遷移
- 通知一覧の表示(時系列降順、「全件」「未読のみ」タブ + 20 件 / ページ ページネーション)
- 行クリックによる既読化 + 通知種別に応じた業務画面へのリダイレクト(chat 通知 → chat ルーム / Q&A 通知 → Q&A スレッド / 面談予約・キャンセル通知 → 面談一覧)
- 「全件既読にする」ボタンによる自分宛の全未読通知の一括既読化

> TopBar 通知ベル + 未読件数バッジは本チケットでは扱わない(`S-A-05` で実装)。提供 PJ のベル Blade は既存だが、Basic 段階ではクリック時の `/notifications` 遷移のみで未読バッジ表示なし。

### 通知発火(4 種類の業務イベント)

> **配信チャネル方針**: 全 4 種類とも **データベース通知 + メール 両方** で配信(ロール組み合わせを問わず統一、チャネル抑制なし)

- chat 新着メッセージ通知(送信者を除く全 chat ルームメンバーに配信)
- Q&A 回答受信通知(スレッド投稿者に配信、回答者 = 投稿者の自己回答時はスキップ)
- 面談予約通知(担当コーチに配信、受講生本人には発火しない)
- 面談キャンセル通知(キャンセル実行者の相手方に配信、自己通知はしない)

### 共通の振る舞い

- 受信者の利用状態が「受講中」でない(退会済 / 修了済 / 招待中)場合は配信スキップ
- 管理者は本 4 種いずれの通知も受信対象から除外
- メール件名は「【Certify LMS】」プレフィックスで統一

## スコープ外

- **JSON API**(認証なし `/api/v1/notifications`) — `S-B-05` で扱う
- **Sanctum Cookie 認証 + JS フロント通知ポップオーバー + リアルタイム push** — `S-A-05` で扱う
- **管理者からのお知らせ配信機能**(配信フォーム / 一斉配信 / 配信履歴) — `S-B-09` で扱う
- **面談リマインダー通知**(前日 18:00 / 1 時間前のリマインダー)— `S-B-10` で扱う(Schedule Command + 同期発火)
- **管理者宛通知** — 設計上 admin は通知の発火対象から除外
- **通知種別 × チャネルごとの ON/OFF 設定 UI** — MVP では全通知が DB + メール固定送信
- **通知の論理削除 / 物理削除 UI** — 既読化のみ提供、削除動線は持たない
- **モバイルプッシュ通知** — 教育 PJ スコープ外
- **メール配信の Queue 非同期化** — 本チケットでは同期送信、Queue Worker 運用は別チケット範囲

## 受け入れ条件

- [ ] 認証済受講生 / コーチがサイドバー「通知」項目をクリックして `/notifications` にアクセスすると自分宛の通知が時系列降順で表示され、「全件」「未読のみ」タブで絞り込みできる(タブ状態は 20 件 / ページのページネーション遷移後も保持される)
- [ ] 通知行クリックで未読通知が既読化され、通知種別に応じた業務画面(chat ルーム / Q&A スレッド / 面談一覧)へリダイレクトされる
- [ ] 「全件既読にする」ボタン押下で自分宛の全未読通知が一括既読化され、通知一覧に戻りフラッシュ表示 + 未読件数が 0 になる
- [ ] chat ルームで新規メッセージが投稿された際に送信者を除くルームメンバーに DB + メール通知が発火する
- [ ] Q&A スレッドに新規回答が投稿された際にスレッド投稿者に DB + メール通知が発火する(回答者 = スレッド投稿者の自己回答時は発火しない)
- [ ] 受講生が面談を予約した際に担当コーチに DB + メール通知が発火し、受講生本人には発火しない
- [ ] 当事者(受講生 or 担当コーチ)が予約済面談をキャンセルした際に相手方に DB + メール通知が発火し、自己通知は発火しない
- [ ] 受信者の利用状態が「受講中」でない場合は配信スキップされ、管理者は本 4 種いずれの通知も受信しない(`notifications` テーブルに INSERT されない)
- [ ] 他人宛通知の既読化 URL を直叩きすると 403、未認証ユーザーが通知一覧画面にアクセスするとログイン画面にリダイレクトされる
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

> **本セクションは「参考」、受講生ごとに異なる実装を許容**(AC を満たせば実装手段は問わない)。ただし **「(必須)」マーカー付きサブセクション**(インターフェース / データモデル > 初期データ Seeder)は AC・採点・動作確認のベース、ここに記載した内容を正確に実装する。

### インターフェース(必須)

| HTTP | パス | 認可 | 振る舞い |
|---|---|---|---|
| GET | `/notifications` | 認証ユーザー(自分宛のみ自動絞り込み、管理者は 0 件) | 通知一覧画面(時系列降順、「全件」「未読のみ」タブ + 20 件 / ページ、未読件数を画面上部に表示) |
| POST | `/notifications/{notification}/read` | 認証ユーザー(本人通知のみ、他人宛は 403) | 単一通知の既読化、成功時に通知種別に応じた業務画面へリダイレクト |
| POST | `/notifications/read-all` | 認証ユーザー(自分宛 × 未読のみ対象) | 全未読通知の一括既読化、`/notifications` リダイレクト + フラッシュ「すべての通知を既読にしました。」 |

**ミドルウェア**: 全ルートに `auth` 適用、ロール制約なし(管理者は通知の発火対象外で常に 0 件表示)。入口はサイドバー「通知」項目のみ(TopBar 通知ベル + 未読件数バッジは `S-A-05` で扱う)。

### データモデル

| エンティティ (Model) | 主要属性 | 関係性 | 制約 |
|---|---|---|---|
| 通知(Laravel 標準 `Notification` モデル / `notifications` テーブル) | 受信者(`notifiable_id` / `notifiable_type`) / 通知種別(`type` = 発火クラス FQCN) / 通知データ(`data` JSON、タイトル / プレビュー / 遷移先 / 種別固有メタ) / 既読化日時(`read_at`、未読時 NULL) | User 1-N 通知(`Notifiable` trait 経由の polymorphic morph) | ULID 主キー(`ulidMorphs('notifiable')`) / SoftDelete 不採用(既読化のみ、削除動線なし) |

**インデックス用途**:

- 受信者 × 未読絞り込み(`notifiable_id` × `notifiable_type` × `read_at` の複合 INDEX)
- 作成日時降順(一覧並び替え用)

**通知データ JSON の構造**: 共通キー(通知種別 / タイトル / プレビュー / 遷移先ルート + パラメータ)+ 種別固有メタ(送信者名 / スレッドタイトル / 受講生名 / 面談日時 等)。通知発火時にスナップショット化し、一覧描画時の関連 Model 再 fetch を不要にする。

**初期データ Seeder(必須)** (`NotificationSeeder`):

- 固定 student × 全種別(chat 受信 / Q&A 回答受信 / 面談キャンセル を最低 1 件ずつ、既読 / 未読を半々)
- 固定 coach × 全種別(chat 受信 / 面談予約 / 面談キャンセル を最低 1 件ずつ、既読 / 未読を半々)
- デモ受講生 × 数件(chat / Q&A 回答 の混在、既読 / 未読を交互)
- デモコーチ × 数件(面談予約 / chat の混在、既読 / 未読を交互)
- 作成日時の散らし: 過去 1〜数日に散らしてページネーション(20 件 / ページ)動作を確認可能に
- 副作用抑制: `Notification::send()` ではなく直接 INSERT し、Mail / Queue 副作用を発生させない
- DatabaseSeeder 順序: `UserSeeder` / `EnrollmentSeeder` / `MentoringSeeder` / `ChatSeeder` / `QaBoardSeeder` の **後**(既存ドメインデータを参照して通知行を生成)

### 異常系

**入力検証**(FormRequest クラス名 + ルール記法):

- 通知一覧 FormRequest (`Notification\IndexRequest`):
  - `tab`: `nullable` / `string` / `in:全件,未読のみ`(範囲外は既定値にフォールバック、エラー表示なし)
  - `page`: `nullable` / `integer` / `min:1`
- 単一既読化 / 全件既読化: パスパラメータ / 認証ユーザーのみで完結、入力検証は最小限

**業務例外**:

- 他人宛通知の既読化 URL を直叩き → 403(`NotificationPolicy::update` 拒否)
- 未認証ユーザーが通知一覧にアクセス → ログイン画面リダイレクト
- 受信者の利用状態が「受講中」でない(退会済 / 修了済 / 招待中)→ 配信スキップ(`notifications` テーブルに INSERT されない)
- 管理者: 本 4 種いずれも受信対象外 → 配信スキップ

### 設計判断

- **発火フックは業務トランザクション確定後に配置**: 模範解答 PJ は発火を業務 Action 内 `DB::afterCommit` で呼ぶ構成。業務トランザクション内で発火しないことで「業務 UPDATE ロールバック + 通知だけ残る」事故を回避。Basic 受講生は Controller 内同期発火に簡略化しても振る舞いは満たせる
- **chat 通知の Mail 抑制**: `ChatMessageReceivedNotification` の `via()` で `mailEnabled=false` 時に `mail` を配列から除外する責務分担(発火側が判断、Notification は反映のみ)
- **通知データ JSON のスナップショット化**: 発火時に送信者名 / スレッドタイトル等を `data` JSON に焼き込み、一覧描画時の eager load を不要にする。発火後のドメイン変更(送信者改名等)は通知行に反映されないが、通知ログとしては受信時点情報が正という考えと一致
- **受信者集合解決の発火側責務**: 受信者の利用状態フィルタ(受講中以外スキップ)/ 管理者除外 / 自己除外 / Q&A 自己回答スキップ / 面談予約はコーチのみ / 面談キャンセルは相手方 等の判定を各発火ラッパー Action が個別に所有する設計。Notification クラスは「受け取ったら配信する」責務だけに専念
- **テスト観点**: 各発火条件(chat 送信者除外 / Q&A 自己回答スキップ / 面談予約はコーチのみ / 面談キャンセルは相手方)の Feature テスト + 受信者の利用状態フィルタ(InProgress 以外スキップ)+ 管理者除外の網羅が本チケット固有の観点

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 通知配信のチャネルは? | データベース + メールの 2 チャネル固定。受信者種別 / 通知種別ごとの ON/OFF 設定 UI は持たない |
| Q&A の自己回答(投稿者本人が回答)で通知は飛ぶ? | 飛ばない。回答者 = スレッド投稿者の場合は配信をスキップする |
| 面談予約通知は受講生にも飛ぶ? | 飛ばない。受講生は予約 UI で即時確認できるためコーチ宛のみ発火 |
| 面談キャンセル通知は誰宛? | キャンセル実行者の **相手方**。受講生キャンセル → コーチ宛 / コーチキャンセル → 受講生宛(自己通知はしない) |
| 受信者が退会済 / 修了済の場合は? | 配信スキップ。利用状態が「受講中」のユーザーのみに配信する |
| 管理者宛通知は? | 一切発火しない。本 4 種のいずれも管理者は受信対象から外す |
| TopBar 通知ベル + 未読件数バッジは本チケット範囲? | 範囲外。`S-A-05` で扱う(バッジの View Composer 含む全実装が Advance 側)。本チケットでは入口はサイドバー「通知」項目のみ |
| 行クリック時の既読化と遷移はどちら先? | 既読化日時を先に確定してから業務画面へリダイレクト。遷移先ルートが未登録なら通知一覧に戻る |
| 全件既読化の対象範囲は? | 自分宛 × 未読 のみ。他人宛は対象外 |
| 通知の削除動線は? | 持たない。既読化のみ。削除は将来要件 |
| 発火タイミングを業務トランザクション内で書く? | 推奨は業務トランザクション確定後(`DB::afterCommit` 等)。同期送信のためメール失敗で業務 UPDATE がロールバックされる事故を避ける |
| 発火点を Controller 内に書いて Action は使わない実装でも OK? | OK。Basic 受講生は Controller method 内で受信者集合を解決して直接通知を発火する実装で振る舞いが満たせれば OK |
| メール件名の推奨は? | 「【Certify LMS】<通知タイトル>」プレフィックス統一。例: 「【Certify LMS】◯◯ さんから新着メッセージ」「【Certify LMS】◯◯ さんから面談予約が入りました」(適切な日本語であれば文言の細部は採点対象外) |
| メール本文の構成は? | 挨拶 + 通知の要約 + 「業務画面を開く」ボタンを含む簡潔な構成。専用 Mailable クラスは作らず Notification の Mail 組立 API のみで完結 |
| フラッシュ文言の推奨は? | 「すべての通知を既読にしました。」等、操作結果が分かる日本語(細部は採点対象外) |
| 同一通知の重複防止(同じ chat メッセージで 2 回送る等)は? | 本チケット範囲外。冪等性 MVP 外 |
| 認可拒否時の HTTP ステータスは 403 / 404? | 未認証はログイン画面リダイレクト / 認証済だが他人宛通知の既読化を試みると 403 |
| 受講生 / コーチ / 管理者は他者の通知を見れる? | 見れない。一覧は自分宛のみ自動絞り込み、単一既読化も本人通知のみ許可で他人宛は 403 |
| 通知データの中に業務 Model の名前を埋めるのは? | 通知発火時点でスナップショット化(送信者名 / スレッドタイトル 等)。発火後にユーザー名等が変わっても通知行は変わらない(通知ログとしては受信時点情報が正という考え) |
