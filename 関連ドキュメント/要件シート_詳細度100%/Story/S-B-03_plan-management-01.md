## 背景・目的

プランのデータ構造(`plans` テーブル)+ 初期データ + 管理画面の Blade + 招待時 / プラン延長時にプランを参照する受講生側ロジックは既に含まれているが、管理画面側の認可・ルーティング・バリデーションのロジックが欠落しており画面が動作しない(404 になる)。運営側はプランの新設・受講期間 / 面談回数の改定・終売を直接操作できず、受講生招待時のプラン指定の前提も立ち上がらない。

本チケットでは、管理者(admin)がプランを CRUD でき、下書き → 公開中 → アーカイブ の状態遷移で招待時に選べるプランを運営側でコントロールできる管理画面を完成させる。公開中・アーカイブ済のプラン、および受講者が紐づくプランは削除を禁止し、招待・受講中ユーザーとの参照整合性を守る。プラン受講モデルの前提となるマスタ管理画面であり、本チケットが揃わないと招待発行 / プラン延長 / 受講生ダッシュボードのプラン情報パネル等の後続フローの検証ができない。

## ユーザーストーリー

- **管理者(admin)として**、受講プランを CRUD したい。なぜなら、運営側で受講期間 / 初期付与面談回数 / 公開状態を自由にコントロールしたいから。
- **管理者として**、プランを下書きから公開へ、または招待画面の選択肢から外すためにアーカイブへ遷移させたい。なぜなら、プランのライフサイクル(新設 → 提供 → 終売)を自前でハンドリングしたいから。
- **管理者として**、受講者が紐づいているプランや、公開中 / アーカイブ済のプランが誤って削除されない仕組みを期待する。なぜなら、受講中ユーザーの参照やプラン履歴が孤立して監査ができなくなるから。
- **受講生(student) / コーチ(coach)として**、管理者専用のプラン管理画面に直接アクセスしても拒否されることを期待する。なぜなら、運営側のマスタ管理画面に他ロールが入れる構成は事故の元だから。

## 要件

### プラン(管理者のみ)

- 一覧: プラン名キーワード検索 + 状態フィルタ(下書き / 公開中 / アーカイブ)+ ページネーション、各行に契約中の受講者数を表示
- 詳細: 基本情報 + 紐づく受講者一覧 + メタ情報(作成者 / 最終更新者 / 作成日時)を表示
- 新規作成: 初期状態は **下書き** で作成、作成後は詳細画面へ遷移
- 編集: 基本情報(プラン名 / 説明 / 受講期間 / 初期付与面談回数 / 並び順)を更新。状態は本フォームでは変更不可(状態遷移は別操作)
- 削除: 下書き状態 **かつ** 受講者が紐づいていないプランのみ物理削除可。公開中 / アーカイブ済、または受講者紐づきありのプランは削除不可(参照整合性を守る)

### 状態遷移(管理者のみ)

- 公開する(下書き → 公開中)
- アーカイブ(公開中 → アーカイブ)。アーカイブ後は招待画面 / プラン延長画面の選択肢から外れるが、受講中ユーザーの参照は維持される
- 下書きへ戻す(アーカイブ → 下書き)。誤アーカイブの取り戻し / 再提供準備の動線
- 公開中 → 下書きへの直接戻しはできない(一度アーカイブを経由する)
- 各遷移は遷移元の状態が一致しない場合は実行できない

### 入力検証(新規作成 / 編集)

- プラン名: 必須 / 100 文字以内
- 説明: 任意 / 2000 文字以内
- 受講期間(日): 必須 / 1〜3650 の整数
- 初期付与面談回数: 必須 / 0〜1000 の整数
- 並び順: 任意 / 0 以上の整数

### アクセス制御(機能群共通)

- プラン管理画面の全操作は管理者のみ。受講生 / コーチはアクセス不可

### 表示順

- 一覧は公開中のプランを優先表示

## スコープ外

- **プラン価格情報の保持** — 決済は LMS 外で完結する方針のため、価格カラム自体を持たない(確定済)
- **受講生招待時のプラン指定 / プラン情報の初期化**(受講プラン割当・受講期間・初期面談回数の確定)— 招待・オンボーディングフローの責務(本チケットでは扱わない)
- **プラン延長**(受講期間 + 面談回数の加算)— 同梱、管理者のユーザー詳細画面の延長動線から呼び出す(本チケットでは扱わない)
- **期限満了による自動卒業** — 同梱、本チケットでは扱わない
- **プラン履歴の表示**(プラン割当 / 延長 / 期限満了の履歴) — 既存の履歴記録ロジックを利用するのは別画面の責務(本チケットの詳細画面では受講者一覧のみ表示)
- **期限切れ判定 / 残日数算出** — 既存の判定ロジックを別 Feature(受講生ダッシュボード等)が利用する
- **受講生 / コーチ向けのプラン情報表示** — ダッシュボード / プロフィール側の責務
- **並び順をドラッグ&ドロップで操作する UI** — 並び順は数値入力で十分

## 受け入れ条件

- [ ] 管理者がプラン管理の一覧を開くと、プラン名キーワード(部分一致)と状態(下書き / 公開中 / アーカイブ)で絞り込みでき、各行に契約中の受講者数が表示され、公開中を優先する並び順で表示される
- [ ] 管理者が新規作成すると **下書き** 状態でプランが作成され、作成したプランの詳細画面にリダイレクト + フラッシュ表示される
- [ ] 管理者が編集すると基本情報が更新され、状態(下書き / 公開中 / アーカイブ)は変更されずに詳細画面へリダイレクト + フラッシュ表示される
- [ ] 管理者が下書き状態かつ受講者が 0 名のプランを削除すると物理削除され、一覧画面にリダイレクト + フラッシュ表示される
- [ ] 公開中 / アーカイブ状態のプラン、または下書きでも受講者が 1 名以上紐づくプランを削除しようとすると 409 が返り、元画面に戻ってフラッシュエラーが表示される(削除されない)
- [ ] 公開(下書き → 公開中)/ アーカイブ(公開中 → アーカイブ)/ 下書きへ戻す(アーカイブ → 下書き)を正しい遷移元状態から実行すると状態が遷移し詳細画面へリダイレクト + フラッシュ表示され、遷移元状態が一致しない操作は 409 + フラッシュエラーになる
- [ ] 受講生 / コーチがプラン管理の任意のルート(一覧 / 詳細 / 作成 / 編集 / 削除 / 状態遷移)にアクセスすると 403 が返る
- [ ] 新規作成 / 編集フォームにてバリデーションが行われ、ルール違反時に日本語のエラーメッセージが表示されるか:
  - プラン名: 必須 / 100 文字以内
  - 説明: 任意 / 2000 文字以内
  - 受講期間(日): 必須 / 1〜3650 の整数
  - 初期付与面談回数: 必須 / 0〜1000 の整数
  - 並び順: 任意 / 0 以上の整数
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### インターフェース(必須)

| HTTP | パス | 認可 | 振る舞い |
|---|---|---|---|
| GET | `/admin/plans` | 管理者のみ | 一覧(キーワード + 状態フィルタ + ページネーション、フィルタ引き継ぎ、各行に受講者数) |
| GET | `/admin/plans/create` | 管理者のみ | 新規作成フォーム表示 |
| POST | `/admin/plans` | 管理者のみ | 新規作成(`status=draft` 固定 INSERT)、`/admin/plans/{plan}` リダイレクト + フラッシュ「プランを作成しました。」 |
| GET | `/admin/plans/{plan}` | 管理者のみ | 詳細(基本情報 + 紐づく受講者一覧 + メタ情報 + 状態遷移ボタン) |
| GET | `/admin/plans/{plan}/edit` | 管理者のみ | 編集フォーム表示 |
| PUT | `/admin/plans/{plan}` | 管理者のみ | 更新(`status` 不変)、`/admin/plans/{plan}` リダイレクト + フラッシュ「プランを更新しました。」 |
| DELETE | `/admin/plans/{plan}` | 管理者のみ | 削除(下書き × 受講者なしのみ物理削除)、成功時 `/admin/plans` リダイレクト + フラッシュ「プランを削除しました。」。状態違反 / 受講者紐づきありは 409 + 元画面 + フラッシュエラー |
| POST | `/admin/plans/{plan}/publish` | 管理者のみ | 公開(下書き → 公開中)、詳細リダイレクト + フラッシュ「プランを公開しました。」。下書き以外は 409 |
| POST | `/admin/plans/{plan}/archive` | 管理者のみ | アーカイブ(公開中 → アーカイブ)、詳細リダイレクト + フラッシュ「プランをアーカイブしました。」。公開中以外は 409 |
| POST | `/admin/plans/{plan}/unarchive` | 管理者のみ | 下書きへ戻す(アーカイブ → 下書き)、詳細リダイレクト + フラッシュ「プランを下書きへ戻しました。」。アーカイブ以外は 409 |

> 認可拒否は 403(受講生 / コーチが直接画面を開いても 403、404 にはしない)。

**ミドルウェア**: 全ルートに `auth` + `role:admin` + `admin` プレフィックスを適用(管理者専用エリア)。

### データモデル

> **既存テーブル**(同梱、変更不要)。受講生は新規マイグレーション / Model を作成せず、下記の構造を理解した上で Controller / FormRequest / 認可を組む。

| エンティティ (Model) | 主要属性 | 関係性 | 制約 |
|---|---|---|---|
| 受講プラン (`Plan`) | プラン名 / 説明 / 受講期間(日) / 初期付与面談回数 / 状態 / 並び順 / 作成者 / 最終更新者 | 受講者(`User.plan_id`)を hasMany / 作成者・最終更新者(`User`)に belongsTo / プラン履歴(`UserPlanLog`)を hasMany | ULID 主キー / 作成者・最終更新者 FK は `restrictOnDelete` / **SoftDelete 不採用 = 物理削除のみ**(Draft / Published / Archived を status 列で表現するマスタ系のため) |

**Enum**:

- プラン状態 (`PlanStatus`): 下書き (`Draft`) / 公開中 (`Published`) / アーカイブ (`Archived`)。`label()` が「下書き」「公開中」「アーカイブ」を返す

**インデックス用途**:

- 状態 × 並び順(`(status, sort_order)` 複合、一覧のフィルタ + 並び替え用、既設)

**初期データ Seeder(必須)** (`PlanSeeder`、同梱):

- 公開中 × 3(受講者紐づきあり: 1 ヶ月 / 3 ヶ月 / 6 ヶ月プラン)+ 下書き × 1(受講者なし)+ アーカイブ × 1(受講者なし)で状態を網羅
- 動作確認の対応: 状態フィルタ(3 状態)/ 状態遷移ボタンの活性条件 / 削除ガード(下書き × 受講者なし = 削除可、公開中・アーカイブ = 409)を実機確認できる
- DatabaseSeeder 順序: `UserSeeder` の後(admin + 受講生を参照してプランを作成・紐づけ)

**Controller** (`app/Http/Controllers/`)
- `PlanController` — 一覧 / 詳細 / 新規 / 編集 / 削除(`index` / `show` / `create` / `store` / `edit` / `update` / `destroy`)
- `PlanStatusController` — 状態遷移(`publish` / `archive` / `unarchive`)

**FormRequest** (`app/Http/Requests/Plan/`)
- `IndexRequest` — キーワード / 状態フィルタの検証 + `viewAny` 認可
- `StoreRequest` / `UpdateRequest` — 基本情報の入力検証 + `create` / `update` 認可

**Action** (`app/UseCases/Plan/`、※ Advance 範囲、Basic 受講生は Controller 内完結も可)
- `IndexAction` — フィルタ適用 + 受講者数(`withCount('users')`)+ 状態優先 + 並び順で paginate
- `ShowAction` — 作成者 / 最終更新者 / 受講者を Eager Load

- `UpdateAction` — 基本情報更新(`status` は変更しない、最終更新者を更新)
- `DestroyAction` — 下書き かつ 受講者なしのみ物理削除(2 段ガード)
- `PublishAction` / `ArchiveAction` / `UnarchiveAction` — 遷移元検証 + `status` 更新 + 最終更新者更新

**Policy** (`app/Policies/`)
- `PlanPolicy` — 全操作(`viewAny` / `view` / `create` / `update` / `delete` / `publish` / `archive` / `unarchive`)を管理者のみ true

**Model + Enum** (`app/Models/`, `app/Enums/`)
- `Plan`(`scopePublished` / `scopeOrdered` を保持)/ `PlanStatus`

**View**(既存、ロック対象)
- `resources/views/plan/management/{index,show,create,edit}.blade.php`

**Migration / Seeder**(いずれも既存)
- `database/migrations/*_create_plans_table.php` / `*_add_plan_columns_to_users_table.php` / `*_create_user_plan_logs_table.php`
- `database/seeders/PlanSeeder.php`

**例外** (`app/Exceptions/Plan/`)
- `PlanNotDeletableException`(削除不可、409 / `forStatusViolation` + `forUsersAttached`)/ `PlanInvalidTransitionException`(状態遷移違反、409 / `forPublish` + `forArchive` + `forUnarchive`)

**Routes** (`routes/web.php`)
- `admin` プレフィックス + `auth` + `role:admin` 配下に `Route::resource('plans', PlanController::class)->parameters(['plans' => 'plan'])->names('admin.plans')`(CRUD 7 ルート)+ 状態遷移 3 ルート(`admin.plans.publish` / `.archive` / `.unarchive` → `PlanStatusController`)

### 異常系

**入力検証**(FormRequest クラス名 + ルール記法):

- 新規作成 / 編集 (`Plan\StoreRequest` / `Plan\UpdateRequest`):
  - `name`: `required` / `string` / `max:100`
  - `description`: `nullable` / `string` / `max:2000`
  - `duration_days`: `required` / `integer` / `min:1` / `max:3650`
  - `default_meeting_quota`: `required` / `integer` / `min:0` / `max:1000`
  - `sort_order`: `nullable` / `integer` / `min:0`
- 一覧 (`Plan\IndexRequest`):
  - `keyword`: `nullable` / `string` / `max:100`
  - `status`: `nullable` / `Rule::enum(PlanStatus::class)`

**業務例外**(状態ベースガード + HTTP ステータス):

- 公開中 / アーカイブ状態のプラン削除 (`PlanNotDeletableException::forStatusViolation`) → 409
- 受講者が紐づくプランの削除 (`PlanNotDeletableException::forUsersAttached`) → 409
- 遷移元状態と一致しない状態遷移 (`PlanInvalidTransitionException::forPublish` / `forArchive` / `forUnarchive`) → 409
- 409 は HTML 経由で元画面に戻り `<x-flash>` に日本語エラーを表示(`Handler` が redirect + flash 変換、JSON 経由は 409 + body)

### 設計判断

- **Controller 2 分割**: CRUD を `PlanController`、状態遷移を `PlanStatusController` に分けて責務を明確化(状態遷移は CRUD と独立した業務操作)。1 Controller に集約しても振る舞いは満たせる
- **認可と整合性チェックの分離**: `PlanPolicy` は「管理者か」の真偽のみ判定し、「下書き × 受講者なしのみ削除可」「遷移元状態の一致」のような状態・参照ベースのガードは Action / Controller 内で具象例外(409)として実装する。Policy に状態判定を混ぜると認可責務が肥大化するため
- **例外メッセージは例外クラスが所有**: 削除不可 2 種(状態違反 / 受講者紐づき)・遷移違反 3 種(公開 / アーカイブ / 下書き戻し)のメッセージは `static` ファクトリで提供し、呼出側に文字列責務を持たせない
- **作成者 / 最終更新者の記録**: 新規作成時に作成者 + 最終更新者を、更新・状態遷移時に最終更新者を記録する
- **一覧の受講者数 / 並び順**: 受講者数は `withCount('users')` で 1 クエリ集計(行ごとの count を避ける)。並び順は「公開中 → 下書き → アーカイブ」優先 + 並び順 + 作成日時(MySQL は `FIELD()`、SQLite は `CASE` 式で同等順序)
- **テスト観点**: 各エンドポイント × 各ロール(管理者 true / コーチ・受講生 403)の認可網羅 + 削除 2 段ガード(状態違反 / 受講者紐づき)の 409 + 状態遷移 3 種の遷移元違反 409 + 新規作成時の `status=draft` 初期化 が本チケット固有の観点

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 受講期間(日) / 初期付与面談回数の上限・下限は? | 受講期間は 1〜3650 日、初期付与面談回数は 0〜1000 回 |
| 説明 / 並び順は必須? | いずれも任意。並び順は省略時 0 |
| プランに価格は持たせる? | 持たない。決済は LMS 外で完結し、LMS 内では受講期間 + 初期付与面談回数の組み合わせのみマスタとして管理する |
| 認可は誰がアクセス可能? | 管理者のみ。受講生 / コーチはプラン管理画面の全操作で 403 |
| 認可拒否時の HTTP ステータスは 403 / 404? | 403(受講生 / コーチが直接画面を開いても 403 を返す。404 では返さない) |
| 公開中 / アーカイブのプランを削除しようとしたら? | 409 を返す。下書き状態のみ削除可 |
| 下書き状態でも受講者が紐づいているプランを削除しようとしたら? | 409 を返す。下書き状態 **かつ** 受講者が 1 名も紐づいていないプランのみ削除可 |
| 公開中 → 下書きの直接戻しはできる? | 不可。一度アーカイブを経由し、アーカイブから下書きへ戻す |
| 削除は SoftDelete / 物理削除? | 物理削除。下書き × 受講者なし の前提を担保しているため、論理削除は採用しない |
| 公開中のプランの編集はできる? | 可。基本情報(プラン名 / 説明 / 受講期間 / 初期付与面談回数 / 並び順)は公開後も変更可。既に受講中のユーザーには影響しない(新規招待時に新しい値で初期化される) |
| 一覧の並び順は? | 「公開中 → 下書き → アーカイブ」優先 + 並び順 昇順 + 作成日時 降順 が望ましい。「並び順 昇順 → 作成日時 降順」だけでも振る舞い OK |
| 検索のヒット範囲は? | プラン名の部分一致のみ(説明は対象外) |
| ページネーションは何件 / ページ? | 20 件推奨(規模が小さいので受講生判断で OK)。Laravel 標準の `withQueryString()` でフィルタ状態を引き継ぐ実装は備えるが、プランは少数のため画面上はページ送りが現れないことが多い。ページネーションの正しさは受け入れ条件(実機確認)ではなく横断コード品質で確認する |
| 作成者 / 最終更新者は誰の ID? | 作成時は作成した管理者、その後の更新時(編集・状態遷移含む)は更新した管理者 |
