## 概要

日次・分次で動く定期実行処理(プラン期間満了ユーザーの修了・受験日超過受講登録の自動失敗・終了時刻超過面談の自動完了)が、対象レコードを全件メモリに読み込んでから処理する実装になっており、対象件数が増えるとピークメモリが線形に増加する。`chunk()` / `chunkById()` / `cursor()` の使い分けを判断したうえで分割処理に置き換え、件数規模に依らないメモリ使用量で完走できるようにする。

## 背景・目的

- **現状の問題**: 対象テーブル(ユーザー / 受講登録 / 面談)が将来的に万単位に拡大することを想定すると、全件読み込みで処理するとプロセスがピークメモリで詰まり、OOM(Out of Memory)で定期実行が落ちるリスクがある。
- **達成したい状態**: 件数が増えても定期実行が完走し、メモリ使用量が件数に依らず一定範囲に収まる状態。`chunk()` / `chunkById()` / `cursor()` の中から処理特性に応じた手段を選択する判断力を養う。
- **教材的価値**: Laravel の代表的なメモリ最適化手段 3 種(`chunk` / `chunkById` / `cursor`)の **使い分けの判断軸**(処理対象を更新するか、メモリ常駐を許容するか、件数規模 等)を学ぶ。とくに「WHERE 条件カラムを処理中に更新する場合は `chunk()` だと取りこぼしが発生する → `chunkById()` が必須」というハマりどころが本チケットの肝。

## やること

- 対象 3 件の定期実行コマンドの一括読み込みを分割処理に置き換える
  1. プラン期間が満了した受講生を修了状態に自動遷移する日次バッチ
  2. 目標受験日を過ぎた受講登録を不合格に自動遷移する日次バッチ
  3. 終了時刻を過ぎた面談を完了状態に自動遷移する分次バッチ
- 各箇所で `chunk()` / `chunkById()` / `cursor()` の中から適切な手段を選んで採用する(処理が状態カラムを更新するかどうかが判断の主軸)
- 既存の定期実行コマンドのテストが通り続けることを確認したうえで、件数を増やしたメモリ計測 or 大量データの完走確認を追加する

## やらないこと

- 既に分割処理になっている箇所(招待リマインダ配信 / チャットメンバー同期 等)の改修
- 「全件を 1 トランザクションで原子的に処理する」業務要件を持つ箇所(期限切れ招待の cascade 退会など)を分割処理に変更すること(原子性が崩れるため対象外)
- 行レベルロック(`lockForUpdate`)を伴う一括取得を分割処理に変更すること(ロックの範囲が崩れるため対象外)
- 通知配信 / メール送信のキュー化(Queue 化は別スコープ)

## 受け入れ条件

- [ ] 対象 3 件の定期実行コマンドが、全件読み込み(`->get()` + `foreach`)ではなく分割処理(`chunk*()` 系)で実装されている
- [ ] 対象 3 件は処理中に WHERE 条件カラム(状態)を更新するため、`chunk()` ではなく `chunkById()`(または同等の取りこぼし防止策)が選択されている
- [ ] 1 チャンクの件数が固定値で指定されている(例: 100 件 / チャンク)
- [ ] 既存の定期実行コマンドのテスト(処理対象の絞り込み・状態遷移・出力メッセージ)が変更前後で同じ結果で pass する(振る舞い不変)
- [ ] 件数規模を増やしてもメモリ使用量が線形増加しないことを示す確認(多件数での完走確認 or `memory_get_peak_usage(true)` を用いた計測テスト)が追加されている

## 実装方針

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の設計を組み立てる。

### 変更対象と変更前後の状態

- **変更対象ファイル候補(3 件)**:
  1. `app/Console/Commands/GraduateExpiredUsersCommand.php`(プラン期間満了ユーザーの修了遷移)
  2. `app/Console/Commands/FailExpiredEnrollmentsCommand.php`(受験日超過 受講登録の不合格遷移)
  3. `app/Console/Commands/Mentoring/AutoCompleteMeetingsCommand.php`(終了超過 面談の完了遷移)
- **変更前(提供 PJ の状態)**: 各コマンドの取得クエリが `->get()` で全件読み込みになっており、その配列を `foreach` で順次処理する形(対象件数分のレコードが常時メモリに常駐)
- **変更後(理想形)**: 各コマンドの取得クエリが `->chunkById(100, function ($items) use (...): void { foreach ($items as $item) { ... } })` 相当の分割処理になっている。1 チャンク分(100 件)だけがメモリに載り、チャンク処理後は GC 対象になる
- **`chunkById()` を選ぶ理由**: 上記 3 件は `foreach` 内で対象レコードの状態カラム(WHERE 条件カラム)を更新する。`chunk()`(offset ベース)だと、処理済みレコードが対象外になりオフセットがずれて取りこぼしが発生する。`chunkById()`(主キーカーソルベース)なら、状態更新で対象から外れても次のチャンクの取得位置がずれずに最後まで網羅できる

### 計測指標と目標値(Performance)

| 項目 | 内容 |
|---|---|
| 計測手法 | テスト内: 多件数(例: 250 件)を投入して定期実行コマンドを `artisan` 経由で呼び、完走することを確認 + `memory_get_peak_usage(true)` を任意で計測 / 手動: `php -d memory_limit=64M artisan {command}` で全件読み込み版(変更前)が落ち、分割処理版(変更後)が完走することを比較 |
| 計測指標 | 定期実行 1 回あたりのピークメモリ使用量 / 完走可否 |
| 改善前(Before) | 対象件数 N で「Eloquent モデル 1 件あたり数 KB × N」が常時メモリに常駐。N=10,000 で **10〜20 MB 規模のピーク消費**(関連リレーションを Eager Loading するとさらに増加)。`memory_limit` が低い実行環境で OOM が出やすい |
| 改善後(After 目標) | 1 チャンク 100 件のみがメモリに載るため、ピークは件数 N に依らず **数百 KB〜1 MB 程度**(リレーション込みでも数 MB に収まる)。対象件数が万単位でも完走する |

### テスト方針

| 種別 | 観点 |
|---|---|
| 振る舞い不変 | 既存の定期実行コマンドテスト(`tests/Feature/Commands/GraduateExpiredUsersCommandTest.php` / `FailExpiredEnrollmentsCommandTest.php` / `Mentoring/AutoCompleteMeetingsCommandTest.php`)が変更前後で同じ結果で pass(対象の絞り込み・状態遷移・出力メッセージ・他レコードへの非影響) |
| メモリ計測(任意) | 1 チャンク件数(例: 100)を超える件数(例: 250 件)を投入し、コマンドが完走し全件が処理される(`->assertExitCode(0)` + 期待件数の状態遷移)ことを assert。`memory_get_peak_usage(true)` を呼び出し、しきい値内に収まることを確認するテストを任意で追加(しきい値は実行環境依存のため厳密 assert は不要、目安として記録) |

### 採用技術と判断理由

- **採用技術**: Eloquent の `chunkById(N, function ...): void`(主キーカーソルベースの分割処理)
- **判断理由**:
  1. 対象 3 件はいずれも `foreach` 内で状態カラムを更新するため、`chunk()` だと取りこぼしが発生する。`chunkById()` が必須(他の選択肢ではない)
  2. `cursor()` は「1 件ずつメモリに載せて流す」ストリーミング用途。本チケットの対象は「対象レコードに対して副作用(状態遷移 + ログ記録 + 通知)を持つ処理」であり、1 チャンク単位で副作用をまとめて扱える `chunkById()` の方が見通しが良い
  3. 1 チャンク件数は 100 が目安(Laravel コミュニティの一般的な選択)。極端に大きい値(1000+)はメモリ効果が薄れ、小さい値(10 以下)はクエリ往復回数が増えて遅くなる
- **`chunk` / `chunkById` / `cursor` の使い分けの一般則**(受講生に伝える材料):
  - `chunk()`: 分割処理。**WHERE 条件カラムを処理中に更新しない**場合に使う(更新するとオフセットがずれて取りこぼし)
  - `chunkById()`: 分割処理。**WHERE 条件カラムを処理中に更新する**場合に必須。主キーで順序付ける制約あり
  - `cursor()`: ストリーミング読み。**1 件ずつメモリに載せて流す**用途(レポート出力 / CSV エクスポート 等、副作用を持たない読み専用)

### 改善対象コードメモ

- 模範解答 PJ では対象 3 コマンドのいずれも `chunkById(100, function ($items) use (...): void { foreach ($items as $item) { ... } })` で実装されており、Step 4 ではこの分割処理を `->get(); foreach (...)` の全件読み込みに巻き戻す
- 関連: `app/UseCases/User/GraduateAction.php`(プラン期間満了時の修了処理 Action)/ `app/UseCases/Enrollment/FailAction.php`(受験日超過時の不合格 Action 相当)/ `app/UseCases/Meeting/AutoCompleteAction.php`(面談自動完了 Action) — これら Action 自体は分割処理対象ではなく、コマンド側のループ取得方法のみが本チケットの対象
- 対象外として残す箇所(変更しない): `app/UseCases/Auth/ExpireInvitationsAction.php`(全件 1 トランザクション原子性が業務要件)/ `app/Services/SessionCloseService.php` の `closeStaleSessions`(`lockForUpdate` で対象一括ロックが必須)

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 対象 3 件はなぜ `chunk()` ではなく `chunkById()` を使うのか? | 3 件とも `foreach` 内で WHERE 条件カラム(状態)を更新する。`chunk()`(offset ベース)は対象から外れたレコードでオフセットがずれて取りこぼしを起こす。`chunkById()`(主キーカーソル)は更新で対象から外れても次の取得位置がずれず、安全に網羅できる |
| 1 チャンク件数は何件が良いか? | 100 件が目安(Laravel コミュニティで一般的)。極端に大きい値だとメモリ効果が薄れ、小さい値だとクエリ往復が増えて遅くなる。受講生の判断で 50〜500 の範囲内なら可 |
| `cursor()` を使う場面は本チケットに含まれるか? | 本チケットの対象 3 件は副作用(状態遷移 + ログ + 通知)を持つため `chunkById()` が適。`cursor()` は「読み専用で 1 件ずつメモリに載せて流す」場面(CSV エクスポート 等)が向く。受講生は両者の使い分けを理解しておけば、本チケットでは `chunkById()` を選ぶ判断ができる |
| 既に分割処理になっている定期実行処理もあるか? | ある(招待リマインダ配信・チャットメンバー同期 等)。それらは本チケットの対象外で、振る舞いを維持する |
| 全件読み込みのままで良い箇所はあるか? | ある。「全件を 1 トランザクションで原子的に処理する」業務要件(期限切れ招待 cascade)や「行レベルロックで対象一括ロック」が必要な処理(滞留学習セッションのクローズ)は分割処理にすると整合性が崩れるため対象外。本チケットの「やらないこと」にも明記 |
| メモリ計測テストのしきい値はどう設定するか? | しきい値は実行環境依存(PHP メモリ確保の挙動 + Eloquent の関連ロードに左右される)で厳密 assert は難しい。受講生には「件数を増やしても完走する」確認(`->assertExitCode(0)` + 期待件数の状態遷移)を主軸にし、`memory_get_peak_usage(true)` の絶対値は参考記録に留めることを推奨 |
