## 概要

定期実行処理(プラン期間満了による受講生の自動修了 / 目標受験日を過ぎた受講登録の自動失敗 / 終了時刻を過ぎた面談の自動完了)が、対象レコードを全件メモリに読み込んでから処理する実装になっており、対象件数が増えるとピークメモリが線形に増加する。対象テーブルが将来的に万単位へ拡大すると、メモリ枯渇で定期実行が落ちるリスクがある。

本チケットでは、これらの一括読み込みを「一定件数ずつに区切って処理する分割処理」に置き換え、件数規模に依らないメモリ使用量で完走できる状態にする。あわせて、対象 3 件はいずれも処理中に「絞り込み条件となる状態」を更新するため、分割方式の選び方を誤ると処理対象を取りこぼす——という分割処理の典型的な落とし穴に対する判断軸(処理対象を更新するか / メモリ常駐を許容するか / 件数規模)を扱う。

## 要件

### 分割処理への置き換え(対象 3 件)

- プラン期間が満了した受講生を修了状態に自動遷移する日次バッチ
- 目標受験日を過ぎた受講登録を不合格(学習中止)に自動遷移する日次バッチ
- 終了時刻を過ぎた面談を完了状態に自動遷移する分次バッチ

### 分割方式の選択

- 対象 3 件はいずれも処理中に「絞り込み条件となる状態」を更新するため、処理を進めても取りこぼしが起きない分割方式を選択する
- 1 回に読み込む件数を固定する(チャンクサイズの指定)

### 動作確認

- 既存の定期実行処理の振る舞い(対象の絞り込み / 状態遷移 / 出力)を変えず、件数を増やしても全件が処理されることを確認する

## スコープ外

- 既に分割処理で実装されている定期実行処理(面談リマインダー配信 / チャットメンバー同期 等)の改修
- 対象全件を 1 つのトランザクションで原子的に処理する業務要件を持つ処理(期限切れ招待の一括失効 + 連動退会 等)の分割処理化(原子性が崩れるため)
- 行レベルロックを伴う一括取得(滞留学習セッションの強制クローズ 等)の分割処理化(ロック範囲が崩れるため)
- 通知配信 / メール送信の非同期化(キュー化は別スコープ)

## 受け入れ条件

- [ ] 対象 3 件の定期実行コマンドが、対象を全件一度に読み込まず一定件数ずつに区切って処理する実装になっており、1 回の読み込み件数を超える件数(例: 数百件)を投入してもコマンドが完走し、対象全件が期待どおり状態遷移する
- [ ] 処理中に絞り込み条件となる状態を更新しても、分割方式の選択ミスによる処理漏れが発生せず、対象全件が取りこぼしなく処理される
- [ ] 本チケットの対象コマンドに対するテスト (Feature 等) が実装されている

## 実装方針(参考)

> **粒度**: 業務語彙 + 技術名(変更対象ファイルパス・クラス名・メソッド名)を併記。具体的なコード片 / SQL クエリは書かない。「どこを巻き戻すか」が機械的に判断できる粒度にする。

### 変更内容

- **変更対象ファイル**(3 件、いずれも `app/Console/Commands/` 配下の `handle()`):
  - `app/Console/Commands/GraduateExpiredUsersCommand.php` — プラン期間満了ユーザーの修了遷移(ループ内で `App\UseCases\Plan\GraduateUserAction` を呼び `status` を `in_progress` → `graduated` に更新)
  - `app/Console/Commands/FailExpiredEnrollmentsCommand.php` — 受験日超過 受講登録の不合格遷移(ループ内の `DB::transaction` で `status` を `learning` → `failed` に更新 + `EnrollmentStatusChangeService` / `DefaultEnrollmentService` を呼出)
  - `app/Console/Commands/Mentoring/AutoCompleteMeetingsCommand.php` — 終了超過 面談の完了遷移(ループ内で `App\UseCases\Meeting\AutoCompleteMeetingAction` を呼び `status` を `reserved` → `completed` に更新)
- **変更前**(提供 PJ で受講生が直面する状態): 各コマンドの取得クエリが `->get()` で全件読み込みになっており、その結果を `foreach` で順次処理する(対象件数分の Eloquent モデルが常時メモリに常駐)
- **変更後**(理想形): 各コマンドの取得クエリが `->orderBy('id')->chunkById(100, function ($items): void { foreach ($items as $item) { ... } })` の主キーカーソルベース分割処理になっている(1 チャンク 100 件のみメモリに載り、チャンク処理後は GC 対象)
- **採用技術と判断理由**: Eloquent の `chunkById(N, ...)`(主キーカーソルベース分割)を採用。対象 3 件はいずれも `foreach` 内で WHERE 条件カラム(`status`)を更新するため、オフセットベースの `chunk()` では処理済みレコードが対象外になりオフセットがずれて**取りこぼし**が発生する → `chunkById()` が必須(他の選択肢ではない)。`cursor()` は「1 件ずつメモリに載せて流す」読み専用ストリーミング用途で、状態遷移 + ログ + 通知などの副作用を持つ本処理には不向き。チャンクサイズ 100 は Laravel コミュニティの一般値(過大だとメモリ削減効果が薄れ、過小だとクエリ往復が増える)。`chunk` / `chunkById` / `cursor` の使い分けの一般則 — `chunk()`: WHERE 列を処理中に更新しない分割 / `chunkById()`: WHERE 列を処理中に更新する分割(主キー順序付け制約あり)/ `cursor()`: 副作用を持たない読み専用ストリーミング
- **テスト観点**: 既存の各コマンド Feature テスト(`tests/Feature/Commands/GraduateExpiredUsersCommandTest.php` / `FailExpiredEnrollmentsCommandTest.php` / `AutoCompleteMeetingsCommandTest.php`)で対象絞り込み・状態遷移・出力メッセージが変更前後で不変。加えて 1 チャンク件数を超える件数(例: 250 件)を投入して終了コード + 期待件数の状態遷移を検証する大量データ完走テストを追加(`memory_get_peak_usage(true)` の絶対値は実行環境依存のため厳密 assert せず参考記録に留める)

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 対象 3 件はなぜ単純な分割(`chunk()`)ではダメなのか? | 3 件とも処理中に絞り込み条件となる状態カラムを更新する。オフセットベースの分割だと、処理済みで対象から外れたレコードのぶんだけ取得位置がずれて取りこぼしが起きる。主キーカーソルベース(`chunkById()`)なら、状態更新で対象から外れても次チャンクの取得位置がずれず全件を網羅できる |
| 1 回に読み込む件数は何件が良いか? | 100 件が目安。過大だとメモリ削減効果が薄れ、過小だとクエリ往復が増えて遅くなる。50〜500 程度なら受講生の判断で可 |
| `cursor()` を使う場面は本チケットに含まれるか? | 含まない。`cursor()` は読み専用で 1 件ずつ流すストリーミング用途(CSV エクスポート等)向け。本チケットの対象は状態遷移 + ログ + 通知などの副作用を伴うため分割処理(`chunkById()`)が適する。両者の使い分けを理解しておけば、本チケットでは分割処理を選べる |
| 全件読み込みのまま残すべき処理もあるか? | ある。「対象全件を 1 トランザクションで原子的に処理する」業務要件を持つ処理(期限切れ招待の一括失効 + 連動退会)や、「行レベルロックで対象を一括ロックする」処理(滞留学習セッションの強制クローズ)は、分割すると整合性・ロック範囲が崩れるため対象外。スコープ外にも明記 |
| 件数を増やすテストのしきい値はどう決めるか? | メモリ使用量の絶対値は実行環境に依存し厳密 assert は難しい。「1 チャンク件数を超える件数を投入しても完走し全件が処理される」(終了コード + 期待件数の状態遷移)を主軸にし、ピークメモリは参考記録に留めることを推奨 |
