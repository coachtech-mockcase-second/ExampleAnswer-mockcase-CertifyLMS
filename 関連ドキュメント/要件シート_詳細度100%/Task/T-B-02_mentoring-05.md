# T-B-02 コーチダッシュボードの担当受講生一覧の N+1 解消

## 概要

コーチ(coach)のダッシュボードに表示される担当受講生一覧で、各受講生の関連情報(受講生氏名 / 担当資格名 / 最終活動日時)の取得が受講生数に応じて個別クエリを発火する N+1 状態になっている。リレーションの一括取得と最終活動日時の集約取得を組み合わせて、担当受講生数に依らずクエリ本数を定数に抑える。

## 要件

- コーチダッシュボードの担当受講生一覧の N+1 を解消する(受講生 / 担当資格 / 最終活動日時の一括取得)
- 最終活動日時は関連テーブル全件の読み込みではなく、最大値の集約取得で求める
- 担当受講生件数を増やしてもクエリ本数が線形増加しない状態にする
- 一覧の表示内容・並び順・絞り込み仕様は不変に保つ

## スコープ外

- 管理者のユーザー管理画面の N+1 解消 — 本チケットでは扱わない（別 Task）
- ダッシュボードの他カード(プラン情報 / 通知 / 学習進捗ゲージ 等)の最適化
- キャッシュ層の導入
- 一覧表示項目・並び順・絞り込み仕様の変更(振る舞いは不変)

## 実装方針(参考)

> **粒度**: 業務語彙 + 技術名(変更対象ファイルパス・クラス名・メソッド名)を併記。具体的なコード片 / SQL クエリ / メソッド完全例は書かない。

### 変更内容

- **変更対象ファイル**: `app/UseCases/Dashboard/FetchCoachDashboardAction.php`(Basic 受講生が Controller 内完結で実装している場合は `app/Http/Controllers/DashboardController::index()` 内のコーチ向けデータ取得処理が対象)。計測対象画面は `GET /dashboard`(コーチでアクセスするとコーチ向けビュー = 担当受講生一覧を描画)
- **変更前**(提供 PJ の状態): 担当受講生(`Enrollment`)一覧の取得クエリで `->with(['user', 'certification'])` と `->withMax('learningSessions as last_activity_at', 'started_at')` が抜けており、Blade で `$enrollment->user->name` / `$enrollment->certification->name` / `$enrollment->last_activity_at` を参照する箇所が件数分の遅延ロードを発火する(担当受講生 N 人で 本体 1 + 受講生個別 N + 担当資格個別 N + 最終活動日時個別 N = 1 + 3N 本、N=10 で 31 本 / N=20 で 61 本)
- **変更後**(模範解答 PJ の完成形): 担当資格スコープの受講登録取得クエリに `->with(['user', 'certification'])`(BelongsTo 2 本を IN クエリ各 1 本に集約)+ `->withMax('learningSessions as last_activity_at', 'started_at')`(関連テーブルの最大値を本体クエリのサブクエリとして集約)を追加。本体 1 + 受講生 IN 1 + 担当資格 IN 1 + 最終活動日時集約(本体に統合)= **3 本固定**。担当受講生件数に依存しない
- **採用技術と判断理由**: 担当資格 / 受講生は BelongsTo のため `with()` の IN クエリ 1 本ずつで一括取得。最終活動日時は関連テーブル(学習セッション)の最大値であり、関連レコード全件を Eager Loading する必要はない — `withMax` 系で本体クエリのサブクエリ / JOIN として組み込めば最大値だけが取れる(全件取得よりメモリ・転送量とも大幅に軽い)。別 Task(管理者ユーザー一覧 × プラン情報の単純 BelongsTo Eager Loading)に対し、本チケットは **複数リレーション + 関連テーブルからの集約取得(`withMax`)** の複合パターンとして配置。ページネーションは導入しない(コーチ 1 名あたりの担当受講生は通常少数で全件取得で問題ない規模)
- **計測(PR 動作確認に併記)**: `DB::enableQueryLog()` + `DB::getQueryLog()` の本数カウント / Laravel Debugbar(ローカル目視)。Before 1 + 3N(N=10 で 31 本)→ After 3 本固定。「担当受講生件数に依存しない定数か」が本質的な合否判定
- **テスト観点**: 既存ダッシュボードテスト(`tests/Feature/UseCases/Dashboard/FetchCoachDashboardActionTest.php` / `tests/Feature/Http/Dashboard/*`)を改修後も pass させ振る舞い不変を担保(担当受講生表示内容 / 担当外受講生の非表示 / 最終活動日時の正確性)+ 担当受講生 10 件程度を生成して `DB::listen()` でクエリをカウントし、件数が N に対して線形にならない上限本数を assert(既存 `tests/Feature/Http/Dashboard/DashboardQueryCountTest.php` のパターン踏襲)

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| クエリ本数の目標値は? | 担当受講生件数に依らない定数本数(目安 5 本以下)。本体 + 受講生 IN + 担当資格 IN + 集約(本体に統合)程度の構成 |
| 最終活動日時を関連テーブルから取るのに `with('learningSessions')` ではなく集約取得を使うのはなぜ? | `with('learningSessions')` は対象受講登録の全学習セッションを Eager Loading してから PHP 側で最大値を計算するため、関連レコードが大量にあるとメモリと SQL 取得量の両方で重くなる。`withMax` は SQL の集約関数(MAX)を本体クエリに組み込むため、最大値だけが取れて転送量も最小 |
| 担当資格スコープ(コーチが見える受講生)はどう絞り込まれている? | コーチが担当する資格の ID 群を取得し、その資格に属する受講登録(学習中 / 修了)に絞る。担当外資格の受講生は元々取得対象に含まれない(本最適化の影響範囲外) |
| ページネーションは導入する? | 本チケットでは導入しない。コーチ 1 名あたりの担当受講生は通常少数で、全件取得でも問題のない規模。将来的に大量になる場合は別途検討 |
| 他の N+1 解消 Task との違いは? | 同じ N+1 解消だが対象 / 解法が異なる。管理者のユーザー一覧 × プラン情報の単純 BelongsTo Eager Loading に対し、本チケットはコーチ視点の担当受講生一覧 × 複数リレーション + 関連テーブルからの集約取得(`withMax`)。受講生は両方を通じて「N+1 解消の引き出しの広さ」を学ぶ |
