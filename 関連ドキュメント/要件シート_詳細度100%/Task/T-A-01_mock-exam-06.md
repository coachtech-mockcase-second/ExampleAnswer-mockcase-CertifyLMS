## 概要

管理者(admin)が閲覧する模試マスタ一覧で、一覧に並ぶ模試ごとに関連情報(所属資格 / 作成者 / 更新者 / 問題数)を個別クエリで取得しており、模試の件数に比例して発行クエリが増える(N+1)。一覧表示が重くなる原因になっている。関連情報を一括取得(Eager Loading + 件数集計)へ書き換えてこれを解消する。表示される一覧の内容そのものは変えない。

## 要件

- 模試マスタ一覧の取得を、模試の件数が増えても発行クエリ数が一定に保たれるよう一括取得へ書き換える
- 関連情報(所属資格 / 作成者 / 更新者 / 問題数)は模試集合に対してまとめて取得する(個別クエリの都度発火をなくす)
- 返却される一覧(模試の並び・各行の表示項目)は改修の前後で完全に同一に保つ

## スコープ外

- 一覧のキャッシュ化 — 本チケットのスコープ外、別 Task で個別判断
- インデックス追加 / スキーマ変更 — 既存インデックスで捌ける範囲のクエリ最適化のみ
- 他の一覧画面(受験履歴 / 弱点分析 等)のパフォーマンス(別チケットで扱う)
- 返却される一覧の内容(並び・表示項目)を変える変更
- ページネーション仕様の変更

## 受け入れ条件

- [ ] 模試マスタ一覧で、模試の件数が増えても発行クエリ数が増えない(所属資格 / 作成者 / 更新者 / 問題数が一括取得される)
  - 確認方法（テスト）: 同梱の `tests/Feature/Http/MockExam/MockExamIndexQueryCountTest.php::test_index_query_count_does_not_grow_with_mock_exam_count`(修正前は失敗)が pass する

## 実装方針(参考)

### 変更内容

- **対象**: `app/UseCases/MockExam/IndexAction.php`(模試マスタ一覧の取得。入口 `MockExamController::index`、模試マスタの管理一覧画面)
- **変更前→後**: 一覧取得が関連の Eager Loading を持たず、各模試の所属資格(`certification`)/ 作成者(`createdBy`)/ 更新者(`updatedBy`)/ 問題数(`mockExamQuestions` 件数)を行ごとに都度クエリ(1+4N、一覧 20 件なら 80 本超) → `->with(['certification', 'createdBy', 'updatedBy'])->withCount('mockExamQuestions')` で関連を一括取得し、一覧件数に依らず発行クエリ数を一定にする
- **判断理由**: belongsTo 関連(`certification` / `createdBy` / `updatedBy`)は `with()` で N+1 を回避、子の件数(`mockExamQuestions`)は `withCount()` で集計列を 1 クエリに畳む。既存インデックス(各 FK)で一括取得が効くためスキーマ追加は不要。表示は同一で内部のクエリ効率のみ改善する
- **テスト**: 模試件数を増やしても発行クエリ数が一定であることをクエリログで検証する新規テストを追加(`DB::enableQueryLog()` + 件数を変えた 2 ケースの発行本数比較)+ 既存の一覧表示テストの pass(振る舞い不変)

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 一覧に出る関連情報は? | 所属資格名 / 作成者・更新者(管理ユーザー)/ 問題数。これらを Eager Loading + 件数集計で一括取得する |
| キャッシュは導入する? | しない(本 Task のスコープ外)。N+1 の解消のみ |
| 表示内容(並び・項目)は変えて良い? | いいえ、完全に同一(改修前後で一覧の見た目・値が一致)。クエリ効率の改善のみ |
| インデックス追加 / マイグレーションは必要? | 不要。既存の外部キーインデックスで一括取得が効く |
| 既存の一覧テストは触って良い? | 既存ケースはそのまま pass させる(振る舞い不変の保証)。N+1 解消の検証は新規テストメソッドとして追加する |
