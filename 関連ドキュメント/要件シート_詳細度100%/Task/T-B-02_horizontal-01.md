## 概要

定期実行処理(プラン期間満了による受講生の自動修了 / 目標受験日を過ぎた受講登録の自動失敗 / 終了時刻を過ぎた面談の自動完了)は、対象を一定件数ずつに区切って処理してはいるものの、その分割方式が処理中の状態更新と両立しないため、対象が 1 回の読み込み件数を超えると一部を取りこぼす。対象 3 件はいずれも処理の中で絞り込み条件となる状態カラムを更新するため、オフセットベースの分割では処理済み件数分だけ取得位置がずれ、後続の対象を取りこぼしてしまう。対象テーブルが将来的に万単位へ拡大すると、取りこぼしが顕在化して自動処理が漏れるリスクがある。

本チケットでは、この分割処理を状態更新と両立する方式(主キーカーソルベースの分割)に修正し、件数規模に依らず・メモリ使用量を一定に保ちつつ・全件を取りこぼしなく処理できる状態にする。あわせて、分割方式の選び方(処理対象を更新するか / メモリ常駐を許容するか / 件数規模)という分割処理の典型的な落とし穴に対する判断軸を扱う。

## 要件

### 分割処理への置き換え(対象 3 件)

- プラン期間が満了した受講生を修了状態に自動遷移する日次バッチ
- 目標受験日を過ぎた受講登録を不合格(学習中止)に自動遷移する日次バッチ
- 終了時刻を過ぎた面談を完了状態に自動遷移する分次バッチ

### 分割方式の選択

- 対象 3 件はいずれも処理中に「絞り込み条件となる状態」を更新するため、処理を進めても取りこぼしが起きない分割方式を選択する
- 1 回に読み込む件数を固定する(チャンクサイズの指定)

### 動作確認

- 既存の定期実行処理の振る舞い(対象の絞り込み / 状態遷移 / 出力)を変えず、件数を増やしても全件が処理されることを確認する

## スコープ外

- 既に分割処理で実装されている定期実行処理(面談リマインダー配信 / チャットメンバー同期 等)の改修
- 対象全件を 1 つのトランザクションで原子的に処理する業務要件を持つ処理(期限切れ招待の一括失効 + 連動退会 等)の分割処理化(原子性が崩れるため)
- 行レベルロックを伴う一括取得(滞留学習セッションの強制クローズ 等)の分割処理化(ロック範囲が崩れるため)
- 通知配信 / メール送信の非同期化(キュー化は別スコープ)

## 受け入れ条件

- [ ] 対象 3 件の定期実行コマンドが、1 回の読み込み件数を超える件数(例: 数百件)を投入してもコマンドが完走し、処理中に絞り込み条件となる状態を更新しても取りこぼしが起きず、対象全件が期待どおり状態遷移する
- [ ] 同梱 `tests/Feature/Commands/ScheduleCommandChunkTest.php`(対象 3 コマンドの全件処理を検証、修正前は失敗)が pass する

## 実装方針(参考)

### 変更内容

- **対象**(3 件、いずれも `app/Console/Commands/` 配下の `handle()`): `GraduateExpiredUsersCommand.php`(プラン満了ユーザー修了遷移、`Plan\GraduateUserAction` で `status` を `in_progress` → `graduated`)/ `FailExpiredEnrollmentsCommand.php`(受験日超過 受講登録の不合格遷移、`DB::transaction` で `status` を `learning` → `failed` + `EnrollmentStatusChangeService` / `DefaultEnrollmentService` 呼出)/ `Mentoring/AutoCompleteMeetingsCommand.php`(終了超過 面談の完了遷移、`Meeting\AutoCompleteMeetingAction` で `status` を `reserved` → `completed`)
- **変更前→後**: 各コマンドの取得クエリが `->orderBy('id')->chunk(100, fn ($items) => ...)` のオフセットベース分割(処理中に WHERE 列 `status` を更新すると取得位置がずれて後続チャンクを取りこぼす) → `->orderBy('id')->chunkById(100, fn ($items) => ...)` の主キーカーソルベース分割(状態更新で対象から外れても取得位置がずれず、1 チャンク 100 件のみメモリに載り処理後は GC 対象)
- **判断理由**: 対象 3 件はいずれも `foreach` 内で WHERE 列(`status`)を更新するため、オフセットベースの `chunk()` では処理済みレコード分だけ位置がずれて取りこぼしが発生 → `chunkById()` 必須。`cursor()` は副作用なしの読み専用ストリーミング用途で本処理(状態遷移 + ログ + 通知)に不向き。チャンクサイズ 100 は一般値(過大はメモリ削減効果減・過小はクエリ往復増)
- **テスト**: 1 チャンク件数を超える件数を投入して全件が取りこぼしなく状態遷移することを検証する大量データ完走テスト `tests/Feature/Commands/ScheduleCommandChunkTest.php`(対象 3 コマンド分。オフセット分割では取りこぼして失敗 / 主キーカーソル分割で pass)+ 既存各コマンド Feature テスト(`tests/Feature/Commands/{GraduateExpiredUsers,FailExpiredEnrollments,AutoCompleteMeetings}CommandTest.php`)の pass(対象絞り込み・状態遷移・出力が不変)。`memory_get_peak_usage(true)` の絶対値は環境依存のため参考記録に留める

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 対象 3 件はなぜ単純な分割(`chunk()`)ではダメなのか? | 3 件とも処理中に絞り込み条件となる状態カラムを更新する。オフセットベースの分割だと、処理済みで対象から外れたレコードのぶんだけ取得位置がずれて取りこぼしが起きる。主キーカーソルベース(`chunkById()`)なら、状態更新で対象から外れても次チャンクの取得位置がずれず全件を網羅できる |
| 1 回に読み込む件数は何件が良いか? | 100 件が目安。過大だとメモリ削減効果が薄れ、過小だとクエリ往復が増えて遅くなる。50〜500 程度なら受講生の判断で可 |
| `cursor()` を使う場面は本チケットに含まれるか? | 含まない。`cursor()` は読み専用で 1 件ずつ流すストリーミング用途(CSV エクスポート等)向け。本チケットの対象は状態遷移 + ログ + 通知などの副作用を伴うため分割処理(`chunkById()`)が適する。両者の使い分けを理解しておけば、本チケットでは分割処理を選べる |
| 全件読み込みのまま残すべき処理もあるか? | ある。「対象全件を 1 トランザクションで原子的に処理する」業務要件を持つ処理(期限切れ招待の一括失効 + 連動退会)や、「行レベルロックで対象を一括ロックする」処理(滞留学習セッションの強制クローズ)は、分割すると整合性・ロック範囲が崩れるため対象外。スコープ外にも明記 |
| 件数を増やすテストのしきい値はどう決めるか? | メモリ使用量の絶対値は実行環境に依存し厳密 assert は難しい。「1 チャンク件数を超える件数を投入しても完走し全件が処理される」(終了コード + 期待件数の状態遷移)を主軸にし、ピークメモリは参考記録に留めることを推奨 |
