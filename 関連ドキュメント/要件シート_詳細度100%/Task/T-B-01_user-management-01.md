## 概要

管理者(admin)のユーザー管理一覧画面で、各行に表示するプラン名の取得が一覧件数に応じて個別クエリを発火する N+1 状態になっている。これを一括取得(eager loading)に切り替え、一覧件数に依らずクエリ本数を定数に抑える。

## 背景・目的

- **現状の問題**: ユーザー管理一覧は 1 ページ 20 件で表示されるが、各行ごとに「プラン名」を取得するクエリが個別に発火する(最大で 20 件分の追加クエリ)。ユーザー数の増加に伴い管理画面の表示クエリが線形に増え、運用時の表示遅延・DB 負荷の原因になる。
- **達成したい状態**: ユーザー件数に依らず、一覧表示時のクエリ本数を少数の定数に収める(一括取得)。今後のユーザー増にも耐えられるようにする。
- **教材的価値**: 「N+1 とは何か」「Eager Loading で何本に減らせるか」を Before / After 計測値で実感する典型題材。

## やること

- ユーザー管理一覧表示時の N+1 を解消する(プラン情報を一括取得に変更)
- Before / After のクエリ本数を計測し、PR に記載する
- クエリ本数を検証する自動テストを追加し、後戻り(N+1 再発)を防ぐ

## やらないこと

- コーチダッシュボードの担当受講生一覧の N+1 解消 — `T-B-02` で扱う
- キャッシュ層の導入(`Cache::remember()` 等)
- ユーザー一覧画面の表示項目・フィルタ・ソート・認可仕様の変更(振る舞いは不変)
- 招待モーダル内のプラン選択リストの最適化(一覧自体とは別クエリで影響範囲外)

## 受け入れ条件

- [ ] 管理者のユーザー管理一覧表示時、ユーザー件数に依らずクエリ本数が定数(目安 5 本以下)に収まっている
- [ ] 一覧取得処理に `with(['plan'])` 相当の Eager Loading が実装されている(プラン情報の一括取得)
- [ ] 既存の一覧画面の表示内容(プラン名 / 残り日数 / ロール / 状態 / 最終ログイン / ページネーション件数 / 並び順 / 各種フィルタ)が変更前と一致する(振る舞い不変)
- [ ] ユーザー管理一覧のクエリ本数の上限を検証する自動テストが追加されている(`DB::listen` 等で発行クエリをカウントし上限を assert)

## 実装方針

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の設計を組み立てる。

### 主要 URL

| メソッド | パス | 振る舞い |
|---|---|---|
| GET | `/admin/users` | ユーザー管理一覧(キーワード / ロール / 状態フィルタ + ページネーション) |

### 変更対象と変更前後の状態

- **変更対象ファイル候補**: `app/UseCases/User/IndexAction.php`(Basic 受講生が Controller 内完結で実装している場合は `app/Http/Controllers/UserController.php::index()` 内の一覧クエリが対象)
- **変更前(提供 PJ の状態)**: 一覧取得クエリで `plan` リレーションの Eager Loading が無く、Blade で `$user->plan?->name` を参照する箇所が件数分の遅延ロードを発火している
- **変更後(理想形)**: 一覧取得クエリに `->with('plan')` を追加し、`plan_id` の IN クエリ 1 本でまとめて取得する形にする
- **影響範囲**: 一覧クエリのみ。招待モーダル用の別プラン一覧(画面下部)は元々別クエリで発火するため対象外

### 計測指標と目標値(Performance)

| 項目 | 内容 |
|---|---|
| 計測手法 | `DB::enableQueryLog()` + `DB::getQueryLog()` の本数カウント(テスト内)/ Laravel Debugbar(ローカル目視) |
| 計測指標 | ユーザー管理一覧表示 1 回あたりの SQL 発行本数 |
| 改善前(Before) | ページネーション 20 件の場合、本体 SELECT 1 + COUNT 1 + プラン個別取得 最大 20 = 最大 **22 本**(プラン未割当のユーザーが含まれると減るが、ユーザー件数に対して線形増加) |
| 改善後(After 目標) | 本体 SELECT 1 + COUNT 1 + プランの IN クエリ 1 = **3 本**。ユーザー件数に依存しない定数 |

### テスト方針

| 種別 | 観点 |
|---|---|
| 振る舞い不変 | 既存の Feature テスト(`tests/Feature/Http/User/IndexTest.php`)が全件 pass(キーワード / ロール / 状態フィルタ / ページネーション / ソート順 / 認可) |
| 性能(N+1 検知) | 多件数(例: 受講生 10 件以上)を用意した上で `DB::listen()` でクエリをカウントし、上限本数(例: 5〜10 本以下)を assert。N+1 が再発したらこのテストが落ちる |

### 採用技術と判断理由

- **採用技術**: Eloquent の `with()` Eager Loading
- **判断理由**:
  1. 表示で参照されるリレーションは「プラン名」のみで他に集計表示なし。`with('plan')` 1 本で十分(過剰な Eager Loading を避ける)
  2. キャッシュ層(`Cache::remember()`)は導入しない。プラン情報は管理画面で頻繁に編集される性質があり、無効化コストが Eager Loading の効果を上回る

### 改善対象コードメモ

- 改善対象の主要ファイル: `app/UseCases/User/IndexAction.php`(模範解答 PJ では一覧クエリの paginate 直前に `->with('plan')` がある状態が正、Step 4 でその 1 行を取り除いた状態を提供する)
- 関連: `app/Models/User.php`(`plan()` BelongsTo リレーション)/ `resources/views/user/management/index.blade.php`(`$u->plan?->name` の参照箇所)/ `tests/Feature/Http/User/IndexTest.php`(既存テスト一式)/ `tests/Feature/Http/Dashboard/DashboardQueryCountTest.php`(`DB::listen()` でのクエリ本数 assert パターンの参考)

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| クエリ本数の目標値は厳密に何本か? | ページネーション 20 件で「ユーザー件数に依らない定数本数」が満たせていれば良い(目安: 5 本以下)。本体 SELECT + COUNT + IN クエリ程度の構成 |
| 計測手法はどれを使うべきか? | 自動テストでは `DB::enableQueryLog()` または `DB::listen()` を推奨。ローカルでの目視確認には Laravel Debugbar を併用すると見やすい |
| 招待モーダルのプラン一覧も最適化すべきか? | 本チケットの対象外。モーダル用のプラン一覧は元々独立した一覧取得で N+1 ではない(共通の Eloquent Query で発火 1 本) |
| クエリ本数 assert テストはどこに置くべきか? | `tests/Feature/Http/User/` 配下の新規ファイル or 既存テストに 1 メソッド追加。`DashboardQueryCountTest` が類似パターンの参考になる |
| `with('plan')` を使う代わりに各行で `User::find()` を回避する手段はあるか? | 本質的には Eager Loading で問題ない。受講生が独自に「JOIN クエリ + select で 1 本にまとめる」等の代替案を提案した場合も、クエリ本数の受け入れ条件を満たし振る舞いが不変なら可 |
