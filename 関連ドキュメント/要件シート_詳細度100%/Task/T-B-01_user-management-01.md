| Feature | user-management |

| 依存チケット | (なし) |

## 概要

管理者(admin)のユーザー管理一覧画面は 1 ページ 20 件で表示されるが、各行に表示するプラン名の取得が一覧件数に応じて個別クエリを発火する N+1 状態になっている(最大で 20 件分の追加クエリ)。ユーザー数の増加に伴い管理画面の表示クエリが線形に増え、運用時の表示遅延・DB 負荷の原因になる。

本チケットでは、プラン情報を一括取得(Eager Loading)に切り替え、ユーザー件数に依らず一覧表示時のクエリ本数を少数の定数に抑える。「N+1 とは何か」「Eager Loading で何本に減らせるか」を Before / After のクエリ本数で実感する典型題材。

## 要件

- ユーザー管理一覧表示時の N+1 を解消する(プラン情報を一括取得に変更)
- Before / After のクエリ本数を計測し、PR に記載する
- クエリ本数を検証する自動テストを追加し、N+1 の再発を防ぐ

## スコープ外

- コーチダッシュボードの担当受講生一覧の N+1 解消(本チケットでは扱わない)
- キャッシュ層の導入(`Cache::remember()` 等)
- ユーザー一覧画面の表示項目・フィルタ・ソート・認可仕様の変更(振る舞いは不変)
- 招待モーダル内のプラン選択リストの最適化(一覧自体とは別クエリで影響範囲外)

## 受け入れ条件

- [ ] 管理者のユーザー管理一覧を表示した際、表示されるユーザー件数を増やしても発行クエリ本数が増加せず、定数本数(目安 5 本以下)に収まる
- [ ] 本チケットの機能に対するテスト(ユーザー件数を増やしてもクエリ本数が一定であることの検証を含む)が実装されている

## 実装方針(参考)

> **粒度**: 業務語彙 + 技術名(変更対象ファイルパス・クラス名・メソッド名)を併記。具体的なコード片 / SQL クエリは書かない。「どこを巻き戻すか」が機械的に判断できる粒度にする。

### 変更内容

- **変更対象ファイル**: `app/UseCases/User/IndexAction.php`(管理者ユーザー一覧取得)。Basic 受講生が Controller 内完結で実装している場合は `UserController::index` 内の一覧クエリが対象
- **変更前(提供 PJ の状態)**: 一覧取得クエリにプランの Eager Loading が無く、一覧 Blade(`resources/views/user/management/index.blade.php`)で各行のプラン名(`$u->plan?->name`)を参照する箇所が件数分の遅延ロードを発火している
- **変更後(理想形)**: 一覧取得クエリにプランの Eager Loading(`->with('plan')`)を追加し、`plan_id` の IN クエリ 1 本でまとめて取得する
- **採用技術と判断理由**: Eloquent の `with()` Eager Loading。表示で参照するリレーションはプラン名のみで集計表示は無いため `with('plan')` 1 本で十分(過剰な Eager Loading を避ける)。キャッシュ層(`Cache::remember()`)は導入しない(プラン情報は管理画面で頻繁に編集される性質があり、無効化コストが Eager Loading の効果を上回る)。Before: ページネーション 20 件で 本体 SELECT 1 + COUNT 1 + プラン個別取得 最大 20 = 最大 **22 本**(ユーザー件数に線形)/ After: 本体 SELECT 1 + COUNT 1 + プラン IN クエリ 1 = **3 本**(ユーザー件数に非依存)
- **テスト観点**: 多件数(受講生 10 件以上)を用意し `DB::listen()` でクエリをカウントして上限本数を assert(N+1 が再発すると落ちる)。既存の一覧 Feature テスト(`tests/Feature/Http/User/IndexTest.php`)の全 pass で振る舞い不変を担保。クエリ本数 assert は `tests/Feature/Http/Dashboard/DashboardQueryCountTest.php` が参考になる

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| クエリ本数の目標値は厳密に何本か? | ページネーション 20 件で「ユーザー件数に依らない定数本数」が満たせていれば良い(目安: 5 本以下)。本体 SELECT + COUNT + IN クエリ程度の構成 |
| 計測手法はどれを使うべきか? | 自動テストでは `DB::enableQueryLog()` または `DB::listen()` を推奨。ローカルでの目視確認には Laravel Debugbar を併用すると見やすい |
| 招待モーダルのプラン一覧も最適化すべきか? | 本チケットの対象外。モーダル用のプラン一覧は元々独立した一覧取得(公開中プランの取得 1 本)で N+1 ではない |
| クエリ本数 assert テストはどこに置くべきか? | `tests/Feature/Http/User/` 配下の新規ファイル or 既存テストに 1 メソッド追加。`DashboardQueryCountTest` が類似パターンの参考 |
| `with('plan')` 以外の代替手段は使ってよいか? | 本質的には Eager Loading で問題ない。受講生が独自に「JOIN + select で 1 本にまとめる」等の代替案を提案した場合も、クエリ本数の受け入れ条件を満たし振る舞いが不変なら可 |
