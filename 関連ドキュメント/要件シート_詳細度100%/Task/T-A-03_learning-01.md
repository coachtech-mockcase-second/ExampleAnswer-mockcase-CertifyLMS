## 概要

受講生の学習進捗(教材の Section / Chapter / Part / 資格レベルでの完了数・完了率)を算出するロジックが、受講登録詳細画面・ダッシュボードなど複数の場所に重複して書かれている。これらを専用の Service クラスへ集約し、呼び出す側からは Service の高レベルメソッドを呼ぶだけで進捗集計が完結するように分離するリファクタリング。重複した集計ロジックを 1 箇所に集め、見通しと保守性を高める(集計仕様の変更時に 1 箇所だけ直せばよくなる)。

## 要件

- 学習進捗の集計(Section / Chapter / Part / 資格 の完了数・完了率の算出)を専用 Service クラスへ集約する
- 複数の場所(受講登録詳細 / ダッシュボード等)に重複していた集計の呼び出しを、すべて Service の高レベルメソッド呼び出しへ置き換える
- 集計結果(各画面に表示される進捗の数値)は改修の前後で完全に同一に保つ
- 集約した Service に対し、集計ロジックの振る舞いを検証できるテストを用意する

## スコープ外

- 進捗集計の仕様変更(完了判定の条件 / 集計対象の変更 等)— 構造の整理のみ
- 進捗のキャッシュ化 — 本チケットのスコープ外、別 Task で個別判断
- 集計クエリの最適化 / インデックス追加 — 構造の集約のみ(クエリ効率は別チケット)
- 振る舞いを変える変更 — 表示される進捗の数値はすべて同一に保つ

## 受け入れ条件

- [ ] 学習進捗の集計が専用 Service に集約され、それを利用する各画面(受講登録詳細 / ダッシュボード等)の側に重複した集計ロジックが残っていない
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている(集約した Service の単体テストを含む)

## 実装方針(参考)

### 変更内容

- **対象**: `app/Services/ProgressService.php`(進捗集計の集約先)。利用側 = `app/Http/Controllers/EnrollmentController.php`(受講登録詳細)/ `app/UseCases/Learning/ShowEnrollmentAction.php`(受講登録詳細の取得)/ `app/UseCases/Dashboard/FetchStudentDashboardAction.php`(受講生ダッシュボード)
- **変更前→後**: Section / Chapter / Part / 資格 の完了数・完了率を算出するロジックが上記利用側に重複インライン(各所で同等の集計を直書き)で `ProgressService` 未存在 → `ProgressService`(`summarize` / `sectionRatio` / `batchCalculate` 等)へ集約し、各利用側は Service の高レベルメソッドを呼ぶだけにする。集計の重複を 1 箇所へ
- **判断理由**: 同一の多階層集計が複数画面に重複すると、集計仕様の変更時に修正漏れ・画面間の不整合を生む。Service へ集約して単一責務化し、利用側は「進捗が欲しい」という意図だけを表明する。振る舞い(算出される数値)は不変で、構造のみ整理する。参考実装 `app/Services/` 配下の既存集計 Service
- **テスト**: 集約した `ProgressService` の単体テスト(Section / Chapter / Part / 資格 各階層の完了数・完了率、境界ケース = 0 完了 / 全完了 / 一部完了)を用意 + 既存の受講登録詳細・ダッシュボード表示テストの pass(振る舞い不変)

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 集約先の Service は 1 クラス? | 1 クラス(`ProgressService`)。進捗集計という単一責務に集約する |
| 集計結果(数値)は変えて良い? | いいえ、完全に同一(改修前後で各画面の進捗の数値が一致)。構造の集約のみ |
| キャッシュ・クエリ最適化は含む? | 含まない(本 Task のスコープ外)。重複した集計の Service 集約のみ |
| 集約した Service のテストはどう書く? | 各階層(Section / Chapter / Part / 資格)の完了数・完了率を、完了 0 件 / 全完了 / 一部完了の境界で検証する単体テストを用意する |
| 利用側(受講登録詳細 / ダッシュボード)の既存テストは触る必要は? | 既存の表示テストはそのまま pass させる(振る舞い不変の保証)。集約に伴う検証は Service の単体テストで担保する |
