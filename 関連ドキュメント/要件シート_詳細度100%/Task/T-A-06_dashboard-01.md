# T-A-06 管理者ダッシュボード集計のキャッシュ化

## 概要

管理者(admin)のダッシュボードは全体 KPI(受講中 / 合格 / 不合格の件数)と資格別の修了率を表示するが、これらは全受講登録を毎回走査して集計するため、受講登録・資格が増えるほど表示のたびに重いクエリが走る。管理者は運用モニタリングで頻繁にダッシュボードを開くため、同じ重い集計が短時間に何度も再計算され、DB 負荷と表示遅延の原因になる。

本チケットでは、この集計結果を一定時間キャッシュして連続表示での再計算を避ける。あわせて、受講状態が変わった時にはキャッシュを無効化し、古い集計値が表示され続けないようにする。「重い集計をどこでキャッシュするか」「いつ無効化するか(時間経過か、状態変化か)」を判断する題材。

## 要件

- 管理者ダッシュボードの全体 KPI と資格別修了率の集計結果を一定時間キャッシュし、短時間の連続表示で同じ集計を再実行しない
- 受講状態の遷移(合格 / 不合格 / 退会 等)が起きた後にダッシュボードを開くと、キャッシュが無効化され最新の集計値に更新される
- キャッシュの保存時間(TTL)は設定値で調整できる

## スコープ外

- 受講生 / コーチダッシュボードのキャッシュ化 — 本チケットは管理者ダッシュボードの集計のみ
- 新規受講登録 / 資格の公開・非公開での即時無効化 — これらは保存時間の失効による反映で許容し、即時無効化は受講状態の遷移のみ
- 外部キャッシュストア(Redis / Memcached 等)の導入 — 既定のキャッシュ保存先で完結
- キャッシュのグループ一括無効化(タグ機能)— 既定の保存先が非対応のため使わない
- 集計クエリ自体の最適化(インデックス追加 / クエリ書き換え)— 本チケットは再計算の回避(キャッシュ)が対象

## 受け入れ条件

- [ ] 管理者ダッシュボードの全体 KPI / 資格別修了率が一定時間キャッシュされ、短時間に連続表示しても重い集計クエリが再実行されない
  - 確認方法（テスト）: 同梱の `tests/Feature/UseCases/Dashboard/AdminDashboardCacheTest.php::test_admin_kpi_is_served_from_cache_on_second_fetch`(キャッシュ未実装では失敗)が pass する
- [ ] 受講状態の遷移後にダッシュボードを開くと、キャッシュが無効化され最新の集計値が表示される(古い件数が残らない)
  - 確認方法（テスト）: 同梱の `tests/Feature/UseCases/Dashboard/AdminDashboardCacheTest.php::test_cache_is_invalidated_on_enrollment_status_change`(キャッシュを入れても無効化を実装しないと失敗)が pass する

## 実装方針(参考)

### 変更内容

- **変更対象ファイル**:
  - `app/Services/EnrollmentStatsService.php`(全体 KPI / 資格別修了率の集計を所有)— `adminKpi()` / `completionRateByCertification()` の集計結果を `Cache::remember` でキャッシュ
  - `app/Services/EnrollmentStatusChangeService.php`(受講状態遷移の記録)— 状態遷移の記録時に集計キャッシュを `Cache::forget` で無効化
  - `config/dashboard.php`(新規)— 集計キャッシュのキーと TTL の設定
- **変更前**: 管理者ダッシュボードを開くたびに、全体 KPI(`status` 別 `GROUP BY`)と資格別修了率(`certifications` JOIN + `GROUP BY`)を毎回集計。表示回数分だけ重いクエリが走る
- **変更後**: 集計を所有する `EnrollmentStatsService` が `Cache::remember(key, ttl, ...)` で結果をキャッシュし、保存時間内の連続表示はキャッシュから返す。受講状態が遷移する `EnrollmentStatusChangeService::recordStatusChange()` で該当キーを `Cache::forget` し、合格・不合格・退会などで集計が変わった時は次回表示で再計算させる
- **採用技術と判断理由(Why)**:
  - **キャッシュは集約 Action ではなく集計を所有する Service 層に置く**: ダッシュボードの集約処理は薄い組み立て役に保ち、「重い計算とそのキャッシュ」は計算の所有者(`EnrollmentStatsService`)に凝集させる。表示経路が増えてもキャッシュ判断が 1 箇所に収まり、集約処理が肥大化しない
  - **無効化は「保存時間(TTL)＋状態遷移時の明示無効化」のハイブリッド**: 既定のキャッシュ保存先(file / array)はタグ非対応のためキーを直接 `forget` する。集計に最も効く変化(受講状態の遷移 = 件数・合格率が変わる)は遷移チョークポイントで即時無効化し、影響の小さい変化(新規受講登録 / 資格公開)は保存時間の失効で取り込む。管理者 KPI は数分の鮮度ずれを許容できるため、全書き込み経路に無効化を張り巡らせるよりシンプルで堅牢
- **テスト観点**: キャッシュヒット(2 回目の表示前にデータを直接増やしても件数が変わらない)と、状態遷移後の無効化(最新件数に更新される)を Feature テストで検証する。保存時間の数値そのものは検証しない(設定値)

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| キャッシュの保存時間は? | 設定値(既定 5 分)。`config/dashboard.php` の TTL 設定で調整できる |
| なぜ集約処理(Action)ではなく Service にキャッシュを置く? | 重い集計を所有するのは集計 Service。ダッシュボードの集約役は薄く保ち、計算とキャッシュを同じ層に凝集させると、表示経路が増えてもキャッシュ判断が分散しない |
| 新規受講登録した直後は KPI に反映される? | 即時ではない。受講状態の遷移(合格 / 不合格 / 退会)では即時無効化されるが、新規登録や資格公開は保存時間の失効(既定 5 分)で反映される。管理者 KPI は数分の鮮度ずれを許容する |
| 受講生 / コーチのダッシュボードもキャッシュする? | 本チケットの対象外。管理者ダッシュボードの全体集計のみ |
| キャッシュをタグでまとめて無効化しないのはなぜ? | 既定のキャッシュ保存先(file / array)がタグ非対応のため。該当キーを直接無効化する |
