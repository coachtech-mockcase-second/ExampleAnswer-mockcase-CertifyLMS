# B-B-05 在籍中ユーザーと同じメールアドレスで重複招待ができてしまう

| 難易度 | Basic |

## 概要

管理者(admin)が、すでに在籍している(受講中 / 卒業の)ユーザーと同じメールアドレスに対して新規招待を発行できてしまう。本来は在籍中ユーザーと同一メールアドレスへの招待は弾かれるべきだが、重複招待が成立し、招待メールも送られてしまう。

## 再現手順

**前提**: 管理者でログイン済み。あるメールアドレスで在籍中(受講中 または 卒業)のユーザーが既に存在する。

1. ユーザー招待フォームで、その在籍中ユーザーと同じメールアドレスを指定して招待を発行する
2. → エラーにならず招待が発行され、招待メールが送信される
3. 同一メールアドレスに対して在籍中ユーザーと招待が二重に存在する状態になる

## 期待する結果

在籍中(受講中 / 卒業)のユーザーと同じメールアドレスへ招待を発行しようとすると、重複として弾かれ(409)、招待は作成されず、エラーが表示される。

## 実際の結果

在籍中ユーザーと同じメールアドレスでも招待が成立し、重複した招待レコードの作成と招待メール送信が行われてしまう。

## 受け入れ条件

- [ ] 在籍中(受講中 / 卒業)のユーザーと同じメールアドレスで招待を発行しようとすると 409 が返り、招待レコードは作成されず招待メールも送信されない
- [ ] 在籍ユーザーが存在しない新規メールアドレスへの招待は従来どおり成功する(正常系が維持されている)
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### 原因

- **主要ファイル**: `app/UseCases/Auth/IssueInvitationAction.php`(招待発行)。※ Action 内のため Basic 範囲外。入力検証側 / Controller 内完結で実装する受講生はそれぞれの実装箇所が対象
- **仕込み内容**: 招待発行の入力検証(`Invitation\StoreRequest`)にはメール重複の `unique` ルールが無く、重複チェックは Action 側ガードが唯一。本来は「在籍中(受講中 / 卒業)の同一メールユーザーが存在すれば `EmailAlreadyRegisteredException`(409)を投げる」ガードがあるが、それが削除されているため重複招待が素通りし、招待メールも送られる。招待中(`invited`)の同一メールユーザーは再利用される設計(再発行 UX)なので重複対象ではない
- **修正範囲**: 「在籍中ユーザーと同一メールアドレスなら 409」の重複チェックを戻す(Action 内ガード、または入力検証での重複検査)。`InvitationController::store` → `Invitation\StoreAction`(ラッパー)→ `Auth\IssueInvitationAction` を辿り、在籍中ユーザー重複チェックが抜けていることを発見する流れ

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 重複と見なす対象の状態は? | 在籍中(受講中 / 卒業)のユーザー。これらと同一メールアドレスへの招待は弾く |
| 招待中(invited)の同一メールユーザーがいる場合は? | 重複エラーにはしない。招待の再発行(既存の招待中ユーザーを再利用 / 旧招待の取り消し → 再発行)の動線として扱う(本 Bug の対象外) |
| 退会済みユーザーと同一メールアドレスの場合は? | 退会時にメールアドレスがリネームされるため、退会前のメールアドレスは再利用可能(重複対象にならない) |
| 重複時の HTTP ステータスは 409 / 422 のどちらか? | 409(競合)。入力形式の不備ではなく、状態の競合として扱う |
| 重複時に招待メールは送られるか? | 送られない。重複チェックで弾かれた時点で招待作成・メール送信ともに行われない |
