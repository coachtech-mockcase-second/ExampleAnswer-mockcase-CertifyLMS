# B-A-01 面談予約で同一コーチ・同一時刻枠が二重予約される

## 概要

複数の受講生がほぼ同時に同じ時刻枠の面談を予約し、同じ担当コーチが割り当たった場合、本来 1 件しか成立しないはずの予約が 2 件とも成立し、そのコーチが同時刻に二重予約(ダブルブッキング)される。コーチは同じ時間に複数の面談を抱えることになり、受講生の一方は予約したはずの面談を受けられない。

## 再現手順

**前提**: 2 名の受講生(student)がそれぞれ残面談回数を持ち、同じ資格を受講中。その資格の担当コーチがある時刻枠に面談可能で、両受講生の予約画面にその枠が「空き」として表示されている。

1. 2 名の受講生が、同じ時刻枠の予約をほぼ同時に確定する(逐次に操作する限り再現せず、並行 HTTP クライアント / 並行テストで再現する)
2. → 両方のリクエストが「面談を予約しました」と完了表示される
3. コーチの面談一覧を開くと、同じ時刻に 2 件の面談が入っている(同一コーチ・同一時刻のダブルブッキング)

## 期待する結果

同じコーチの同じ時刻枠には、並行予約でも 1 件しか成立しない。2 件目は 409(空きコーチなしエラー)で拒否され、同一コーチが同時刻に二重予約されることはない。

## 実際の結果

予約候補のコーチを選ぶ際の「予約済みコーチを除外する判定」は、すでに確定(コミット)済みの予約しか検知できない。2 つの並行リクエストが互いの確定前に同じ「空き」を読むと、両方ともこの除外をすり抜ける。さらにデータベース側に「同じコーチ・同じ時刻は 1 件のみ」を保証する仕組みがないため、両方の予約がそのまま登録され、二重予約が成立する。

## 受け入れ条件

- [ ] 同じコーチの同じ時刻枠に複数の予約がほぼ同時に走っても、予約は 1 件しか成立しない。2 件目以降は「空きコーチなし」として 409 で拒否され、同一コーチが同時刻に二重予約されない
  - 確認方法（テスト）: 同梱の `tests/Feature/Http/Meeting/MeetingControllerTest.php::test_store_blocks_double_booking_for_same_coach_and_slot`(修正前は失敗)が pass する
- [ ] 単発の正常な予約・残数 0 での予約拒否・キャンセル → 返却 などの単発系の挙動は従来どおり維持されている

## 実装方針(参考)

### 原因

- **主要ファイル**: `database/migrations/2026_05_25_000000_create_meetings_table.php`(面談テーブル定義)。面談テーブルに「同じコーチ・同じ時刻の面談は 1 件まで」を保証する一意制約 (`(coach_id, scheduled_at)` の UNIQUE) が無く、これが二重予約を許す根本原因。
- **関連ファイル**: `app/UseCases/Meeting/StoreAction.php`(面談予約の総合処理)。予約候補コーチを絞る `findAvailableCoaches` の「予約済みコーチ除外」判定 (`whereDoesntHave('meetingsAsCoach', ...)`) は、すでに確定済みの予約しか見ないため、互いの確定前に同じ空きを読む並行リクエスト同士の競合は防げない。面談登録 (`Meeting::create`) を一意制約違反 (`UniqueConstraintViolationException`) として捕捉し 409 (`MeetingNoAvailableCoachException`) へ変換する箇所もあるが、一意制約自体が無いと発火しない。
- **修正範囲**: 面談テーブルに「コーチ × 時刻」の一意制約を追加し、`StoreAction` の面談登録を一意制約違反で捕捉して 409 に変換する(既にある変換処理が活きる)。並行リクエストを直列化する排他ロック (`lockForUpdate`) で実装してもよいが、一意制約のほうが素直。

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| なぜ普通のクリック操作では再現しにくい? | 逐次に予約する限り、先の予約が確定(コミット)してから後の予約が候補コーチを読むため、予約済みコーチが除外されて二重予約にならない。二重予約は「2 つの予約が互いの確定前に同じ空きを読む」ほぼ同時のリクエストでのみ起きる。再現は並行 HTTP クライアント(`curl` 並走 等)/ 並行テストで行う |
| 予約済みコーチの除外判定があるのに、なぜ一意制約も要る? | 除外判定はアプリ側で「読んだ時点で予約済みか」を見るだけなので、2 つの並行リクエストが互いの確定前に同じ「空き」を読むと両方すり抜ける。データベースの一意制約は予約の状態(予約済み / キャンセル済み 等)を問わず「同じコーチ・同じ時刻は物理的に 1 件まで」を保証する最終防衛線で、これが無いと並行時の二重予約を止められない |
| 修正は一意制約と排他ロックのどちらが正解? | 受け入れ条件(同一コーチ・同一時刻の二重予約が起きない)を満たせばどちらでも可。一意制約(コーチ × 時刻)が素直で推奨。排他ロックで予約処理を直列化する方法でも防げる |
| この二重予約を自動テストでどう再現する? | 同じコーチ・同じ時刻で「キャンセル済み」の面談を 1 件あらかじめ作っておき、その枠に予約を実行する。キャンセル済みは候補コーチの除外判定をすり抜ける一方、一意制約は状態を問わず効くため、「一意制約があれば 409 / 無ければ二重登録」という形で並行を起こさずに競合を決定論的に再現・確認できる |
| 二重予約された状態は画面で見える? | 見える。コーチの面談一覧に同じ時刻の面談が 2 件並ぶ。ただしバグの再現自体はほぼ同時の並行リクエストが要るため、動作確認はテスト(または並行スクリプト)で担保する |
| 二重予約を拒否したときのエラー文言は? | 「指定された時刻には空きコーチがいません。別の時刻をお選びください。」等、空きコーチなしと同じ案内(409)。先に予約が成立した側はそのまま残り、後から来た側がこの案内を受け取る(適切な日本語であれば文言の細部は採点対象外) |
| 残面談回数の会計(残数がマイナスになる等)はこのチケットの対象? | 対象外。本チケットは「同一コーチ・同一時刻の枠の二重予約」を防ぐことに限る。残数管理は別の仕組みで担保されている |
