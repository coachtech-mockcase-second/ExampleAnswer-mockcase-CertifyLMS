## 概要

招待メールに含まれる招待 URL(トークン)が、一度オンボーディングを完了した後も使い回せてしまう。本来は一度使った招待 URL は使用済みになって再利用できないはずだが、同じ URL から何度でもオンボーディングをやり直せる(名前・パスワードの再設定が繰り返せる)状態になっている。

## 再現手順

**前提**: 有効な招待 URL を 1 通受け取っている(招待中ユーザー)。

1. 招待 URL を開き、オンボーディング(名前・パスワード設定等)を完了する
2. 同じ招待 URL をもう一度開く
3. → 使用済みエラーにならず、オンボーディング画面が再び開けてしまい、再度登録内容を上書きできる

## 期待する動作

一度オンボーディングを完了した招待 URL に再アクセスすると、使用済みとして拒否され(410)、オンボーディングを再実行できない。

## 実際の動作

オンボーディング完了後も招待が「未使用」の状態のまま残るため、同じ URL から何度でもオンボーディングが可能で、登録内容を繰り返し上書きできてしまう。

## 受け入れ条件

- [ ] オンボーディングを完了した招待 URL に再アクセスすると 410 が返り、オンボーディングを再実行できない
- [ ] 未使用の有効な招待 URL からのオンボーディングは従来どおり完了できる(正常系が維持されている)

## 実装方針

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の調査・修正方針を組み立てる。

### 主要 URL

| メソッド | パス | 振る舞い |
|---|---|---|
| GET | `/onboarding/{invitation}` | オンボーディング画面表示(署名付き URL) |
| POST | `/onboarding/{invitation}` | オンボーディング完了(名前・パスワード等の確定) |

### 原因箇所メモ

- 原因の主要ファイル: `app/UseCases/Auth/OnboardAction.php`(オンボーディング完了)
- 関連: `InvitationStatus`(`pending` / `accepted` / 等)。`App\Exceptions\Auth\InvalidInvitationTokenException`(410)。OnboardAction 冒頭に「招待が未使用(`pending`)状態か / 期限内か」の検証ガードがあり、完了処理で招待を `accepted` に更新して使用済み化する設計
- 仕込み内容(Step 4 引き算): 完了処理内の「招待を使用済み(`accepted` + 受諾日時)に更新する」ブロックを削除する。これにより招待が `pending` のまま残り、冒頭ガードを毎回通過して再オンボードが可能になる
- 受講生が辿るべき修正範囲: オンボーディング完了時に招待を使用済み(受諾済み)状態へ更新する処理を戻す。`OnboardingController::store` から `Auth\OnboardAction` を辿り、完了後に招待状態が更新されていない(使い回せる)ことを発見する流れ

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 使用済み招待 URL に再アクセスしたときの HTTP ステータスは? | 410(Gone)。一度きりのトークンが消費済みであることを示す |
| 招待 URL には有効期限もあるのか? | ある(署名付き URL + 招待の有効期限)。本 Bug の対象は「期限内でも完了後に使い回せる」点(使用済み化漏れ)であり、期限切れ判定とは別 |
| オンボーディング完了後、ユーザーはどうなるか? | 自動的にログインしてダッシュボードへ。完了した招待は使用済みになり再利用不可 |
| 同じメールアドレスに再招待したい場合は? | 管理者が改めて招待を発行する(新しい招待 URL が発行される)。既存の使用済み URL は復活しない |
