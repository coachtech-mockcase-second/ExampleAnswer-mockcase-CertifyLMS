## 概要

管理者(admin)がユーザー管理の一覧を開くと、退会済みのユーザーが通常の一覧にも混ざって表示される。本来、退会済みユーザーは退会フィルタを指定したときだけ表示されるべきだが、フィルタなし・在籍中フィルタなどでも退会済みユーザーが一覧に出てしまう。

## 再現手順

**前提**: 管理者でログイン済み。退会済みのユーザーが 1 件以上存在する(初期データに退会済み受講生が含まれる)。

1. ユーザー管理の一覧画面を、状態フィルタなし(または「受講中」フィルタ)で開く
2. → 在籍中ユーザーに混じって退会済みユーザーが一覧に表示される
3. 「退会済」状態でフィルタしたときは退会済みユーザーが表示される(この挙動は正しい)

## 期待する結果

状態フィルタが「退会済」のときだけ退会済みユーザーが一覧に含まれる。フィルタなし・在籍中などのフィルタでは、退会済みユーザーは一覧に表示されない。

## 実際の結果

状態フィルタの指定に関わらず退会済みユーザーが常に一覧の対象になり、通常の一覧(フィルタなし / 在籍中)にも退会済みユーザーが混入する。

## 受け入れ条件

- [ ] 状態フィルタなし、および在籍系の状態(受講中 / 卒業)で絞り込んだとき、退会済みユーザーが一覧に表示されない
- [ ] 状態フィルタを「退会済」にしたときのみ退会済みユーザーが一覧に表示される(正常系が維持されている)
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

> **粒度**: 業務語彙で原因を要約 + 主要ファイルパス・クラス名・メソッド名を併記。修正コード片 / SQL 詳細は書かない(コードリーディングは受講生の作業)。

### 原因

- **主要ファイル**: `app/UseCases/User/IndexAction.php`(管理者ユーザー一覧の取得クエリ)。※ Action 内のため Basic 範囲外。Controller 内完結で実装する受講生は `UserController::index` の一覧取得処理が対象
- **仕込み内容**: `User` は SoftDelete 採用で、退会済みユーザーは退会状態(`UserStatus::Withdrawn`)+ 論理削除済(`deleted_at` セット)のため通常クエリではグローバルスコープにより除外される。本来は「状態フィルタが退会のときだけ `withTrashed()`(論理削除済みを含める)を適用する」条件分岐だが、その条件が外れて `withTrashed()` が一覧取得の冒頭で常時呼ばれており、全フィルタで退会済みユーザーが混入する
- **修正範囲**: `withTrashed()` を「退会フィルタが指定されたときのみ」適用する条件分岐に戻す。`UserController::index` から `User\IndexAction` を辿り、`withTrashed()` が無条件に呼ばれていることを発見する流れ。一覧取得冒頭の 1 箇所の修正で完結

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 退会済みユーザーは論理削除 / 物理削除のどちらか? | 論理削除(SoftDelete)。退会時に退会状態 + 論理削除され、過去の受講履歴の参照整合を保つ |
| 退会済みユーザーはどの画面で見られるべきか? | ユーザー管理一覧で状態フィルタを「退会済」にしたときのみ。通常の一覧には出さない |
| 退会済みユーザーの詳細画面は開けるべきか? | 開ける(詳細は論理削除済みも取得対象)。本 Bug の対象は「一覧への混入」であり、詳細閲覧の可否は別論点 |
| 在籍中(受講中 / 卒業)フィルタのときの挙動は? | 退会済みは一覧に含めない。在籍系フィルタは論理削除されていないユーザーのみ対象 |
