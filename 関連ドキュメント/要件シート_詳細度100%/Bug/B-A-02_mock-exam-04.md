# B-A-02 模試の採点結果が常に不合格になる

## 概要

受講生(student)が模試を提出すると、本来であれば合格点を超える解答(例: 5 問中 4 問正解 = 80%)でも、採点結果が **常に不合格扱い** になる。得点率の計算で百分率への変換が抜けているため、`0.80` のような小数値で保存されてしまい、合格判定(得点率 ≧ 合格基準点)が常に false になる。

## 再現手順

**前提**: 受講生でログイン済み、受講中の資格に紐づく公開模試(合格基準点 60% / 問題数 5 問を想定)を 1 つ受験開始済み。

1. 模試を受験中の状態にする(各問に解答)
2. 5 問中 4 問について正解の選択肢を選び、提出する
3. → 採点結果画面に遷移する
4. 採点結果を確認すると、得点率が `0.80`(または `0.80%`)と表示され、判定が **不合格** になっている
5. 全問正解(5 / 5)でも同様に小さな小数値となり、不合格判定になる

## 期待する結果

得点率は「正解数 ÷ 全問題数」の **百分率(0〜100 のスケール)** で算出され、小数第 2 位まで保持される。5 問中 4 問正解なら `80.00`、5 問中 5 問正解なら `100.00`、3 問正解なら `60.00`。合格基準点(0〜100 の整数)以上で合格、未満で不合格と判定される。

## 実際の結果

得点率が 0〜1 のスケール(`0.00` 〜 `1.00`)で保存されてしまい、0〜100 スケールの合格基準点と比較できない。全問正解(値としては `1.00`)でも `1.00 ≧ 60` は false となり、合格基準点が 0 でない限り常に不合格と判定される。採点結果画面の表示も `0.80` のような違和感のある値になる。

## 受け入れ条件

- [ ] 【受講生】得点率の百分率保存と合否判定
  1. 模試提出時の得点率が 0〜100 の百分率で保存される
  2. 合格基準点以上で合格・未満で不合格と判定される
  3. 全問正解は `100.00` で合格、合格基準点ぴったりも合格、全問不正解・未解答は `0.00` で不合格
- [ ] 【受講生】集計表示での得点率スケール一貫性
  1. 受験履歴一覧・模試試験結果画面・弱点分析ヒートマップ等の集計表示で、得点率が 0〜100 スケールで一貫表示される(0〜1 の小数として誤表示されない)
- [ ] 【受講生】採点結果画面の表示
  1. 模試を提出すると採点結果画面に遷移する
  2. 各問題の正答 / 誤答 / 未解答が一覧表示される
- [ ] 同梱 `tests/Feature/UseCases/MockExamSession/GradeActionTest.php::test_grades_session_with_mixed_correct_and_wrong_answers`(修正前は赤)が pass する

## 実装方針(参考)

### 原因

- **主要ファイル**: `app/UseCases/MockExamSession/GradeAction.php`(採点ロジック本体)。呼出経路は受験画面の「提出する」ボタン → `MockExamSessionController::submit` → `MockExamSession\SubmitAction`(提出 + 採点 + ターム再判定を 1 トランザクションで束ねる)→ `GradeAction`
- **仕込み内容**: 得点率の算出で百分率への変換が欠落しており、正答率が 0〜1 のスケール(`0.80` 等)で保存される。合格判定は得点率と合格基準点スナップショット(`MockExamSession.passing_score_snapshot`、0〜100 の整数)を比較するため、スケール不一致で常に不合格側に倒れる。波及として、得点率を 0〜100 スケール前提で利用する弱点ヒートマップ / 合格可能性スコア(`WeaknessAnalysisService`)の数値も同時にずれる
- **修正範囲**: `GradeAction` 内の得点率算出を百分率スケールに直すだけで完結する。スキーマ(`score_percentage` カラムは `decimal(5,2)` で `100.00` まで表現可能)は変更不要。修正時は弱点ヒートマップ / 合格可能性スコアが 0〜100 スケールを前提にしている点も含めて整合を点検する

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 得点率はどのスケール(0〜1 か 0〜100 か)で保存しますか? | 0〜100 の百分率スケール。`80%` なら `80.00` を保存する。合格基準点が 0〜100 の整数なので、それと直接比較できるスケールに揃える |
| 小数の精度はどこまで保持しますか? | 小数第 2 位まで。`100.00` まで格納し、四捨五入で丸める |
| 0 問の模試が来た場合の挙動は? | ゼロ除算を防ぐため得点率を `0.00`、合否を不合格として返す。模試には通常 1 問以上ある前提だが防御的に残す |
| 合格基準点ぴったりの得点率は合格 / 不合格どちら? | 合格(以上判定)。「合格基準点 60% で 60% ぴったり = 合格」とする運用 |
| 既に採点済みの過去セッションの得点率はどうしますか? | 本チケットの責務範囲は新規提出時の採点ロジック修正のみ。既存の壊れたデータ(`0.80` 等)は初期データ再投入(`migrate:fresh --seed`)で復旧する範囲を想定。本番運用想定のデータ補正バッチは別チケット扱い |
| 弱点ヒートマップ / 合格可能性スコアも壊れていますか? | これらは得点率を 0〜100 スケールで利用するため、本 Bug の影響を間接的に受ける。本 Bug を直すと自動的に正常化するので、別途修正は不要 |
| 提出 → 採点を別々のトランザクションに分けてもよいですか? | いいえ。提出(状態更新)・採点(得点率 / 合否更新)・ターム再判定を 1 トランザクションで束ねる既存の振る舞いを維持してください。途中失敗での部分書き込みを避ける目的 |
| 二重提出するとどうなりますか? | 受験中以外のセッションを提出しようとすると 409 で弾かれる。本 Bug の修正範囲ではこの提出ガードには触れないこと |
| 得点率の表示単位(`%` の有無)は? | 画面側で `%` を付ける運用。保存する値は数値だけ(`80.00`)で、`%` 文字は持たせない |
