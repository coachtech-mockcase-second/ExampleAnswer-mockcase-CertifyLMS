## 概要

受講生(student)が模試を提出すると、本来であれば合格点を超える解答(例: 5 問中 4 問正解 = 80%)でも、採点結果が **常に不合格扱い** になる。得点率の計算で **百分率への変換が抜けている** ため、`0.80` のような小数値で保存されてしまい、合格判定(`得点率 >= 合格点`)が常に false になる。

## 再現手順

**前提**: 受講生でログイン済み、受講中の資格に紐づく公開模試(合格点 60% 想定、問題数 5 問)を 1 つ受験開始済み。

1. 模試を受験中の状態にする(各問に解答)
2. 5 問中 4 問について正解の選択肢を選び、提出する
3. → 採点画面に遷移する
4. 採点結果を確認すると、得点率が **`0.80`(または `0.80%`)** と表示され、判定が **不合格** になっている
5. 全問正解(5 / 5)でも同様に小さな小数値となり、不合格判定になる

## 期待する動作

得点率は「正解数 ÷ 全問題数」の **百分率(0〜100 のスケール)** で算出され、小数第 2 位まで保持される。5 問中 4 問正解なら `80.00`、5 問中 5 問正解なら `100.00`、3 問正解なら `60.00`。合格点(合格基準点、0〜100 の整数)以上で合格、未満で不合格と判定される。

## 実際の動作

得点率が **0〜1 のスケール**(`0.00` 〜 `1.00`)で保存されてしまい、合格点(0〜100 のスケール)と比較できない。結果として全問正解(値としては `1.00`)でも `1.00 >= 60` は false となり、合格点が 0 でない限り常に不合格と判定される。受験画面の表示も `0.80` のような違和感のある値になる。

## 受け入れ条件

- [ ] 全問題に正解した場合、得点率が `100.00`(百分率スケール)で保存される
- [ ] 合格点を上回る正答率で提出すると合否判定が **合格** になり、合格点を下回ると **不合格** になる(境界として「合格点ぴったり」も合格扱い)
- [ ] 1 問も解答していない / 全問不正解の場合、得点率は `0.00`、判定は不合格になる
- [ ] 採点結果画面 / 受験履歴一覧 / 弱点分析ヒートマップ等の集計表示で、得点率が `0〜100` のスケールで一貫して表示される(0〜1 の小数として誤表示されない)
- [ ] 模試の状態遷移(受験中 → 提出済 → 採点完了)・問題ごとの正誤記録・ターム再判定などの周辺挙動は従来どおり維持されている

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の調査・修正方針を組み立てる。

| GET | `/mock-exam-sessions/{session}` | 採点結果表示(得点率・合否・弱点ヒートマップ)。再描画でバグの影響を視覚的に確認できる |

### 原因箇所メモ

- 原因の主要ファイル: `app/UseCases/MockExamSession/GradeAction.php`(採点ロジック本体)
- 関連:
  - `app/UseCases/MockExamSession/SubmitAction.php`: 提出 + 採点 + ターム再判定を 1 トランザクションでまとめる Action。`GradeAction` を内部で呼ぶ。`MockExamSession` の `lockForUpdate()` で二重提出を防止
  - `app/Models/MockExamSession`: `score_percentage` カラム(`decimal(5,2)`、cast: `decimal:2`)/ `passing_score_snapshot` カラム(`unsignedTinyInteger`、cast: `integer`、受験開始時に `MockExam.passing_score` のスナップショット)/ `pass` カラム(boolean)を保持
  - `app/Models/MockExamAnswer`: 各問の解答ログ。`selected_option_id` + `is_correct` を `GradeAction` が更新
  - `app/Models/MockExamQuestionOption`: 選択肢マスタ。`is_correct` boolean を持つ
  - 関連集計: `app/Services/WeaknessAnalysisService`(弱点ヒートマップ + 合格可能性スコア)は `score_percentage` を 0〜100 スケールで利用するため、本 Bug があると弱点分析側にも数値ズレが波及する
- 仕込み内容(Step 4 引き算): `GradeAction::__invoke` の得点率算出行から **百分率変換の `* 100`** を削除する。具体的には `round($totalCorrect / $totalQuestions * 100, 2)` を `round($totalCorrect / $totalQuestions, 2)` に書き換える(全問題数 0 件のガードはそのまま `0.0` を返す)。これにより `score_percentage` が `0.80` 等の小数値で保存され、合格判定 `$scorePercentage >= (float) $session->passing_score_snapshot` が常に false 側に倒れる
- 受講生が辿るべき修正範囲: 受験画面の「提出する」ボタンから採点までの経路(`MockExamSessionController::submit` → `MockExamSession\SubmitAction` → `MockExamSession\GradeAction`)を辿り、得点率の算出式で百分率変換が抜けている 1 行を特定して戻す。**スキーマ側(`score_percentage` カラム精度)は変更不要**(`decimal(5,2)` で `100.00` まで表現可能)。修正時は弱点ヒートマップ / 合格可能性スコアが 0〜100 スケールを前提にしている点も含めて整合性を点検する

### 採用技術と判断理由

- **採用技術**: PHP の `intdiv` ではなく **`round($numerator / $denominator * 100, 2)` で百分率 + 小数 2 桁丸め**。`decimal(5,2)` カラムに `0〜100.00` の範囲で格納
- **設計判断**:
  1. **百分率スケール(0〜100)で統一**: `MockExam.passing_score` / `MockExamSession.passing_score_snapshot` が 0〜100 の整数(合格点 = 「60 点 / 100 点」の意味)であり、`score_percentage` も同じスケールに揃える。0〜1 の小数スケールにすると弱点分析 / 合格可能性スコア / Blade 表示すべてに換算ロジックが必要になり、可読性とテスト容易性が下がる
  2. **`round(..., 2)` の採用**: 小数 2 桁で十分(`decimal:2` cast と整合)。`number_format` ではなく `round` を使うのは「文字列表現」ではなく「数値計算用」だから(平均算出 / 合否判定 / 弱点 70% 閾値計算で再利用)
  3. **0 問模試のガード**: `$totalQuestions > 0` の三項演算子で `0.0` を返す(ゼロ除算回避)。模試マスタには通常 1 問以上ある想定だが、防御的に残す
  4. **`passing_score_snapshot` でのスナップショット**: 採点時に `MockExam.passing_score` を直接参照せず、受験開始時に `MockExamSession.passing_score_snapshot` へ焼き込む既存設計を維持。マスタ側の合格点変更の影響を受けない冪等性が担保される
  5. **採点とターム再判定のトランザクション分離は不採用**: `SubmitAction` の `DB::transaction()` 内で「セッション状態更新 → `GradeAction` 採点 → `TermJudgementService::recalculate` 再判定」が原子的に走る既存設計を維持。採点だけ成功して合否が DB に乗らない / 合否は更新されたが受講生のターム判定が古いまま、といった不整合を避ける

### テスト方針

- **既存テスト**(`tests/Feature/UseCases/MockExamSession/GradeActionTest`)が `60% 合格 / 50% 不合格 / 33% 不合格` を `score_percentage` の値で assert しているため、本 Bug の仕込み(`* 100` 削除)を入れた直後はこのテストが落ちる。修正完了で再度通ることをもって退行 0 を確認できる
- **追加観点**(Bug 固有検証として残す):
  - 全問正解(`$totalCorrect === $totalQuestions`)で `score_percentage === 100.00` / `pass === true`
  - 0 問正解 で `score_percentage === 0.00` / `pass === false`
  - 合格点ぴったり(例: 合格点 60 / 5 問中 3 問正解 = 60.00)で `pass === true`
  - 合格点直下(例: 合格点 70 / 5 問中 3 問正解 = 60.00)で `pass === false`
- **動作確認動画**: 「合格圏内のセッションを提出 → 合格画面が表示される」「合格点未満のセッションを提出 → 不合格画面が表示される」の 2 ケースを画面録画として PR に添付
- **波及確認**: 弱点ヒートマップ / 合格可能性スコア(`WeaknessAnalysisService`)が 0〜100 スケールを前提にしているため、修正前は弱点判定 / 合格可能性バンドが歪む。修正後にダッシュボード / 模試結果画面で表示が正常化することをスクショ or 画面確認で点検

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 得点率はどのスケール(0〜1 か 0〜100 か)で保存しますか? | 0〜100 の百分率スケール。`80%` なら `80.00` を保存する。合格点(`passing_score_snapshot`)が `0〜100` の整数なので、それと直接比較できるスケールに揃える |
| 小数の精度はどこまで保持しますか? | 小数第 2 位まで。`decimal(5,2)` で `0.00 〜 100.00` を格納し、四捨五入で丸める |
| 0 問の模試が来た場合の挙動は? | ゼロ除算を防ぐため得点率を `0.00`、合否を `false` として返す。模試マスタには通常 1 問以上ある前提だが防御的に残す |
| 合格点ぴったりの得点率は合格 / 不合格どちら? | 合格(`>=` 判定)。「合格点 60% で 60% ぴったり = 合格」とする運用 |
| 既に Graded された過去セッションの得点率はどうしますか? | 本チケットの責務範囲は **新規提出時の採点ロジック修正のみ**。既存の壊れたデータ(`0.80` 等)は提供 PJ の Seeder と本修正 + `migrate:fresh --seed` で再投入される範囲で復旧する想定。本番運用想定でのデータ補正バッチは別チケット扱い |
| 弱点ヒートマップ / 合格可能性スコアも壊れていますか? | これらは `score_percentage` を 0〜100 スケールで利用するため、本 Bug の影響を間接的に受けます。本 Bug を直すと自動的に正常化するので、別途修正は不要 |
| 提出 → 採点を別トランザクションに分離してもよいですか? | いいえ。提出(`status` 更新)・採点(`score_percentage` / `pass` 更新)・ターム再判定(`Enrollment.current_term`)を 1 トランザクションで束ねる既存設計を維持してください。途中失敗での部分書き込みを避ける目的です |
| 二重提出の防止はどう実装されていますか? | `SubmitAction` 冒頭で対象セッションを `lockForUpdate()->findOrFail()` で取得し、`status === InProgress` 以外なら例外(`MockExamSessionNotInProgressException`、HTTP 409)を返します。本 Bug の修正範囲ではこのロジックには触らないでください |
| 得点率の表示単位(`%` の有無)は? | 表示側(Blade)で `%` を付ける運用。DB に保存する値は数値だけ(`80.00`)。`%` 文字を DB に持たないこと |
