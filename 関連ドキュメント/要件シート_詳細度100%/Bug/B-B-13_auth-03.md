# B-B-13 オンボーディングで確認用パスワードが一致しなくても登録できる

## メタ情報

| 項目 | 値 |
|---|---|
| チケット ID | `B-B-13` |
| Feature 連番 | `auth-03` |

| サブカテゴリ | データ(バリデーション) |

| 依存チケット | (なし) |

## 概要

招待を受けたユーザーがオンボーディングでパスワードを設定する際、パスワードと確認用パスワードが一致していなくても登録が通ってしまう。確認用入力の突き合わせが行われないため、入力ミス(typo)に気づけないまま意図しないパスワードが設定される。

## 再現手順

**前提**: 有効な招待 URL を受け取った招待中ユーザー。

1. オンボーディング画面で、名前と「パスワード」「確認用パスワード」を入力する
2. 「パスワード」と「確認用パスワード」にわざと異なる値を入力して送信する
3. → 入力エラーにならず、オンボーディングが完了してしまう

## 期待する結果

「パスワード」と「確認用パスワード」が一致しない場合は入力エラー(422)になり、オンボーディングが完了しない。両者が一致したときのみ登録が進む。

## 実際の結果

確認用パスワードとの一致チェックが行われず、両者が異なっていても登録が成立する。入力ミスを検出できないまま、利用者の意図しないパスワードが設定される。

## 受け入れ条件

- [ ] 「パスワード」と「確認用パスワード」が一致しない場合、422 が返りオンボーディングが完了しない
- [ ] 両者が一致している場合は従来どおりオンボーディングが完了する(正常系が維持されている)
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の調査・修正方針を組み立てる。

### 原因

- **主要ファイル**: `app/Http/Requests/Auth/OnboardingRequest.php`(オンボーディング入力検証 FormRequest)。確認用入力欄の表示名(「パスワード（確認）」)は `attributes()` に定義済み。
- **仕込み内容**: パスワードの検証ルールから「確認用入力との一致(`confirmed`)」ルールだけが抜けている(`required` / `min:8` 等は残る)。`confirmed` は送信値 `password` と `password_confirmation` を突き合わせるルールで、これが無いと不一致でも検証を通過する。一見動くが確認用の突き合わせだけが効かない。
- **修正範囲**: パスワードの入力検証ルールに「確認用入力との一致(`confirmed`)」を戻す。FormRequest の `rules()` 1 箇所の修正で完結(Basic 範囲内)。

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 不一致時の HTTP ステータスは? | 422(入力検証エラー)。入力画面にエラーメッセージを表示して登録を止める |
| パスワードの他の制約は? | 必須・最小文字数(8 文字以上)などの基本ルールは維持。本 Bug の対象は「確認用入力との一致」のみ |
| 確認用パスワードの項目名は? | 「パスワード（確認）」。エラーメッセージにこの表示名が使われる(文言の細部は採点対象外) |
| 一致チェックはどの層で行うべきか? | 標準は入力検証(FormRequest)の層で確認用一致(`confirmed`)ルールを使う。受講生が Controller で独自に突き合わせても振る舞いを満たせば可 |
