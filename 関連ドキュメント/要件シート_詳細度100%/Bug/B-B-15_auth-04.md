## 概要

コーチ(coach)が、本来は管理者(admin)だけがアクセスできる管理画面(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 等)に入れてしまう。管理者専用エリアのロールガードが緩んでおり、管理者向けの運営機能がコーチに露出している。

## 再現手順

**前提**: コーチでログイン済み。

1. コーチのまま、管理者専用の管理画面(例: ユーザー管理一覧)の URL にアクセスする
2. → 403 で弾かれず、管理者専用画面が表示される
3. 同様に他の管理者専用画面(招待管理 / プラン管理 / 資格マスタ管理 等)も開けてしまう
4. (対比)受講生(student)では同じ画面は 403 で弾かれる

## 期待する動作

管理者専用の管理画面には管理者だけがアクセスでき、コーチ・受講生がアクセスすると 403 が返る。

## 実際の動作

管理者専用エリアにコーチが入れてしまい、管理者向けの運営機能(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 等)がコーチに露出する。

## 受け入れ条件

- [ ] コーチが管理者専用の管理画面にアクセスすると 403 が返る
- [ ] 管理者は管理者専用の管理画面に従来どおりアクセスできる(正常系が維持されている)
- [ ] 受講生は従来どおり管理者専用の管理画面にアクセスすると 403 が返る

## 実装方針

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の調査・修正方針を組み立てる。

### 主要 URL

| メソッド | パス | 振る舞い |
|---|---|---|
| GET | `/admin/users` | ユーザー管理一覧(管理者専用。コーチ・受講生は 403 であるべき) |
| (同グループ) | `/admin/invitations` ほか、`/admin/plans` / `/admin/certifications` 等 | 管理者専用の運営機能(同じロールガードで保護) |

### 原因箇所メモ

- 原因の主要ファイル: `routes/web.php`(管理者専用ルート群のミドルウェア指定) + `app/Http/Middleware/EnsureUserRole.php`(ロール存在確認)
- 関連: 同ファイルには「admin 専用」(`role:admin`)と「admin + コーチ共有」(`role:admin,coach`、資格閲覧 / 教材管理用)のルート群が併存する。教材管理側でコーチが担当資格のみ操作できるのは Policy で別途絞り込む設計(B-B-01 参照)。本件はそれとは別で、管理者専用エリアそのものへのロールガード
- 仕込み内容(Step 4 引き算): 管理者専用ルート群のロール指定を `role:admin` から `role:admin,coach` に書き換える(隣接する共有ルート群からのコピペミスを模す)。これによりコーチが管理者専用エリアに入れてしまう(受講生は引き続き弾かれる)
- 受講生が辿るべき修正範囲: 管理者専用ルート群のロールガードを「管理者のみ」に戻す。コーチでは管理者画面に入れるが受講生では弾かれる、という挙動から、ルート群のロール指定にコーチが混入していることを発見する流れ。ロール存在確認はミドルウェア(`EnsureUserRole`)の役割で、教材管理の担当資格スコープ(Policy)とは層が異なる点を区別する

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 管理者専用エリアにコーチ / 受講生がアクセスしたときの HTTP ステータスは? | 403。ロールが満たないアクセスは Forbidden |
| 教材管理にコーチがアクセスできるのは矛盾しないか? | 矛盾しない。教材管理は「管理者 + コーチ共有」エリアで、コーチは担当資格のみ Policy で絞られる(B-B-01)。本件の対象は「管理者専用」エリア(ユーザー管理 / 招待管理 / プラン管理 等)で、コーチは入れてはいけない |
| ロール判定はどの層で行うべきか? | ルートに適用するロール存在確認ミドルウェアの層。「担当資格のみ」などリソース固有の絞り込みは Policy の層で、役割が分かれる |
| 受講生でも再現するか? | しない。仕込みは管理者専用ルート群に「コーチ」を混入させるもので、受講生は引き続き弾かれる。コーチでのみアクセスできてしまう |
