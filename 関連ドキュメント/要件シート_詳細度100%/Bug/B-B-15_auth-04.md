# B-B-15 コーチが管理者専用の管理画面にアクセスできてしまう(ロールガード漏れ)

## メタ情報

| 項目 | 値 |
|---|---|

| Feature 連番 | `auth-04` |
| Feature | auth(ロール認可ミドルウェア) |
| 種別 | Bug |
| サブカテゴリ | 認可・認証(Middleware チェック漏れ) |
| 難易度 | Basic |
| 工数 (h) | 3 |

## 概要

コーチ(coach)が、本来は管理者(admin)だけがアクセスできる管理画面(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 等)に入れてしまう。管理者専用エリアのロールガードが緩んでおり、管理者向けの運営機能がコーチに露出している。

## 再現手順

**前提**: コーチでログイン済み。

1. コーチのまま、管理者専用の管理画面(例: ユーザー管理一覧)の URL にアクセスする
2. → 403 で弾かれず、管理者専用画面が表示される
3. 同様に他の管理者専用画面(招待管理 / プラン管理 / 資格マスタ管理 等)も開けてしまう
4. (対比)受講生(student)では同じ画面は 403 で弾かれる

## 期待する結果

管理者専用の管理画面には管理者だけがアクセスでき、コーチ・受講生がアクセスすると 403 が返る。

## 実際の結果

管理者専用エリアにコーチが入れてしまい、管理者向けの運営機能(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 等)がコーチに露出する。

## 受け入れ条件

- [ ] コーチが管理者専用の管理画面(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 等)にアクセスすると 403 が返る
- [ ] 管理者は管理者専用の管理画面に従来どおりアクセスでき、受講生は従来どおり 403 が返る(正常系・既存挙動が維持されている)
- [ ] 本チケットの機能に対するテスト (Unit / Feature 等) が実装されている

## 実装方針(参考)

### 原因

- **主要ファイル**: `routes/web.php`(管理者専用ルート群のミドルウェア指定)+ `app/Http/Middleware/EnsureUserRole.php`(`role:...` で指定したロールにアクセスユーザーの role が含まれるかを検査し、未充足なら 403)。
- **仕込み内容**: 管理者専用ルート群(ユーザー管理 / 招待管理 / プラン管理 / 資格マスタ管理 / カテゴリ管理 / 面談パック管理 / お知らせ配信 等)のロール指定が `role:admin` から `role:admin,coach` に書き換わっている(隣接する「管理者 + コーチ共有」ルート群からのコピペミスを模す)。`EnsureUserRole` 自体は正しく、指定ロールにコーチが含まれるためコーチが通過する。受講生はどちらの指定にも含まれず引き続き 403。
- **修正範囲**: 管理者専用ルート群のロール指定を `role:admin`(管理者のみ)に戻す。`routes/web.php` の該当ルートグループ 1 箇所の修正で完結。「管理者 + コーチ共有」エリア(資格マスタ閲覧 / 教材管理。コーチは担当資格のみ Policy + scope で絞る)は別グループで、こちらは `role:admin,coach` のままが正しい。

## 補足

### 想定 Q&A

| 質問 | 回答 |
|---|---|
| 管理者専用エリアにコーチ / 受講生がアクセスしたときの HTTP ステータスは? | 403。ロールが満たないアクセスは Forbidden |
| 教材管理にコーチがアクセスできるのは矛盾しないか? | 矛盾しない。教材管理 / 資格マスタ閲覧は「管理者 + コーチ共有」エリアで、コーチは担当資格のみ Policy + scope で絞られる。本件の対象は「管理者専用」エリア(ユーザー管理 / 招待管理 / プラン管理 等)で、コーチは入れてはいけない |
| ロール判定はどの層で行うべきか? | ルートに適用するロール存在確認ミドルウェアの層。「担当資格のみ」などリソース固有の絞り込みは Policy の層で、役割が分かれる |
| 受講生でも再現するか? | しない。仕込みは管理者専用ルート群に「コーチ」を混入させるもので、受講生は引き続き弾かれる。コーチでのみアクセスできてしまう |
