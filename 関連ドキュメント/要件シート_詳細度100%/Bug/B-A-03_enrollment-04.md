## 概要

受講生(student)が模試の受験セッションを作成 → キャンセル(未開始のまま破棄)した場合、本来は学習タームが **基礎ターム** に戻るべきだが、誤って **実践ターム** のままになる。一度模試を作成・キャンセルしただけで、進行中の模試が無くなっても永遠に「実践ターム」と判定される(教習所メタファーの「教習生 → 卒業検定挑戦中」が戻らない)。

## 再現手順

**前提**: 受講生でログイン済み、受講中の資格に紐づく公開模試が 1 つ以上ある。受講登録の現在のタームは「基礎ターム」。

1. 模試一覧から任意の模試を選び「受験セッションを作成する」を実行する → タームが「実践ターム」に切り替わる(これは正常)
2. 作成された受験セッション(未開始)の詳細画面で「キャンセル」を実行する → セッションがキャンセル状態に遷移
3. ダッシュボード or 受講登録詳細を開いて「現在のターム」を確認する
4. → 進行中・提出済・採点済の模試が 1 件も存在しないにもかかわらず、ターム表示が **実践ターム** のままになっている

## 期待する動作

受講生が抱える受験セッションの状態が **受験中 / 提出済 / 採点完了** のいずれでもなくなった時点で(= 進行中の模試が 1 件も存在しない状態に戻った時点で)、学習タームは **基礎ターム** に再判定される。未開始のまま破棄したセッション(キャンセル状態)は「進行中の模試」として扱わない。

## 実際の動作

タームの再判定対象に **キャンセル状態のセッションも含まれている** ため、過去に 1 回でも模試セッションを作成・キャンセルした受講生は、進行中の模試が 1 件も無くても「実践ターム」と判定され続ける。基礎ターム表示への巻き戻しが永続的に起きない。

## 受け入れ条件

- [ ] 模試セッションを「作成 → キャンセル」したのち、進行中の模試が 1 件も無い状態では学習タームが **基礎ターム** に再判定される(受験セッションのキャンセル直後にダッシュボード / 受講登録詳細で確認できる)
- [ ] 受験中 / 提出済 / 採点完了 の模試セッションが 1 件でも残っている場合は、引き続き **実践ターム** のままになる(キャンセル済セッションとの混在ケースも含む)
- [ ] 模試セッションを「作成 → 受験開始(進行中に遷移)」した瞬間に学習タームが **実践ターム** に切り替わる(既存挙動が維持されている)
- [ ] 受講登録の現在のタームと再判定結果が一致する場合、不要な更新クエリが走らない(既存挙動が維持されている)

## 実装方針

> **参考設計の一例**。受け入れ条件を満たせれば実装手段は問わない。受講生は提供 PJ コード + ヒアリングで自分の調査・修正方針を組み立てる。

| POST | `/mock-exam-sessions/{session}/submit` | 受験中セッションを提出 + 採点 → タームを再判定するトリガ |

### 原因箇所メモ

- 原因の主要ファイル: `app/Services/TermJudgementService.php`(学習ターム判定の集約 Service、`recalculate(Enrollment): TermType`)
- 関連:
  - `app/Enums/TermType`: `BasicLearning`(基礎ターム、`basic_learning`)/ `MockPractice`(実践ターム、`mock_practice`)の 2 値 Enum。デフォルトは `basic_learning`
  - `app/Models/Enrollment`: `current_term` カラム(`string(30)` default `basic_learning`、`TermType::class` に cast)。タームの保持先
  - `app/Models/MockExamSession`: `status` カラム(`MockExamSessionStatus` cast、`not_started` / `in_progress` / `submitted` / `graded` / `canceled` の 5 値)
  - 呼出元: `app/UseCases/MockExamSession/StartAction.php`(NotStarted → InProgress)/ `SubmitAction.php`(InProgress → Submitted → Graded)/ `DestroyAction.php`(NotStarted → Canceled)が同一トランザクション内で `TermJudgementService::recalculate` を呼ぶ
- 仕込み内容(Step 4 引き算): `TermJudgementService::recalculate` 内の MockExamSession 抽出クエリ `whereIn('status', ['in_progress', 'submitted', 'graded'])` に **`'canceled'` を誤って追加** し `whereIn('status', ['in_progress', 'submitted', 'graded', 'canceled'])` にする。これにより一度キャンセルしたセッションを抱える受講生はキャンセル後も「進行中の模試あり」と判定され、基礎タームに戻らない
- 受講生が辿るべき修正範囲: 「キャンセル直後に確認するとタームが実践ターム表示のまま」をきっかけに `MockExamSessionController::destroy` → `MockExamSession\DestroyAction` → `TermJudgementService::recalculate` のフローを辿り、ターム判定の `whereIn(...)` に含めるべきでない状態(`canceled`)が混入していることを発見する流れ。「未開始(`not_started`)もターム判定対象外」「終端のうち `graded` は対象 / `canceled` は対象外」の境界整理が学習ポイント

### 採用技術と判断理由

- **採用技術**: Eloquent の `whereIn('status', [...])` で「ターム判定対象状態」を明示列挙し、`exists()` で 1 件でもあれば実践ターム / 0 件なら基礎ターム判定
- **設計判断**:
  1. **判定対象状態の集合**: 「受講生がいま実際に模試と向き合っている / 結果を保持している」状態 = `in_progress`(受験中) / `submitted`(提出済、採点待ち) / `graded`(採点完了) の 3 状態のみ。`not_started`(作成だけして未着手) と `canceled`(明示的に破棄)は **ターム判定対象外**
  2. **`graded` を含める理由**: 教習所メタファーでは「卒業検定挑戦中 = 受験前 / 受験中 / 受験後の結果確認まで」を「実践ターム」とみなす。採点完了後も再受験準備期間として実践タームに留める設計
  3. **`canceled` を含めない理由**: キャンセルは「実践タームに足を踏み入れたが破棄した」状態なので基礎タームへの巻き戻し対象。判定対象に含めると一度キャンセルした受講生が永久に実践タームになる(=本 Bug)
  4. **`not_started` を含めない理由**: セッション作成だけで実践タームに入れると「予約だけして放置」のユーザーが基礎ターム表示に戻れない。実際の「実践ターム入り」のトリガは `StartAction`(NotStarted → InProgress)に揃える
  5. **`exists()` を採用**: 件数取得(`count() > 0`)よりも `exists()` の方が SQL 最適化(LIMIT 1)で軽量。判定だけが目的なので件数は不要
  6. **クラス分割の余地はあるが採用しない**: 状態集合を Enum や定数に切り出す案もあるが、本 Service の中だけで使う 1 配列に対して過剰抽象になる。`status` Enum の `isActiveForTermJudgement(): bool` メソッドを生やす案も同様に「YAGNI」の範囲

### テスト方針

- **既存テスト**(`tests/Unit/Services/TermJudgementServiceTest`)が以下を網羅: セッション 0 件 → 基礎ターム / `not_started` のみ → 基礎ターム / `in_progress` あり → 実践ターム / `submitted` あり → 実践ターム / `graded` あり → 実践ターム / 同タームの場合に updated_at が更新されない。本 Bug の仕込みを入れると `not_started` テストは通るが、**`canceled` ケース** がカバーされていない(現行テスト未追加)
- **追加観点**(Bug 固有検証として残す):
  - `canceled` のみ存在 → **基礎ターム** に判定されること(本チケットで明示)
  - `in_progress` + `canceled` の混在 → **実践ターム**(進行中があれば従来どおり)
  - `graded` + `canceled` の混在 → **実践ターム**(graded があれば従来どおり)
  - 既に実践ターム保持 → 進行中なくなった瞬間に基礎タームに UPDATE される(`current_term` の DB 値が変わる)
- **動作確認動画**: 「作成 → キャンセル → ダッシュボードでターム表示が基礎タームに戻る」のフローを画面録画として PR に添付。修正前後で差分が明確になるよう、修正前(常に実践タームのまま)と修正後(基礎タームに戻る)の 2 ケース録画を併記
- **退行確認**: `MockExamSession\StartAction` / `SubmitAction` / `DestroyAction` 経由のターム再判定がそれぞれ正しい方向に動くこと(`Tests\Feature\Http\MockExamSession\LifecycleTest` 等)、`Enrollment.current_term` の cast / default 値 / Enum 整合性が崩れていないこと

## 補足

### 想定ヒアリング Q&A

| 質問 | 回答 |
|---|---|
| 「実践ターム」と判定する MockExamSession の状態は何ですか? | `in_progress`(受験中) / `submitted`(提出済) / `graded`(採点完了)の 3 状態。これらが 1 件でも存在すれば実践ターム |
| `not_started`(未開始) は実践タームに含めないのですか? | 含めません。実践ターム入りのトリガは「受験開始」(`StartAction` で NotStarted → InProgress) の瞬間。セッション作成だけで実践タームに入れると、作って放置しただけの受講生も基礎ターム表示に戻れなくなります |
| `canceled` を含めないのは何故ですか? | キャンセルは「実践タームに足を踏み入れたが破棄した」状態。判定対象に入れると一度キャンセルした受講生が永久に実践ターム扱いになります(=本 Bug)。教習所メタファーで「卒業検定の挑戦履歴を破棄したら、教習生(基礎ターム)に戻る」自然な振る舞い |
| `graded`(採点完了) を含めるのは何故ですか? | 教習所メタファーで「卒業検定挑戦中 ⊇ 結果確認」とみなし、採点完了後も再受験準備の期間として実践タームに留めます。受験履歴の有無を「いま実践フェーズか」の判定材料にする設計 |
| 一度実践タームに入った後、基礎タームに戻ることはありますか? | あります。進行中(`in_progress` / `submitted` / `graded`)の模試が 0 件になった瞬間に基礎ターム判定が走ります。例: 唯一の `graded` セッションが管理者により物理削除された場合、次のターム再判定タイミング(StartAction / SubmitAction / DestroyAction の起動時)で基礎タームに戻ります |
| ターム判定はどのタイミングで走りますか? | MockExamSession の状態変化を伴う Action(受験開始 / 提出 / キャンセル)から、状態変更と同一トランザクション内で自動呼び出しされます。受講生が手動でリフレッシュを要求する画面は存在しません |
| 受講生が複数の資格を受講している場合、片方の資格でのキャンセルが他資格のタームに影響しますか? | しません。タームは **受講登録(Enrollment)ごと** に独立して保持されます(`Enrollment.current_term`)。受講生 × 資格の組み合わせごとに別判定 |
| 状態判定対象を Enum メソッド(例: `MockExamSessionStatus::isActiveForTerm(): bool`)に切り出してもいいですか? | 振る舞いが受け入れ条件を満たせば実装手段は問いません。模範解答 PJ では Service 内に配列直書きで保持しており、ターム判定に閉じた表現で過剰抽象を避けています |
| 同じタームが連続して算出された場合、毎回 UPDATE が走るのは性能的に問題ありませんか? | 既存実装は「`current_term` と新判定が一致したら UPDATE をスキップ」する分岐を持っています。本 Bug の修正範囲ではこの分岐には触らないでください |
