# 30%要件・評価シート 配布運用ガイド

30%版要件シート（スプシ + チケット Docs）と評価シート（スプシ）の**修正・版管理・検証**の運用手順。セットアップとコマンド一覧は `README.md`、設計経緯は issue #67 を参照。

## 全体像（SSoT と派生ビュー）

```
リポジトリ md（SSoT・手編集はここだけ）
├ 要件シート_詳細度30%/{Story,Bug,Task}/*.md ──(sync_docs.py)──→ チケット Docs 41 件
├ 要件シート_詳細度30%/概要.md ────────┐
│                                      ├─(build_sheets.py)──→ 模擬案件_CertifyLMS_要件シート（詳細度30%）
│   （チケット Doc リンク = manifest）──┘        タブ: シート1 概要 / シート2 チケット一覧
└ 評価シート.md ──────────(build_sheets.py)──→ 模擬案件_CertifyLMS_評価シート
                                                 タブ: 総合評価 / Story / Bug / Task / 横断品質 / 誤答リスト
```

- **Doc・スプシは派生ビュー**。Google 上で直接編集しない（編集はすべて md → 同期ツールで反映）。
- `manifest.json` が Doc ID / スプシ ID / バージョン / 差し替え履歴を持つ。**スプシ・Doc の URL は再 build / push では変わらない**。

## LMS 複製の特性（この運用の前提）

- LMS はテスト開始時に**マスタスプシを生徒ごとに複製**する（スナップショット）。→ **マスタスプシの修正は、それ以降に開始した生徒にしか反映されない**。
- 一方、スプシ内の**チケット Doc リンクは複製後も同じ URL** を指す。→ **Doc の内容更新（push）は過去に開始した生徒にも反映される**。
- この非対称を使い分けるのが版管理の核:

| 修正の種類 | 手段 | 過去に開始した生徒 | 以降の生徒 |
|---|---|---|---|
| チケットの軽微修正 | `push`（同一 Doc の内容置換） | **反映される** | 反映される |
| チケットの実質変更 | `supersede`（新 Doc + 旧凍結） | 旧版のまま（旧 URL 生存） | 新版 |
| 概要・一覧・評価シートの修正 | `build`（スプシ再生成） | 反映されない | 反映される |

## チケット md を修正したとき（最頻の運用）

> Claude Code では **Skill `/ticket-doc-sync <ID>`** が以下（判定の提案 → 実行 → リンク同期 → 検証）を一括で行う。

**軽微 / 実質の判断基準**（issue #67 確定）:

| 判定 | 対象 | 手段 |
|---|---|---|
| **軽微** | 誤字・表記ゆれ・書式・ニュアンス調整（仕様・採点に影響しない） | `push` |
| **実質** | 要件・受け入れ条件相当の振る舞い・スコープ・インターフェース・**依存チケット・工数** の変更 | `supersede` |

**迷ったら supersede**（過去生徒の採点前提を壊さない側に倒す）。

```bash
cd 関連ドキュメント/スプシ配布/scripts

# 軽微: 同一 URL のまま内容置換
.venv/bin/python sync_docs.py push S-B-02
.venv/bin/python sync_docs.py verify --content

# 実質: 新 Doc + 旧 Doc を _archive へ凍結 + マスタスプシのリンク差し替え
.venv/bin/python sync_docs.py supersede S-B-02 --reason "受け入れ条件の変更: ..."
.venv/bin/python build_sheets.py links-sync
.venv/bin/python build_sheets.py verify
```

- 実質変更は従来どおり **100%版・評価シートとの派生整合**（CLAUDE.md 修正波及表）を同一作業単位で行う。評価シート.md が動いたら `build evaluation` も実行する。
- `supersede` 後の旧 Doc は `_archive/` で完全凍結（無改変・URL 生存）。差し替え理由は manifest の `history` に残る。

## 概要.md / 評価シート.md を修正したとき

```bash
.venv/bin/python build_sheets.py build requirement   # 概要.md・チケット一覧の変更
.venv/bin/python build_sheets.py build evaluation    # 評価シート.md の変更（md 内部検算 + スプシ検算つき）
```

- build は**タブ全面再生成**（スプシ上の手編集は消える。マスタを手編集しない理由）。URL・スプシ ID は不変。
- 評価シートの配点・行の変更は**過去に開始した生徒のシートに反映されない**。進行中の生徒に適用したい修正は採点者へ個別連絡する（例: 修正依頼スプシ経由）。

## チケットの追加・退役

- **追加**: 100% → 30% md を生成 → `push <新ID>` → 概要.md のチケット一覧・件数・工数を更新 → `build requirement`（+ 評価シート.md に AC を追加して `build evaluation`）。
- **退役**: 30% md を削除 → Doc をブラウザで `_archive/` へ移動 → manifest の該当エントリを削除 → 概要.md・評価シート.md を更新して両方 build。`verify` が orphan（manifest にあるが md がない）を検知する。

## 検証コマンド

| コマンド | 見るもの |
|---|---|
| `sync_docs.py verify` | ① md がすべて同期済みか（push/supersede 忘れ） |
| `sync_docs.py verify --content` | ① + ② Doc の手編集検知 + ③ Doc 内容と md の突合 |
| `build_sheets.py verify` | 評価スプシの集計値 ⇔ 評価シート.md 集計の検算 + チケット一覧のリンク一致 |

定期点検（採点者対応の後など）は `verify --content` + `build_sheets.py verify` の 2 本で足りる。

## マスタの配置・共有・LMS 登録

- マスタ 2 枚とチケット Docs フォルダは **雛形シートフォルダ**（既存コースのマスタ置き場）直下に置く。移動は `build_sheets.py place`（スコープ制約で失敗する場合はブラウザでドラッグ。**移動しても ID・URL は不変**）。
- スプシの共有は **`build_sheets.py share --role <reader|writer>`** で明示的に付与する（build は共有設定を変更しない）。既存コースのマスタ慣例は「リンクを知っている全員: 編集者」。Docs トップフォルダは「リンクを知っている全員: **閲覧者**」（init 時に付与済み。**生徒に書かせない**）。
- LMS へのマスタ登録・生徒複製の設定はスプシ URL で行う（manifest の `spreadsheets.*.url`）。

## トラブルシュート

- **verify ② が「手編集されています」**: Google 上で誰かが Doc を編集した合図。diff を `verify --content` で確認し、取り込むなら md に反映して push、破棄なら push で上書き。
- **links-sync が「スプシに行が無い」**: チケット追加後に `build requirement` を実行していない。
- **push/supersede 直後に ② が出る**: 旧バグ（変換後処理の modifiedTime 更新）は settle 待ちで解消済み。再発したら数秒おいて `push` を再実行すれば自己修復する。
- **token 失効（`invalid_grant`）**: GCP の OAuth 同意画面がテスト公開のままだとリフレッシュトークンが 7 日で失効する。`scripts/token.json` を削除して任意のコマンドを実行 → ブラウザで再認証。恒久対応は同意画面の公開ステータスを「本番」へ変更（drive.file は非機密スコープのため審査不要）。
