# 確定済みの意思決定

このファイルは、議論を経て確定した方針・決定事項を記録するものです。
ここに記載された内容は原則として覆さない前提で進行します。

---

## プロジェクト方針

- 既存の「SkillHub(LearnHub)」要件定義はあるが、方針は覆してよい
- 模擬案件としての目的達成を最優先で最適な形を設計する
- 本模擬案件は受講生の卒業判断基準として使用する
- 卒業面談で100問の口頭試問を別途実施するため、模擬案件側でのAI対策は不要

## 技術・環境

- 技術スタック・環境構築は模擬案件（4つ目のテスト）BookShelfと同じ（Laravel 10, PHP 8.2, MySQL 8.0, Docker/Sail, Tailwind CSS, Fortify）
- フロントエンド(Blade)は全て実装済みで提供
- バックエンドは一部実装済み / 一部バグあり / 一部未実装の状態で提供

## ボリューム

- 実装ボリュームはBookShelfの80%目標（コードリーディング時間を考慮）

## 基本/応用の切り分け

- 基本要件は教材（pj-ct-newtext）の習得範囲内で解けるもの
- 応用要件は教材範囲外OK
- Basicブランチ = 基本要件、Advanceブランチ = Basic + 応用要件

## 出題カテゴリ（5カテゴリ）

1. **バグフィックス**（壊れている → 直す）: 調査力・デバッグ力
2. **既存機能の改修**（動いている → 拡張する）: コードリーディング・既存コードとの整合性
3. **新機能開発**（存在しない → 作る）: 設計力・実装力
4. **リファクタリング**（動くが汚い → 改善する）: コード品質の判断力
5. **テスト追加**（テストがない → 書く）: 仕様理解力・品質意識

## 応用要件の内容

- Sanctum認証付きAPI開発を含む
- 最後のトリとしてGemini APIチャットボットを含む
- リファクタリング応用にパフォーマンスチューニング（インデックス、キャッシュ、スロークエリ）を含む

## 設計方針

### Bladeを読んで仕様を逆算する
- 受講生はBladeを読んで「バックエンドが何を返すべきか」を自分で把握する
- ルート、変数、リレーションをコードから読み取るスキルを鍛える

### コードリーディングの自然な組み込み
- コードリーディングを独立したチケットにはしない（実務的でないため）
- 各実装チケットの中に、既存コードを読んで仕様を把握するプロセスを自然に組み込む
- コードリーディングを前提とした機能開発チケットも設ける（お手本機能を参考に実装）

### 既存テストの提供
- 提供プロジェクトに既存テストを含める
- お手本機能のテストを提供し、テストの読み方を学ぶ
- バグフィックス時に「既存テストを壊さないこと」を条件にする
- 一部のバグは「テストが落ちている状態で提供し、テストを通す」形式にする

### 実務でよくあるバグを幅広く含める
- コードの実装ミスだけでなく、データ起因のバグ（0除算、null参照、特定条件でのみ発生）も含める
- シンプルなCRUDが動かないバグも入れる（入口として）

## 運用

- チケット単位でブランチを切りPRを出す
- 提供Bladeは coachtech-prepared-file Organizationに配置

## 要件書

- 100%要件書を先に作成し、そこから削って50%要件書を作成する
- 50%と100%の差分は「あなたのタスク」セクションとしてヒアリング項目を構造化する
- 作業中は docs/requirements/ に配置し、最終的に @関連ドキュメント/ に移動する

## ドキュメント管理

- スペック駆動で管理する
- ファイルは段階的に作成する（空ファイルを事前に作らない）

```
                                  ↓

                                  ↓
                    Phase 7 → Phase 8 (Advance確定)
                                  ↓
                    Phase 9 → Phase 10 → Phase 11 → Phase 12
```

| Phase | 入力 | 出力 |
|-------|------|------|

| 2 | specs | 仮PJコード |

| 4 | 仮PJ, tickets | 提供版コード, provided-state.md 確定 |
| 5 | 提供版, Basic ticket specs | Basic完全手順書 |

### フェーズ詳細

#### Phase 1: 設計
- **設計対象は完成版の全体像**（仮PJ・提供版はそのサブセット）
- テーマ決定 → `specs/overview.md`
- DB設計 → `specs/database.md`
- ルーティング設計 → `specs/routes.md`
- チケット大枠設計（ID・カテゴリ・タイトル・概要・Basic/Advance区分）
  - ※チケット詳細はPhase 3で、実コードを見てから書く（修正容易性のため）

#### Phase 2: 仮プロジェクト構築
- specsに基づいて全ページが正常に動作する素の状態を作る
- 含むもの: 全Blade、全ルート、モデル、マイグレーション、Seeder、認証、お手本機能
- 含まないもの: 受講生タスクの実装（新機能のルート定義はBlade表示用に必要）
- リファクタ対象のコード（foreachなど）はこの段階で自然に生まれる

#### Phase 3: チケット詳細設計
- 実際のコードベースを見ながら各チケットの詳細仕様を記述
- `specs/tickets/*.md` 作成
- `specs/provided-state.md` ドラフト

#### Phase 4: 提供版構築
- チケット仕様に基づいてバグ仕込み・スタブ化
  - バグ: 特定のコードを意図的に壊す（with()削除、バリデーション削除等）
  - 新機能: メソッドの中身をスタブに置換
  - リファクタ・テスト: そのまま（操作不要）
- `specs/provided-state.md` 確定

#### Phase 5: Basic完全手順書作成
- 提供版を出発点として、Basicチケットの全手順を記述
- カテゴリ別+順序付き構成
- 環境構築～Basicチケット実装をカバー

#### Phase 6: Basic完成版構築（通しプレイ①）
- Basic完全手順書に沿ってExampleAnswer上で実装
- 手順書の不備を修正しながら進行 → **Basicブランチ確定**
- 教材化メモを残す（ハマりポイント、補足すべき概念、受講生がつまずきそうな箇所）

#### Phase 7: Advance完全手順書作成
- Basic完成版をベースに、Advanceチケットの手順を追加
- Advance固有の実装手順を記述（Sanctum API、Gemini等）

#### Phase 8: Advance完成版構築（通しプレイ②）
- Advance完全手順書に沿ってBasic上に実装
- 手順書の不備を修正しながら進行 → **Advanceブランチ確定**
- 教材化メモを残す

#### Phase 9: 評価項目作成
- 全完成版コードとチケット仕様をもとに評価項目・評価シートを作成
- BookShelf参考: 大項目別の配点、詳細な評価基準

#### Phase 10: 要件書作成
- 完成版・提供版・評価項目を踏まえて100%要件書を作成
  - 環境構築手順、機能要件、基本設計、非機能要件、テスト要件を含む（BookShelf構造参考）
- 100%から削って50%要件書を作成
  - ヒアリングポイントを組み込み、「行間を読む」設計

#### Phase 11: 復習教材作成
- 確定済み完全手順書 + Phase 6/8の教材化メモをベースにチャプター化
- 各章: 学習目標 → エンジニアの思考プロセス → 実装 → How to（BookShelf構造参考）
- Basic用とAdvance用を作成

#### Phase 12: 最終通しプレイ
- AssignedProjectをclone、50%要件書に基づいて実装テスト
- 受講生体験の検証、ドキュメント調整

### 旧3ステップ構築手順との対応

旧ステップの「各ステップが1種類の操作に集中する」設計思想は維持:
- 旧ステップ1（仮PJ構築）→ Phase 2
- 旧ステップ2（破壊のみ）→ Phase 4
- 旧ステップ3（実装のみ）→ Phase 6 + Phase 8（Basic/Advanceの2段階に分離）

### Basic/Advance 2段階ループ

- BookShelfの実績（基本手順書→基本実装→応用手順書→応用実装）に基づく
- 手順書作成→通しプレイのペアを2回回すことで:
  - Basicを先に確定し、Advanceの土台を安定させる
  - 各段階で手順書が検証される

### ドキュメント作成タイミングの方針

各ドキュメントは最適なタイミングで作成:
- **仕様（specs）**: 大枠はコード前（Phase 1）、詳細はコード後（Phase 3）→ 修正容易性
- **手順書**: 提供版の後、実装の前（Phase 5/7）→ 実装が手順書の検証になる
- **評価項目**: 全実装後（Phase 9）→ 何を評価するか確定してから
- **要件書**: 評価項目の後（Phase 10）→ 評価と整合した要件に
- **復習教材**: 要件書の後（Phase 11）→ 全情報が揃った状態で最終品質。Phase 6/8の教材化メモで実装時の知見を保全

### リポジトリへの配置

- ExampleAnswer: Phase 2→4→6→8の全工程を行う。最終状態が完成版（模範解答）
- AssignedProject: Phase 4の提供版をコピー、git initで履歴リセット
- Preparedblade: 最後にExampleAnswerの resources/ をコピー

### 出戻り（修正）フロー

各フェーズで問題が発見された場合の影響範囲と対応方針。

#### 修正コスト指標

各フェーズの成果物に対する「修正すべきではない度」（= 修正時の波及コスト）:

| Phase | 成果物 | 修正コスト | 理由 |
|-------|--------|-----------|------|
| 1 | specs | ★☆☆☆☆ 低い | ドキュメントのみ、下流にまだコードがない |
| 2 | 仮PJコード | ★★☆☆☆ やや低い | コードはあるが、まだ破壊・手順書と連動していない |
| 3 | チケット詳細 | ★★☆☆☆ やや低い | ドキュメントのみ、提供版構築前なら安い |
| 4 | 提供版コード | ★★★☆☆ 中程度 | 手順書・完成版との整合が必要になり始める |
| 5 | Basic手順書 | ★★★☆☆ 中程度 | Phase 6（通しプレイ）で検証される前提なので修正は想定内 |
| 6 | Basic完成版 | ★★★★☆ 高い | Advanceの土台。修正するとPhase 7/8にも波及 |
| 7 | Advance手順書 | ★★★☆☆ 中程度 | Phase 8で検証される前提 |
| 8 | Advance完成版 | ★★★★★ 非常に高い | 全ドキュメント（評価・要件・教材）の基盤 |
| 9-11 | 各ドキュメント | ★★★★☆ 高い | 相互参照が多く、1つの修正が連鎖する |

#### 各フェーズの出戻りパターンと対応

**Phase 2（仮PJ構築）→ Phase 1（設計）への出戻り**
- 発生例: 実装してみたらDB設計やルーティングに不備が見つかった
- 対応: specsを修正して仮PJ構築を続行。コストは低い
- 修正範囲: specs + 仮PJコード

**Phase 3（チケット詳細）→ Phase 1-2 への出戻り**
- 発生例: チケット詳細を書く中で、仮PJの構造変更が必要と判明
- 対応: specs修正 → 仮PJコード修正 → チケット詳細続行
- 修正範囲: specs + 仮PJコード + チケット詳細（ドラフト段階）

**Phase 4（提供版）→ Phase 3 への出戻り**
- 発生例: バグ仕込みやスタブ化の際にチケット仕様の見直しが必要
- 対応: チケット詳細修正 → 提供版構築続行
- 修正範囲: tickets/*.md + 提供版コード

**Phase 6（Basic通しプレイ）→ Phase 4-5 への出戻り**
- 発生例: 手順通りに実装したが動かない / 提供版の状態が不適切
- 対応: 手順書修正は想定内（Phase 6の目的）。提供版の修正が必要な場合はPhase 4に戻る
- 修正範囲: 手順書（想定内） / 提供版コード + provided-state.md（要注意）
- ⚠️ 提供版修正が必要な場合はPhase 3のチケット仕様との整合も確認すること

**Phase 8（Advance通しプレイ）→ Phase 6-7 への出戻り**
- 発生例: Advance実装がBasic完成版と衝突 / Advance手順に不備
- 対応: Advance手順書修正は想定内。Basic完成版の修正が必要な場合は要注意
- 修正範囲: Advance手順書（想定内） / Basic完成版修正（高コスト: Phase 5手順書も連動修正）
- ⚠️ Basic完成版の修正が必要な場合は、Phase 5手順書・Phase 4提供版との整合も確認すること

**Phase 9-11（ドキュメント作成）→ Phase 8以前への出戻り**
- 発生例: 評価項目や要件書を書く中で、完成版の実装漏れや仕様不整合を発見
- 対応: 原則としてドキュメント側で吸収する。コードの修正は最小限に留める
- ⚠️ コード修正が不可避な場合: 完成版修正 → 手順書修正 → 評価項目修正 → 要件書修正の連鎖が発生
- 判断基準: 「受講生の学習体験に致命的な影響があるか」で修正要否を判断

**Phase 12（最終通しプレイ）→ 各フェーズへの出戻り**
- 発生例: 受講生体験として不適切な箇所を発見
- 対応: ドキュメント調整で吸収できる範囲に留める
- ⚠️ コード修正は原則行わない。行う場合はPhase 4-8の全成果物への波及を覚悟すること

#### 出戻り防止の原則

1. **各フェーズの完了前に入力元との整合を確認する**（特にPhase 4, 6, 8）
2. **Phase 6/8の通しプレイが出戻りの最大の検出機会**。ここで徹底的に検証する
3. **Phase 8完了後のコード修正は極力避ける**。ドキュメント側での吸収を優先する
4. **修正が必要な場合は、影響を受ける全下流成果物をリストアップしてから着手する**
