# Certify LMS 詳細度 100% チケット作成 Skill

## 入力

`$ARGUMENTS`: チケット ID (1 件 or カンマ区切り複数件)。

- 単一: `S-B-02`
- 複数: `S-B-02,S-B-03,S-B-04` (1 ターン内で連続生成)
- なければユーザーに確認

## 出力

`関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_{feature略称-連番}.md` を作成 + `_review-log.md` 追記 + README 進捗トラッカー更新。

## 必須読み込み

実行前に以下を Read:

1. **`.claude/rules/ticket-spec.md`** — 詳細化規約 (paths frontmatter で auto-load、未ロードなら明示 Read)。**セクション構成 / AC 3 ブロック構造 / AC 圧縮原則 / 件数目安 / 禁止される記載 / ロール表記 / Q&A 粒度** の全書き方規約が集約されている。**最重要**
2. **`関連ドキュメント/要件シート_詳細度100%/README.md`** — チケット一覧(該当 ID の基本情報: タイトル / Feature / サブカテゴリ / 工数)+ 進捗トラッカー(更新先)
3. **`関連ドキュメント/要件ブレインストーミング.md` § 2** — スコープ決定情報(提供 PJ で何を渡すか / 受講生は何を実装するか)。`【UI のみ: テーブル + Seeder データ提供、admin CRUD UI + Policy を受講生が実装】` のような括弧書きが README に転記されておらず本ファイルにのみ存在。Bug の「Step 4 仕込み方」列も本ファイル
4. **`templates/{story|bug|task}.md`** (Skill 配下) — テンプレ(セクション骨格 + 各サブセクションの記述例)
5. **`関連ドキュメント/要件シート_詳細度100%/Story/S-B-01_qa-board-01.md`** — 詳細化済み Story 参考実例
6. **対応する `docs/specs/{feature}/{requirements,design,tasks}.md`** — Feature 完成形 SDD
7. **対応する `模範解答プロジェクト/app/` 配下の実装** — Feature コード本体
   - spec と実装に乖離があれば、**実装を最新 / 正** として扱う
8. **再生成モード時のみ**: 既存版チケット `関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_*.md` を入力資産として Read(`## 実装方針` / `## 補足` を保持、`## 受け入れ条件` のみ新規約で再生成)

補助参照: `関連ドキュメント/要件シート_詳細度100%/_review-log.md`(模範解答 PJ レビュー履歴、再生成モードでは既存判定 ✅ / ⚠️ / 🕐 / 対応完了 を尊重)

複数チケット指定時、上記 1-5 は 1 回読めば全チケットで共有可能。チケット個別の 6-7 は Feature が異なれば都度 Read。

## プロセス

### Step 0: モード判定(新規生成 / 再生成)

`関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_*.md` が既に存在する場合 → **再生成モード**:

- 既存版を入力として活用、`## 実装方針` / `## 補足` / `## 概要` / `## 背景・目的` / `## ユーザーストーリー` / `## やること` / `## やらないこと` / `## メタ情報` は保持(必要に応じて軽微な修正)
- `## 受け入れ条件` のみ新規約準拠で再生成
- Step 3(模範解答 PJ 懐疑的レビュー)はスキップ(既存判定がある前提)、Step 5 と Step 6 に集中

既存ファイルがなければ新規生成モード(Step 1-8 を全実行)。

### Step 1: チケット情報取得

README のチケット一覧から該当 ID のタイトル / Feature / サブカテゴリ / 工数を取得。不足情報(Bug の「Step 4 仕込み方」等)は要件ブレインストーミング.md § 2 を参照。

### Step 2: テンプレ選択

種別に応じて `templates/{story|bug|task}.md` を Read。

### Step 3: 模範解答 PJ 懐疑的レビュー

模範解答 PJ は受講生が理想的な実装を学ぶ教材であるべきため、本チケット抽出前に「現在の実装がベストか?」を問い直す。引き算で受講生に渡す前に模範解答 PJ を理想形に修正しておくことで、受講生は正しい設計を学べる。

`模範解答プロジェクト/` 配下の対応 Feature コードを Read して検証(画面分割の妥当性 / 認可粒度 / レイヤー配置 / API 設計 / ロール間共通化 等)。

判定:
- ✅ 適切 → 現状実装に沿って実装方針を記載
- ⚠️ 修正必要 → 詳細化を一時停止、模範解答 PJ を先に修正してから再開
- 🕐 Phase D で一括処理 → 結果を `_review-log.md` に追記、詳細化は次に進む

レビュー結果を `_review-log.md` に追記(`S-B-01` の既存エントリ形式に倣う)。

### Step 4: 詳細仕様の組み立て

選択したテンプレ(`templates/{種別}.md`)のセクション骨格に従って各セクションを埋める。**記述粒度・記述ルールはすべて `ticket-spec.md` に集約されている** ので、特に以下を熟読:

- 「セクション構成」(種別別の骨格)
- 「実装方針の性質」(参考粒度、業務語彙ベース、詳細値は禁止)
- 「実装粒度の記載範囲」(セクション別の許可範囲、業務語彙への置き換え、LMS ドメイン語彙)
- 「禁止される記載」(別チケット言及 / 過去案件言及 / 構築側メタ情報 / 各セクション冒頭の説明文 / 採点運用情報 など)

**実装方針の記述ソース**(構築側内部、チケットには書かない):
- `## 実装方針(参考)` の各サブセクションは、模範解答 PJ の該当実装コードを読んでその振る舞いとハイレベル設計を業務語彙で記述(spec は参考、実装が正)
- 模範解答 PJ の実装が「ベストではない」と判断した場合は `_review-log.md` に判定(⚠️ / 🕐)を記録した上で改修後の姿で記述(✅判定なら現状実装に沿って記述)

### Step 5: 受け入れ条件 (= 採点項目)

**受け入れ条件 = 採点項目**。各項目 = 評価シート ① の 1 採点行 と 1:1 対応する。

`ticket-spec.md`「3. 受け入れ条件の書き方」を必ず参照。要点:

- **happy path 中心 + 重要なエッジケースのみ**(全パターン羅列はしない、BookShelf 評価シート風)
- **フラット箇条書き**(3 ブロック構造は採用しない)
- **件数目安**: Story 通常 4-7 / 規模大 6-10 / Bug 1-3 / Task 3-5
- 認可は機能群ごとに 1 件、入力検証は該当時のみ 1 件、いずれも他の AC と同列にフラットに並べる
- **圧縮原則 9 パターン**(A〜I)に従って統合

**Skill 側の運用チェック**:
- 生成後に `- [ ]` の数をカウント(`grep -c "^- \[ \]" {ファイル}` 等)し、種別別目安と照合
- 目安超過 → 圧縮原則を再度当てはめて削減
- 妥当な根拠(規模大 Story / 工数大 Task 等)があれば超過 OK

### Step 6: ロール表記 / 想定ヒアリング Q&A

`ticket-spec.md`「ロール表記ルール」「想定ヒアリング Q&A の粒度」に従う。

- ロール初出時に「受講生(student) / コーチ(coach) / 管理者(admin)」フル表記、以降日本語のみ
- Q&A は **要件・仕様レベルのみ**(設計判断 / 実装詳細は書かない)、**フラットなテーブル 1 個**
- 種別別の必要性: **Story = 必須** / **Bug・Task = 任意**(仕様確認の余地がなければ省略可)
- コーチが受講生のヒアリング応答時に「正しい仕様」を即答できる材料を整備

### Step 7: ファイル作成 + 進捗トラッカー更新

- ファイル作成: `関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_{feature略称}-{連番}.md`
- README 進捗トラッカー更新:
  - 該当種別 × 難易度 のカウントを +1
  - 完成済みリストに `✅ {ID} / {feature略称-連番} {タイトル}(YYYY-MM-DD 詳細化)` を追加

### Step 8: 完了報告

ユーザーに以下を簡潔に報告:

- 作成したファイルパス
- 模範解答 PJ レビュー結果 (✅ / ⚠️ / 🕐)
- 進捗トラッカー (X / 40 件 完成)
- AC 件数(規約目安との比較)
- Basic 制約注記の有無 (Bug で「※」を使った場合等)

## 並列処理 (複数チケット連続生成)

複数チケット ID をカンマ区切りで指定された場合、1 ターン内で全チケットを順番に生成する連続実行モード。

```
ユーザー入力: /ticket-detail-100p S-B-02,S-B-03,S-B-04
Skill 動作:
  1. 必須読み込み 1-5 を 1 回実行 (全チケットで共有)
  2. 各チケット ID について Step 0-8 を順番に実行 (Step 5, 6 でのチケット固有 spec / 実装読み込みは個別)
  3. 全件完了後、まとめて報告 (各チケットのファイルパス + レビュー結果 + 進捗 + AC 件数集計)
```

規模目安: 1 ターン内で context 圧迫しない範囲 (2-5 件程度)。それ以上は複数ターンに分割推奨。

## 注意事項

### Bug の Basic 例外注記の書き方

Action / Service ファイル内のバグが構造上避けられない Basic Bug の場合、チケット冒頭(概要直下 or メタ情報下)に以下のような注記を入れる:

```markdown
> **※ Basic 範囲外への例外注記**: 本 Bug は構造上 `app/UseCases/{Feature}/{Action}.php` 内の修正が必要となる。Basic 受講生は通常 Controller 範囲で実装するが、本チケットでは模範解答 PJ の Action 構造を踏襲しているため、修正対象に Action ファイルが含まれる。受講生が Controller 内完結で実装している場合は、対応する Controller method を修正対象とする。
```

### 同 Feature 内チケットの整合

同 Feature に複数チケットがある場合(例: `notification-01` / `notification-02` / `notification-05`)、依存関係(`S-B-01` qa-board → `S-B-04` notification 基盤 等)を意識して実装方針の整合を取る。

### 模範解答 PJ + spec の Read 範囲(Step 3 補助)

該当 Feature のディレクトリ全体を Grep / Read。少なくとも:

**実装 (`模範解答プロジェクト/`)**:
- `app/Models/{Entity}.php` / `app/Enums/`
- `app/Http/Controllers/` 該当
- `app/UseCases/{Feature}/` 全 Action
- `app/Services/{Feature}*.php`(該当時)
- `app/Policies/` 該当
- `app/Http/Requests/` 該当
- `resources/views/{feature}/` 全 Blade
- `database/migrations/` 該当 + `database/seeders/` 該当

**spec**:
- `docs/specs/{feature}/{requirements,design,tasks}.md` 全 3 ファイル

乖離時の優先: spec と実装に違いがあれば、**実装を最新 / 正** として扱う。spec は当初設計の参考。
