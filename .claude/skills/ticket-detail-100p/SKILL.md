# Certify LMS 詳細度 100% チケット作成 Skill

## 入力

`$ARGUMENTS`: チケット ID (1 件 or カンマ区切り複数件)。

- 単一: `S-B-02`
- 複数: `S-B-02,S-B-03,S-B-04` (1 ターン内で連続生成)
- なければユーザーに確認

## 出力

`関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_{feature略称-連番}.md` を作成 + `_review-log.md` 追記 + README 進捗トラッカー更新。

## 必須読み込み

実行前に以下を Read:

1. **`.claude/rules/ticket-spec.md`** — 詳細化規約(paths frontmatter で auto-load、未ロードなら明示 Read)。**書き方規約はすべてここに集約**(セクション構成 / 記述粒度 / 受け入れ条件 / 想定 Q&A / Basic 制約 / ロール表記 / 模範解答 PJ レビュー基準 / 30% 版変換ルール 等)
2. **`関連ドキュメント/要件シート_詳細度100%/README.md`** — チケット一覧(該当 ID の基本情報: タイトル / Feature / サブカテゴリ / 工数) + 進捗トラッカー(更新先)
3. **`関連ドキュメント/要件ブレインストーミング.md` § 2** — スコープ決定情報(提供 PJ で何を渡すか / 受講生は何を実装するか) + Bug の「Step 4 仕込み方」列(README には転記されていない情報を含む)
4. **`templates/{story|bug|task}.md`**(Skill 配下) — テンプレ骨格(セクション骨格 + 各サブセクションのプレースホルダー)
5. **対応する `docs/specs/{feature}/{requirements,design,tasks}.md`** — Feature 完成形 SDD
6. **対応する `模範解答プロジェクト/app/` 配下の実装** — Feature コード本体(spec と乖離時は **実装が正**)
7. **再生成モード時のみ**: 既存版チケット `関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_*.md` — 保持する固定セクションを把握

補助参照: `関連ドキュメント/要件シート_詳細度100%/_review-log.md`(再生成モードでは既存判定 ✅ / ⚠️ / 🕐 を尊重)

複数チケット指定時、上記 1-4 は 1 回読めば全チケットで共有可能。チケット個別の 5-6 は Feature が異なれば都度 Read。

## プロセス

### Step 0: モード判定

`関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_*.md` が既に存在する場合 → **再生成モード**。

- **保持セクション**: `## メタ情報` / `## 概要`(Bug/Task) / `## 背景・目的`(Story) / `## ユーザーストーリー`(Story) / `## 要件`(Story/Task) / `## スコープ外`(Story/Task) / Bug の `## 再現手順` 〜 `## 実際の結果`
- **再生成セクション**: `## 受け入れ条件` / `## 実装方針(参考)` / `## 補足 > ### 想定 Q&A`
- Step 3(模範解答 PJ 懐疑的レビュー)はスキップ(既存判定がある前提)

既存ファイルがなければ新規生成モード(Step 1-8 を全実行)。

### Step 1: チケット情報取得

README のチケット一覧から該当 ID のタイトル / Feature / サブカテゴリ / 工数を取得。不足情報(Bug の「Step 4 仕込み方」等)は要件ブレインストーミング.md § 2 を参照。

### Step 2: テンプレ選択

種別に応じて `templates/{story|bug|task}.md` を Read。

### Step 3: 模範解答 PJ 懐疑的レビュー

規約 9 に従い、模範解答 PJ コードを Read して判定(✅ / ⚠️ / 🕐)。結果を `_review-log.md` に追記。

### Step 4: 詳細仕様の組み立て

テンプレ骨格に従い各セクションを記述。**記述粒度・記述ルールはすべて規約 ticket-spec.md に集約済**、規約を都度参照しながら記述する。

### Step 5: 受け入れ条件 (= 採点項目)

規約 3 章に従う。生成後に `- [ ]` の数をカウント(`grep -c "^- \[ \]" {ファイル}`)し、規約 3.3 件数目安と照合。目安超過時は規約 3.4 圧縮原則(5 パターン)を適用。

### Step 6: ロール表記 / 想定ヒアリング Q&A

規約 5(Q&A 粒度) / 7(ロール表記)に従う。

### Step 7: ファイル作成 + 進捗トラッカー更新

- ファイル作成: `関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_{feature略称}-{連番}.md`
- README 進捗トラッカー更新:
  - 該当種別 × 難易度 のカウントを +1
  - 完成済みリストに `✅ {ID} / {feature略称-連番} {タイトル}(YYYY-MM-DD 詳細化)` を追加

### Step 8: 完了報告

ユーザーに以下を簡潔に報告:

- 作成したファイルパス
- 模範解答 PJ レビュー結果(✅ / ⚠️ / 🕐)
- 進捗トラッカー(X / Y 件 完成)
- AC 件数(規約目安との比較)
- Basic 制約注記の有無(Bug で「※」を使った場合等、規約 6.1 参照)

## 並列処理(複数チケット連続生成)

複数チケット ID をカンマ区切りで指定された場合、1 ターン内で全チケットを順番に生成する連続実行モード。

```
ユーザー入力: /ticket-detail-100p S-B-02,S-B-03,S-B-04
動作:
  1. 必須読み込み 1-4 を 1 回実行(全チケットで共有)
  2. 各チケット ID について Step 0-8 を順番に実行(Step 5-6 でのチケット固有 spec / 実装読み込みは個別)
  3. 全件完了後、まとめて報告
```

規模目安: 1 ターン内で context 圧迫しない範囲(2-5 件程度)。それ以上は複数ターンに分割推奨。

## 模範解答 PJ + spec の Read 範囲(Step 3 補助)

該当 Feature のディレクトリ全体を Grep / Read。最低限:

**実装** (`模範解答プロジェクト/`):

- `app/Models/{Entity}.php` / `app/Enums/`
- `app/Http/Controllers/` 該当
- `app/UseCases/{Feature}/` 全 Action
- `app/Services/{Feature}*.php`(該当時)
- `app/Policies/` 該当
- `app/Http/Requests/` 該当
- `resources/views/{feature}/` 全 Blade
- `database/migrations/` 該当 + `database/seeders/` 該当

**spec**:

- `docs/specs/{feature}/{requirements,design,tasks}.md` 全 3 ファイル

乖離時の優先: spec と実装に違いがあれば、**実装を最新 / 正** として扱う。spec は当初設計の参考。
