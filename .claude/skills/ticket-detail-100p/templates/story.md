## 概要

> 何を作るか(What) を 1〜2 文で簡潔に。

## 背景・目的

> **なぜこのチケットがあるか** を以下 3 点で記述:
>
> - **現状の問題**: 現状どんな課題・ギャップがあるか(受講生・コーチ・運営の困りごと / 既存機能の制約 等)
> - **達成したい状態**: 解決後にどうなっているか(定性的な成果像)
> - **価値・優先度**: なぜ今やるべきか(この機能がプロダクトにもたらす価値)
>
> 背景が薄いとチケットの妥当性評価や代替案検討ができず、受講生も「なぜこれを作るのか」の本質を理解せず手を動かす羽目になる。

## ユーザーストーリー

> Connextra 形式(`{ロール}として、{何を}したい。なぜなら、{目的}から`)。ロール別に 1 行ずつ。Why を明示する装置として 30% 版でも保持する。

- **{ロール B}として**、{何を}したい。なぜなら、{目的}から。

## やること

> **エンティティ単位 / 機能単位** で `###` 小見出しを置き、各操作にロール権限と認可挙動を明示する。受講生が 30% 版でも実装範囲とロールごとの認可を一目で把握できる構造にする。
>
> **書き方の原則**:
>
> - **エンティティ単位 / 機能単位** で `###` 小見出し(例: 「{Entity}」「閲覧・検索」「共通の振る舞い」等)
> - 各操作を `- **{操作名}**:` で始めて、**ロール権限 + 認可挙動**(403 / 404 / 409 等)を 1 行で明示
> - 技術レイヤー(フロント/バック)分割は禁止
> - 受け入れ条件に書かれている振る舞いはすべてどこかの機能ブロックに含めて、**やること ↔ 受け入れ条件 の網羅性** を担保
> - 細かい振る舞い詳細(フラッシュ文言・リダイレクト先パス・具体的な HTTP メソッド)は受け入れ条件側で記述、やることは **機能の枠組みとロール権限の概観**
> - **設計粒度の記載は禁止**: カラム名(`name` / `description` / `status` 等) / バリデーション詳細(文字数上限 / 数値範囲 等) / Enum 機械値(`status=draft` / `'open'` 等) / URL パス詳細(`/admin/meeting-packs/*` 等) / SQL の並び順詳細(`sort_order ASC` 等) は **実装方針側** に書く。やることは業務語彙(「面談パック」「下書き」「公開中」「アーカイブ」「基本情報を更新」等)のみで構成し、コード由来の機械的識別子は含めない
