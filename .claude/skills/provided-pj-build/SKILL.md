# provided-pj-build

完成済みの **模範解答プロジェクト**(全チケット実装済)から各チケット分を **引き算** し、**提供プロジェクト**(= 受講生 Template 用の「作るべき機能だけが欠けた、動く既存 PJ」)をクラスタ単位で構築する自己完結ワークフロー。直列実行。

## なぜ引き算方式か

足し算(空 PJ に積む)より整合性が高い。完成形が動く状態を起点に、要件シートの diff 指示どおりに「削る / バグ化する / 巻き戻す」だけなので、提供 PJ は常に「模範解答 − 受講生担当分」という明確な状態になり、模範解答との `diff` で過不足を客観検査できる。

## このワークフローの中核的な落とし穴(最重要)

**「要件シートどおりに機械的に仕込む」だけでは不十分。** 模範解答は防御多層・既存パターンの塊なので、チケットが想定した症状と実装が食い違うことが頻繁に起こる。実際に observed したパターン:

- **別層マスク**: 認可ミドルウェアを緩めても、FormRequest/Policy が同じ role を二重ガードしていて症状が出ない(赤テスト 0 件)
- **症状ズレ**: バリデーションを抜くと「静かに不正データ」ではなく DB 一意制約違反で 500 になる(チケットは「成立する」と記述)
- **テスト欠落(silent gap)**: バグを仕込んでも、その挙動を assert する同梱テストが模範解答に存在せず、赤にならない

だから本ワークフローは **引き算 = チケット↔コードの整合性監査** でもある。各仕込みで「**チケットが書いた症状が実際に再現するか(Playwright)**」と「**赤になるはずの同梱テストが実際に赤か**」を必ず確認し、食い違ったら黙って進めず **reconcile**(Phase 3)する。これがこの skill の存在意義の半分。

## 入力

- **`$ARGUMENTS`**: 着手するクラスタ(名前 or 番号)。例: `01 auth` / `content-management` / `next`(次の未チェッククラスタを選ぶ)。無ければ Phase 0 でユーザーに確認。
- クラスタ = Feature 群でまとめた引き算単位(共有ファイルを 1 回で読み切れる粒度)。プロジェクトの「クラスタ別チェックリスト」(下記 SSoT)で定義される。

## 必須読み込み(SSoT — これらを正とし、記憶や憶測で進めない)

引き算の判断根拠は **プロジェクトの SSoT** から読む(プロジェクトが変われば中身も変わる。汎用 skill なので「どこに何があるか」は対象 PJ の CLAUDE.md / 要件シートが教える):

1. **CLAUDE.md「Step 4 引き算戦略」表** — 種別 × 変換タイプ(新規機能=ロジック削除 / バグ=バグ化 / 改修・リファクタ=巻き戻し)の大原則。実装ディレクトリ名(例: `模範解答プロジェクト/` `提供プロジェクト/`)もここ。
2. **各チケットの 実装方針 > コンポーネント節の「既存」マーカー** — 要件シート(詳細度100%)の各チケット。`既存 / 既存・提供 / 既存、ロック対象` = 残す / 無印 = build 対象 = 削除。**Story 引き算の唯一の機械判定基準**。
3. **README「Step 4 仕込み方」列** — Bug の仕込みファイル・行レベル指示(どのファイルのどの処理をバグ実装に置換するか)。要件シートの README にある。
4. **Step 4 運用方針 / クラスタ別チェックリスト** — 進行管理(GitHub issue や `project_step4_strategy` メモ等)。クラスタ順・各クラスタのチケット・検証ゲート・運用ルール・確定済みの判断・保留事項を集約。**着手前に必ず最新を確認**(過去の壁打ちメモを鵜呑みにしない)。
5. **`.claude/rules/` 配下** — 既存パターン(Controller 薄い / Action・Service 配置 / テスト規約 / Blade API 契約等)を把握するため。「何が既存パターンか」を知らないと「無印=削除」の線引きを誤る。

> SSoT 階層: **模範解答のコードが ground truth**。チケット実装方針 / README は **引き算の手引き**。食い違ったら手引きに黙って従わず Phase 3 で reconcile する。

## 全体フロー(Phase 0 → 5)

```
Phase 0  クラスタ着手準備   (クラスタ選択 → ブランチ → 共有ファイル読み切り → 提供PJ runnable 準備)
   ↓
Phase 1  種別別 引き算実装  (1チケット=1コミット、順序: Story gut → Task revert → Bug inject)
   ↓        └ 詳細は references/by-ticket-type.md
Phase 2  検証ゲート         (静的 + 同梱テスト赤/緑 + Playwright 症状再現 + 全ロール巡回500ゼロ)
   ↓        └ 詳細は references/verification-gate.md / references/runnable-provided-pj.md
Phase 3  不整合 reconcile   ★ Phase 2 で「症状が出ない/テストが赤にならない」を検出したら必ず通る
   ↓        └ 詳細は references/reconcile.md
Phase 4  テスト citation    (Bug/perf-Task の受け入れ条件に「同梱 path::method が pass」採点行を追記、欠落テストは模範解答に追加)
   ↓
Phase 5  PR + 記録 + 完了報告 (クラスタ毎ブランチ → main へ PR、issue/メモ更新、構造化報告)
```

> Phase 2 と Phase 3 はループ。仕込み → 検証 → 食い違い検出 → reconcile → 再仕込み/再検証。クラスタ内のチケットを 1 件ずつこのループで確定させる。

---

## Phase 0: クラスタ着手準備

1. **クラスタ選択**: SSoT 4(チェックリスト)から次の未チェッククラスタを選ぶ(`$ARGUMENTS` で指定された場合はそれ)。そのクラスタのチケット一覧・各チケットの種別(Story/Bug/Task)・サブカテゴリを把握。
2. **共有ファイルの読み切り**: クラスタ内チケットが触る共有ファイル(`routes/web.php` / 共通 Service / Policy 群等)を**先に読み切る**。クラスタ単位にする理由がこれ(後のチケットで同じファイルを再読しない)。
3. **ブランチ**: クラスタ毎にブランチを切る(例: `step4/02-content-learning`)。提供PJ の実体は前クラスタまでの成果に依存するので、**前クラスタが main にマージ済みなら `origin/main` から、未マージ(PR レビュー待ち等)なら前クラスタのブランチ(例 `origin/step4/01-auth`)から分岐**。後者では前クラスタの注入赤も引き継ぐ(Phase 2 でベースライン扱い)。`gh pr view <前クラスタPR> --json state` で状態確認。実運用は「PR レビュー待ちで次クラスタに着手」が普通なので、未マージ分岐がむしろ既定と考える。
4. **提供 PJ を runnable にする**: 症状再現(Phase 2)に提供 PJ の実行環境が要る。提供 PJ は vendor/.env 無しで配布されるので、模範解答の vendor を流用して単体起動できるようにする。手順は **references/runnable-provided-pj.md**。
5. **着手宣言**: クラスタ名・対象チケット・想定する変換種別を一言共有してから Phase 1 へ。

---

## Phase 1: 種別別 引き算実装

**1 チケット = 1 コミット**(受講生の 1チケット=1PR と追跡性を揃える。1 つのバグが複数ファイル[例: 6 Policy / 3 Action]にまたがっても **1 チケット分は 1 コミット**にまとめる)。クラスタ内の順序は **Story gut → Task revert → Bug inject**(土台を削ってから巻き戻し、最後にバグ注入。依存の少ない順)。

種別ごとの変換規則の詳細は **references/by-ticket-type.md** を読む。要点:

- **Story(機能開発)**: コンポーネント節「既存」マーカーの部品(Blade / JS / 表示系 ViewComposer)は残し、無印(Controller method / Action / Service / Repository / FormRequest / Policy)を削除。**新規機能はドメインデータ層(機能固有 migration / Model / Enum / Factory / 専用 Seeder)も削除**(答えだし回避)。既存機能の拡張は **その Story が足した差分だけ巻き戻し**。
- **Bug(バグ修正)**: README「Step 4 仕込み方」列どおり指定箇所をバグ実装に置換。**同梱テストは残す**(修正前は赤 = バケット①、受講生が直すと緑)。
- **Task(改修)**: 実装方針どおり巻き戻し。perf Task = 同梱テスト残す(赤) / refactor Task = テスト削除(受講生が書く)。

各仕込みで **不要になった import / `@throws` / 死にコードは整理**するが、**「正しい意図」を述べた DocBlock・業務ルール記述・チケットが手掛かりにしている表示名等は温存**(受講生が修正時に辿れるように)。コミットは何をなぜ削った/仕込んだかを本文に書く。

> Pint hook がプロジェクト側にあると編集後に自動整形される(無い環境では手で規約準拠を保つ)。

---

## Phase 2: 検証ゲート

「消しすぎ / 消し足りない / 仕込み誤り」を客観判定する。詳細は **references/verification-gate.md**。骨子:

1. **静的**: `composer dump-autoload` / `artisan about`(boot) / `route:list`(削除 Controller を指す死にルート検出) / `npm run build`。
2. **DB 整合**: `migrate:fresh --seed` 緑(削除テーブルへの Seeder/FK/Factory 参照切れを検出)。
3. **同梱テストの赤/緑**(バケット①の核):
   - 仕込んだ Bug / perf-Task の **代表テストが実際に赤になる**ことを確認(`path::method` を名指しして実行)。
   - Story / refactor のテストは削除されている(② バケット)ことを確認。
   - **他のテストが想定外に赤くなっていない**(仕込みが局所的か)。広域な仕込み(ミドルウェア等)は全テスト実行で blast radius を見る。
4. **Playwright で症状再現**(本ゲートの心臓): チケットの「再現手順 / 実際の結果」どおりの症状が、提供 PJ を実際にブラウザ操作して **再現するか** を確認。観察可能なバグはここで必ず目視する(例: 卒業ユーザーでログイン → プラン機能が 403 にならず表示される)。Playwright MCP(`mcp__playwright__*`)を使う。
5. **全ロール巡回 500 ゼロ**: 主要ロール(admin/coach/student 等)で主要ルートを巡回し、引き算が boot/描画を壊していない(500 が出ない)ことを確認。

> **3 と 4 が一致しないときが reconcile の合図**: 「テストは赤になったが症状が違う」「症状は出るがテストが赤にならない」「どちらも起きない」→ Phase 3 へ。

---

## Phase 3: 不整合の reconcile(★ この skill の核)

Phase 2 で「チケットの想定どおりにならない」を検出したら、**黙って先に進めない**。原因を切り分けて対処する。詳細・実例は **references/reconcile.md**。判断の枝:

- **A: チケットが誤り**(実装フローの想定違い・症状記述が不正確) → 100% / 30% 要件シート + README を **事実に合わせて修正**(記録を残す)。バグ/テスト/修正自体は有効ならコードは触らない。
- **B: 模範解答が理想から乖離 / テスト不足** → 模範解答を改善(欠落テストを追加 / 設計を整える)。**テスト追加・設計変更はユーザー確認**(相談ポリシー)。追加したテストは提供 PJ にも反映され赤で出る。
- **C: 仕込みが構造的に不成立 / A・B 判断が曖昧** → **停止してユーザーに相談**。保留して別途再設計する選択肢を含む(該当チケットをクラスタから外し commit を drop)。

reconcile した内容(どちらに倒したか・理由)は必ず SSoT(issue / メモ)と該当チケットに記録する。Step 4 はこの整合性監査の記録が成果物の一部。

---

## Phase 4: テスト citation(C案)

Bug / perf-Task のチケットは「振る舞い観察ベースの受け入れ条件」に加えて、**「同梱 `path::method`(修正前は赤)が pass する」採点行を 1 行**持つ(客観オラクル + 振る舞い行の併記。詳細はプロジェクトの `ticket-spec.md` §3.6)。本クラスタで仕込んだ Bug / perf-Task について:

1. **赤になる代表テストを名指し**して、そのチケットの受け入れ条件に採点行を追記。
2. **代表テストが模範解答に無ければ(Phase 3-B の silent gap)、模範解答に追加**してから citation(模範=緑 / 提供=赤 を実機確認)。
3. refactor Task は同梱の赤テストが無い(振る舞い不変)ので対象外 — 構造はコード確認(grep)で見る。

> 監査ツール/サブエージェントに「テストの有無」を委ねた場合、query-count / cache / order 等の見落としが起きやすい。**GAP 判定は必ず実 grep / 実行で再検証**する(エージェント出力を鵜呑みにしない)。

---

## Phase 5: PR + 記録 + 完了報告

1. **PR**: クラスタ毎ブランチを main へ PR(出戻りの局所化)。前クラスタがマージ済み前提。
2. **検証の実機証跡**: Phase 2/4 の結果(テスト件数・赤/緑・Playwright 症状再現・巡回 500 ゼロ・reconcile した不整合)を PR 本文に構造化。
3. **記録更新**: SSoT のクラスタ別チェックリスト(issue)をチェック + 進捗コメント。`project_step4_strategy` 等のメモも更新。reconcile / 保留した項目は次セッションが追えるよう明記。
4. **構造化完了報告**: 仕込んだチケット一覧(種別 × ファイル × 仕込み内容)/ 検証結果 / 保留・再設計に回した項目 / 次クラスタ。

---

## 相談ポリシー(重要)

**承認なしで進める**: SSoT が明示している機械的な引き算 / typo・整形 / コミット粒度の判断 / 観察可能で素直に再現するバグの仕込み。

**ユーザーに必ず確認する**(`AskUserQuestion`):
- Phase 3-B(模範解答コードの修正 / テスト追加 / 設計変更を伴う)
- Phase 3-C(仕込み不成立 / 保留・再設計の判断)
- チケット↔コードの食い違いで、A(チケット修正)か B(コード修正)か判断が割れる場合
- Blade ロック hook 有効化や、basic/advance ブランチ分割など **後戻りしにくい一括操作**
- 純粋なバグ修正(模範解答コードを正しく直すだけ)以外で、設計判断(責務分離・SoftDelete 採否等)を伴う変更

確認は 2〜4 択を提示し、推奨案を 1 番目 + `(推奨)`、各択に挙動差/トレードオフを書く。

## 制約

- **編集対象は提供 PJ**(模範解答は ground truth として原則保持)。例外: Phase 3-B のテスト追加・設計改善はユーザー確認の上で模範解答に入れ、提供 PJ にも反映。
- **クラスタ単位 / 1チケット=1コミット / Story→Task→Bug 順** を守る。
- **同梱テストの所在を種別で守る**: Bug・perf-Task は残す(赤)、Story・refactor は削除(受講生記述)。
- 検証用に作った提供 PJ の `.env` / vendor / build 等は gitignore 配下(成果物・PR に混入させない)。検証用スクショも repo 外へ退避。
- 全クラスタ完了後の最終工程(Blade ロック hook 有効化 / basic・advance 分割等)は本 skill の範囲外 or 別途すり合わせ(プロジェクトの方針に従う)。

## 完了基準(クラスタ単位)

- クラスタ内 全チケットの引き算が完了(または reconcile で保留に切り分け済み)、1チケット=1コミット
- 検証ゲート通過: 静的 OK / `migrate:fresh --seed` 緑 / `npm run build` 緑 / 巡回 500 ゼロ
- 仕込んだ Bug・perf-Task の **代表テストが赤**(修正で緑)を実機確認、Story・refactor のテストは削除済
- **チケットの症状が Playwright で再現**することを確認(観察可能なもの)
- 不整合は Phase 3 で reconcile 済み(A: チケット修正 / B: コード改善 / C: 保留)、記録済み
- Bug・perf-Task チケットに「同梱 `path::method` pass」採点行を追記済(C案)
- PR 作成 + issue/メモ更新 + 構造化完了報告
