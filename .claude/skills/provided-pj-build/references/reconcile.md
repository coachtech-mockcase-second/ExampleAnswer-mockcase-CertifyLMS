# 不整合の reconcile(チケット↔コードの食い違い)

引き算は **チケット↔コードの整合性監査** でもある。模範解答は防御多層・既存パターンの塊なので、「要件シートどおりに仕込んだのに、チケットが書いた症状が出ない / 赤になるべきテストが赤にならない」が頻繁に起こる。**黙って先に進めず**、原因を切り分けて対処する。

## SSoT 階層(食い違い時の判断軸)

- **模範解答のコードが ground truth**(実際に動いてテストが通っている事実)
- チケット実装方針 / README「Step 4 仕込み方」は **引き算の手引き**(設計時の想定。コードと食い違うことがある)
- 食い違ったら手引きに盲従しない。下の A/B/C に切り分ける。

## 検出トリガ(Phase 2 で気づく)

- **症状が出ない**: 仕込んだのにチケットの「実際の結果」が再現しない
- **テストが赤にならない**: 仕込んだ Bug の赤になるべき同梱テストが緑のまま
- **別症状**: チケットは「静かに不正」と書くが実際は 500 等

## 3 つの枝

### A: チケットが誤り(実装フローの想定違い・症状記述が不正確)

バグ/テスト/修正自体は有効だが、チケットの **再現手順 / 実際の結果 / 期待結果の記述が実装と食い違う**。
→ **要件シート(100% / 30%)+ README を事実に合わせて修正**(記録を残す)。コードは触らない。

### B: 模範解答が理想から乖離 / テスト不足

バグは観測可能で正当だが、**それを検証する同梱テストが模範解答に無い**(silent gap)、または模範解答の設計が理想から乖離している。
→ **模範解答を改善**(欠落テストを追加 / 設計を整える)。追加したテストは提供 PJ にも反映され赤で出る(バケット①成立)。
→ **テスト追加・設計変更はユーザー確認**(相談ポリシー)。`§9 模範解答 PJ 懐疑的レビュー` の精神。

### C: 仕込みが構造的に不成立 / A・B 判断が曖昧

仕込んでも観測可能な意味のある症状にならない(構造的に不可能)、またはチケットの前提自体が実装と矛盾していて A か B か判断が割れる。
→ **停止してユーザーに相談**。該当チケットをクラスタから保留(commit を rebase で drop)し、別途再設計する選択肢を含む。

## 実例(このワークフローで実際に起きたパターン)

### 別層マスク → C(保留)or B(テスト追加)

**症状**: 認可ミドルウェアを `role:admin` → `role:admin,coach` に緩めても、コーチが管理画面に入れない・赤テスト 0 件。
**原因**: admin の各エンドポイントが FormRequest/Policy で `role===Admin` を**二重ガード**(defense-in-depth)。ミドルウェアを緩めても別層で 403。`User\IndexTest` のコーチ→403 が仕込み下でも緑 = 403 は Policy 由来と実証できる。観測可能な漏れは二次ガードの無い 1 エンドポイントのみ。
**対処**: チケット前提(「ミドルウェアが唯一の役割門」)と実装(多層)が設計レベルで矛盾 → ユーザー判断で **保留・別途再設計**(C)。代替案: ①role 判定をミドルウェア単層化 ②二次ガードの無いエンドポイントにテスト追加して観測可能化 ③別の認可バグへ差替。
**教訓**: 1 層を狙うバグは、**別層が同じ判定をしていないか**を必ず確認(`grep` で Policy/FormRequest の role 判定を探す)。

### 検証削除 → DB 制約 500(症状ズレ)→ C or A

**症状**: 在籍重複チェックのガードを抜くと、チケットは「重複招待が成立・メール送信」と書くが、実際は `User::create` で email 一意制約違反の **500**。
**原因**: users.email が UNIQUE で、既存ユーザーへの招待再利用は別条件でのみ起こる設計 → 在籍ユーザーは新規作成パスに行き必ず制約違反。「静かな重複」は**構造的に不可能**。
**対処**: バグ/テスト/修正は機能的に有効だが症状記述が不正確 → A(チケットを 500 症状に修正)か、500 クラッシュが題材として不適なら C(別バグへ差替)。ユーザー判断。
**教訓**: 「検証(unique 前段)を抜く」系は、**背後に DB 制約があると 500 になる**(静かな不正データにならない)。チケットの想定症状を疑う。

### silent テスト gap → B(模範解答にテスト追加)

**症状**: バリデーション(`confirmed` 等)を抜いても赤になるテストが無い。一見「テストファイルはある」が、その挙動を assert するメソッドが無い(例: 全テストの確認用パスワードが一致値で `confirmed` 自体が未テスト)。
**原因**: 模範解答のテストカバレッジ漏れ。バグは観測可能で正当(Playwright で症状再現できる)が、同梱の赤テストが無い。
**対処**: B。模範解答に当該挙動を assert するテストを追加(模範=緑 / 提供=赤 を実機確認)→ バケット①成立 + カバレッジ向上。チケットに citation。
**教訓**: 「機能のテストファイルが在る」≠「バグが壊す具体的挙動を assert している」。**メソッド本文の assert を読み**、「この assert は仕込み時に落ちるか」を確認。落ちないなら gap。

### DB インデックスによる暗黙整列マスク → A(テスト強化) or B(コード確認)

**症状**: 一覧の明示整列(`->ordered()` 等の `ORDER BY`)を削っても、一覧が昇順のまま表示され、並び順テストも緑のまま(bucket① 不成立)。
**原因**: `(parent_id, order)` のような複合インデックスがあると、`WHERE parent_id=?` のクエリが MySQL でそのインデックスを使い、`ORDER BY` 無しでも **偶然 order 昇順で返る**(PK を逆順に挿入しても揃う)。コード上は欠落(latent bug)だが実機で観測不能。
**対処**: A(模範解答テストを強化 = `ORDER BY` 欠落で確実に赤になる結合テストに。逆順挿入 + index が効かない並びで **viewData の順序**を assert)→ 模範解答変更につき要承認 / B(チケットを「コード確認: `->ordered()` の有無を grep」に倒し citation を諦める)。
**教訓**: 並び順 Bug は Model の scope Unit だけでは `ORDER BY` 削除を検出できない。**viewData の並びを見る Action/Controller 結合テスト**で赤化を確認する(C案 citation の盲点。`feedback_ordering_integration_test_gap` の罠)。

### Bug の仕込み先が gut する Story のコード内 → 部分 gut or 相談(C)

**症状**: ある Bug の仕込み先(README が指す原因ファイル)が、同クラスタ or 既マージの **Story の gut 対象コード**そのもの。Story を gut するとそのファイルが消え、Bug を仕込む先が提供 PJ に存在しなくなる(Bug が Story に依存しているとき顕在化)。
**原因**: Bug チケットは「既存コードを直す」前提だが、その既存コードを別の Story が「受講生が作る」build 対象にしている。両立しない。
**対処**: その Story を **部分 gut** — Bug の仕込み先 + それを観察できる最小サーフェス(該当 Controller method / View / route / 認可)だけバグ込みで残し、残りを build 対象として削除。残すサーフェスが他の build 対象ルートを参照するなら feature-aware ガードで休眠化(下記)。部分 gut が不自然 / 判断が割れるなら **C: 相談**(Bug 保留 / 別バグへ差替 / コード確認型へ再設計)。プロジェクトに確定済みの切り分け方針(過去の設計メモ等)があればそれを正とし、現行 SSoT ルールと食い違う場合は最新確認の上で踏襲。
**教訓**: Phase 0 で「Bug の仕込み先 ∈ gut する Story」を先に洗う(依存チケット欄が手掛かり)。後で気づくと gut をやり直すことになる。

### 新規機能のデータ層が基盤に織り込まれている → feature-aware 化 or 相談(B/C)

**症状**: 「新規機能 = データ層(table/Model/Enum/Seeder/Factory)含め全削除」のはずが、その **データ層が KEPT な基盤コードから参照**されている — 基盤テーブルの FK / 基盤 Model のリレーション / 基盤 Service の Eager Load / 基盤テスト / 既存(残す)Blade。単純削除すると `migrate:fresh` の FK 切れ・未定義クラス参照・基盤ページ 500 になる。
**原因**: 模範解答が「新規機能のデータ層」を基盤に密結合させた(チケットは独立した新規データ層と記述)。チケット記述とコードの結合度が食い違う(§9 懐疑的レビューの拾い漏れ)。
**対処**: まず **基盤に同型の cross-feature 参照が既に decouple されていないか**を探す(別 Feature のテーブルを `class_exists` / 動的クラス名 / `Route::has` / 遅延 FK migration で feature-aware にしている箇所)。あれば **その確立パターンに揃えて当該データ層も decouple** し、gut しても基盤が休眠する(存在下では透過)状態にする → 提供 PJ で削除可能に。同型パターンが無い / decouple が広範すぎるなら **B: データ層を基盤として残す**(schema 提供の give-away を許容)か **C: 相談**。模範解答を feature-aware 化するなら Phase 3-B(要確認・両スイート検証)。
**教訓**: 新規機能 gut の前に「このデータ層は誰が参照しているか」を grep(FK / リレーション / Service / テスト / Blade)。基盤が参照していれば単純削除は不可。

### citation テストがバグの「失敗ケース」を踏まず偶然 PASS → テスト強化(B)

**症状**: Bug を仕込んだのに、citation 候補の同梱テストが **赤にならず緑のまま**。テスト自体は存在し該当挙動を assert もしている(assert 欠落の silent gap とは別)。
**原因**: テストの入力値がバグの「誤った固定値」と **偶然一致**。例:「status フィルタが常に `draft` 固定」になるバグに対し、テストが `draft` で絞り込み検証していると、誤動作でも結果が一致して PASS する(「下書きを選んだときだけ偶然正しく見える」)。テストは在るが **失敗ケースを踏んでいない**。
**対処**: B。テストを **バグの失敗が顕在化する入力**に強化(フィルタ値バグなら、固定値とは **異なる値**で絞り込みその結果が出ることを assert)。模範解答=緑 / 提供=赤 を実機確認してから citation。
**教訓**: Phase 4「赤を実機確認」は形式でなく必須の forcing function。citation を名指したら **実際に赤になるか必ず実行**し、緑のままなら入力値の偶然一致を疑う。

## reconcile を支える検出技法

- **赤テストの名指し + 実行**: 「赤になるはず」のテストを `path::method` で特定し実際に実行。赤にならなければ即 reconcile 入り(この skill の Phase 4 = C案 citation がこの forcing function を兼ねる)。
- **症状の Playwright 再現**: テストが緑でも、チケットの症状が実機で出るか/出ないかで A・B・C を切り分け。
- **別層 grep**: 認可/検証バグは、狙った層以外(Policy / FormRequest / DB 制約 / downstream の `abort_unless`)が同じ判定をしていないか grep で確認。
- **監査ツールの結果を鵜呑みにしない**: サブエージェント/grep に「テストの有無」を委ねると query-count / cache / order / filter / quota テストを見落としやすい。GAP 判定は**必ず実 grep / 実行で再検証**。

## 記録

reconcile した内容(どの枝に倒したか・理由・代替案)は **SSoT(issue / メモ)と該当チケット**に必ず記録する。Step 4 の成果物は「引き算されたコード」だけでなく「チケット↔コードの整合性監査の記録」でもある。保留(C)した項目は次セッションが追えるよう、元の仕込み内容と再設計案を残す。
