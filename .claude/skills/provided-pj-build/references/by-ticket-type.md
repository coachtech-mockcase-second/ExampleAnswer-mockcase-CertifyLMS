# 種別別 引き算変換の規則

各チケットを種別 × サブカテゴリで変換する。判断は **CLAUDE.md「Step 4 引き算戦略」表** と **各チケットの実装方針 > コンポーネント節「既存」マーカー** と **README「Step 4 仕込み方」列** を正とする。本ファイルはその適用手順。

## 共通原則

- **「既存」マーカーが SSoT**: コンポーネント節で `既存 / 既存・提供 / 既存、ロック対象` と記された部品は **残す**。無印 = **build 対象 = 削除**。受講生が「コードリーディング + ロジック/API/JS 実装」を担当する設計なので、Blade・JS・表示系の足場は残し、頭脳(ロジック層)を抜く。
- **不要コードの整理**: 削除/巻き戻しで参照されなくなった import / use / `@throws` / 死にコードは消す。ただし **受講生の手掛かりになる記述は温存**:
  - クラス/メソッドの DocBlock が述べる「正しい意図・契約」(コードが満たさなくなっても残す = 「ドキュメントはこう言っているのに実装が伴わない」が良いデバッグ信号)
  - チケットが言及している表示名・ラベル・業務ルール(例: `attributes()` の項目名)
- **1 チケット = 1 コミット**。コミット本文に「何を・なぜ削った/仕込んだか」+「修正前は赤になる同梱テスト」を書く。

## Story(機能開発)

### 新規機能の構築(自己完結ページ)

機能まるごと差分 → **データ層含めて全削除**(答えだし回避):

- 削除: Controller method / Action(UseCase) / Service / Repository / FormRequest / Policy
- 削除: **ドメインデータ層** = 機能固有の migration / Model / Enum / Factory / 専用 Seeder
- 削除: 機能テスト(② バケット = 受講生が書く。削除クラス参照で起動エラーになるため残せない)
- **残す**: コンポーネント節「既存」マーカーの Blade / JS / 表示系 ViewComposer 等の足場

ルート: 自己完結機能は完全削除で到達不能化(ナビは `Route::has()` 自己ガードが効く)。**横断参照される名前**だけ coming-soon プレースホルダで温存(`Route::view(..., 'placeholders.coming-soon')`)。

> データ層を例外的に残すのは「フレームワーク標準スキーマ × 残すプラミングがそれに依存」する時だけ(例: 全ページ実行の topbar Composer が触るテーブル)。その場合は **模範解答側を feature-aware に小改修**して、テーブル/ルート未作成でも 500 しないようにしてから削る。

### 既存機能の拡張

土台(提供 PJ の所与)は残し、**この Story が足した差分だけ巻き戻す**(diff スタイル):

- 追加された列 / テーブル / リレーション / ロジックのうち、**この Story が足した分**を特定して戻す
- 例: 「面談予約(既存) に Google Calendar 連携(この Story)を足した」→ 面談予約は残し、GCal 連携分のコード/設定だけ削除

「何がこの Story の差分か」は実装方針の差分記述で判断。土台まで削らない(隣の Story/既存機能を壊す)。

## Bug(バグ修正)

**README「Step 4 仕込み方」列** がファイル・行レベルの SSoT。指定箇所を「正しい実装」から「バグった実装」に置換する。

- **同梱テストは残す**: 修正前は赤(バケット①)。受講生がバグを直すと緑。**テストファイルは削除しない**。
- 仕込みは **局所的に**。指定ファイルの指定箇所のみ。広域に波及させない。
- 仕込み後、**チケットの「実際の結果」どおりの症状が出るか**を必ず確認(Phase 2 の Playwright 症状再現)。出ない/別症状なら Phase 3 reconcile。

仕込みの典型:
- ガード/バリデーション削除(例: 在籍重複チェック・`confirmed` ルール・状態遷移の `abort_unless`)
- 状態更新の欠落(例: status を遷移させる 1 行を消す)
- 認可指定の緩め(例: `role:admin` → `role:admin,coach`)
- 計算/比較の誤り(例: スコアの百分率変換漏れ)

> Bug は既存コードを直すチケットなので、難易度ラベルや「Controller で代替可」等の選択肢提示はチケットに書かない(Story と違う)。仕込みも「原因箇所を淡々とバグ化」する。

## Task(リファクタリング・既存改修)

実装方針 > 変更内容 が巻き戻しの SSoT。リファクタ後 → リファクタ前の状態に戻す。

- **perf Task**(N+1 / キャッシュ / chunk 等): 最適化を巻き戻す(N+1 を復活 / キャッシュを外す等)。**同梱の性能テスト(query-count / cache 等)は残す**(最適化前は赤)。
- **refactor Task**(Action 分離 / Service 集約等): 重複・密結合状態に意図的に戻す。**テストは削除**(振る舞い不変なので残すと緑のまま = 信号にならない。受講生が書く)。構造は **コード確認(grep)** で採点する。
- **依存元 Story が gut 済みの refactor**: 依存先のコードが既に消えていれば実質 no-op(前方指示として記録)。

## 削除/巻き戻しの「やりすぎ/足りない」検出

- **やりすぎ**: `route:list` の死にルート(削除 Controller を指す)/ `migrate:fresh --seed` の FK・Factory・Seeder 参照切れ / 残すべき「既存」マーカー部品まで消した
- **足りない**: 削除対象のロジックが残っている(答えだし)/ Story のデータ層が残っている

模範解答との `diff`(vendor/node_modules/.env 等除外)で、引き算の delta が「狙ったチケット分のみ」かを確認できる。
