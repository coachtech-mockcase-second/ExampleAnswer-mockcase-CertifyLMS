# Certify LMS 詳細度 100% チケット仕様 (規約)

本 rules は `関連ドキュメント/要件シート_詳細度100%/` 配下でチケットを詳細化する際に参照する **規約集**。Skill `ticket-detail-100p` が起動時に参照する。

ナビ・進捗・チケット一覧は `関連ドキュメント/要件シート_詳細度100%/README.md` を参照。

## 受け入れ条件と詳細仕様の役割分担

本シートで最も重要な原則。受け入れ条件 (採点項目) と 詳細仕様 (実装方針) の役割を明確に分離する。

### 振る舞いと文言の分離

| 区分 | 内容 | 例 |
|---|---|---|
| **受け入れ条件 (採点項目)** | 振る舞いのみ。具体文言は含めない | 「投稿成功時に作成されたスレッド詳細画面にリダイレクトされる」「投稿成功時にフラッシュメッセージが表示される(文言は適切な日本語であれば良い)」「コーチ / 管理者が投稿しようとすると 403 が返る」 |
| **詳細仕様 (実装方針) 内** | 推奨文言を提示 (ヒアリング応答用) | 「フラッシュ文言例: 『質問を投稿しました』」「エラーメッセージ例: 『タイトルは必須です。』」 |
| **採点ルール** | 振る舞いを客観判定。文言の細部は適切な日本語であれば減点なし | 「『投稿しました』も『投稿成功』も OK」 |

**例外** (具体性を保つ振る舞いの一部):

- リダイレクト先 URL のパス: 「`/qa-board/{thread}` にリダイレクト」(振る舞いの一部)
- 認可挙動の HTTP ステータス: 403 / 404 / 409 等 (振る舞いの一部)

### セクション構成と Step 4 引き算範囲

3 種別ともセクション骨格を **`## 受け入れ条件` → `## 実装方針` → `## 補足`** で統一する。`## 実装方針` 配下サブセクションを種別ごとに使い分ける。

| 種別 | 独立トップレベルセクション | `## 実装方針` 配下サブセクション | `## 実装方針` の Step 4 引き算 |
|---|---|---|---|
| **Story** | `## Seeder 設計`(新規テーブル時のみ、`## 受け入れ条件` 直前) | `### 主要 URL` / `### データモデル` / `### バリデーション` / `### 認可設計` / `### API 仕様` / `### テスト観点` / `### アーキテクチャ判断` / `### 関連ファイルメモ` | ✅ **対象**(模範解答 PJ から該当実装を削除して提供 PJ を作る、新規 Story なら Blade だけ残る) |
| **Bug** | (なし) | `### 主要 URL`(バグ発生画面) / `### 原因箇所メモ`(コーチ用、受講生に直接教えない、Step 4 仕込み内容も整理) | ❌ **対象外**(既にバグが仕込まれている状態の説明 + コーチ用ヒント材料。30% 版変換時に削除 / 抽象化) |
| **Task** | (なし) | `### 主要 URL`(計測対象 / リファクタ対象、Refactoring で対象画面なしなら省略可) / `### 変更対象と変更前後の状態` / `### 計測指標と目標値`(Performance) / `### テスト方針` / `### 採用技術と判断理由` / `### 改善対象コードメモ`(コーチ用、受講生に直接教えない) | ✅ **対象**(模範解答 PJ をリファクタ前 / 最適化前の状態に巻き戻す) |

**Story の Seeder 設計**: 採点動作確認の前提データ。**Step 4 引き算対象外**(提供 PJ にも残す方針。模範解答 PJ の Seeder が 100% シートと不一致なら別ロジックで提供 PJ に投入)。

**主要 URL の位置(3 種別共通)**: `## 実装方針` 配下の最初のサブセクションに置く。Bug は既に提供 PJ にある画面の参照点(引き算対象外)、Story では新規実装対象 / Task では計測 / リファクタ対象。

### 実装方針の性質

- 実装方針は **参考設計の一例**。普通の設計(Design)として適切なフォーマットで記述する(BookShelf シートに無理やり寄せない)
- 振る舞いが受け入れ条件を満たせば実装手段は問わない
- **模範解答 PJ の実装をベースに記述**(spec 当初設計と乖離があれば実装が正)。模範解答 PJ が適切でないと判断した場合は **`_review-log.md` に判定(⚠️ / 🕐) を記録した上で記述** する(下記「模範解答 PJ 懐疑的レビュー」セクション参照)
- 各サブセクション内で「**必須**(振る舞いに必要)/ **受講生判断**(推奨選択肢)」を区別:
  - 必須: URL パス / テーブル名 / 必須カラム / 必須バリデーションルール / Policy 判定方針(ロール × 当事者)
  - 受講生判断: 命名詳細 / インデックス追加判断 / クラス分割 / メッセージ文言 / Seeder 件数

### 実装粒度の記載範囲

実装粒度(テーブル名 / クラス名 / SQL 詳細 / Laravel 実装語彙 等)を記載できるのは **`## Seeder 設計`(Story 新規テーブル時のみ)** と **`## 実装方針` 配下** のみ。それ以外のセクション(概要 / 背景・目的 / ユーザーストーリー / やること / やらないこと / 再現手順 / 期待する動作 / 実際の動作 / 補足)は **業務語彙のみ** で記述する。

#### セクション別の許可範囲

| セクション | 実装粒度の記載 | 備考 |
|---|---|---|
| 概要 / 背景・目的 / ユーザーストーリー / やること / やらないこと | ❌ 業務語彙のみ | LMS ドメイン語彙(後述)は許容 |
| 再現手順 / 期待する動作 / 実際の動作(Bug) | ❌ 業務語彙のみ | 同上 |

| **実装方針 配下**(主要 URL / データモデル / バリデーション / 認可設計 / API 仕様 / テスト観点 / アーキテクチャ判断 / 関連ファイルメモ / 原因箇所メモ 等) | ✅ 実装粒度すべて OK | コーチ・採点者向け参考設計 |
| 受け入れ条件 | △ 一部許容 | リダイレクト先 URL のパス / HTTP ステータス(403 / 404 / 409 / 200 / 201 / 204 等) / 状態名(下書き / 公開中 等) は **振る舞いの一部** として許容(規約「振る舞いと文言の分離」参照)。テーブル名・クラス名・SQL 演算子・バリデーションルール名は NG |
| 補足 内の想定ヒアリング Q&A | △ 仕様レベルの振る舞い記述に限る | HTTP ステータス・URL パスは振る舞いの一部として OK、テーブル名・クラス名・Laravel 実装語彙は NG。コーチが受講生の仕様質問に「正しい振る舞い」で応答する場面なので、業務語彙寄り |

#### LMS ドメイン語彙(業務語彙として扱う)

`product.md` / spec で一級語彙として定義されている LMS ドメインの英語識別子は、業務語彙の一部として扱い、業務語彙化対象外:

- **教材階層**: Part / Chapter / Section(spec で業務語彙として定着、提供 PJ コードでも英語のまま)
- **エンティティ名**: 演習問題 / 教材内画像 / 出題分野マスタ / 面談パック / 質問スレッド / 回答 など(日本語表記推奨、Q&A や受け入れ条件で説明として英語名を併記するのは OK)
- **状態名**: 下書き / 公開中 / アーカイブ / 受講中 / 修了 / 退会 など(日本語表記、Enum 値の英語は実装方針内のみ)
- **ロール**: 受講生(student) / コーチ(coach) / 管理者(admin)(初出のみ括弧書きで英名、以降日本語、コード片 `student` / `coach` / `admin` は実装方針内のみ)
- **Feature 略称**: qa-board / mock-exam / mentoring / notification など(spec の Feature 名、業務語彙の一部として扱う)
- **HTTP ステータス**: 200 / 201 / 204 / 403 / 404 / 409 / 422 / 500 など(振る舞いの一部として常時許容)
- **SoftDelete**(振る舞い概念): 「論理削除」「SoftDelete」両方 OK(振る舞いとしての概念で、Trait 名と区別)

### Basic / Advance 制約

| 範囲 | Basic | Advance |
|---|---|---|
| Controller / FormRequest / Policy / Middleware / Fortify Action | ✅ | ✅ |
| Notification (DatabaseChannel + Mail) / `DB::transaction` / Eloquent (with / withCount / scope 等) | ✅ | ✅ |
| **Action クラス (UseCases)** | ❌(教材範囲外・初出) | ✅ |
| **Service クラス** | ❌(教材範囲外・初出) | ✅ |
| Blade ロジック | ❌ | ❌ |
| 外部 API / Pusher Broadcasting / Sanctum | ❌ | ✅ |

**運用ルール**:

- Basic チケットの詳細仕様 (実装方針) は **Basic 範囲内** で記載 (**Controller 内完結** を前提とする)
- Action / Service の使用は **受講生の判断で推奨**(チャレンジするなら歓迎)、ただし必須ではない
- 模範解答 PJ は Advance 含む完成形 (Action / Service 多用) のため、Basic チケットの想定実装と異なる箇所あり (詳細化時に Basic 範囲で書き直す)
- **Bug 例外**: Action / Service ファイル内のバグなど、構造上どうしても Basic 範囲外を参照する必要がある場合、チケット内に「**※**」記号で例外説明を明記してから記載

## 想定ヒアリング Q&A の粒度

各チケットの `## 補足` 配下 `### 想定ヒアリング Q&A` セクションに含める内容は **要件・仕様レベルの「正しい振る舞い確認」のみ** とする。

### 含める粒度(仕様確認用)

- **バリデーション仕様**: 最大文字数、必須 / 任意、形式制約、上下限値
- **認可振る舞い**: 誰がアクセス可能か、HTTP ステータス(403 / 404 / 409 等)、当事者判定の規則
- **振る舞い詳細**: フラッシュ表示の有無、リダイレクト先、副作用(通知発火・状態遷移)、SoftDelete / 物理削除の選択
- **文言の推奨例**: フラッシュ・エラーメッセージの推奨文(コーチが受講生に提示する応答材料、文言細部は採点対象外)
- **データの境界**: 公開停止資格は 404 / 担当外資格は 403 など、状態 × ロールで振る舞いが分岐する条件

### 含めない粒度(設計判断・実装詳細)

- **クラス分割**: Action / Service を使うべきか、Controller を分けるか
- **命名**: ファイル名 / クラス名 / メソッド名
- **DI / Eloquent 詳細**: `with()` の具体的指定、Repository パターン採否
- **テスト場所 / 種別**: Feature / Unit どちらに書くか、ファイルパス
- **HTTP メソッド選択**: POST / PATCH のどちらか(振る舞いに直接影響しないなら実装詳細)

設計判断 / 実装詳細はコーチが受講生のヒアリングに応じて **個別判断**(受け入れ条件を満たせば実装手段は問わないため、Q&A 化して画一回答を渡すと教育上 NG)。

### 種別別の必要性

| 種別 | 想定ヒアリング Q&A |
|---|---|
| **Story** | **必須**(詳細化のメインで、30% 版で「やること」「受け入れ条件」が抽象化された際にコーチが即答する材料) |
| **Bug** | **任意**。30% 版で「期待する動作」「実際の動作」が抽象化された際の「正しい振る舞い確認」用。本文セクションで仕様が完全にカバーされていれば省略可 |
| **Task** | **任意**。Refactoring は振る舞い不変 / Performance は計測値中心で仕様確認の余地が小さい。必要なら書く |

### フラットなテーブル構造

3 階層(必須回答 / 実装判断 / 補足)構造は廃止。**フラットなテーブル 1 個** で記述する。「実装判断(設計レベル)」階層は本規約で扱わないため不要、「補足」階層は仕様確認に統合する。

## 受け入れ条件の書き方原則

**受け入れ条件 = 採点項目**。各項目 = 評価シート ① の 1 採点行 と 1:1 対応する。

- **振る舞いベース** (Story / Bug): ユーザー外面 (操作 / 表示 / 副作用) で記述、コード内部実装は含めない
- **Task の例外**: Task は画面動作で判定不能 (Refactoring は振る舞い不変 / Performance は計測値が中心) なので、PR の file Changes で判定する **構造記述** / **Before/After 計測値** で代替可。採点フローも「PR の file Changes を確認」が中心
- **1 項目 1 振る舞い**: 複合表現は分割
  - ❌ 「投稿成功時にリダイレクト + フラッシュ表示。コーチが投稿すると 403」(3 振る舞いが混在)
  - ✅ 「投稿成功時に詳細画面にリダイレクトされる」「投稿成功時にフラッシュが表示される」「コーチが投稿すると 403」(3 項目に分割)
- **文言は含めない**: 具体的なフラッシュ文言・エラーメッセージ文言・タイトル等は受け入れ条件に書かない (実装方針内に推奨例として提示)
- **MECE**: 同じ振る舞いを複数項目に分散させない
- **採点者目線**: 採点者が「✅ / ❌」を 1 回判断で付けられる粒度
- **例外**: 強く結合した一連の振る舞い (操作 → 即時の応答) は 1 項目に統合可

### 種別別の項目数目安

| 種別 | 受け入れ条件項目数 |
|---|---|
| Story | 8〜12 (規模大なら 15-19) |
| Bug | 1〜3 (Bug 固有 AC のみ。共通 DoD = テスト pass / PR 7 セクション / 動画 は AC に書かず評価シート ③ に集約) |
| Task | 3〜5 |

### 横断品質との分離

「テスト pass」「PR 7 セクション完備」「動画記述」は **チケット個別の受け入れ条件には書かない**(全チケット共通の横断採点として評価シート ③ に集約)。

### やる ↔ 受け入れ条件 の網羅性

Story の `## やること` 内に書いた振る舞いは、すべて `## 受け入れ条件` に対応行が存在することを詳細化時に確認する。「やる」だけに書いて AC 漏れがあると採点項目から落ちる。

- やる > エンティティ単位の操作(投稿 / 編集 / 削除 等)→ AC で個別の振る舞いに分解(認可挙動 / 副作用 / 表示等)
- やる > `### 共通の振る舞い`(横断ルール・特殊ケース)→ AC に必ず 1 行以上の対応行
- AC の各項目 → やる のどこかの機能ブロックに居場所が必要(逆方向の網羅性も成立)

## ロール表記ルール

LMS 内のロールは以下のように表記する:

- **受講生(student)** / **コーチ(coach)** / **管理者(admin)**

各チケット内の **初出時にフル表記**(日本語 + 括弧書きの英名)、以降は日本語のみ(「受講生」「コーチ」「管理者」)。コード片(バッククォート内)、Enum 値、`auth:` ミドルウェア名等は英のまま。

> 注: 本シート使用側の「採点者」「コーチ(PM 役)」は LMS ロールとは別概念(README ロール責務表参照)。LMS のコーチと PM 役のコーチは同一人物の場合が多いが、責務として分離する。

## 依存チケットの記録方向

メタ情報「依存チケット」は **「このチケットが依存する(= 先に存在しないと困る)チケット」** を記録する。**「このチケットに依存される」方向は記録しない**(依存される側を網羅メンテするコストが高い + 採点者目線では入口が分かれば十分)。

- 例: `S-B-01`(qa-board)は `S-B-04`(notification 基盤)に依存 → S-B-01 のメタ情報に `S-B-04` と書く。S-B-04 側には書かない
- なしの場合は `(なし)` と明記

## 模範解答 PJ 懐疑的レビュー

> **なぜ懐疑的レビューが必要か**: 模範解答 PJ は **受講生が理想的な実装を学ぶ教材** であるべきため、本チケットを抽出する前に「現在の実装がベストか?」を問い直す。引き算で受講生に渡す前に模範解答 PJ を理想形に修正しておくことで、受講生は正しい設計を学べる。

各チケット詳細化時に、対応する模範解答 PJ 実装に対して **「現在の実装がベストか?」を必ず一度問い直す**。

検証ポイント: 画面分割の妥当性 / 認可粒度 / レイヤー配置 / API 設計 / ロール間共通化

判定を `関連ドキュメント/要件シート_詳細度100%/_review-log.md` に記録:

- ✅ 適切 — 現状で問題なし
- ⚠️ 修正必要 — 詳細化を一時停止して模範解答 PJ を修正してから再開(引き算前に修正する方が出戻りが少ない)
- 🕐 Phase D で一括処理 — 他 Feature と横断的に判断する必要、詳細化完了後にまとめて対応

## 採点必要レイヤー区分

100% 版に書く / 採点対象 の判定基準。詳細化時の迷いを最小化する。

| レイヤー | 100% 版での記載位置 | 採点 |
|---|---|---|
| **A. ユーザー外面の振る舞い** | | |
| 画面パス(URL) | ✅ Story / Bug / Task 共通: `## 実装方針` > `### 主要 URL`(Task の Refactoring で対象画面なしなら省略可) | ✅ 必須 |
| 画面表示要素 | ✅ 受け入れ条件 | ✅ 必須 |
| リダイレクト先 URL | ✅ 受け入れ条件 (パスを含む) | ✅ 必須 |
| フラッシュメッセージ表示 | ✅ 受け入れ条件 (表示有無のみ、文言含めない) | ✅ 必須 |
| フラッシュメッセージ文言 | ✅ 実装方針 > 推奨文言例 (Q&A 連携) | △ (適切な日本語なら OK、減点なし) |
| 認可挙動(403 等) | ✅ 受け入れ条件 (HTTP ステータス含む) | ✅ 必須 |
| 副作用(通知発火 / 状態遷移) | ✅ 受け入れ条件 | ✅ 必須 |
| **B. データ・スキーマ最低限** | | |
| 新規テーブル名(Story 新規のみ) | ✅ 概要 + 実装方針 > データモデル | ✅(新規 Story のみ) |
| **C. ヒアリング集約(Q&A・実装方針)** | | |
| バリデーション詳細 (ルール) | ✅ 実装方針 > バリデーション + 補足 > Q&A | △ (振る舞いで採点、文言不問) |
| エラーメッセージ文言 | ✅ 実装方針 > バリデーション + 補足 > Q&A | △ (適切な日本語なら OK) |

## 30% 版変換 (Phase F)

今は考慮しない。Phase F で変換ルールを別途定義する(`## 実装方針` + `## 補足` セクションを削除、`## やること` / `## 受け入れ条件` を抽象化辞書で一般化する想定)。
