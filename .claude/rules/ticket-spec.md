# Certify LMS 詳細度 100% チケット仕様 (規約)

本 rules は `関連ドキュメント/要件シート_詳細度100%/` 配下でチケットを詳細化する際に参照する **規約集**。Skill `ticket-detail-100p` が起動時に参照する。

ナビ・進捗・チケット一覧は `関連ドキュメント/要件シート_詳細度100%/README.md` を参照。

## 受け入れ条件と詳細仕様の役割分担

本シートで最も重要な原則。受け入れ条件 (採点項目) と 詳細仕様 (実装方針) の役割を明確に分離する。

### 振る舞いと文言の分離

| 区分 | 内容 | 例 |
|---|---|---|
| **受け入れ条件 (採点項目)** | 振る舞いのみ。具体文言は含めない | 「投稿成功時に作成されたスレッド詳細画面にリダイレクトされる」「投稿成功時にフラッシュメッセージが表示される(文言は適切な日本語であれば良い)」「コーチ / 管理者が投稿しようとすると 403 が返る」 |
| **詳細仕様 (実装方針) 内** | 推奨文言を提示 (ヒアリング応答用) | 「フラッシュ文言例: 『質問を投稿しました』」「エラーメッセージ例: 『タイトルは必須です。』」 |
| **採点ルール** | 振る舞いを客観判定。文言の細部は適切な日本語であれば減点なし | 「『投稿しました』も『投稿成功』も OK」 |

**例外** (具体性を保つ振る舞いの一部):

- リダイレクト先 URL のパス: 「`/qa-board/{thread}` にリダイレクト」(振る舞いの一部)
- 認可挙動の HTTP ステータス: 403 / 404 / 409 等 (振る舞いの一部)

### 詳細仕様の参考性

- 詳細仕様 (実装方針) は **参考**。**COACHTECH 受講生ごとに正解が異なることを許容**
- 振る舞いが受け入れ条件を満たせば実装手段は問わない
- 各サブセクション内で「**必須** (振る舞いに必要) / **受講生判断** (推奨選択肢)」を区別:
  - 必須: テーブル名 / 必須カラム / 必須バリデーションルール / Policy 判定方針 (ロール × 当事者)
  - 受講生判断: 命名詳細 / インデックス追加判断 / クラス分割 / メッセージ文言 / Seeder 件数

### Basic / Advance 制約

| 範囲 | Basic | Advance |
|---|---|---|
| Controller / FormRequest / Policy / Middleware / Fortify Action | ✅ | ✅ |
| Notification (DatabaseChannel + Mail) / `DB::transaction` / Eloquent (with / withCount / scope 等) | ✅ | ✅ |
| **Action クラス (UseCases)** | ❌(教材範囲外・初出) | ✅ |
| **Service クラス** | ❌(教材範囲外・初出) | ✅ |
| Blade ロジック | ❌ | ❌ |
| 外部 API / Pusher Broadcasting / Sanctum | ❌ | ✅ |

**運用ルール**:

- Basic チケットの詳細仕様 (実装方針) は **Basic 範囲内** で記載 (**Controller 内完結** を前提とする)
- Action / Service の使用は **受講生の判断で推奨**(チャレンジするなら歓迎)、ただし必須ではない
- 模範解答 PJ は Advance 含む完成形 (Action / Service 多用) のため、Basic チケットの想定実装と異なる箇所あり (詳細化時に Basic 範囲で書き直す)
- **Bug 例外**: Action / Service ファイル内のバグなど、構造上どうしても Basic 範囲外を参照する必要がある場合、チケット内に「**※**」記号で例外説明を明記してから記載

## 受け入れ条件の書き方原則

**受け入れ条件 = 採点項目**。各項目 = 評価シート ① の 1 採点行 と 1:1 対応する。

- **振る舞いベース** (Story / Bug): ユーザー外面 (操作 / 表示 / 副作用) で記述、コード内部実装は含めない
- **Task の例外**: Task は画面動作で判定不能 (Refactoring は振る舞い不変 / Performance は計測値が中心) なので、PR の file Changes で判定する **構造記述** / **Before/After 計測値** で代替可。採点フローも「PR の file Changes を確認」が中心
- **1 項目 1 振る舞い**: 複合表現は分割
  - ❌ 「投稿成功時にリダイレクト + フラッシュ表示。コーチが投稿すると 403」(3 振る舞いが混在)
  - ✅ 「投稿成功時に詳細画面にリダイレクトされる」「投稿成功時にフラッシュが表示される」「コーチが投稿すると 403」(3 項目に分割)
- **文言は含めない**: 具体的なフラッシュ文言・エラーメッセージ文言・タイトル等は受け入れ条件に書かない (実装方針 (参考) 内に推奨例として提示)
- **MECE**: 同じ振る舞いを複数項目に分散させない
- **採点者目線**: 採点者が「✅ / ❌」を 1 回判断で付けられる粒度
- **例外**: 強く結合した一連の振る舞い (操作 → 即時の応答) は 1 項目に統合可

### 種別別の項目数目安

| 種別 | 受け入れ条件項目数 |
|---|---|
| Story | 8〜12 (規模大なら 15-19) |
| Bug | 共通 DoD 3 + Bug 固有 AC 1〜2 |
| Task | 3〜5 |

### 横断品質との分離

「テスト pass」「PR 7 セクション完備」「動画記述」は **チケット個別の受け入れ条件には書かない**(全チケット共通の横断採点として評価シート ③ に集約)。

## ロール表記ルール

LMS 内のロールは以下のように表記する:

- **受講生(student)** / **コーチ(coach)** / **管理者(admin)**

各チケット内の **初出時にフル表記**(日本語 + 括弧書きの英名)、以降は日本語のみ(「受講生」「コーチ」「管理者」)。コード片(バッククォート内)、Enum 値、`auth:` ミドルウェア名等は英のまま。

> 注: 本シート使用側の「採点者」「コーチ(PM 役)」は LMS ロールとは別概念(README ロール責務表参照)。LMS のコーチと PM 役のコーチは同一人物の場合が多いが、責務として分離する。

## 依存チケットの記録方向

メタ情報「依存チケット」は **「このチケットが依存する(= 先に存在しないと困る)チケット」** を記録する。**「このチケットに依存される」方向は記録しない**(依存される側を網羅メンテするコストが高い + 採点者目線では入口が分かれば十分)。

- 例: `S-B-01`(qa-board)は `S-B-05`(notification 基盤)に依存 → S-B-01 のメタ情報に `S-B-05` と書く。S-B-05 側には書かない
- なしの場合は `(なし)` と明記

## 模範解答 PJ 懐疑的レビュー

> **なぜ懐疑的レビューが必要か**: 模範解答 PJ は **受講生が理想的な実装を学ぶ教材** であるべきため、本チケットを抽出する前に「現在の実装がベストか?」を問い直す。引き算で受講生に渡す前に模範解答 PJ を理想形に修正しておくことで、受講生は正しい設計を学べる。

各チケット詳細化時に、対応する模範解答 PJ 実装に対して **「現在の実装がベストか?」を必ず一度問い直す**。

検証ポイント: 画面分割の妥当性 / 認可粒度 / レイヤー配置 / API 設計 / ロール間共通化

判定を `関連ドキュメント/要件シート_詳細度100%/_review-log.md` に記録:

- ✅ 適切 — 現状で問題なし
- ⚠️ 修正必要 — 詳細化を一時停止して模範解答 PJ を修正してから再開(引き算前に修正する方が出戻りが少ない)
- 🕐 Phase D で一括処理 — 他 Feature と横断的に判断する必要、詳細化完了後にまとめて対応

## 採点必要レイヤー区分

100% 版に書く / 採点対象 の判定基準。詳細化時の迷いを最小化する。

| レイヤー | 100% 版での記載位置 | 採点 |
|---|---|---|
| **A. ユーザー外面の振る舞い** | | |
| 画面パス(URL) | ✅ 実装方針 > 主要画面・操作 (Story/Task) または 補足 > 主要 URL (Bug) | ✅ 必須 |
| 画面表示要素 | ✅ 受け入れ条件 | ✅ 必須 |
| リダイレクト先 URL | ✅ 受け入れ条件 (パスを含む) | ✅ 必須 |
| フラッシュメッセージ表示 | ✅ 受け入れ条件 (表示有無のみ、文言含めない) | ✅ 必須 |
| フラッシュメッセージ文言 | ✅ 実装方針 > 推奨文言例 (Q&A 連携) | △ (適切な日本語なら OK、減点なし) |
| 認可挙動(403 等) | ✅ 受け入れ条件 (HTTP ステータス含む) | ✅ 必須 |
| 副作用(通知発火 / 状態遷移) | ✅ 受け入れ条件 | ✅ 必須 |
| **B. データ・スキーマ最低限** | | |
| 新規テーブル名(Story 新規のみ) | ✅ 概要 + 実装方針 > データモデル | ✅(新規 Story のみ) |
| **C. ヒアリング集約(Q&A・実装方針)** | | |
| バリデーション詳細 (ルール) | ✅ 実装方針 > バリデーション + 補足 > Q&A | △ (振る舞いで採点、文言不問) |
| エラーメッセージ文言 | ✅ 実装方針 > バリデーション + 補足 > Q&A | △ (適切な日本語なら OK) |
| 認可ルール詳細(Policy 方針) | ✅ 実装方針 > 認可設計 + 補足 > Q&A | △ (挙動で採点) |
| **D. 書かない(振る舞いで採点)** | | |
| Controller / Service / Action クラス名 | ❌ 実装方針 内では言及可(参考) | ❌(命名は ② 横断で採点) |
| Policy メソッド名 | ❌ 実装方針 内では言及可(参考) | ❌ |
| テストファイル名 / パス | ❌ | ❌(pass で採点) |
| **E. パフォーマンス Task 専用** | | |
| Before/After 計測値 | ✅ 受け入れ条件 + 実装方針 > Before/After 計測方針 | ✅ 必須 |
