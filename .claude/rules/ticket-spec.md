# Certify LMS 詳細度 100% チケット仕様 (規約)

本 rules は `関連ドキュメント/要件シート_詳細度100%/` 配下でチケットを詳細化する際に参照する規約集。Skill `ticket-detail-100p` から起動時に参照される。

ナビ・進捗・チケット一覧は `関連ドキュメント/要件シート_詳細度100%/README.md` を参照。

---

## 1. セクション構成

種別ごとのセクション骨格は **`templates/{story,bug,task}.md`** を参照(本規約ではセクション内の記述ルールのみ規定)。

種別ごとの主な違い:

| 種別 | 冒頭セクション | 受け入れ条件 | 工数規模 |
|---|---|---|---|
| **Story** | 背景・目的 + ユーザーストーリー | 3 ブロック(振る舞い / 認可 / 入力検証) | 通常 5-9h、規模大 10h+ |
| **Bug** | 概要 + 再現手順 + 期待する結果 + 実際の結果 | 単一フラット 1-3 件 | 2-8h |
| **Task** | 概要 | 3 ブロック(認可・入力検証は該当時のみ)、Performance は計測値中心 | 3-6h |

---

## 2. 記述粒度の原則

**チケット全体を業務語彙 + ハイレベル設計に揃える**(実務要件粒度)。詳細値(カラム名 / 型 / FK / Enum 値 / クラス名 / バリデーション機械値 / SQL クエリ)はチケット本体に書かず、必要なら以下に隔離する:

- **`## 補足 > ### 想定 Q&A`** — コーチがヒアリング応答に使う詳細仕様(文字数制限・状態定義・フラッシュ文言例 等)
- **`## 実装方針 > ### 関連ファイル`** — コードリーディング起点となる具体ファイルパス・クラス名

実装方針は「**参考設計の一例**」であり、振る舞いが受け入れ条件を満たせば実装手段は問わない。

### 2.1 セクション別の記載範囲

| セクション | 実装粒度 | 備考 |
|---|---|---|
| 背景・目的 / ユーザーストーリー / 概要 / やること / やらないこと / 再現手順 / 期待する結果 / 実際の結果 | ❌ 業務語彙のみ | LMS ドメイン語彙(後述 2.3)は許容 |
| 受け入れ条件 | △ 一部許容 | URL パス / HTTP ステータス / 状態名 / ロール識別子 は振る舞いの一部として OK。テーブル名・クラス名は NG |
| **実装方針 配下**(主要 URL / 認可方針 / データモデル / Seeder 設計 / API 仕様 / 変更対象 / 採用技術 / 原因箇所メモ) | ❌ 業務語彙 + ハイレベル設計のみ | 「面談パック: 名前 / 面談回数 / 状態」のような業務語彙ベース。カラム名・型・FK 詳細は禁止 |
| 実装方針 > 関連ファイル | ✅ ファイルパス・クラス名 OK | コードリーディング起点として具体パスを列挙 |
| 補足 > 想定 Q&A | ✅ 詳細仕様 OK(コーチ即答用) | 業務語彙寄り、Laravel 実装語彙は NG |

### 2.2 禁止される記載

| カテゴリ | 例 | 代替(業務語彙) |
|---|---|---|
| テーブル名 / カラム名 | `qa_threads` / `meeting_packs.status` | 質問スレッド / 公開状態 |
| 具体的なクラス名 / メソッド名 | `MeetingPackController` / `AiChatConversationPolicy` / `assignedCoach()` / `EnsureActiveLearning::handle()` | Policy / Controller / Middleware(技術名のみ) |
| Enum 値 / SQL / バリデーション機械値 | `status=draft` / `where(...)` / `max:200` | 下書き / 公開中の絞り込み / 入力検証あり |
| ファイルパス | `app/Policies/PartPolicy.php` | (記載しない、必要なら「関連ファイル」内のみ) |
| 別チケット言及(依存以外) | 「`S-B-04` で扱う通知基盤を利用」 | チケット間の交差参照は依存チケット欄のみ |
| 過去案件への言及 | 「BookShelf 流の...」「BookShelf 既習パターン」 | Certify LMS 単体で完結させる |
| 構築側メタ情報 | 「Step 4 引き算範囲」「Phase D」「`_review-log.md` 参照」 | 受講生・採点者の視野外、書かない |
| 各セクション冒頭の説明文 | `## 受け入れ条件\n\n以下に...` の 1 文 | セクション見出しで意味が伝わる、冗長 |
| 採点運用情報 | 「採点者は ✅ / ❌ で判定」「評価シート ① に転記」 | 評価シート側で管理 |

> **Laravel 技術名(`Policy` / `FormRequest` / `Service` / `Middleware` / `Controller` / `Action` / `Repository` / `Seeder` 等)の扱い**:
> - **実装方針内**: 記載 OK(受講生が学習済みの基礎技術。「認可は Policy で実装」「バリデーションは FormRequest で実装」等で参考設計を示す)
> - **受け入れ条件**: **記載しない**(振る舞いベースで書く。採点者は画面・動作・HTTP レスポンスで判定するため、内部で `Policy` を使っているか手動チェックかは関係ない)
> - **具体的なクラス名・メソッド名**(`AiChatConversationPolicy` 等)は受講生ごとに命名が異なるため、AC・実装方針ともに NG

### 2.3 LMS ドメイン語彙(業務語彙として常時許容)

`product.md` / spec で一級語彙として定着している以下は、業務語彙の一部として扱う:

- **教材階層**: Part / Chapter / Section
- **エンティティ名**: 演習問題 / 教材内画像 / 出題分野マスタ / 面談パック / 質問スレッド / 回答 等(日本語推奨)
- **状態名**: 下書き / 公開中 / アーカイブ / 受講中 / 修了 / 退会 等(日本語、Enum 値の英語は禁止)
- **ロール**: 受講生(student) / コーチ(coach) / 管理者(admin)
- **Feature 略称**: qa-board / mock-exam / mentoring / notification 等
- **HTTP ステータス**: 200 / 302 / 403 / 404 / 409 / 422 / 500 等
- **SoftDelete**(振る舞い概念): 「論理削除」「SoftDelete」両方 OK

### 2.4 実装方針サブセクションの粒度ガイド

「設計と実装の中間レベル」= **参考設計**として書く。各サブセクションの含める / 含めない の指針:

| サブセクション | 含める | 含めない |
|---|---|---|
| **主要 URL** | 業務操作名 × URL パスのテーブル + ルート全体に適用するミドルウェア概念(認証 / ロール / レート制限 等) | HTTP メソッド名(GET/POST 等)、Controller クラス名、ルート name |
| **認可方針** | ロール × 状態 × 当事者のレベルで「誰が何にアクセス可・拒否される」 | Policy クラス名、メソッド名、`$this->authorize()` 等の実装 |
| **データモデル**(新規時) | エンティティ業務語彙名 + 主要属性(業務語彙) + リレーション概念 + SoftDelete 採否(振る舞い概念として) | テーブル名、カラム名、型、FK、インデックス、Migration コード |
| **Seeder 設計**(新規時) | 状態別件数バリエーション + 採点動作確認用途 | Seeder クラス名、Factory 詳細、`DatabaseSeeder` 追加順序 |
| **API 仕様**(該当時) | エンドポイント概念 + 認証要否 + リクエスト / レスポンスの大枠 | JSON 構造詳細、Resource クラス名、FormRequest クラス名 |
| **原因箇所メモ**(Bug、コーチ向け) | 業務語彙 + エンティティ名でバグ箇所を示唆 + 受講生が辿るべき修正範囲のヒント | ファイルパス、クラス名、修正コード片(→ 関連ファイルへ) |
| **変更対象と変更前後の状態**(Task) | 業務語彙で変更対象 + 変更前 + 変更後 | クラス名、メソッド名、`with()` クエリ詳細 |
| **採用技術と判断理由**(Task) | 技術概念 + 判断軸(なぜそれを選ぶか) | SQL 詳細、Eloquent クエリ完全例 |
| **関連ファイル** | コードリーディング起点となる主要ファイルパス・クラス名 | (本サブセクション内は制限なし、コーチ事前準備用) |

## 3. 受け入れ条件の書き方

**受け入れ条件 = 採点項目**。各項目 = 評価シート ① の 1 採点行 と 1:1 対応する。

「仕様の網羅記述」ではなく「**採点者が ✅ / ❌ を 1 回で判断できる項目リスト**」として書く(BookShelf 評価シートが参考)。

### 3.1 書き方の基本

- **happy path 中心**: 正常系の主要な振る舞いを 1 行 = 1 採点項目で記述
- **重要なエッジケースのみ独立**: AI 失敗フォールバック / 機能 OFF / 特殊な状態遷移 など、教材的に重要なものに絞る(全パターン羅列はしない)
- **網羅性より採点しやすさ**: 全 HTTP ステータスを書き並べない。「拒否される」「アクセスできない」レベルに丸める
- **具体的に書く(抽象表現を避ける)**: 「適切な〜」「正しく実装され」「アクセス拒否」のような曖昧語は使わず、**画面・動作・HTTP レスポンスとして観察できる事象**(画面非表示 / エラーメッセージ表示 / 403 / リダイレクト 等)で記述
  - ❌ 「機能 OFF / API キー未設定 / 上限超過時に、適切な案内とともに利用が拒否される」(3 件が混ざる + 「適切な」が曖昧)
  - ✅ 「機能 OFF の場合、サイドバー / ウィジェットが非表示になり、AI 相談ルートにアクセスしても画面が表示されない」/「受講生が日次送信上限を超えてメッセージを送信した際に、上限超過の旨のエラーメッセージが表示され、送信が拒否される」
  - ❌ 「入力検証が正しく実装され、エラー時に適切なメッセージが表示される」(「正しく」「適切な」が曖昧)
  - ✅ 「会話タイトル(空 / 長すぎる)/ メッセージ本文(空 / 長すぎる)を入力した際に、バリデーションエラーが表示され、保存・送信されない」
  - ❌ 「コーチ / 管理者の AI 相談ルートへのアクセスが拒否される」(「拒否」だけだと何が起きるか不明)
  - ✅ 「コーチ / 管理者 / 学習中以外の受講生が AI 相談ルートにアクセスすると 403 が返る」
- **技術名は AC に書かない**: `Policy` / `FormRequest` / `Middleware` 等の技術名は採点判定に不要(採点者は振る舞いを観察で判定するため)。実装方針内では記載 OK
- **URL パスを含める**(採点しやすさのため): `/ai-chat/conversations/{conversation}` 等
- **HTTP ステータスは適宜含める**: 採点判定に有用なら 403 / 404 / 422 等を書いて OK
- **文言は含めない**: フラッシュ・エラーメッセージ文言は実装方針 or Q&A に推奨例として提示。バリデーション機械値(`max:200` 等)も書かず、業務語彙(「長すぎる」)で表現
- **文体は固定しない**: 「〜される」(断定形)/「〜できる」/「〜こと」/「〜されるか」(疑問形)を自然に使い分け OK
- **フラット構造**: 3 ブロック構造は採用しない。機能の主要な振る舞い → エッジケース → 認可 → 入力検証 の順で **箇条書きを 1 段でフラットに並べる**(順序は規模・自然さに応じて柔軟)

### 3.2 採点項目に含める要素

チケットごとに該当するものをフラットに列挙する:

| 種類 | 例 |
|---|---|
| **正常系 happy path** | 「`/ai-chat` にアクセスした際に、最新の会話詳細画面に遷移する(0 件なら空状態画面)」 |
| **重要なエッジケース** | 「AI 応答に失敗した場合、受講生メッセージは保存され、エラー状態のメッセージから再送信できる」 |
| **副作用 / 状態遷移** | 「会話を削除した際に、配下のメッセージも連動削除される」 |
| **認可**(該当時のみ、機能群ごとに 1 件) | 「会話のオーナーのみが自分の会話を閲覧・編集・削除・メッセージ送受信でき、他の受講生が直アクセスすると 403 が返る」 |
| **入力検証**(該当時のみ) | 「会話タイトル(空 / 長すぎる)/ メッセージ本文(空 / 長すぎる)を入力した際に、バリデーションエラーが表示され、保存・送信されない」 |

> 認可の書き方: 同じ判定で複数機能をカバーする場合は 1 件にまとめる(機能ごとに繰り返さない)。振る舞いベースで「誰が何をできる / 他がアクセスすると 403 が返る」のように観察可能な事象で表現する。

### 3.3 種別別の件数目安

| 種別 | 件数目安 |
|---|---|
| Story 通常 | **4-7 件** |
| Story 規模大 | **6-10 件** |
| Bug | **1-3 件** |
| Task | **3-5 件** |

**合計目標**: 40 チケットで **約 200 件**(平均 5 件 / チケット、BookShelf 比例)。

### 3.4 圧縮原則(統合チェックリスト)

AC を 2 つ以上書こうとしているとき、以下に該当するなら **1 項目に統合**:

| # | パターン | 統合方向 |
|---|---|---|
| A | 同一業務操作の側面分解(成功 / 副作用 / 通知 を別 AC) | 1 行統合(「成功時に〜され、〜される」) |
| B | 複数機能で同じ判定の認可拒否 | 機能群レベルで 1 項目 |
| C | バリデーションエラーをルール別に細分化 | 「入力検証が正しく実装され、エラー時に適切なメッセージが表示される」1 行 |
| D | UI トリガーを細分化(クリック / ESC / 外側) | 1 行統合 |
| E | 同型状態遷移 N 種(公開 / アーカイブ / 下書き戻し) | 1 行統合 |
| F | 機能 OFF スイッチを細分化 | 1 行統合(「機能 OFF / API キー未設定 / 上限超過時に適切な案内とともに拒否される」) |
| G | 同一動線の表示要素を細分化(未読バッジ / 空状態 / 99+) | 1 行統合 |
| H | 横断採点項目(動画 / テスト pass / 振る舞い不変) | AC に書かず ② / ③ に集約(3.5 参照) |
| I | 同操作のバリエーション(アップロード / 削除 / MIME / サイズ) | 1-2 行に集約 |

**統合判断**: 「同じ業務操作の異なる側面 / 同じ実装ファイル内で完結 / 1 PR で連動」のすべてに該当すれば統合する。

### 3.5 横断採点項目は AC に書かない

以下は **チケット個別 AC には書かない**(全チケット共通の横断採点として評価シート ② / ③ に集約):

- テスト pass / PR 7 セクション完備 / 動画記述
- Pint / 命名 / N+1 / Clean Architecture からの逸脱
- 振る舞い不変(Task)

### 3.6 共通原則

- **MECE**: 同じ振る舞いを複数項目に分散させない
- **採点者目線**: 採点者が「✅ / ❌」を 1 回判断で付けられる粒度
- **強結合な振る舞いは 1 項目に統合**: 「成功時に保存 + リダイレクト + フラッシュ表示」は 1 行
- **分割の例外**: 1 行に詰めすぎて読みづらいときは分割 OK(ただし件数目安を超えるなら統合を再検討)

## 4. やること の書き方

- 機能・操作を **業務語彙の箇条書き** でシンプルに列挙(「○○を作成する」「○○を表示する」)
- サブセクション分けは規模により任意。**業務語彙ベースの大カテゴリ**(「面談パックの管理」「面談予約フロー」)で分ける。技術用語(「エンティティ」「リソース」)は NG
- **認可挙動・HTTP ステータス・状態遷移詳細は記載しない**(認可は実装方針 > 認可方針、振る舞いは受け入れ条件で網羅)
  - ✅ 「会話の新規作成」
  - ❌ 「会話作成: 受講生のみ可、他は 403」

### 4.1 やる ↔ 受け入れ条件 の網羅性

「やること」内の業務操作は、すべて `## 受け入れ条件` に対応行が存在することを詳細化時に確認:

- やる の業務操作 → AC 振る舞いブロックで 1 業務操作 1 項目
- 認可規則 → AC 認可ブロックで機能群統合
- 入力検証 → AC 入力検証ブロックで 1 行統合
- AC の各項目 → やる のどこかに居場所(逆方向の網羅性)

## 5. 想定 Q&A の粒度

`## 補足 > ### 想定 Q&A` は **要件・仕様レベルの「正しい振る舞い確認」のみ** を含む。形式は **フラットなテーブル 1 個**。

### 5.1 含める / 含めない

| 含める(仕様確認) | 含めない(設計判断・実装詳細) |
|---|---|
| バリデーション仕様(最大文字数 / 必須 / 任意 / 上下限値) | クラス分割(Action / Service を使うべきか) |
| 認可振る舞い(誰がアクセス可能 / HTTP ステータス / 当事者判定) | 命名(ファイル名 / クラス名 / メソッド名) |
| 振る舞い詳細(フラッシュ表示 / リダイレクト先 / 副作用 / SoftDelete 採否) | DI / Eloquent 詳細(`with()` 指定 / Repository 採否) |
| 文言の推奨例(フラッシュ・エラーメッセージ) | テスト場所 / 種別 / ファイルパス |
| データの境界(状態 × ロールで分岐する条件) | HTTP メソッド選択(POST / PATCH のどちら) |

設計判断 / 実装詳細はコーチが受講生のヒアリングに応じて **個別判断**(受け入れ条件を満たせば実装手段は問わない)。

### 5.2 種別別の必要性

| 種別 | 必要性 |
|---|---|
| Story | **必須**(30% 版抽象化時にコーチが即答する材料) |
| Bug | **任意**(本文セクションで仕様が完全にカバーされていれば省略可) |
| Task | **任意**(Refactoring は振る舞い不変 / Performance は計測値中心で仕様確認の余地が小さい) |

## 6. Basic / Advance 制約

| 範囲 | Basic | Advance |
|---|---|---|
| Controller / FormRequest / Policy / Middleware / Fortify Action | ✅ | ✅ |
| Notification(DatabaseChannel + Mail) / `DB::transaction` / Eloquent(with / withCount / scope) | ✅ | ✅ |
| Action クラス(UseCases) | ❌ | ✅ |
| Service クラス | ❌ | ✅ |
| Blade ロジック | ❌ | ❌ |
| 外部 API / Pusher Broadcasting / Sanctum | ❌ | ✅ |

**運用ルール**:

- Basic チケットの実装方針は **Basic 範囲内**(Controller 内完結が前提)
- Action / Service の使用は受講生判断で推奨(チャレンジするなら歓迎、必須ではない)
- 模範解答 PJ は Advance 含む完成形のため、Basic チケットの想定実装と異なる箇所あり(詳細化時に Basic 範囲で書き直す)
- **Bug 例外**: Action / Service ファイル内のバグなど構造上 Basic 範囲外を参照する必要がある場合、チケット内に「**※**」記号で例外説明を明記

## 7. ロール表記ルール

- LMS ロール: **受講生(student) / コーチ(coach) / 管理者(admin)**
- 初出時にフル表記(日本語 + 括弧書きの英名)、以降は日本語のみ
- コード片(バッククォート内)、Enum 値、`auth:` ミドルウェア名 等は英のまま

> 「採点者」「コーチ(PM 役)」は LMS ロールとは別概念(README ロール責務表参照)。

## 8. 依存チケットの記録方向

メタ情報「依存チケット」は **「このチケットが依存する(= 先に存在しないと困る)チケット」** のみ記録。「このチケットに依存される」方向は記録しない(メンテコスト高 + 採点者目線では入口が分かれば十分)。

- 例: `S-B-01`(qa-board)は `S-B-04`(notification 基盤)に依存 → `S-B-01` のメタ情報に `S-B-04` と書く
- なしの場合は `(なし)` と明記

## 9. 模範解答 PJ 懐疑的レビュー

模範解答 PJ は **受講生が理想的な実装を学ぶ教材** であるべきため、チケット詳細化時に「現在の実装がベストか?」を必ず一度問い直す。引き算で受講生に渡す前に模範解答 PJ を理想形に修正しておくことで、受講生は正しい設計を学べる。

**検証ポイント**: 画面分割の妥当性 / 認可粒度 / レイヤー配置 / API 設計 / ロール間共通化

判定を `関連ドキュメント/要件シート_詳細度100%/_review-log.md` に記録:

- ✅ **適切** — 現状で問題なし
- ⚠️ **修正必要** — 詳細化を一時停止して模範解答 PJ を修正してから再開
- 🕐 **Phase D で一括処理** — 他 Feature と横断的に判断する必要、詳細化完了後にまとめて対応
