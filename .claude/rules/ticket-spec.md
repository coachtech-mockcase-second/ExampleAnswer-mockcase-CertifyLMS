# Certify LMS 詳細度 100% チケット仕様 (規約)

本 rules は `関連ドキュメント/要件シート_詳細度100%/` 配下でチケットを詳細化する際に参照する規約集。Skill `ticket-detail-100p` から起動時に参照される。

ナビ・進捗・チケット一覧は `関連ドキュメント/要件シート_詳細度100%/README.md` を参照。

---

## 1. セクション構成

種別ごとのセクション骨格は **`templates/{story,bug,task}.md`** を参照(本規約ではセクション内の記述ルールのみ規定)。

種別ごとの主な違い:

| 種別 | 冒頭セクション | 受け入れ条件 | 実装方針(参考)サブセクション | 工数規模 |
|---|---|---|---|---|
| **Story** | 背景・目的 + ユーザーストーリー | 4-7 件(規模大 6-10) | インターフェース / データモデル / コンポーネント / 異常系 / 設計判断 | 通常 5-9h、規模大 10h+ |
| **Bug** | 概要 + 再現手順 + 期待する結果 + 実際の結果 | 1-3 件 | 原因 | 2-8h |
| **Task** | 概要 | 1-3 件 | 変更内容 | 3-6h |

---

## 2. 記述粒度の原則

セクションごとに語彙レイヤーを使い分ける(2026-05-27 改訂)。Step 4 引き算 SSoT 化のため **実装方針 配下では業務語彙 + 技術名(クラス名 / Enum 英語キー / 機械値)の併記を必須**化した。

### 2.1 セクション別の記載範囲

| セクション | 業務語彙 | 技術語彙 | 備考 |
|---|---|---|---|
| 背景 / ユーザーストーリー / 概要 / 要件 / スコープ外 / 再現手順 / 期待結果 / 実際結果 / Q&A | ✅ 必須 | ❌ NG | LMS ドメイン語彙(2.2)は許容、技術カテゴリ名(`Policy` / `FormRequest` 等)も最小限 |
| 受け入れ条件 | ✅ 必須(振る舞い観察ベース) | △ URL / HTTP ステータス / ロール識別子 + **確認方法（テスト）サブ項目(3.6)内のテスト実名**のみ OK、クラス名・テーブル名・Enum 英語キー・機械値は NG | 採点者は画面・動作で判定(自動テストで判定する行のみ確認方法（テスト）で補う) |
| **実装方針 配下 全 5 サブセクション** | ✅ 推奨(クラス名と併記) | ✅ **必須** — Model / Enum / Controller / Action / Service / Repository / Policy / Notification / FormRequest / 例外クラス名 + Enum 英語キー + バリデーションルール記法(`required` / `max:200` 等) | Step 4 引き算 SSoT として機能させる |
| 実装方針 > コンポーネント | レイヤー別の業務語彙責務 | ✅ クラス名 + ファイルパス + 1 行責務 | クラス列挙のSSoT(他セクションで再列挙しない) |

**技術名併記の記法例**(実装方針 配下で使用):

- コンポーネント節: `会話管理 Controller (\`AiChatConversationController\`): HTTP 受付 → 認可委譲 → Action 呼出`
- データモデル節: `AI 相談会話 (\`AiChatConversation\`)` / `Enum 発言者ロール (\`AiChatMessageRole\`): 受講生 (\`User\`) / AI (\`Assistant\`)`
- エラーハンドリング節: `会話作成 FormRequest (\`AiChat\\StoreRequest\`)` / `業務例外 (\`EnrollmentNotActiveException\`): 受講登録が学習中でない場合は 403`

業務語彙が採点者・コーチ向けの読みやすさ、技術名が受講生・実装者向けの 1:1 対応(模範解答 PJ コードとのリンク)を担保する。

### 2.2 LMS ドメイン語彙(業務語彙として常時許容)

`product.md` / spec で一級語彙として定着している以下は業務語彙として扱う(背景・要件・受け入れ条件・Q&A で使用):

- **教材階層**: Part / Chapter / Section
- **エンティティ名**: 演習問題 / 教材内画像 / 出題分野マスタ / 面談パック / 質問スレッド / 回答 等(日本語推奨)
- **状態名**(業務観察用): 下書き / 公開中 / アーカイブ / 受講中 / 修了 / 退会 等(画面・採点で使う表記)
- **ロール**: 受講生(student) / コーチ(coach) / 管理者(admin)
- **Feature 略称**: qa-board / mock-exam / mentoring / notification 等
- **HTTP ステータス**: 200 / 302 / 403 / 404 / 409 / 422 / 500 等(実装方針・異常系・Q&A・API 直叩き系 AC で使用。画面 UAT の受け入れ条件にはコードを書かず表示事象で書く → 3.1)
- **SoftDelete**(振る舞い概念): 「論理削除」「SoftDelete」両方 OK

### 2.3 セクション横断の禁止記載

セクションスコープに関わらず以下は **全セクションで禁止**:

| カテゴリ | 例 | 代替 |
|---|---|---|
| 別チケット言及(依存以外) | 「`S-B-04` で扱う通知基盤を利用」 | チケット間の交差参照は依存チケット欄のみ |
| 過去案件への言及 | 「BookShelf 流の...」「BookShelf 既習パターン」 | Certify LMS 単体で完結 |
| **COACHTECH 固有の文脈** | 「Pro 生として...」「COACHTECH 受講生として」「教材としての位置づけ」 | 「実務で頻出する」「実務級」のように LMS の価値視点に言い換え |
| 構築側メタ情報 | 「Step 4 引き算範囲」「Phase D」「提供 PJ / 提供プロジェクト」「模範解答 PJ」「`_review-log.md` 参照」 | 受講生・採点者の視野外、書かない。**読み手から見れば「提供 PJ」と「模範解答 PJ」の区別はなく、ただ 1 つの既存プロジェクトがあるだけ**。既存ファイル / テーブルを指す技術注記は「既存」(例: `**View**(既存、ロック対象)`)、背景での現状説明は「現状」or 主語を直に述べる(例: 「提供 PJ の受講生は学習を進められるが」→「受講生は学習を進められるが」) |
| 構築側ドキュメント参照 | 「`backend-http.md` 準拠」「`backend-usecases.md` の規約」「`docs/specs` / `steering` 参照」 | `.claude/rules/*.md`(backend-* / frontend-*)・`docs/specs/`・`docs/steering/` は受講生に渡らない構築側専用。設計判断の根拠は本文に残し、ファイル名の引用だけ落とす(例: 「(`backend-exceptions.md` 準拠)」→ 削除 / 「`backend-http.md`「ロール別 namespace 禁止」と整合」→「ロール別に namespace を分けない」と概念で書く) |
| 各セクション冒頭の説明文 | `## 受け入れ条件\n\n以下に...` の 1 文 | 冗長 |
| 採点運用情報 | 「採点者は ✅ / ❌ で判定」「評価シート ① に転記」「(手順は採点者ガイド)」「確認方法（コード / メール / 準備）: …」 | 評価シート側・採点者ガイド側で管理。**例外: 確認方法（テスト）のみ AC 直下に併記できる**(3.6)。それ以外の確認手段(コード確認 / メール確認 / 再現準備)はチケットに書かず採点者ガイドが持つ |

### 2.4 実装方針サブセクションの粒度ガイド(外部→内部の潜行順)

サブセクション命名 + 内容 + **順序を以下に揃える**。全 5 サブセクション:

| 順 | サブセクション | 含める | 必須 |
|---|---|---|---|
| 1 | **インターフェース (必須)** | エンドポイント表(HTTP × URL × **認可** × 振る舞い、リダイレクト先・フラッシュ文言例含む) + ミドルウェア + 機能 OFF 時挙動(該当時) | 常時 (必須) |
| 2 | **データモデル**(新規テーブル時のみ) | エンティティ表(Model 名 + 主要属性業務語彙 + 関係性 + **制約**(FK / SoftDelete / 削除戦略)) + Enum 別ブロック(Enum クラス名 + 英語キー + 業務語彙併記) + インデックス用途 + **初期データ Seeder(必須)** 独立ブロック | 新規時のみ |
| 3 | **コンポーネント** | レイヤー別マッピング — Controller / FormRequest / Action / Service(新規時)/ Repository(新規時)/ Policy / Notification(該当時)/ Model + Enum / View / Migration + Seeder / 例外 / Routes。**各レイヤーにクラス名 + ファイルパス + 1 行責務**。Action 等 Basic 範囲外要素には「※」注記 | 常時 |
| 4 | **異常系** | 入力検証(FormRequest クラス名 + `required` / `max:200` 等の機械的ルール記法のみ、推奨日本語文言は Q&A に集約)+ 業務例外(例外クラス名 + 状態ベースガード + HTTP ステータス) + 外部 API フォールバック(該当時) | 常時 |
| 5 | **設計判断** | 特殊設計判断 + 採用根拠(なぜを書く、何をは他セクションで)+ **テスト観点**(標準パターン以外の特殊観点があれば 1〜2 行で内包) | 該当時のみ |

> **順序の論理**(外部→内部の潜行順): 直前の「受け入れ条件」=振る舞いの観察可能事象、直後の「インターフェース」=振る舞いの技術契約で連続性を保つ。次に「何が永続化される(データモデル)→ どこに何の実装がある(コンポーネント)→ 異常時はどうなる(異常系)→ なぜこの作り(設計判断)」と外側から内側へ潜る。

**設計原則**:

- **MECE な SSoT 集約**: 同じ情報を 2 箇所に書かない。各情報の唯一の置き場:
  - **認可ルール** → インターフェース表の `認可` 列のみ(認可マトリクスは独立節を持たない)
  - **クラス名 + ファイルパス + 責務** → コンポーネント節のみ(他セクションで再列挙しない)
  - **削除戦略 / 外部キー / SoftDelete** → データモデル エンティティ表の `制約` 列のみ
  - **エラー文言推奨例(日本語)** → 想定 Q&A のみ(異常系節は機械的ルール記法のみ)
  - **Basic 共通前提**(JS なし / XSS 防御 / Pint 整形 等) → **書かない**(`.claude/rules/` 既存ガイドに委譲)
- **Step 4 引き算 SSoT 化**: コンポーネント節を読めば「どのファイルを削るか / どのレイヤーに仕込むか / どこに巻き戻すか」が機械的に判断できる粒度
- **データモデル節は Model 名併記 + Enum 英語キー併記を必須**(業務語彙だけだとテーブル/Enum の対応が推測勝負になる)
- **設計判断節は「なぜ」のみ**(「何を」は他セクションに)。本チケット固有のトレードオフのみ記述、自明な構成や Basic 共通前提は書かない
- **テスト戦略の独立節は持たない**: 標準的な Feature テストパターンは AC 「本チケットの機能に対するテストが実装されている」で担保。本チケット固有の特殊観点(認可マトリクス網羅 / 並行性検証 / 計算境界 / Before/After 計測 等)があれば設計判断節に 1〜2 行で内包

**「(必須)」マーカー付きサブセクション**(以下 2 つは AC・採点・動作確認のベース、正確に実装):

- **インターフェース(必須)**: URL パス・HTTP メソッド・認可・振る舞いは AC と直結
- **データモデル > 初期データ Seeder(必須)**: AC の動作確認用データパターンを正確に実装

#### Bug 専用サブセクション

| サブセクション | 含める | 含めない |
|---|---|---|
| **原因** | 原因の主要ファイル(パス + クラス名) + 仕込み内容要約(業務語彙 + クラス名・メソッド名併記) + 修正範囲(レイヤー / メソッド名 OK) | 修正コード片 / SQL クエリ詳細 |

> Bug の **Step 4 仕込み方詳細は README の「Step 4 仕込み方」列が SSoT**(構築側メタ情報なのでチケット本体には書かない、規約 2.3)。チケット本体の「原因」は採点者・コーチ向けの業務理解用に簡潔な技術概要を記載する。

#### Task 専用サブセクション

| サブセクション | 含める | 含めない |
|---|---|---|
| **変更内容** | 変更対象ファイル(パス + クラス名、レイヤー別グルーピング可) + 変更前後の業務語彙 + 採用技術概念 + 判断理由(Why)+ テスト観点(該当時のみ、Before/After 計測手順 / モック化境界 等の特殊観点) | コード完全例 |

> Task は実装方針サブセクションを **「変更内容」のみ** に集約(Bug の「原因」のみと対称)。Story の 5 構造(インターフェース / データモデル / コンポーネント / 異常系 / 設計判断)に対し、Task は本質的に既存構造の変更が主目的のため、What と Why が密接で分離する意味が薄い。Why(採用根拠 + 代替案比較)とテスト観点はすべて変更内容内に統合。

## 3. 受け入れ条件の書き方

**受け入れ条件 = 採点項目**。各項目 = 評価シート ① の 1 採点行 と 1:1 対応する。

「仕様の網羅記述」ではなく「**採点者が ✅ / ❌ を 1 回で判断できる項目リスト**」として書く(BookShelf 評価シートが参考)。

### 3.1 書き方の基本

- **採点者目線で 1 採点単位**: 採点者が「✅ / ❌」を 1 回判断で付けられる粒度。MECE(同じ振る舞いを複数項目に分散させない)
- **happy path 中心 + 重要なエッジケースのみ独立**: 正常系の主要な振る舞いを 1 行 = 1 採点項目。AI 失敗フォールバック / 機能 OFF / 特殊な状態遷移 など教材的に重要なものに絞る(全パターン羅列しない)
- **観察可能事象で具体化(抽象表現を避ける)**: 「適切な〜」「正しく〜」「拒否される」のような曖昧語は使わず、画面・動作で観察できる事象(画面非表示 / エラーメッセージ表示 / アクセス不可 / リダイレクト 等)で記述
  - ❌ 「機能 OFF / API キー未設定 / 上限超過時に、適切な案内とともに利用が拒否される」(3 件混在 + 「適切な」が曖昧)
  - ✅ 「機能 OFF の場合、サイドバー / ウィジェットが非表示になり、AI 相談ルートにアクセスしても画面が表示されない」
  - ❌ 「コーチ / 管理者の AI 相談ルートへのアクセスが拒否される」(「拒否」だけだと何が起きるか不明)
  - ✅ 「コーチ / 管理者 / 学習中以外の受講生は AI 相談ルートにアクセスできない」
- **認可 AC はアクセス可否で書く(振る舞い UAT 粒度)**: 権限のないロール・状態の禁止は「〜以外はアクセスできない / 操作できない / 閲覧できない」で書く(HTTP ステータスや「エラー画面」の種別は書かない — ブラウザで出る画面は 403 / 404 / 案内画面と多様で、`APP_DEBUG` でも見え方が変わるため)。UI に到達経路がない非 GET リクエストの拒否 / 偽装送信への耐性 / 同じ状態への再操作の冪等性は、採点者がブラウザで再現できないため AC にしない(要件・実装方針には残し、受講生テスト + 横断品質で担保する)。**Blade(@can 含む)は完成形で提供され、受講生が実装するのは Policy / Controller の `authorize` なので、「ボタンの出し分け」だけを認可の判定根拠にしない**(@can は提供済みで、Policy 未実装でも全員非表示になり得る)。認可は「権限のない経路で実際にアクセス・操作できないこと」で見る。確認手段(他人のリソースの URL や管理画面・編集ページを開いて入れないこと)は採点者ガイドが持つ
- **強結合な振る舞いは 1 項目に統合**: 「成功時に保存 + リダイレクト + フラッシュ表示」は 1 行(圧縮原則は 3.4)
- **技術名は AC に書かない**: `Policy` / `FormRequest` / `Middleware` 等の技術名は採点判定に不要(振る舞い観察ベース)。実装方針内では記載 OK
- **専門用語を避け平易な日本語で(認知負荷の最小化)**: 「冪等」「べき等」「no-op」「TOCTOU」「busy」等の専門語は AC に書かず、観察できる事象で言い換える(例:「冪等な no-op として成功」→「同じ操作を繰り返してもエラーにならず、状態も変わらない」)。採点者が辞書を引かずに判定できる語彙に統一する
- **括弧 `()` を多用しない**: 補足を `(…)` で詰め込むと読みにくい。文を分ける / 主文に織り込む / 確認方法サブ項目へ逃がす。1 行あたり括弧は 1 個程度を目安
- **確認方法サブ項目は同梱テスト citation の行のみ(3.6)**: Bug / パフォーマンス Task の同梱テスト pass で判定する行に限り、AC 行の下にインデント 1 段で `- 確認方法（テスト）: 同梱の path::method(修正前は失敗)が pass する` を足す(実ファイル名・メソッド名必須 = コマンドで機械的に確認できる)。それ以外(メール送信 / N+1 等のコード成果 / 再現に準備が要る振る舞い / 受講生が書くテスト)はサブ項目を付けない — 判定手段は採点者ガイドが持つ。画面・URL 直叩きで確認できる大多数はもとよりサブ項目なし
- **バリデーション AC は業務語彙レベルで網羅 + 1 行統合**: 機械値ルール記法(`max:200` 等)は実装方針 > エラーハンドリングに集約、AC は「画面で日本語エラー表示 + 業務語彙制約一覧」を 1 行に
  ```
  - [ ] {画面名} にて、{入力項目群} にバリデーションが行われ、ルール違反時に日本語のエラーメッセージが表示されるか:
    - タイトル: 必須 / 200 文字以内
    - 本文: 必須 / 5000 文字以内
    - 配信対象タイプ: 必須 / 全受講生・資格指定・ユーザー指定のいずれか
  ```
- **HTTP ステータスコードは AC に書かない**: 403 / 404 / 409 / 422 等のコードはブラウザでは見えないため書かない。認可は「アクセスできない / 操作できない / 閲覧できない」(アクセス可否、上記)、状態ガードによる操作拒否は「〜できず、フラッシュエラーが表示される」(操作結果が画面に出る)、フォーム検証は「日本語のバリデーションエラーが表示される」で書く。再現困難な異常系(外部 API 失敗 / 生成失敗 等)は観察できる結果(「連携情報が保存されず未連携のまま」「レコードもファイルも作られない」)で書き、判定は同梱テスト / コードに回す。**例外: API 直叩き系 AC(「API を直叩きすると 401」等、HTTP クライアントで確認することが自明な行)のみコード記載可**(3.6)。URL パスは適宜含める。実装方針・異常系・Q&A でのコード記載は OK
- **文言は含めない**: フラッシュ・エラーメッセージ文言は実装方針 or Q&A に推奨例として提示。バリデーション機械値も書かず、業務語彙(「長すぎる」)で表現
- **文体は固定しない**: 「〜される」「〜できる」「〜こと」「〜されるか」を自然に使い分け
- **フラット構造**: 機能の主要な振る舞い → 重要なエッジケース → 副作用 / 状態遷移 → 入力検証 → 認可 の順で **箇条書きを 1 段でフラットに並べる**(順序は規模・自然さに応じて柔軟)
- **分割の例外**: 1 行に詰めすぎて読みづらいときは分割 OK(ただし件数目安を超えるなら統合を再検討)

### 3.2 採点項目に含める要素

チケットごとに該当するものをフラットに列挙する:

| 種類 | 例 |
|---|---|
| **正常系 happy path** | 「`/ai-chat` にアクセスした際に、最新の会話詳細画面に遷移する(0 件なら空状態画面)」 |
| **重要なエッジケース** | 「AI 応答に失敗した場合、受講生メッセージは保存され、エラー状態のメッセージから再送信できる」 |
| **副作用 / 状態遷移** | 「会話を削除した際に、配下のメッセージも連動削除される」 |
| **入力検証**(該当時のみ) | 「{画面} にて {項目群} にバリデーションが行われ、ルール違反時に日本語のエラーメッセージが表示されるか: タイトル(必須 / 200 文字以内)...」 |
| **テスト**(種別で扱いが変わる→3.6) | Story / リファクタ Task:「本チケットの機能に対するテストが実装されている」を常時追加(受講生が書く、網羅性は問わない) / **Bug・パフォーマンス Task:「同梱 `path::method`(修正前は赤)が pass する」採点行を 1 行必ず追加**(観察可能な振る舞い行と併記。代表テストが無ければ模範解答に追加してから記載) |
| **認可**(該当時のみ、機能群ごとに 1 件) | 「会話のオーナーのみが自分の会話を閲覧・編集・削除・メッセージ送受信でき、他の受講生が直アクセスするとエラー画面が表示される」 |

> 認可の書き方: 同じ判定で複数機能をカバーする場合は 1 件にまとめる(機能ごとに繰り返さない)。

### 3.3 種別別の件数目安

| 種別 | 件数目安 |
|---|---|
| Story 通常 | **4-7 件** |
| Story 規模大 | **6-10 件** |
| Bug | **1-3 件** |
| Task | **1-3 件**(目的 + 該当時の検証項目 + テスト実装) |

**合計目標**: 40 チケットで **約 200 件**(平均 5 件 / チケット、BookShelf 比例)。

### 3.4 圧縮原則(5 パターン)

AC を 2 つ以上書こうとしているとき、以下に該当するなら **1 項目に統合**:

| # | パターン | 例 |
|---|---|---|
| 1 | **業務操作の側面分解** — 同一業務操作の成功 / 副作用 / 通知 / 表示要素を別 AC にしない | 「成功時に〜され、〜される」/「未読バッジ / 空状態 / 99+ 表示」を 1 行に |
| 2 | **認可判定の機能群統合** — 複数機能で同じ判定の認可拒否は 1 項目 | 「コーチ / 管理者 / 学習中以外の受講生が AI 相談ルートにアクセスするとエラー画面」(機能群レベルで 1 件) |
| 3 | **同一フィールド・同一機能の選択肢列挙** — 同一フィールド内のルール / 同型状態遷移 N 種 / 機能 OFF スイッチ / 同操作のバリエーション | 「タイトル(空 / 長すぎる / 形式不正)」/「公開 / アーカイブ / 下書き戻し」/「機能 OFF / API キー未設定 / 上限超過時に拒否」を 1 行に。**異種フィールド or 異種チェック種別が混在する場合は分割** |
| 4 | **単一動線の UI トリガー差分** — クリック / ESC / 外側クリック等の閉じる操作 | 「開閉できる」レベルで 1 行に |
| 5 | **横断採点項目** — PR セクション / 動画 / Pint / N+1 / 振る舞い不変(Task)等 | AC には書かず評価シート ②(横断評価)に集約(3.5 参照) |

**統合判断**: 「同じ業務操作の異なる側面 / 同じ実装ファイル内で完結 / 1 PR で連動」のすべてに該当すれば統合する。

### 3.5 横断採点項目と AC の整理

以下は **チケット個別 AC には書かない**(全チケット共通の横断採点として評価シート ②(横断評価)に集約):

- PR 5 セクション完備 / 動画記述
- Pint / 命名 / Clean Architecture からの逸脱、**全チケット横断で N+1 が新規混入していないか**のチェック
- 振る舞い不変そのものの再記述(Task)/ Before-After 計測値(Task Performance、計測値は実装方針 > 変更内容に書く)
- テストの **網羅性 / カバレッジ / 全 pass**(評価シート ② で扱う)

> **Performance / リファクタ Task の成果そのものは AC に残す**(② へ送らない): 「N+1 が解消されている」「区切り読みになっている」「外部連携が Service に集約され直接参照がない」等は当該 Task の達成目標なので AC に記載し、同梱テストがあるものは確認方法（テスト）を付す(3.6)。コード確認で判定する成果はサブ項目を付けず採点者ガイド側で扱う。② へ送るのは「他チケットで新たに N+1 を作っていないか」等の **横断逸脱チェック** のみ。

**例外: テスト関連は AC に含める(種別で形が変わる、3.6)**。Story / リファクタ Task は「本チケットの機能に対するテストが実装されている」を 1 行追加(受講生が書くため、抽出ユニット・モックの新規テストが成果)。**Bug・パフォーマンス Task は「同梱 `path::method`(修正前は赤)が pass する」採点行を 1 行必ず追加する**(観察可能な振る舞い行と併記。代表テストが模範解答に無ければ追加してから記載 = カバレッジ不足の検出器を兼ねる)。いずれも網羅性・カバレッジ・全 pass は評価シート ② で別途扱う。

### 3.6 「確認方法」サブ項目(テスト専用)と判定手段の所在

**確認方法サブ項目は「同梱テストの citation」専用**(Bug / パフォーマンス Task のみ)。AC 直下に付けてよいのは `- 確認方法（テスト）: 同梱の path::method(修正前は失敗)が pass する` の形だけで、**採点者がコマンド(`sail artisan test --filter=<メソッド名>`)で機械的に確認できる実名を必ず書く**。それ以外の行は画面・URL 直アクセスでの動作確認が既定で、**サブ項目を付けない**。受講生が書くテスト(実名が無くコマンドで特定できない)を確認方法にしない。

**画面で観察できない・再現に準備が要る行の判定手段はチケット本文に書かない**: コード確認(N+1 / リファクタ構造 / chunk 等)、メール確認(Mailpit)、再現準備(DB の直接更新 / 環境設定の変更 / 外部 CLI・実外部サービスの操作)は、行 → 手段の対応を**採点者ガイド(構築側運用文書)が一元的に持つ**。チケットは要件の記述に徹し、採点の段取りを混ぜない。なお「API を直叩きすると 401」のような AC は文言自体が手段を内包する(HTTP クライアントで叩くのは自明)ためそのままでよい。手動再現が難しい振る舞い(決済イベントの再送・逆順到着 等)も、**画面で観察できるオラクル(残数等の数値表示)に評価基準を寄せて通常の UAT 行として書く**(再現手段は採点者ガイドが持つ)。

**テストの扱い = 種別で 2 バケット(ハイブリッド)**。提供 PJ は模範解答 PJ から各チケットの実装を引き算して作るため、テストの所在が種別で分かれる:

| バケット | 種別 | テストの所在 | テスト系 AC / 確認方法 |
|---|---|---|---|
| **① 同梱テスト pass** | **Bug** / **パフォーマンス Task** | 模範解答のテストを提供 PJ に**残す**(修正前は失敗=赤) | **代表同梱テストが pass する採点行を 1 行必ず置く**:「同梱 `path::method`(修正前は赤)が pass する」。観察可能な振る舞い行は動作確認のまま**併記**(テスト行=客観オラクル / 振る舞い行=人間可読・採点者目視)。**代表テストが無ければ模範解答に追加してから書く**(bucket① 成立の必須条件 + テスト不足の検出器を兼ねる)。汎用「テストが実装されている」行は置かない(テストは同梱で受講生実装ではない) |
| **② 受講生が記述** | **Story** / **リファクタ Task** | 同梱不可(Story=削除クラス参照で起動エラー / リファクタ=振る舞い不変で緑のまま)→ 受講生が書く | 「本チケットの機能に対するテストが実装されている」行を常時必須。**個別行に確認方法（テスト）は一切付けない**(受講生のテストは名前が異なりコマンドで特定できず、確認方法として機能しないため)。手動再現が難しい振る舞いは画面で観察できるオラクルに評価基準を寄せて UAT 行として書く。リファクタの構造チェック(コード確認)はチケットに書かず採点者ガイド側で扱う |

> このバケット分けは **Step 4 引き算ルール** も兼ねる: ① のテストは提供 PJ に残す(赤で出す)、② のテストは削る。

**記法**(AC 行直下にインデント 1 段、`- 確認方法（テスト）: …`。実名必須):

```
バケット①(Bug、振る舞い行 + 専用テスト採点行を併記):
- [ ] 状態フィルタなし・在籍系フィルタの一覧に退会済みユーザーが表示されない
- [ ] 同梱 `tests/Feature/Http/User/IndexTest.php`(該当メソッド、修正前は赤)が pass する

バケット①(パフォーマンス Task):
- [ ] コーチダッシュボードの担当受講生一覧で、担当受講生が増えても発行クエリ数が増えない
  - 確認方法（テスト）: 同梱の `tests/Feature/Http/Dashboard/DashboardQueryCountTest.php::test_coach_dashboard_query_count_is_within_limit`(修正前は失敗)が pass する

バケット②(Story・手動再現が難しい振る舞い → 画面オラクルへ寄せた UAT 行、サブ項目なし):
- [ ] 同一の決済完了イベントが再送されても残数加算は 1 回のみで、再送後も画面の残数表示は増えない
```

**運用ルール**:

- **サブ項目なし(画面確認)が大多数**になるよう設計。確認方法（テスト）は「自動テストが判定オラクル」の行に限定
- **同梱テストは実名で書く**: `path::method` まで
- **括弧 `()` を多用しない**(3.1)
- **評価シートへの転記**: 確認方法（テスト）はそのまま評価シートに転記する(自動テスト項目のため、採点後に受講生へ開示されても問題ない)
- **運用文書への参照を書かない**: チケット本文のどこにも「(手順は採点者ガイド)」「採点者ガイド参照」等の運用文書参照を書かない(規約 2.3 の構築側ドキュメント参照の禁止と同じ扱い)

## 4. 「要件」の書き方

> セクション名: 旧「やること」→ **「要件」**、旧「やらないこと」→ **「スコープ外」**(2026-05-26 業界標準寄せ。アジャイル / Jira / DSDM 等の慣例に整合)。

「要件」は **チケットが達成すべき機能スコープを完全に表現する** セクション。機能・認可・入力検証・副作用・エッジケース挙動 を含め、受け入れ条件は **要件に内包される項目を採点者が確認する判定リスト** として書く(要件 ⊇ 受け入れ条件)。

業界標準として User Story は要件の全てを含まないため、「要件」セクションに機能要件 + 必要時の非機能要件を整理する(Mountain Goat / DSDM 流)。

### 4.1 含める内容

- 業務操作(CRUD / 状態遷移 / 一覧表示 等)
- 認可ルール(誰が何をできる、当事者 / ロール制約)
- 入力検証(主要なバリデーション制約: 必須 / 文字数 / 形式 等、業務語彙ベース)
- 副作用(連動操作 / 通知発火 / フラッシュ表示 等)
- エッジケース挙動(機能 OFF / 状態制約 / 整合性チェック 等)
- 非機能要件 — **該当時のみ独立サブセクション化**(性能 / レート制限 / セキュリティ強度 / 運用観測 等。出てこないチケットでは作らない)

### 4.2 含めない(他セクションへ逃がす)

- HTTP ステータスコード(403 / 404 / 422 等) → 受け入れ条件で観察可能事象として書く
- URL パス・ルート名 → 実装方針 > URL
- 技術用語(「Resource」「Policy」「FormRequest」「Action」「Service」 等)→ 実装方針 配下のみ

### 4.3 構造の例

サブセクション分けは規模により任意。**業務語彙ベースの大カテゴリ**(機能群名 + 該当時の制約)で分ける:

```
### 会話の管理(受講生・学習中・オーナーのみ)
- 会話の新規作成 / 詳細表示 / タイトル編集(1〜100 文字) / 削除(配下メッセージも連動削除)
- 最新会話への自動遷移(0 件なら空状態画面)

### メッセージ送受信(会話オーナーのみ)
- 受講生メッセージ送信(1〜2000 文字) / AI 応答取得 / エラー時の再送信

### アクセス制御(機能群共通)
- 学習中受講生のみアクセス可能、コーチ / 管理者 / 学習中以外の受講生はアクセス拒否

### 非機能要件(該当時のみ追加)
- 日次送信上限: 受講生 1 人あたり 50 通 / 日(超過時は送信拒否)
- AI 応答の運用観測メタデータ記録(モデル名 / トークン数 / 応答時間)
```

### 4.4 粒度の例

- ✅ 「会話の新規作成(受講生のみ)」
- ✅ 「会話タイトルの編集(オーナーのみ、1〜100 文字)」
- ✅ 「会話の削除(オーナーのみ、配下メッセージも連動削除)」
- ❌ 「会話作成: 受講生のみ可、他は 403」(HTTP ステータスを含めない、受け入れ条件側へ)
- ❌ 「StoreAction を実装」(技術用語、実装手段を含めない)

### 4.5 要件 ↔ 受け入れ条件 の網羅性

「要件」の各項目は、すべて `## 受け入れ条件` に対応行が存在することを詳細化時に確認(双方向の網羅性、**要件 ⊇ AC** の原則):

- 要件に書いた業務操作 → AC に 1 業務操作 1 項目で対応
- 要件に書いた認可ルール → AC に機能群統合の認可項目で対応
- 要件に書いた**バリデーション制約**(業務語彙)→ AC に統合バリデーション項目で対応(両者とも業務語彙レベル、機械値詳細は実装方針 > エラーハンドリング に集約)
- 要件に書いた副作用 → AC に副作用項目で対応
- 要件に書いた非機能要件 → AC に該当する観察可能事象で対応
- AC の各項目 → 要件のどこかに居場所(AC が要件外の事象を確認していないか)

**3 階層の詳細度階段**(要件 / AC / 実装方針):
- **要件** = 業務語彙で機能スコープ網羅(「タイトル 200 文字以内」「必須」)
- **AC** = 業務語彙で採点項目化(「タイトル / 本文 にバリデーションが行われ、日本語のエラーメッセージが表示される: タイトル必須・200 文字以内、...」)
- **実装方針 > エラーハンドリング** = 機械値ルール記法で設計詳細(`title: required / string / max:200`)

## 5. 想定 Q&A の粒度

`## 補足 > ### 想定 Q&A` は **要件・仕様レベルの「正しい振る舞い確認」のみ** を含む。形式は **フラットなテーブル 1 個**。

### 5.1 含める / 含めない

| 含める(仕様確認) | 含めない(設計判断・実装詳細) |
|---|---|
| バリデーション仕様(最大文字数 / 必須 / 任意 / 上下限値) | クラス分割(Action / Service を使うべきか) |
| 認可振る舞い(誰がアクセス可能 / HTTP ステータス / 当事者判定) | 命名(ファイル名 / クラス名 / メソッド名) |
| 振る舞い詳細(フラッシュ表示 / リダイレクト先 / 副作用 / SoftDelete 採否) | DI / Eloquent 詳細(`with()` 指定 / Repository 採否) |
| 文言の推奨例(フラッシュ・エラーメッセージ) | テスト場所 / 種別 / ファイルパス |
| データの境界(状態 × ロールで分岐する条件) | HTTP メソッド選択(POST / PATCH のどちら) |

設計判断 / 実装詳細はコーチが受講生のヒアリングに応じて **個別判断**(受け入れ条件を満たせば実装手段は問わない)。

### 5.2 種別別の必要性

| 種別 | 必要性 |
|---|---|
| Story | **必須**(30% 版抽象化時にコーチが即答する材料) |
| Bug | **任意**(本文セクションで仕様が完全にカバーされていれば省略可) |
| Task | **任意**(Refactoring は振る舞い不変 / Performance は計測値中心で仕様確認の余地が小さい) |

## 6. Basic / Advance 制約

| 範囲 | Basic | Advance |
|---|---|---|
| Controller / FormRequest / Policy / Middleware / Fortify Action | ✅ | ✅ |
| Notification(DatabaseChannel + Mail) / `DB::transaction` / Eloquent(with / withCount / scope) | ✅ | ✅ |
| Action クラス(UseCases) | ❌ | ✅ |
| Service クラス | ❌ | ✅ |
| Blade ロジック | ❌ | ❌ |
| 外部 API / Pusher Broadcasting / Sanctum | ❌ | ✅ |

**運用ルール**:

- Basic チケットの実装方針は **Basic 範囲内**(Controller 内完結が前提)
- Action / Service の使用は受講生判断で推奨(チャレンジするなら歓迎、必須ではない)
- 模範解答 PJ は Advance 含む完成形のため、Basic チケットの想定実装と異なる箇所あり(詳細化時に Basic 範囲で書き直す)
- **Bug / Task 例外**: Action / Service ファイル内に原因がある バグ / 既存改修 は、原因(Bug)/ 変更内容(Task)の主要ファイルでそのファイルを淡々と示すのみとする。Story と違い「Controller 内完結で実装してよい」の選択肢提示や「Basic 範囲外」の難易度ラベルは書かない(理由は 6.1)

### 6.1 Bug / Task の Action / Service 内原因箇所の書き方

Bug / Task は **既存コードの該当箇所を読んで直す** チケット。原因 / 改修箇所が Action / Service 内にあっても、受講生は既存構造をそのまま使って修正するため、Story と違い「Controller 内完結で実装してよい」という選択肢は存在しない(コードが既に Action / Service にあり、選択の余地がない)。

したがって Bug / Task では、原因箇所が Action / Service にあっても **難易度ラベルや代替実装の案内を書かない**:

- **「Basic 範囲外」等の難易度ラベルを書かない** — Bug / Task の難易度は「原因の理解・修正の認知負荷」で決まり、原因箇所がどの層(Controller / Action / Service)にあるかとは独立。層の所在を難易度として注記しない
- **「Controller 内完結で実装してよい」「対応する Controller method を修正対象とする」の選択肢提示を書かない** — 既存コードの該当箇所(Action / Service)を直すのが Bug / Task。代替実装先を案内しない
- 原因(Bug)/ 変更内容(Task)の **主要ファイルで、原因箇所の Action / Service ファイルを淡々と示す**(クラス名 + ファイルパス + 1 行責務)

#### ❌ 悪い(難易度ラベル + Controller 代替提示)

```markdown
- **主要ファイル**: `app/UseCases/User/IndexAction.php`。※ Action 内のため Basic 範囲外。Controller 内完結で実装する受講生は `UserController::index` が対象
```

#### ✅ 良い(原因箇所を淡々と示すのみ)

```markdown
- **主要ファイル**: `app/UseCases/User/IndexAction.php`(管理者ユーザー一覧の取得クエリ)
```

> **Story は別**: Story は提供時にロジックが削除され受講生がゼロから実装するため、`コンポーネント` 節の inline ※ で「Action は Basic 範囲外(Controller 内完結でも可)」と実装自由度を示してよい(規約 2.4)。Bug / Task は既存コードを直すため、この自由度注記は付けない。

## 7. ロール表記ルール

- LMS ロール: **受講生(student) / コーチ(coach) / 管理者(admin)**
- 初出時にフル表記(日本語 + 括弧書きの英名)、以降は日本語のみ
- コード片(バッククォート内)、Enum 値、`auth:` ミドルウェア名 等は英のまま

> 「採点者」「コーチ(PM 役)」は LMS ロールとは別概念(README ロール責務表参照)。

### 7.1 ユーザーストーリーのペルソナ統一ルール

- **ユーザーストーリーのペルソナは LMS プロダクト内ロール (受講生 / コーチ / 管理者) のみ** を使う
- **禁止ペルソナ**: 開発者(JS フロントエンド開発者 / モバイルアプリ開発者 / バックエンド開発者 等)、COACHTECH 受講生 / API 学習者 / Pro 生候補 等の **構築側メタペルソナ**
- 理由: チケットは LMS プロダクトの要件記述であり、構築側 / 教材視点を混入させると採点者が「これは誰のための機能か」を判定しにくくなる
- API チケット等のようにエンドユーザー操作を伴わないチケットでも、ユーザーストーリーは LMS ロール視点で書く(例: 「受講生として、自分の通知が JSON 形式で取得できる経路があってほしい。なぜなら、将来 JS フロントから非同期に通知を取得できる UX の土台が必要だから」)
- 開発者観点 / 構築側メタ観点 / 「将来のクライアント開発者」視点 は **背景・目的** セクションに統合する

## 8. 依存チケットの記録方向

メタ情報「依存チケット」は **「このチケットが依存する(= 先に存在しないと困る)チケット」** のみ記録。「このチケットに依存される」方向は記録しない(メンテコスト高 + 採点者目線では入口が分かれば十分)。

- 例: `S-B-01`(qa-board)は `S-B-04`(notification 基盤)に依存 → `S-B-01` のメタ情報に `S-B-04` と書く
- なしの場合は `(なし)` と明記

## 9. 模範解答 PJ 懐疑的レビュー

模範解答 PJ は **受講生が理想的な実装を学ぶ教材** であるべきため、チケット詳細化時に「現在の実装がベストか?」を必ず一度問い直す。引き算で受講生に渡す前に模範解答 PJ を理想形に修正しておくことで、受講生は正しい設計を学べる。

**検証ポイント**: 画面分割の妥当性 / 認可粒度 / レイヤー配置 / API 設計 / ロール間共通化

判定を `memo/関連ドキュメント/要件シート_詳細度100%/_review-log.md` に記録(構築ログは gitignore 済みの `memo/` 配下に退避済み):

- ✅ **適切** — 現状で問題なし
- ⚠️ **修正必要** — 詳細化を一時停止して模範解答 PJ を修正してから再開
- 🕐 **Phase D で一括処理** — 他 Feature と横断的に判断する必要、詳細化完了後にまとめて対応

## 10. 全チケット共通の要件外項目

以下は **全チケット共通で要件 / AC に含めない**(模範解答 PJ でも実装しない or 採点対象外)。各チケット個別の「スコープ外」に書く必要なし(規約レベルで全件除外)。

| 領域 | 内容 | 理由 |
|---|---|---|
| **モバイル / レスポンシブ対応** | 画面端余白 / 内部スクロール / ブレークポイント切替 等 | デスクトップ前提、画面サイズ調整は採点対象外 |
| **アクセシビリティ (WCAG / a11y) 準拠** | `role="dialog"` / `aria-*` / フォーカス管理 / `tabindex` / スクリーンリーダー対応 等 | 採点振る舞いから外す。実装側で意識する余地は残すが、AC・要件・Q&A 等で言及しない |
| **UI 操作の細かい仕様** | ESC キー操作 / 外側クリックでのモーダル close / フォーカス遷移 / アニメーション / トランジション 等 | AC に書かず、「開閉できる」レベルで抽象化。実装方針 or Q&A で技術メモとして残すのは OK |
| **ローディング表示 / スピナー** | API フェッチ中のスピナー / プログレスバー / スケルトン 等 | 採点対象外、見栄え調整 |
| **fetch エラー / フォールバック挙動** | 401 / 500 / Network エラー時の画面挙動(空状態 / エラーメッセージ / リトライ 等) | 受講生が実装方法を選べる余地として AC 不要(実装方針内のみ OK) |
| **エラーログ / コンソール出力** | `console.error` / `console.warn` 等の開発者向け出力 | 採点対象外 |
| **国際化 (i18n) / 多言語対応** | 言語切替 / 翻訳ファイル管理 等 | 日本語 UI 前提 |

> 上記項目を **実装方針サブセクション内に技術メモとして残す** のは OK(コーチが受講生のヒアリングに答える材料として)。AC・要件・Q&A から削るのみ。

## 11. 30% 版変換ルール

30% 版は **受講生用** で、コーチへのヒアリング誘導目的で **意図的に曖昧化** する(Certify LMS 固有のコンセプト = 「曖昧要件をヒアリングで詳細化する実務 PJ 参画体験」)。

### 11.1 削除セクション

100% 版から以下を削除し、30% 版にコピーしない:
- `## 受け入れ条件`(採点者専用)
- `## 実装方針(参考)`(コーチ用、設計詳細)
- `## 補足 > ### 想定 Q&A`(コーチガイド)

30% 版に残るのは: `## メタ情報` / `## 背景・目的`(Story) / `## ユーザーストーリー`(Story) / `## 要件` / `## スコープ外` / `## 概要`(Bug/Task) / Bug の `## 再現手順` 〜 `## 実際の結果` のみ。

### 11.2 抽象化辞書(要件セクション内、ヒアリング誘導目的)

| 100% (具体) | 30% (抽象化) |
|---|---|
| 「タイトル(1〜200 文字)」 | 「タイトル(文字数制限あり)」 |
| 「101 文字以上で 422」 | 「長すぎる入力は拒否」 |
| 「100 件以上で「99+」固定表示」 | 「大量の未読がある時の適切な表示」 |
| 「コーチが直アクセスすると 403」 | 「コーチはアクセス拒否」 |
| 「リダイレクト + フラッシュ「○○を作成しました」」 | 「成功時の確認表示」 |
| 「業務操作 × ロール マトリクス」 | 「○○ロールのみ可能」程度のロール記述 |
| 「1〜50 の整数」 | 「適切な範囲の整数」 |

### 11.3 ヒアリング誘導と負担軽減

受講生は 30% 版で曖昧な記述を見て、コーチに具体値を聞き出す。負担軽減策:
- **コーチガイド(想定 Q&A)に境界値・HTTP・認可詳細を集約**: コーチが Q&A をファイル内検索で即答可能な状態を保証(1 往復で解決)
- **文言(エラー / フラッシュ)はヒアリング対象から外す**: 「適切な日本語であれば文言の細部は採点対象外」が原則、コーチが文言の具体例を提示するのは任意
- **項目数の節度**: 機能群あたりヒアリング対象 2-4 項目に収まる粒度を目安に
