# Certify LMS 詳細度 30% チケット変換規約

100% 版チケット（`関連ドキュメント/要件シート_詳細度100%/{Story|Bug|Task}/{ID}_*.md` = SSoT）を入力に、受講生配布用の 30% 版を生成する規約。受講生は曖昧な仕様をコーチ（PM 役）へのヒアリング + 既存コードリーディングで詳細化する。

- **出力**: `関連ドキュメント/要件シート_詳細度30%/{Story|Bug|Task}/{ID}_{feature略称-連番}.md`（100% と 1:1、同一ファイル名）。
- 30% 本文に **ヒアリング誘導の文言は書かない**（抽象化した仕様をそのまま提示するのみ）。
- **`ticket-spec.md`（100% の書き方規約）は参照しない**。本ファイルで自己完結。

## コア原則 — 30%化 = 仕様(WHAT)を一段高い抽象度に言い直す

100% は仕様を「正確にどう振る舞うか」（いつ・誰に・どの条件で・どの値で）まで書く。30% は **同じ仕様を「何ができるか（= できること）」の高さに言い直す**。具体的な値・列挙・条件分岐・文言は、言い直しの結果として表に出なくなる（個別に「消す」のではなく、粒度を上げた一文に畳まれて見えなくなる）。

| 軸 | 30%での扱い |
|---|---|
| **WHY**（背景・目的・価値） | **残す**（具体のまま）。動機を曖昧にしても意味がない |
| **WHAT**（仕様） | **一段高い抽象度に言い直す（主作業）**。「正確にどう振る舞うか」→「何ができるか」へ。下の操作で具体が落ちる |
| **HOW**（実装・設計） | **削除**。例外 = 「必須」サブ節（インターフェース / 初期データ）は残す |

> **「できること」の例**: 「業務イベントが起きると当事者に通知が届く」「受講生は自分宛の通知を一覧で確認できる」。←→ 100% の契約: 「chat 投稿時に送信者を除くメンバーへ DB+メール、面談予約時に担当コーチへ…（4 イベント表）、20 件/ページ、他人宛はアクセスできない」。

## 言い直しで具体が落ちる典型パターン（要件 + インターフェース振る舞いに適用）

「消す」のではなく、仕様を上の抽象度に言い直した結果これらが表に出なくなる:

1. **スカラ値が落ちる**: 「1〜200文字」→「文字数制限がある」／「20件/ページ」→「ページネーションがある」
2. **列挙・マッピングが畳まれる（最頻 = 仕様そのものの抽象化）**: 「chat=送信者除く全員 / Q&A=投稿者 / …」の 4 行表 → 「業務イベントが起きると当事者に通知が届く」。**どのイベント / 誰宛 / 自己除外 がまるごと一文に畳まれる**
3. **条件・分岐が粗くなる**: 「公開中の資格のみ閲覧可」→「閲覧できる資格には条件がある」
4. **文言が落ちる**（フラッシュ / エラーメッセージ）
5. **アクセス可否・ロール可視性が粗くなる**: 「他人宛はアクセスできない」→「他人のものは操作できない」

> **伏せたことを匂わせない**: 言い直した要件は平易に言い切る。「〜には仕様がある」「詳細は別途定義」のように、本当は決まっている仕様をもったいぶって伏せている感を出す注記は **書かない**。未確定は記述の粗さで暗黙に示せばよく、明示的に「ここは曖昧」と注記しない。

## 出力セクション（種別別）

### Story
`## メタ情報`（そのまま）/ `## 背景・目的`（残す・仕様漏れは粗く）/ `## ユーザーストーリー`（残す）/ `## 要件`（**抽象化**）/ `## スコープ外`（残す）/ `## インターフェース`（HTTP・URL・認可の構造は残し、振る舞いは粗く + 末尾に提供済み画面）/ `## 初期データ`（**シナリオベース**、下記の基準）

> 100% の `## 実装方針(参考)` ラッパーは消す。配下のうち **インターフェース(必須) → `## インターフェース`**、**データモデル>初期データSeeder(必須) → `## 初期データ`** に昇格。データモデル本体 / コンポーネント / 異常系 / 設計判断 / 受け入れ条件 / 想定Q&A は **削除**。

> **例外 — 提供済み画面の明示**: コンポーネント節のうち **View（既存）行だけ** は `## インターフェース` 末尾に昇格し、「**画面**: `resources/views/...` に提供済み（本チケットでの変更は不要）」の 1〜2 行で書く。スコープ定義（画面は作らない・触らない）であって HOW ではないため残す。ルール: ① パスは **ディレクトリ単位** に丸める（ファイル個別列挙・partial 列挙はしない。1 ファイルだけならそのまま）② 「ロック対象」等の構築側語彙は除去 ③ 画面を持たないチケット（バッチ・通知発火のみ等）は書かない ④ 受講生が新規に作る画面・JS が混在するチケットは「どこまで提供済みで、どこから実装対象か」の境界を明示する。

> **例外 — コマンド署名の明示**: 画面を持たないバッチ系チケットは、ルート表の代わりに **コマンドの署名（`php artisan` のコマンド名 + オプション名・値）をインターフェースとして記載する**。採点・動作確認の起動点が受講生の命名に揺れると採点コマンドが成立しないため、URL パスと同じく契約として与える（対象抽出の時間窓・件名・スケジュール登録時刻などの振る舞いは粗いままにする）。

> **`## 初期データ` はシナリオベースで書く**（「具体値を落とす」のではなく「業務シナリオへ言い直す」）:
> - **各 AC が観察・検証するために必要なデータ状態**を、業務シナリオ（状態の混在 / ロール / 関係性）で列挙する。例:「公開中 / 下書き / アーカイブの各状態が揃っている」「既読・未読を混在させた通知」「連携済 / 未連携のコーチ」「達成済・未達成の目標を混在」
> - **schema は書かない**（テーブル名・カラム名・FK・具体的なカラム値）。受講生はコードリーディング + ヒアリングで詳細化する
> - 基準は **「seed 一発で、その機能の各 AC を検証できる状態が揃う」**。100% の「データモデル > 初期データ Seeder」が用意する検証状態を、業務シナリオへ **漏れなく** 言い直す（抽象化しすぎて AC 検証状態を落とさない）
> - action で発火する状態（通知の発火・メール送信等、seed 不要なもの）は列挙しない。既存機能で「既に含まれている」初期データに乗る場合はその旨を書く。形式は **1 シナリオ = 1 bullet** を目安に

### Bug
`## メタ情報` / `## 概要`（やや抽象化可）/ `## 再現手順` / `## 期待する結果` / `## 実際の結果`

> = バグ報告書。症状の核は残す。`## 受け入れ条件` / `## 実装方針(参考)>原因` / `## 補足>想定Q&A` は **削除**（原因 = 答え）。必須サブ節なし。

### Task

`## メタ情報` / `## 概要` / `## 要件` / `## スコープ外`

> Task は **エンジニアが書く改修チケット**。Story / Bug のような「仕様の抽象化」はせず、**技術アプローチ語を残す**(Action 分離 / Service 集約 / 一括取得・Eager Loading / キュー・worker・非同期化 / 自動リトライ・失敗ジョブ / chunk / キャッシュ / モック 等)。エンジニアが「何をやるタスクか」を理解できる粒度で書く(技術語を消すと意味不明になる)。
> **削除する**: `## 受け入れ条件` / `## 実装方針(参考)>変更内容` / `## 補足>想定Q&A`、および **正確なクラス名・ファイルパス・メソッド名・コード片・参考実装・テストパス**(例 `MeetingController` / `app/UseCases/Meeting/` / `ShouldQueue` / `queue:work`)。「どこを・どう実装するか」は受講生の作業として残す。
> `N+1` 等の診断ラベルは任意で平易化してよい(例:「クエリがコーチ数に比例して増える」+「一括取得(Eager Loading)に書き換え」)。タイトルも技術ベースに(「N+1 解消」→「クエリ最適化」等)。必須サブ節なし。

## 禁止（生成後に自己点検）

- **答え漏れ**: 原因 / 変更内容 / 想定Q&A / 受け入れ条件 の混入。Bug の直し方・Task の実装手段が読み取れる記述。
- **メタ漏れ**: 構築側語彙（提供PJ / 模範解答 / Step4 / `.claude` / `docs` 参照）。読み手には「1 つの既存プロジェクト」しか見えない。
- **抽象化の非一貫**: 要件で消した仕様が 背景・インターフェース・初期データ に残存していないか。
- **回答不能**: 抽象化で消した仕様は、必ず 100% 側（要件 / インターフェース / 想定Q&A）でコーチが回答可能なこと。不能なら先に 100% へ追記する。
